# Building Production-Grade Tools for AI Agents: What Works After 100 Deployments
# 构建生产级 AI Agent 工具：100 次部署后的实战经验

Most developers who build AI agents make the same mistake: they spend weeks designing the orchestration layer, tuning the system prompt, and picking the right model — then hand the LLM a pile of hastily wrapped API endpoints and wonder why it fails in production. Here's the hard truth from teams shipping agents daily: tool design has a larger impact on agent reliability than prompt engineering. A well-crafted tool prevents hallucinations at the structural level. A poorly crafted tool guarantees them. This article walks through what we've learned from building, deploying, and debugging production AI agents across dozens of real-world workflows. You'll get concrete patterns, working code examples, and the anti-patterns that cost us the most in production incidents.

大多数开发 AI Agent 的开发者都会犯同一个错误：他们花费数周时间设计编排层、调整系统提示词（System Prompt）并选择合适的模型，然后将一堆草率封装的 API 接口丢给大模型（LLM），最后却困惑为什么它在生产环境中表现不佳。来自一线 Agent 开发团队的残酷真相是：工具设计对 Agent 可靠性的影响远大于提示词工程。一个精心设计的工具可以在结构层面防止幻觉，而一个糟糕的工具则会直接导致幻觉。本文将分享我们在构建、部署和调试数十个真实生产级 AI Agent 工作流中总结的经验。你将获得具体的模式、可运行的代码示例，以及那些在生产事故中让我们付出惨重代价的反模式。

### The Contract Between Deterministic and Non-Deterministic Code
### 确定性代码与非确定性代码之间的契约

When you write a function for another developer, you're working between two deterministic systems. Same input, same output. The calling code knows exactly what to expect. An AI tool is a fundamentally different contract. You're writing an interface between deterministic code (your backend service, database, or API) and a non-deterministic consumer (the LLM). The model might:
* Call your tool when you expected it to use something else
* Send malformed arguments because the description was ambiguous
* Retry your tool three times because the error message didn't tell it why it failed
* Ignore your tool entirely because the description didn't explain when to use it

当你为其他开发者编写函数时，你是在两个确定性系统之间工作。相同的输入，相同的输出。调用方代码完全知道会发生什么。而 AI 工具则是一种本质上完全不同的契约。你是在编写一个连接确定性代码（后端服务、数据库或 API）与非确定性消费者（LLM）的接口。模型可能会：
* 在你期望它使用其他工具时调用你的工具
* 因为描述模糊而发送格式错误的参数
* 因为错误信息没有说明失败原因而反复重试三次
* 因为描述中没有说明何时使用而完全忽略你的工具

This means every tool needs five components that traditional APIs never bothered with: a precise name, a rich description, a strict input schema, structured error handling, and a predictable output format. Let's build each one.

这意味着每个工具都需要传统 API 从未关注过的五个组件：精确的名称、丰富的描述、严格的输入模式（Schema）、结构化的错误处理以及可预测的输出格式。让我们逐一构建。

### 1. Naming: The First Signal the LLM Evaluates
### 1. 命名：LLM 评估的第一个信号

The tool name is the first thing the model scans when deciding which tool to call. It functions like a class name in a codebase — it sets expectations before any other signal.

工具名称是模型在决定调用哪个工具时扫描的第一项内容。它就像代码库中的类名一样——在任何其他信号出现之前，它就设定了预期。

```python
# Bad: vague, could mean anything
@mcp_tool(name="process")
def process(data): ...

# Bad: too generic, overlaps with other tools
@mcp_tool(name="get_data")
def get_data(query: str): ...

# Good: specific verb + noun, clear scope
@mcp_tool(name="list_overdue_invoices")
def list_overdue_invoices(customer_id: str): ...

# Good: resource_action pattern
@mcp_tool(name="invoice_send_reminder")
def invoice_send_reminder(invoice_id: str, channel: str): ...
```

Pick one convention — verb_noun or resource_action — and enforce it across every tool on your server. Mixing conventions forces the LLM to learn two mental models, and under load, it will confuse them. We saw a 23% drop in correct tool selection on a production agent when the team had `get_user`, `user_create`, and `delete_file` all coexisting with no pattern.

选择一种约定（例如 `动词_名词` 或 `资源_动作`）并在服务器的所有工具中强制执行。混合使用约定会迫使 LLM 学习两种思维模型，在高负载下，它会混淆两者。我们曾观察到，当团队同时使用 `get_user`、`user_create` 和 `delete_file` 且没有任何统一模式时，生产环境 Agent 的工具选择准确率下降了 23%。

### 2. Descriptions: Embedded Prompt Engineering
### 2. 描述：嵌入式提示词工程

The tool description is the most underestimated field in tool design. The LLM reads this to decide when to use the tool and what it will get back. It's prompt engineering baked into the tool definition itself.

工具描述是工具设计中最被低估的字段。LLM 通过阅读描述来决定何时使用该工具以及它将获得什么反馈。这实际上是将提示词工程直接嵌入到了工具定义中。

```python
MISMATCHED_DESCRIPTION = "Searches the database"

GOOD_DESCRIPTION = """\
Full-text search across the company knowledge base.
Use when the user asks to find internal documentation, policies, or technical specs.
Returns up to 10 results ranked by relevance, each with title, snippet, and URL.
Does NOT search emails or chat messages — use search_communications for those."""
```

Notice what the good description does: it says what it does, tells the LLM when to use it, describes the output shape, and explicitly states what it won't do. That last part is critical — explicit negative boundaries prevent the LLM from reaching for the wrong tool when it's close-but-not-right. A real measurement from our deployments: improving tool descriptions alone — no code changes — cut task completion time by 40% and reduced wrong-tool selection by 60%.

注意好的描述做了什么：它说明了功能、告诉 LLM 何时使用、描述了输出格式，并明确指出了它“不会”做什么。最后一点至关重要——明确的负面边界可以防止 LLM 在工具功能“接近但不完全匹配”时误选。我们部署中的真实测量数据显示：仅改进工具描述（无需修改代码），任务完成时间缩短了 40%，错误工具选择率降低了 60%。

### 3. Input Schemas: Never Trust the LLM
### 3. 输入模式：永远不要信任 LLM

Models hallucinate parameter values, confuse types, and invent fields that don't exist. Your tool must validate every input before processing. JSON Schema constraints are your first line of defense:

模型会产生参数值幻觉、混淆类型并虚构不存在的字段。你的工具必须在处理前验证每一个输入。JSON Schema 约束是你的第一道防线：

```python
GOOD_INPUT_SCHEMA = {
    "type": "object",
    "properties": {
        "query": {
            "type": "string",
            "minLength": 1, "maxLength": 500,
            "description": "Natural language search query or exact document title"
        },
        "limit": {
            "type": "integer",
            "minimum": 1, "maximum": 50, "default": 10,
            "description": "Maximum number of results to return"
        },
        "category": {
            "type": "string",
            "enum": ["engineering", "hr", "finance", "legal", "all"],
            "default": "all",
            "description": "Restrict search to a specific document category"
        }
    },
    "required": ["query"],
    "additionalProperties": False
}
```

Enums eliminate entire classes of failures. When environment accepts only "staging" or "production", the LLM can't invent "prod-us-east" and crash your deployment script. We've found that using enums and regex patterns for parameters eliminated 80% of runtime validation errors in production.

枚举（Enum）消除了整类故障。当环境参数只接受 "staging" 或 "production" 时，LLM 就无法虚构出 "prod-us-east" 并导致你的部署脚本崩溃。我们发现，对参数使用枚举和正则表达式可以消除生产环境中 80% 的运行时验证错误。

**Poka-Yoke Parameters**
**防错参数（Poka-Yoke）**

Take it a step further with poka-yoke design — making misuse structurally impossible:
更进一步，采用防错设计（Poka-Yoke），从结构上杜绝误用的可能性：

```python
# Instead of accepting free-text paths that cause path traversal:
{"path": {"type": "string"}} # bad

# Use enums with absolute paths for known configs:
{"config": {
    "type": "string",
    "enum": ["/etc/prod/config.yaml", "/etc/staging/config.yaml"]
}} # good
```

### 4. Error Handling: Errors Are Prompts for the LLM
### 4. 错误处理：错误也是给 LLM 的提示词

When a tool fails, the LLM needs enough information to decide whether to retry, try a different tool, or ask the user for help. Opaque errors like "Internal Server Error" leave the model stranded. MCP has two error mechanisms, and conflating them causes silent failures:

当工具失败时，LLM 需要足够的信息来决定是重试、尝试其他工具，还是向用户寻求帮助。像“内部服务器错误”这样晦涩的错误会让模型陷入困境。MCP 有两种错误机制，混淆它们会导致静默失败：

* **Protocol-level errors (JSON-RPC):** unknown tool, malformed arguments, server unavailable. The call never reached your tool logic.
* **Tool execution errors (isError: true):** the tool ran but failed. The agent can reason about these.

* **协议级错误 (JSON-RPC)：** 未知工具、参数格式错误、服务器不可用。调用从未到达你的工具逻辑。
* **工具执行错误 (isError: true)：** 工具已运行但失败了。Agent 可以针对这些错误进行推理。

```python
# Bad: generic error, LLM cannot reason about what went wrong
return {"error": "Something went wrong"}

# Good: structured error with actionable context
```
