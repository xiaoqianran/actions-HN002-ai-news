# Micro-benchmarks don’t tell the whole story
# 微基准测试并不能说明全部问题

I published a benchmark report comparing 15 HTTP server implementations across C, Rust, Java, Scheme, Go, Node.js, Deno, Bun, Gleam, Common Lisp, Python, Ruby, and Racket. Scheme (pico + io_uring) placed fifth overall at 222k req/s — behind C (h2o), Rust (axum), Java (Vert.x), and Bun, but ahead of Deno, Go, Node.js, and everything else. These are micro-benchmarks. I want to be honest about what they show and what they don’t.

我发布了一份基准测试报告，对比了 15 种 HTTP 服务器实现，涵盖了 C、Rust、Java、Scheme、Go、Node.js、Deno、Bun、Gleam、Common Lisp、Python、Ruby 和 Racket。Scheme (pico + io_uring) 以每秒 22.2 万次请求（req/s）的成绩位列总榜第五——仅次于 C (h2o)、Rust (axum)、Java (Vert.x) 和 Bun，但领先于 Deno、Go、Node.js 以及其他所有实现。这些都是微基准测试。我想坦诚地说明它们展示了什么，以及它们没能展示什么。

### What the numbers say
### 数据说明了什么

A trivial counter endpoint. Single-threaded. Ten seconds per concurrency level. wrk as the load generator. No disk, no database, no parsing, no serialization. The purest measure of “how fast can this runtime shuffle bytes through a socket.”

这是一个简单的计数器端点。单线程。每个并发级别测试十秒。使用 wrk 作为负载生成器。没有磁盘读写、没有数据库、没有解析、没有序列化。这是衡量“运行时在套接字间传输字节速度有多快”最纯粹的指标。

Rust peaks early at 316k req/s (8 connections), then holds a gentle plateau — the hallmark of a mature runtime with a well-tuned scheduler. Scheme peaks later at 222k req/s (16 connections), also plateaus cleanly. Node.js sits flat around 140k from 4 connections onward. FastAPI never breaks 48k — the Python GIL is the ceiling, not uvicorn.

Rust 在 8 个连接时达到 31.6 万 req/s 的峰值，随后保持平稳——这是成熟运行时和调优良好的调度器的标志。Scheme 在 16 个连接时达到 22.2 万 req/s 的峰值，随后也平稳运行。Node.js 从 4 个连接开始就稳定在 14 万左右。FastAPI 从未突破 4.8 万——瓶颈在于 Python 的 GIL，而非 uvicorn。

But the more interesting picture is latency under pressure:
但更有趣的是压力下的延迟表现：

*   **p99 latency (ms) at 256 concurrent connections**
*   **256 并发连接下的 p99 延迟 (毫秒)**
    *   Rust: 1.03
    *   Scheme: 1.39
    *   Node.js: 9.66
    *   FastAPI: 130.03

At 256 connections, Scheme’s p99 is 1.39ms — thirty-five percent higher than Rust’s 1.03ms, but seven times lower than Node.js’s 9.66ms. FastAPI’s p99 explodes to 130ms — a hundred times worse than Scheme. This is where architecture starts to show through the numbers.

在 256 个连接下，Scheme 的 p99 延迟为 1.39 毫秒——比 Rust 的 1.03 毫秒高出 35%，但比 Node.js 的 9.66 毫秒低了七倍。FastAPI 的 p99 延迟激增至 130 毫秒——比 Scheme 差了一百倍。这就是架构开始在数据中体现价值的地方。

### What the numbers don’t say
### 数据没能说明什么

Nothing about correctness. Nothing about how the code reads, how it changes, how it fails. Nothing about what happens when you add a database, a parser, a state machine, an authentication layer. Nothing about the six months after the benchmark, when the system has to evolve under production pressure and the original author is gone.

数据无法说明正确性。无法说明代码的可读性、变更难度或故障表现。无法说明当你加入数据库、解析器、状态机或身份验证层后会发生什么。也无法说明在基准测试结束六个月后，当系统必须在生产压力下演进且原作者离职时会发生什么。

A micro-benchmark is a controlled experiment. It holds everything constant except the variable you’re measuring. That’s its virtue and its limit. The variable here is raw I/O throughput. Everything that makes a real system hard — concurrency bugs, resource leaks, error propagation, compositional complexity — is absent by design.

微基准测试是一项受控实验。除了你正在测量的变量外，它保持一切不变。这是它的优点，也是它的局限。这里的变量是原始 I/O 吞吐量。而让真实系统变得困难的一切因素——并发 Bug、资源泄漏、错误传播、组合复杂性——在设计上都被排除在外了。

### Why it still matters
### 为什么它依然重要

The benchmark exists to answer one narrow question: is the letloop async runtime fast enough that choosing Scheme doesn’t mean accepting a performance tax? The answer is yes. At 222k req/s with a p99 under 1.4ms at high concurrency, pico + io_uring is competitive with production-grade runtimes written in systems languages, ahead of Node.js, and dramatically ahead of Python. That matters because the choice was never about speed. It was about something else.

这项基准测试旨在回答一个狭窄的问题：letloop 异步运行时是否足够快，以至于选择 Scheme 不意味着要承受性能损失？答案是肯定的。在 22.2 万 req/s 的吞吐量下，高并发时的 p99 延迟低于 1.4 毫秒，pico + io_uring 足以与系统语言编写的生产级运行时竞争，领先于 Node.js，并大幅领先于 Python。这很重要，因为选择它从来不仅仅是为了速度，而是为了其他东西。

### Transparent async
### 透明的异步机制

Most async runtimes are colored. In Rust, you write `async fn` and enter a different world — different traits, different lifetimes, different debugging. In JavaScript, you write `async/await` and functions split into two kinds: the ones that return promises and the ones that don’t. In Python, the split is even worse — `asyncio` is essentially a parallel universe that doesn’t compose cleanly with synchronous code.

大多数异步运行时都是“有色”的。在 Rust 中，你编写 `async fn` 就进入了另一个世界——不同的 trait、不同的生命周期、不同的调试方式。在 JavaScript 中，你使用 `async/await`，函数被分为两类：返回 Promise 的和不返回的。在 Python 中，这种分裂更严重——`asyncio` 本质上是一个与同步代码无法完美组合的平行宇宙。

letloop’s async is transparent. You write ordinary Scheme. The runtime handles scheduling via `io_uring` underneath. There is no function coloring, no `await` keyword, no split between “async code” and “normal code.” A procedure that reads from a socket and a procedure that computes a factorial use the same call conventions, the same continuation model, the same debugging tools.

letloop 的异步是透明的。你编写的是普通的 Scheme 代码。运行时在底层通过 `io_uring` 处理调度。没有函数颜色，没有 `await` 关键字，也没有“异步代码”与“普通代码”之分。读取套接字的程序和计算阶乘的程序使用相同的调用约定、相同的延续模型和相同的调试工具。

This is not a cosmetic difference. Function coloring has real costs: It doubles the API surface — every library that does I/O needs sync and async variants, or forces callers into one world. It makes composition harder — mixing sync and async code is where most concurrency bugs live in Python and JavaScript. It raises the floor of understanding — a new developer can’t ignore the async/sync distinction; it’s load-bearing from day one.

这不仅仅是表面上的差异。函数颜色化有实际成本：它使 API 表面积翻倍——每个执行 I/O 的库都需要同步和异步两个版本，或者强迫调用者进入单一世界。它增加了组合难度——在 Python 和 JavaScript 中，大多数并发 Bug 都源于同步与异步代码的混合。它提高了理解门槛——新开发者无法忽视异步/同步的区别；从第一天起，这就是系统的核心负担。

Transparent async removes these costs. The entire system has one calling convention. Composition is ordinary function composition. The floor of understanding is lower. Not zero — `io_uring` itself is complex machinery — but the complexity stays below the API boundary where it belongs, instead of leaking into every function signature.

透明异步消除了这些成本。整个系统只有一种调用约定。组合就是普通的函数组合。理解门槛更低。虽然不是零门槛——`io_uring` 本身就是复杂的机制——但复杂性被限制在 API 边界之下，而不是泄漏到每个函数签名中。

### What’s next
### 未来展望

`chez scheme` + `pico http parser` + `io_uring` is a work in progress. The scheduler is single-threaded. The error handling story is incomplete. These are real gaps, not aspirational roadmap items. But the foundation is sound. The numbers confirm that the architecture doesn’t impose a performance ceiling. What matters now is everything the benchmark doesn’t measure: correctness, composability, comprehensibility. The things that make a system livable for the person who maintains it after the benchmarks are forgotten.

`chez scheme` + `pico http parser` + `io_uring` 仍处于开发阶段。调度器是单线程的。错误处理机制尚不完善。这些是实际存在的差距，而非空谈的路线图。但基础是稳固的。数据证实了该架构没有性能瓶颈。现在重要的是基准测试无法衡量的东西：正确性、可组合性、可理解性。这些才是当基准测试被遗忘后，让维护者能够舒适地使用系统的关键。
