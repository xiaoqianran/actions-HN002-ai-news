# Why AI Engineers Are Moving Beyond LangChain to Native Agent Architectures
# 为什么 AI 工程师正在超越 LangChain，转向原生智能体架构

Frameworks accelerated the first wave of LLM apps, but production demands a different architecture.
框架加速了第一波大语言模型（LLM）应用的发展，但生产环境需要一种不同的架构。

Recently, I’ve been sitting closely with this topic, and it brought back some experiences from working on a couple of projects. Take this scenario: you ship an LLM-powered feature, the demo is clean, and all stakeholders are happy. Then three weeks into production, something breaks in a way nobody can explain. You spend an afternoon staring at logs that tell you what happened but not why. Then it turns out that the framework swallowed the context somewhere between step three and step four, and now you’re reading source code you didn’t write. That is not a bug report; it is a wakeup call about the architecture.
最近，我一直在深入思考这个话题，这让我想起了之前参与几个项目时的一些经历。设想一下：你发布了一个由 LLM 驱动的功能，演示效果很完美，所有利益相关者都很满意。然而在上线三周后，系统出现了一些没人能解释的故障。你花了一个下午盯着日志，日志告诉你发生了什么，却没告诉你原因。最后发现，是框架在第三步和第四步之间“吞掉”了上下文，而现在你不得不去阅读那些并非你编写的源代码。这不仅仅是一个错误报告，更是对架构的一次警示。

Frameworks like LangChain let engineers build LLM-powered systems without first understanding how those systems work under pressure. At first, that sounds like the cavalry has arrived. But trust me, the cost doesn’t show up until you’re deep in a production incident, and now you’re stuck wondering why your agent skipped the verification step it was supposed to run. This post is about that cost and why more engineers, after discovering it, are now building the orchestration layer by themselves.
像 LangChain 这样的框架让工程师无需深入了解系统在压力下的运行机制，就能构建 LLM 驱动的系统。起初，这听起来像是救星降临。但请相信我，只有当你深陷生产事故时，才会意识到其代价——你会被困住，纳闷为什么你的智能体跳过了本该执行的验证步骤。本文旨在探讨这种代价，以及为什么越来越多的工程师在发现这一点后，开始选择自行构建编排层。

### Give LangChain Its Credit
### 给 LangChain 应有的认可

I remember watching a colleague build a working RAG pipeline in about forty minutes sometime in early 2023. He went from the vector store through the retrieval chain, prompt templates, and the LLM call, all connected by lunchtime. Six months prior, that would have been at least a two-week project. Come to think of it, that’s actually how and why LangChain spread so fast. Most engineers hadn’t built LLM applications before. Nobody had strong opinions about the right way to structure a retrieval chain or manage conversation memory and other stuff like that. LangChain showed up with answers that were modular, composable, and documented, and of course, teams grabbed them immediately, including mine. So when I say it creates problems in production, I’m not being dismissive. It was optimized for the phase most teams were in when they adopted it. The problems came later, when the phase changed.
我记得 2023 年初，曾看到一位同事在四十分钟内就搭建好了一个可用的 RAG（检索增强生成）流水线。从向量数据库到检索链、提示词模板，再到 LLM 调用，他在午饭前就全部搞定了。而在六个月前，这至少需要两周的时间。回想起来，这正是 LangChain 传播如此迅速的原因。大多数工程师此前从未构建过 LLM 应用，没人对如何构建检索链或管理对话记忆等问题有强烈的见解。LangChain 提供了模块化、可组合且文档齐全的解决方案，团队自然会立即采用，包括我的团队。因此，当我提到它在生产环境中会引发问题时，我并非在贬低它。它针对的是大多数团队采用它时的那个阶段，而问题是在阶段发生变化后才出现的。

### Where the Abstraction Breaks
### 抽象失效之处

When I was learning object-oriented programming in my sophomore year, one of the first concepts that clicked was abstraction: hiding the internal details of how something works and only exposing what the user needs. LangChain applies that same idea to LLM orchestration. It hides a lot of what’s happening inside your system so you can move faster. But production AI systems demand something that cuts against that: clarity. You need to know exactly what your system did, in what order, with what inputs, and why. Not roughly. Exactly. Abstractions trade that visibility for speed. That’s a fair trade at first, until the hidden complexity becomes the very thing you need to understand.
大二学习面向对象编程时，我最先理解的概念之一就是“抽象”：隐藏内部实现细节，只向用户暴露必要的部分。LangChain 将这一理念应用于 LLM 编排，它隐藏了系统内部的许多运行细节，让你能更快地推进。但生产级 AI 系统需要的是与之背道而驰的东西：清晰度。你需要确切地知道系统做了什么、执行顺序如何、输入是什么以及原因何在。不是大概，而是精确。抽象以可见性换取速度，起初这是一种公平的交易，直到隐藏的复杂性成为你必须理解的核心障碍。

And it shows up in more ways than one.
这种情况以多种方式表现出来：

*   **Debugging is worse than it sounds:** When a multi-step chain gives the wrong output, you’re not just debugging your own code. You’re also trying to understand the framework’s execution flow and what the callback layer was doing behind the scenes. I once spent three hours tracking down a failure that turned out to be a memory module silently cutting out context. The fix itself took four minutes. Finding what caused it took half a day because the abstraction made the actual behavior invisible.
    **调试比听起来更糟糕：** 当一个多步链输出错误结果时，你不仅是在调试自己的代码，还在试图理解框架的执行流以及回调层在后台做了什么。我曾花三个小时追踪一个故障，结果发现是一个内存模块在静默地截断上下文。修复本身只用了四分钟，但找出原因却花了大半天，因为抽象使得实际行为变得不可见。

*   **Observability hits a ceiling:** You can integrate LangSmith and get useful traces, but you’re still seeing things through the framework’s lens, limited to the spans it chooses to expose. When you need visibility into something specific to your business logic, you end up working around the framework’s data model instead of just measuring what actually matters.
    **可观测性触及天花板：** 你可以集成 LangSmith 并获得有用的追踪信息，但你依然是通过框架的视角在观察，受限于它所选择暴露的跨度（spans）。当你需要深入了解特定业务逻辑时，你最终不得不绕过框架的数据模型，而不是直接测量真正重要的数据。

*   **Multi-agent state is where things really fall apart:** The moment you have agents coordinating, one planning, others executing, and another verifying, shared state becomes the real problem. Who created this information, when, and is it still valid? One agent updates memory, another reads a stale version, and the coordinator makes a decision based on context that no longer matches reality. Framework-managed state tends to work just fine for the happy path and quietly breaks down at the edge cases. Production systems live in those edge cases.
    **多智能体状态是真正崩溃的地方：** 一旦涉及智能体协作——一个负责规划，其他负责执行，还有一个负责验证——共享状态就成了真正的问题。谁创建了这些信息？何时创建的？它现在还有效吗？一个智能体更新了内存，另一个读取了过期的版本，协调者基于不再符合现实的上下文做出了决策。框架管理的状态在“理想路径”下运行良好，但在边缘情况下会悄无声息地崩溃。而生产系统恰恰就存在于这些边缘情况中。

*   **Latency accumulates:** Every abstraction layer adds overhead through serialization, validation, callback firing, and internal routing that runs whether you need it or not. In a prototype that overhead is invisible. Under real traffic, it shows up in percentile latency, specifically in the p95 and p99 ranges where users actually feel it. The cost per call might be small, but in an agentic system making four, five, or even six model calls per user request, those small costs compound quickly. At some point, you have to ask whether that overhead is still worth what it buys you.
    **延迟累积：** 每一层抽象都会通过序列化、验证、回调触发和内部路由增加开销，无论你是否需要，这些逻辑都会运行。在原型阶段，这种开销是不可见的；但在真实流量下，它会体现在百分位延迟中，特别是在用户能明显感知的 p95 和 p99 范围内。单次调用的成本可能很小，但在一个每个用户请求都要进行四、五甚至六次模型调用的智能体系统中，这些小成本会迅速累积。在某个时刻，你必须问自己：这种开销是否还值得你所换取的东西？

None of this is impossible to solve inside a framework. But the fixes start to look like working around the framework instead of working with it. And once you get there, it becomes harder to tell what the framework is still giving you.
这些问题并非无法在框架内解决，但修复方案往往看起来像是在“绕过”框架，而不是与框架“协作”。一旦到了这一步，就很难说清楚框架到底还为你提供了什么价值。

### So What Does “Building It Yourself” Actually Look Like?
### 那么，“自行构建”到底是什么样的？

“Native agent architecture” sounds more complex than it really is. It just means writing the orchestration logic yourself as code you own, instead of relying on a framework’s abstraction of it. State is something you define and update explicitly. Tools are clear functions you can test on their own. Memory is code you wrote, so it’s easier to debug, control, and understand what gets stored and how it gets retrieved. The model call is your code, which means you can instrument it directly and trace what matters. Sure, there will be more code upfront. But when something breaks, the failure is in your code and not somewhere inside an execution model written...
“原生智能体架构”听起来比实际更复杂。它仅仅意味着你自己编写编排逻辑作为你拥有的代码，而不是依赖框架的抽象。状态是你显式定义和更新的；工具是你可以独立测试的清晰函数；记忆是你自己编写的代码，因此更容易调试、控制，并且能清楚地理解存储了什么以及如何检索。模型调用也是你的代码，这意味着你可以直接对其进行插桩（instrumentation）并追踪重要信息。当然，前期会有更多的代码量，但当出现故障时，问题出在你的代码中，而不是在某个编写好的执行模型内部……
