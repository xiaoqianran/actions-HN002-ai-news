# Why I'm Learning ROS 2 as a Database Person
# 为什么我作为一个数据库从业者要学习 ROS 2

There's a moment that happens in every robotics company that makes it to production. The pilot worked. The robot does the thing. Now there are twenty on the floor, then fifty, then a fleet, and somebody asks: "What did the sensors look like on unit 17 during those three anomalies last quarter?" That's when the data infrastructure question gets real.
在每一家成功实现产品化的机器人公司，都会经历这样一个时刻：试点项目成功了，机器人能正常工作了。接着，车间里出现了 20 台、50 台，乃至一支机器人车队。这时有人会问：“上个季度那三次异常期间，17 号单元的传感器数据是什么样的？”这时，数据基础设施的问题才真正变得紧迫起来。

As robotics moves from lab to warehouse floor to factory line, that question is going to come up a lot more often. The default answers aren't built for it. I'm not a robotics engineer. Never built a robot. Never written a ROS 2 node or had an argument about MCAP vs. SQLite3 (though I'm about to).
随着机器人技术从实验室走向仓库和生产线，这类问题将会越来越频繁地出现。而现有的默认方案并非为此而生。我不是机器人工程师，从未制造过机器人，也从未写过 ROS 2 节点，更没为 MCAP 和 SQLite3 的优劣争论过（尽管我马上就要开始了）。

What I am is someone who's spent 20+ years watching what happens when storage decisions get made before the questions are fully understood. I've watched teams run hundreds of test iterations and then realize they can't answer "which run had that anomaly" without loading every bag file into memory. That's the shape of regret I'm talking about.
我是一个在过去 20 多年里，目睹了无数次“在完全理解需求前就做出存储决策”所带来后果的人。我见过团队进行了数百次测试迭代，最后却发现，如果不把所有的 bag 文件加载到内存中，他们根本无法回答“到底是哪一次运行出现了异常”这个问题。这就是我所说的“遗憾的形态”。

I work at Tiger Data doing developer relations for TimescaleDB. Time-series data is literally my job. When I started looking at how ROS 2 handles recorded telemetry at scale, I saw something familiar: high-frequency sequential writes, stable append rates, a recording format optimized for replay rather than analysis, no real query story for "compare behavior across 200 runs without loading everything into memory." Oof.
我在 Tiger Data 工作，负责 TimescaleDB 的开发者关系。时序数据就是我的本职工作。当我开始研究 ROS 2 如何大规模处理记录的遥测数据时，我看到了熟悉的一幕：高频顺序写入、稳定的追加速率、一种针对回放而非分析优化的记录格式，以及在“不加载所有数据到内存的情况下对比 200 次运行的行为”这一需求上，缺乏真正的查询方案。哎。

rosbag2 is good at what it does. Record, replay, done. That's the right scope for it. If you've been running ROS 2 long enough to think about what happens after the bag file, you've probably felt the edge of that scope. The problem is that "done recording" is the beginning of a different problem when you're trying to understand what's happening across a fleet, over time, at scale.
rosbag2 在其职责范围内表现出色：记录、回放、结束。这正是它的定位。如果你使用 ROS 2 的时间足够长，开始思考 bag 文件之后该做什么，你可能已经感受到了这个工具的局限性。问题在于，当你试图在大规模、长时间跨度下理解整个车队的运行状况时，“记录完成”恰恰是另一个问题的开始。

There's an open feature request in the rosbag2 repo that names TimescaleDB and InfluxDB by name as candidate options for exactly this. It's been open since July 2024. Zero comments. The community noticed the gap. Nobody filled it. So that's what I'm going to try to figure out, in public.
在 rosbag2 的代码仓库中，有一个悬而未决的功能请求，明确提到了 TimescaleDB 和 InfluxDB 作为解决此问题的候选方案。该请求自 2024 年 7 月以来一直处于开放状态，没有任何评论。社区注意到了这个缺口，但没人去填补它。所以，我打算公开尝试解决这个问题。

I'm going to build a ROS 2 node that writes telemetry directly to TimescaleDB and actually query it. All of it in a public GitHub repo, commit by commit, so the decisions are visible as they get made. Document everything that breaks. The database side I know. The ROS 2 side I'm learning from scratch, and I'll be honest about that difference in real time.
我将构建一个 ROS 2 节点，直接将遥测数据写入 TimescaleDB 并进行实际查询。所有过程都会在一个公开的 GitHub 仓库中进行，逐个提交代码，让决策过程透明可见。我会记录下所有遇到的故障。数据库方面我驾轻就熟，而 ROS 2 方面我正在从零开始学习，我会实时坦诚地记录这种差异。

And to be clear: I don't actually know what any of this looks like at fleet scale. That's not false modesty. That's the whole point. What's coming next: getting the local environment set up (TurtleBot3 is basically the hello world of ROS 2 simulation: it's what the official tutorials use, it publishes the standard topics I care about, and Gazebo runs the whole thing without me needing actual hardware) and a post on what rosbag2 actually stores and why the format question matters once you're past replay.
需要说明的是：我确实不知道这一切在大规模车队环境下会是什么样子。这不是虚伪的谦虚，这正是此行的目的。接下来的计划是：搭建本地环境（TurtleBot3 基本上是 ROS 2 仿真的“Hello World”：它是官方教程使用的模型，发布了我关心的标准话题，而且 Gazebo 可以在不需要真实硬件的情况下运行整个系统），并撰写一篇关于 rosbag2 实际存储内容以及为什么在回放之外格式问题至关重要的文章。

After that, the actual build: schema decisions, type mapping headaches, and a Grafana dashboard over live robot telemetry. The full tutorial comes last, once I've learned enough to actually teach it. If you've ever wanted to weigh in on that rosbag2 feature request, now's a good time. And if you're working on this problem (fleet telemetry pipelines, robot data at scale, the gap between recording and understanding), I want to hear what you're running into.
之后是实际构建：模式设计、类型映射的头疼问题，以及基于实时机器人遥测数据的 Grafana 仪表盘。完整的教程会放在最后，等我学到足够多、足以传授他人时再发布。如果你一直想对那个 rosbag2 功能请求发表意见，现在正是时候。如果你也在处理这个问题（车队遥测流水线、大规模机器人数据、记录与理解之间的鸿沟），我很想听听你遇到了什么挑战。
