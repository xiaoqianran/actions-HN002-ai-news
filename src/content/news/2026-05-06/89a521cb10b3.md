### Hybrid Architecture
### 混合架构

How to engineer a system capable of delivering robust solutions, even when moved into entirely new contexts? You just need to make it solve not a specific special case, but something more generalized — a problem at a higher level of abstraction. But how do we bring this to life?
如何设计一个即使在完全陌生的环境下也能提供稳健解决方案的系统？你只需要让它解决的不是某个特定的特殊案例，而是一个更通用的问题——即更高抽象层面的问题。但我们该如何实现这一点呢？

Let’s divide the problem into layers and solve it using a hybrid: RL commands the high-level strategy, and LP lives at the low-level execution. In doing so, we allow RL to synthesize broader domain knowledge, while LP solves specific, individual cases of packaging.
让我们将问题分层，并使用混合方法来解决：RL（强化学习）负责高层策略，而 LP（线性规划）负责底层执行。通过这种方式，我们让 RL 综合更广泛的领域知识，同时让 LP 处理具体的打包案例。

`action = [num_vehicles_1, .. , num_vehicles_n]`

`action = [num_vehicles_1, .. , num_vehicles_n]`

Due to this “separation of duties,” the RL component is unburdened by the minute, technical trivialities of what parcels go where, or how they are packed. Like a manager detached from the execution details. Ultimately, the RL agent affects the environment indirectly — its grand actions are processed through the LP solver, which then refreshes the environment’s state.
由于这种“职责分离”，RL 组件不再被琐碎的技术细节所困扰，例如包裹去向何处或如何打包。它就像一位脱离执行细节的管理者。最终，RL 智能体通过间接方式影响环境——其宏观动作由 LP 求解器处理，随后刷新环境状态。

Here is how we process the RL agent’s action and pass it into the LP solver.
以下是我们处理 RL 智能体动作并将其传递给 LP 求解器的方法：

```python
def decide_send_LP(self, action: np.array):
    # Parse the RL agent's action array into a dictionary of active destinations
    neighb_action = {v_id: num_v for v_id, num_v in enumerate(action) if num_v > 0}
    if not neighb_action:
        return 0, 0 # No vehicles dispatched
    
    # Get warehouse inventory for parcels that can actually go to the chosen destinations
    available_parcels = self.get_available_parcels(destinations=neighb_action.keys())
    if available_parcels.empty:
        return 0, 0 # No packages to send
    
    # The LP decides which parcels go into the vehicles to maximize volume/profit
    av_vehicles = self.get_available_vehicles()
    parcels_result, edges_result = send_veh(neighb_action, available_parcels, av_vehicles)
    
    # Update the environment state based on the LP's physical execution
    self.process_sent(parcels_result)
    
    # Return costs to the environment (for reward calculation)
    shipment_cost = sum(edges_result.c_cost * edges_result.v_varr_value)
    num_vehicles_sent = edges_result.v_varr_value.sum()
    return shipment_cost, num_vehicles_sent
```

What is happening here? Initially, we must translate the agent’s actions into a digestible format, ensuring the agent actually asked for at least one dispatch. Then, we check if any parcels in the warehouse can be sent. Next, we run linear programming, which packs available packages into available vehicles, choosing not only the class of transport but the specific vehicle, as well as where this parcel will go. And finally, we update the environment’s state based on the LP execution, calculate the shipping costs, and return it to calculate the reward.
这里发生了什么？首先，我们必须将智能体的动作转换为可理解的格式，确保智能体确实请求了至少一次调度。然后，我们检查仓库中是否有包裹可以发送。接下来，我们运行线性规划，将可用包裹装入可用车辆，不仅选择运输类别，还选择具体车辆以及包裹的去向。最后，我们根据 LP 的执行结果更新环境状态，计算运输成本，并将其返回以计算奖励。

Thus, we got the portability — so long as the structure of the task is the same, the system can adapt to any problem within the same class.
因此，我们获得了可移植性——只要任务结构相同，系统就能适应同一类问题中的任何情况。

### Scale-Invariant Observations
### 尺度不变的观测值

Let’s say we got the hybrid architecture. But how to make it survive in various contexts if RL agents’ observation and action spaces are technically fixed at initialization? I achieved that by transforming the observations — I normalized the observation space to make it scale-invariant.
假设我们已经有了混合架构。但如果 RL 智能体的观测空间和动作空间在初始化时是固定的，我们该如何让它在各种环境下生存呢？我通过转换观测值实现了这一点——我归一化了观测空间，使其具有尺度不变性。

Instead of tracking raw counts (e.g., “how many packages were sent”), we track ratios (e.g., “what percentage of the total backlog was sent”). This is a specific technical trick that gives you the “free” transfer of an agent from one task to another by allowing the agent to operate on a higher level of abstraction where absolute numbers are irrelevant. Let’s discuss some examples.
我们不再追踪原始计数（例如“发送了多少包裹”），而是追踪比例（例如“发送了总积压量的百分之几”）。这是一个特殊的技术技巧，它允许智能体在更高抽象层面（绝对数值无关紧要）上运行，从而实现智能体从一个任务到另一个任务的“免费”迁移。让我们讨论一些例子。

```python
def upd_perc_piles_wh(env):
    piles_wh = env.metrics['piles_wh']
    return np.array([piles_wh / env.num_piles])
```

Here, to make the observation scale-invariant, I divide the current warehouse inventory `piles_wh` by the absolute number of packages that will pass through the simulation `env.num_piles`. By doing that, the agent learns to prioritize based on the percentage of the daily workload it is currently holding.
在这里，为了使观测值具有尺度不变性，我将当前仓库库存 `piles_wh` 除以将通过模拟的总包裹数 `env.num_piles`。通过这样做，智能体学会了根据其当前持有的每日工作量百分比来确定优先级。

```python
def upd_warehouse_loading_loading_level_by_directions(env):
    # Get the current physical inventory at this specific node
    parcels = env.get_current_warehouse_parcels()
    if parcels.empty:
        return np.zeros(env.num_vertices)
    
    # Prepare the destinations array
    destinations = parcels['destination'].values.astype(int)
    
    # Get the counts for the destinations
    counts = np.bincount(destinations, minlength=env.num_vertices)
    return counts / len(parcels)
```

First, we pull the current stock of packages at this specific warehouse and verify that it is not empty. Next, we extract the ‘destination’ column as an array of integers, which represent the target warehouse IDs.
