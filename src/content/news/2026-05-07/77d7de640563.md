# Why I Don’t Trust LLMs to Decide When the Weather Changed
# 为什么我不信任大模型（LLM）来判断天气何时发生变化

**A physicist's approach to building production-grade agents**
**一位物理学家构建生产级智能体的思路**

**Fernando Arizmendi | May 6, 2026 | 7 min read**
**Fernando Arizmendi | 2026年5月6日 | 7分钟阅读**

Most weather apps have a simple problem: they show you the forecast, but they don’t tell you when it actually changed. That might sound trivial. It isn’t. Modern numerical weather prediction (NWP) systems — like ECMWF IFS — produce remarkably accurate forecasts at ~9 km resolution, updated every few hours. The data is already very good. The problem is not the forecast. The problem is attention: knowing when a change in that data is actually meaningful.
大多数天气应用都有一个简单的问题：它们向你展示预报，却不会告诉你预报究竟在何时发生了变化。这听起来可能微不足道，但事实并非如此。现代数值天气预报（NWP）系统（如欧洲中期天气预报中心的 IFS）能以约 9 公里的分辨率生成非常精确的预报，并每隔几小时更新一次。数据本身已经非常出色。问题不在于预报，而在于“注意力”：即如何判断数据中的变化是否真的具有意义。

I didn’t learn that from software engineering. I learned it years earlier, studying chaos theory at the Instituto Balseiro. It was there, working through dynamical systems, that I first encountered a slightly unsettling idea: A system can be completely deterministic and still be practically unpredictable. That idea stayed with me. And years later, when I started building AI systems, I realized that many of them were ignoring it.
我并不是从软件工程中学到这一点的。多年前，我在巴尔塞罗研究所（Instituto Balseiro）研究混沌理论时就明白了这一点。正是在那里，通过对动力系统的研究，我第一次接触到一个令人不安的观点：一个系统可以是完全确定性的，但在实践中却依然是不可预测的。这个想法一直伴随着我。多年后，当我开始构建人工智能系统时，我意识到许多系统都忽略了这一点。

### The problem with “vibe-based” deltas
### “基于感觉”的差异分析存在的问题

When I started seeing how developers were building weather agents, I noticed a pattern: Fetch forecast data, feed it into an LLM, and ask: “Did the weather change significantly?” At first glance, this seems reasonable. From a physics perspective, it is problematic — at least for problems where the decision boundary is already well-defined — because it replaces a well-defined threshold with a probabilistic interpretation.
当我开始观察开发者如何构建天气智能体时，我发现了一种模式：获取预报数据，输入大模型，然后问：“天气有显著变化吗？”乍一看，这似乎很合理。但从物理学角度来看，这存在问题——至少对于决策边界已经明确的问题而言是如此——因为它用概率解释取代了定义明确的阈值。

In a chaotic system, significance is not a linguistic judgment — it is a threshold defined on variables like temperature, precipitation, or wind speed. It depends on magnitudes, context, and time horizons. An LLM is a stochastic process. It is very good at generating language, but it is not designed to enforce deterministic boundaries on physical systems. When you ask an LLM whether a forecast “changed significantly,” you’re asking a probabilistic model to approximate a deterministic rule that you could have defined explicitly. That introduces variability exactly where you want consistency.
在混沌系统中，“显著性”不是一种语言判断，而是基于温度、降水或风速等变量定义的阈值。它取决于量级、背景和时间跨度。大模型是一个随机过程。它非常擅长生成语言，但并非旨在对物理系统强制执行确定性边界。当你问大模型预报是否“发生了显著变化”时，你是在要求一个概率模型去近似一个本可以明确定义的确定性规则。这恰恰在你需要一致性的地方引入了变数。

The failure modes are subtle:
*   Trends inferred from phrasing rather than data
*   Inconsistent decisions across similar inputs
*   Outputs that cannot be tested or reproduced
其失效模式非常隐蔽：
*   趋势是从措辞而非数据中推断出来的
*   面对相似输入时决策不一致
*   输出结果无法测试或复现

In many applications, that might be acceptable. In agriculture, energy, and logistics — where a 3°C drop is a phase transition for a crop, a non-linear spike in energy demand, or an operational disruption — it is not. These decisions need to be stable and explainable. Which led me to a simple rule: If you can write an assert statement for it, you probably shouldn’t be using a prompt.
在许多应用中，这或许可以接受。但在农业、能源和物流领域——其中 3°C 的下降可能意味着农作物的物候转变、能源需求的非线性激增或运营中断——这是不可接受的。这些决策必须是稳定且可解释的。这让我得出了一个简单的规则：如果你能为它写一个断言（assert）语句，那么你就不应该使用提示词（prompt）。

### My path to this problem
### 我解决此问题的路径

My career has looked less like a straight line and more like a trajectory in phase space. A Marie Curie PhD in climate dynamics, five years directing R&D at Uruguay’s national meteorology institute — forest fire prevention, seasonal forecasting, climate adaptation — then a shift to production ML at Microsoft and Mercado Libre. That arc gave me something specific: I already understood the physics of the data, the skill horizons of the models, and what “significant change” actually means in a physical system. Not as a software abstraction — as a measurable delta on a variable with known uncertainty bounds.
我的职业生涯看起来不像一条直线，更像相空间中的轨迹。我拥有气候动力学的居里夫人博士学位，曾在乌拉圭国家气象研究所负责研发工作五年（涉及森林火灾预防、季节性预报、气候适应），随后转向微软和 Mercado Libre 从事生产级机器学习。这段经历赋予了我独特的视角：我早已理解数据的物理特性、模型的预测能力边界，以及“显著变化”在物理系统中究竟意味着什么。它不是一种软件抽象，而是具有已知不确定性边界的变量上的可测量差异。

When I started building AI systems, the instinct was immediate: this is a threshold problem. Thresholds belong in code, not in prompts. Skygent is one expression of that perspective — an agent designed not to display forecasts, but to detect meaningful changes in them. The system runs continuously on real forecast data for user-defined events, evaluating changes every few hours and only triggering alerts when predefined conditions are met. In practice, most evaluation cycles result in no alert — only a small fraction of changes cross the significance threshold. That’s the point: signal, not noise.
当我开始构建人工智能系统时，直觉告诉我：这是一个阈值问题。阈值应该存在于代码中，而不是提示词中。Skygent 就是这一理念的体现——它设计的目的不是展示预报，而是检测预报中的有意义变化。该系统针对用户定义的事件，持续运行在实时预报数据上，每隔几小时评估一次变化，仅在满足预定义条件时触发警报。在实践中，大多数评估周期都不会产生警报——只有极少数变化会跨越显著性阈值。这正是重点所在：提取信号，而非噪音。

### The architecture
### 架构

Skygent follows a clean separation across five layers. Only one layer calls the LLM.
Skygent 遵循五个层级的清晰分离。其中只有一层会调用大模型。

**The Deterministic Gatekeeper**
At the core is a Python evaluator. It doesn’t interpret — it calculates. It:
*   Compares consecutive Pydantic-validated forecast snapshots
*   Evaluates deltas against configurable thresholds
*   Incorporates context: event type, variable sensitivity
*   Accounts for forecast horizon using established NWP skill limits
This is where decisions are made. Every alert has a traceable path: which variable changed, by how much, which threshold was crossed. In a corporate or government environment, being able to explain why an alert fired — without saying “the model felt like it” — is not optional.

**确定性守门人 (The Deterministic Gatekeeper)**
核心是一个 Python 评估器。它不进行解释，而是进行计算。它：
*   比较经 Pydantic 验证的连续预报快照
*   根据可配置的阈值评估差异
*   结合背景信息：事件类型、变量敏感度
*   利用既定的 NWP 预测能力限制来考虑预报时效
这是做出决策的地方。每个警报都有可追溯的路径：哪个变量发生了变化、变化了多少、跨越了哪个阈值。在企业或政府环境中，能够解释警报触发的原因——而不是说“模型觉得该触发了”——是必不可少的。

**The Trigger**
An alert fires only if a threshold is crossed. If the delta doesn’t cross the boundary, nothing happens. This is a binary, testable condition — not a judgment call.
**触发器 (The Trigger)**
只有在跨越阈值时才会触发警报。如果差异未跨越边界，则什么也不会发生。这是一个二元的、可测试的条件，而不是一种主观判断。

**The Narrator**
Only after the decision is made does the LLM enter the pipeline. Its role is strictly limited: take structured JSON data, translate it into natural language.
**叙述者 (The Narrator)**
只有在做出决策后，大模型才会进入流程。它的角色被严格限制：接收结构化的 JSON 数据，并将其翻译成自然语言。

*# Structured payload sent to GPT-4o-mini*
*# 发送给 GPT-4o-mini 的结构化负载*
{
  "event_name": "Ana's Wedding",
  "variable": "precipitation_probability_max",
  "from_value": 10.0,
  "to_value": 50.0,
  "delta": 40.0,
  "horizon_days": 5.2,
  "confidence": "medium"
}
*Output: “Rain probability increased from 10% to 50% for your event window. Confidence is medium due to the 5-day forecast horizon.”*
*输出：“在您的活动窗口期内，降水概率从 10% 增加到 50%。由于是 5 天后的预报，置信度为中等。”*

The LLM is not deciding anything. It is explaining.
大模型没有做任何决定，它只是在进行解释。
