# I Let CodeSpeak Take Over My Repository
# 我让 CodeSpeak 接管了我的代码仓库

**What happened when I migrated a 10K+ line project into an AI-native workflow**
**当我将一个 1 万多行的项目迁移到 AI 原生工作流时，发生了什么？**

Programming languages have evolved significantly over time, becoming increasingly abstract and easier for humans to understand. In the early days of computing, programmers worked directly with machine code, manually entering raw binary instructions using punch cards, where holes encoded data and commands for early mainframes. This process was tedious and highly error-prone: a single misplaced hole could break the entire program and force developers to start over from scratch.
编程语言随着时间的推移发生了巨大的演变，变得越来越抽象，也越来越易于人类理解。在计算机发展的早期，程序员直接使用机器码工作，通过打孔卡手动输入原始二进制指令，孔洞编码了早期大型机的数据和命令。这个过程既繁琐又极易出错：一个孔洞位置放错，就可能导致整个程序崩溃，迫使开发者从头开始。

To reduce this friction, assembly language emerged, replacing raw binary code with human-readable mnemonics such as ADD or MOV, which were then translated back into machine instructions. The next major leap came in the 1950s when the first compiled high-level languages such as FORTRAN and COBOL appeared. They introduced syntax closer to mathematical notation and natural language. Later, languages such as ALGOL, C, and C++ brought structured programming, memory abstraction, and object-oriented concepts, dramatically reducing software complexity and improving maintainability.
为了减少这种阻力，汇编语言应运而生，它用人类可读的助记符（如 ADD 或 MOV）取代了原始二进制代码，然后再将其翻译回机器指令。下一个重大飞跃发生在 20 世纪 50 年代，当时出现了第一批编译型高级语言，如 FORTRAN 和 COBOL。它们引入了更接近数学符号和自然语言的语法。后来，ALGOL、C 和 C++ 等语言带来了结构化编程、内存抽象和面向对象的概念，极大地降低了软件复杂性并提高了可维护性。

Modern languages like Python and Java continued this trend by prioritising readability and offering vast ecosystems of built-in libraries, allowing engineers to focus more on solving business problems rather than dealing with hardware-level details. More recently, however, the way we write code has started to shift again with the rise and widespread adoption of AI. This naturally raises the question: will programming languages evolve as well? It feels increasingly plausible that one day we may end up coding in plain English. That’s why I was excited to try one of these potential languages of the future: CodeSpeak, currently available in alpha preview. Let’s see how it works.
Python 和 Java 等现代语言延续了这一趋势，它们优先考虑可读性并提供庞大的内置库生态系统，使工程师能够更专注于解决业务问题，而不是处理硬件层面的细节。然而，最近随着人工智能的兴起和广泛应用，我们编写代码的方式再次开始发生转变。这自然引发了一个问题：编程语言也会随之进化吗？有一天我们最终可能会用纯英语进行编程，这种可能性似乎越来越大。正因如此，我很兴奋能尝试这种未来的潜在语言之一：CodeSpeak（目前处于 Alpha 预览阶段）。让我们看看它是如何工作的。

### Setting everything up
### 环境配置

As usual, before jumping into building, we first need to set everything up. Luckily, the process is pretty straightforward. We can install CodeSpeak using the uv package manager:
像往常一样，在开始构建之前，我们首先需要进行环境配置。幸运的是，这个过程非常简单。我们可以使用 uv 包管理器安装 CodeSpeak：

```bash
uv tool install codespeak-cli
codespeak --version # CodeSpeak CLI 0.4.1
```

Next, we need to log in. Running the following command opens a browser window where you can either authenticate with your Google account or create a dedicated CodeSpeak login and password. After accepting the Terms & Conditions, you’ll be successfully authorised.
接下来，我们需要登录。运行以下命令会打开一个浏览器窗口，你可以选择使用 Google 账号进行身份验证，或者创建一个专门的 CodeSpeak 登录名和密码。在接受条款和条件后，你将成功获得授权。

```bash
codespeak login
```

The final step is setting up your API key, since CodeSpeak follows a Bring Your Own Key policy. At the moment, it only supports an Anthropic API key, which makes sense given how popular Anthropic models are for coding tasks right now. You can generate your key in https://platform.claude.com/settings/keys and export it as an environment variable.
最后一步是设置你的 API 密钥，因为 CodeSpeak 遵循“自带密钥”（Bring Your Own Key）政策。目前，它仅支持 Anthropic API 密钥，考虑到 Anthropic 模型在当前编程任务中的流行程度，这很合理。你可以在 https://platform.claude.com/settings/keys 生成密钥，并将其导出为环境变量。

```bash
export ANTHROPIC_API_KEY=<your-key>
```

And that’s it. We’re ready to start building.
就是这样。我们准备好开始构建了。

### Building with CodeSpeak
### 使用 CodeSpeak 进行构建

CodeSpeak supports greenfield development when you put together a spec and build from it. However, I thought it would be more interesting to test it on a somewhat more mature project (or at least one that is already functional) and see how well it handles an existing codebase. In my previous article, I built a web application for tracking and analysing workout data using spec-driven development. I’ve been actively using it since then and have already found several areas for improvement, so it felt like a good candidate for testing CodeSpeak.
CodeSpeak 支持在编写规范并据此构建时的“绿地开发”（从零开始）。然而，我认为在更成熟的项目（或至少是已经具备功能的项目）上测试它会更有趣，看看它处理现有代码库的能力如何。在上一篇文章中，我使用规范驱动开发构建了一个用于跟踪和分析锻炼数据的 Web 应用程序。从那时起我一直在积极使用它，并已经发现了几个可以改进的地方，因此它非常适合用来测试 CodeSpeak。

### Takeover
### 接管项目

Before using CodeSpeak to implement new features, we first need to generate an initial version of the specifications. Fortunately, CodeSpeak can help with that as well. I started by copying all the code from my fitness app into a new repository. Then, I removed all existing specs and the constitution file to emulate a more realistic scenario when you vibe-coded something quickly and now want to migrate the project into a more structured workflow with CodeSpeak.
在使用 CodeSpeak 实现新功能之前，我们首先需要生成规范的初始版本。幸运的是，CodeSpeak 也可以在这方面提供帮助。我首先将健身应用的所有代码复制到一个新的仓库中。然后，我删除了所有现有的规范和配置文件，以模拟一种更真实的场景：当你快速“凭感觉”写完代码，现在想通过 CodeSpeak 将项目迁移到更结构化的工作流中。

CodeSpeak allows you to migrate only selected parts or modules of a project, but I decided to go all in and move the entire application. The repository contains around 13K lines of code, so while it’s not enormous, it’s still a reasonably sized project for an experiment like this. We can start generating specs with the following command.
CodeSpeak 允许你只迁移项目的特定部分或模块，但我决定“全押”，迁移整个应用程序。该仓库包含约 1.3 万行代码，虽然不算庞大，但对于这样的实验来说，它是一个规模适中的项目。我们可以使用以下命令开始生成规范。

```bash
codespeak takeover
```

CodeSpeak then spends some time analysing your local files. In my case, the whole process took around a minute. After that, it opened a webpage with a proposed structure for the codebase. It suggested splitting my application into four modules: Frontend, Backend API, Data Layer, and Backend Tests. Modules are a well-established programming concept, and it’s easy to see why they’re useful: they create smaller self-contained domains, improve readability, and make it easier to reuse components across projects. So, it’s wonderful that CodeSpeak suggests us to create 4 different specs rather than trying to put everything in one long document.
CodeSpeak 会花一些时间分析你的本地文件。就我而言，整个过程大约花了一分钟。之后，它打开了一个网页，展示了代码库的建议结构。它建议将我的应用程序拆分为四个模块：前端（Frontend）、后端 API（Backend API）、数据层（Data Layer）和后端测试（Backend Tests）。模块是一个成熟的编程概念，很容易看出它们的用处：它们创建了更小的自包含域，提高了可读性，并使跨项目重用组件变得更容易。因此，CodeSpeak 建议我们创建 4 个不同的规范，而不是试图把所有东西都塞进一个长文档中，这非常棒。

In the browser, you can review each module and ask CodeSpeak to split, merge, or rethink it. In my case, the suggested structure looked reasonable, so I decided to keep everything as is and migrate all modules, allowing the entire repository to be managed by CodeSpeak. After selecting all modules and starting the takeover, CodeSpeak processed them one by one: reading files, generating specifications, refining them, and running tests.
在浏览器中，你可以查看每个模块，并要求 CodeSpeak 进行拆分、合并或重新构思。就我而言，建议的结构看起来很合理，所以我决定保持原样并迁移所有模块，让整个仓库由 CodeSpeak 管理。在选择所有模块并开始接管后，CodeSpeak 逐一处理它们：读取文件、生成规范、优化规范并运行测试。
