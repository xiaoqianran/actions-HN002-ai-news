### Vibe coding vs Spec-driven development
### 氛围编程 vs 规范驱动开发

Many of us have already experienced vibe coding: you write a short prompt (for example, “Please add a DAU chart to my web application”), wait for the agent to generate the change, run it locally, and check whether the result matches your expectations. Usually, it doesn’t. So you go back to the same chat, ask the agent to adjust the chart, and keep iterating until the result is good enough.
我们许多人都体验过“氛围编程”：你写一个简短的提示词（例如，“请给我的 Web 应用添加一个日活跃用户（DAU）图表”），等待代理生成更改，在本地运行，然后检查结果是否符合预期。通常情况下，结果并不符合。于是你回到同一个对话框，要求代理调整图表，并不断迭代，直到结果足够好为止。

This approach works reasonably well for simple projects, but it doesn’t scale well, especially when multiple developers are working on the same codebase. The main drawbacks are the lack of best practices and shared conventions. For example, without a structured approach, teams can easily end up with five different ways to run ML model training inside the same DBT pipeline.
这种方法对于简单项目效果尚可，但难以扩展，尤其是当多名开发人员在同一个代码库上工作时。其主要缺点是缺乏最佳实践和共享约定。例如，如果没有结构化的方法，团队很容易在同一个 DBT 流水线中搞出五种不同的机器学习模型训练方式。

Another common issue is that we usually don’t persist the results or reasoning from our conversations with AI agents. As a result, it becomes easy to lose track of why certain decisions were made. For example, an agent might forget why you cleaned up data in a particular way, and the next update could silently introduce a different result. Context decay is also an especially common problem. AI agents are stateless, and when working on larger projects, we often have to start new chats because of context window limitations, effectively starting our communication from scratch.
另一个常见问题是，我们通常不会持久化保存与 AI 代理对话的结果或推理过程。结果就是，我们很容易忘记当初为什么做出某些决定。例如，代理可能会忘记你为什么要以某种特定方式清洗数据，下一次更新可能会悄无声息地引入不同的结果。“上下文衰减”也是一个特别普遍的问题。AI 代理是无状态的，在处理大型项目时，由于上下文窗口的限制，我们经常不得不开启新的对话，实际上等于从零开始沟通。

Spec-driven development (SDD) is much closer to traditional engineering practices. Instead of jumping straight into implementation, we start by doing the hard thinking ourselves: making architectural decisions, defining requirements, and documenting them in a structured markdown specification stored in the repository and updated alongside the project. This creates an important shift: we decouple the specification (what we are building and why) from the implementation (the actual code). SDD addresses many of the core issues of vibe coding by preserving context across sessions (and even across different AI agents) while aligning both humans and agents around the project’s main non-negotiables.
“规范驱动开发”（SDD）更接近传统的工程实践。我们不再直接跳入实施阶段，而是先进行深入思考：做出架构决策、定义需求，并将它们记录在存储于代码库中并随项目同步更新的结构化 Markdown 规范文档中。这带来了一个重要的转变：我们将规范（我们要构建什么以及为什么构建）与实施（实际代码）解耦。SDD 通过在会话之间（甚至在不同的 AI 代理之间）保留上下文，同时让人员和代理围绕项目的核心原则达成一致，从而解决了“氛围编程”的许多核心问题。

### SDD workflow
### SDD 工作流

A typical spec-driven development workflow usually consists of the following stages. The first step is defining the constitution — an agreement on the key decisions for the project. It usually includes several core documents:
一个典型的规范驱动开发工作流通常包含以下阶段。第一步是定义“宪法”——即对项目关键决策达成一致。它通常包括几份核心文档：

*   **Mission** explains the why: why are we building this project, and what are its key goals and features?
*   **使命（Mission）**：解释“为什么”——我们为什么要构建这个项目，它的关键目标和功能是什么？
*   **Tech Stack** documents technical decisions, as well as deployment and update processes.
*   **技术栈（Tech Stack）**：记录技术决策，以及部署和更新流程。
*   **Roadmap** outlines project phases, planned features, and is continuously updated as the project evolves.
*   **路线图（Roadmap）**：概述项目阶段、计划功能，并随着项目的进展不断更新。

规范既可以为新项目创建，也可以为现有项目创建，这使得该方法非常灵活。一旦项目级文档就绪，我们就可以进入功能开发阶段，通常包括：

1.  Understanding what we want to build and writing a detailed specification.
    理解我们要构建的内容并编写详细的规范。
2.  Implementing the changes.
    实施更改。
3.  Validating that the implementation works as expected.
    验证实施结果是否符合预期。
