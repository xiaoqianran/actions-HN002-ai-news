# I Built the Same B2B Document Extractor Twice: Rules vs. LLM
# 我两次构建了同一个 B2B 文档提取器：规则法 vs. 大模型法

A practical comparison between rule-based PDF extraction using pytesseract and an LLM-based approach with Ollama and LLaMA 3, based on a realistic B2B order scenario.
基于真实的 B2B 订单场景，对使用 pytesseract 的基于规则的 PDF 提取与使用 Ollama 和 LLaMA 3 的基于大模型（LLM）的方法进行了实践性对比。

Imagine the following situation: You work in the operations team of a medium-sized company. Every day, your team processes order forms from different B2B customers. All of them arrive as PDFs. And in theory, they all contain the same information: customer ID, purchase order number, delivery date, and the ordered items.
想象一下这种情况：你在一家中型公司的运营团队工作。每天，你的团队都要处理来自不同 B2B 客户的订单表格。所有订单都以 PDF 格式发送过来。理论上，它们都包含相同的信息：客户 ID、采购订单号、交货日期和订购商品。

In practice, however, every document looks slightly different: One customer places the purchase order number in the top-left corner, the next one in the bottom-right corner. Some write “PO Number”, others use “Order ID”, “Order Reference”, or something completely different.
然而在实践中，每份文档看起来都略有不同：有的客户把采购订单号放在左上角，有的则放在右下角。有的写着“PO Number”，有的则使用“Order ID”、“Order Reference”或其他完全不同的表述。

For us humans, this is usually not a problem. We look at the document, understand the context, and immediately recognize which information is meant. For traditional automation systems, however, this becomes difficult: A regex rule can specifically search for “PO Number: “. But what happens if the next customer uses “Order Reference: “ instead?
对于我们人类来说，这通常不是问题。我们看一眼文档，理解上下文，就能立即识别出所需的信息。但对于传统的自动化系统来说，这就变得困难了：正则表达式（regex）规则可以专门搜索“PO Number: ”。但如果下一个客户使用的是“Order Reference: ”该怎么办呢？

That is exactly the problem I recreated for this article. We compare two different approaches for extracting structured data from B2B order forms: A traditional rule-based approach using pytesseract and regex rules; An LLM-based approach using pytesseract, Ollama, and LLaMA 3.
这正是本文所要重现的问题。我们将比较从 B2B 订单表格中提取结构化数据的两种不同方法：一种是使用 pytesseract 和正则表达式的传统规则法；另一种是使用 pytesseract、Ollama 和 LLaMA 3 的基于大模型的方法。

The goal of this article is not to show that LLMs are generally better. They are not always. A much more interesting question is: At what point do traditional extraction pipelines start to reach their limits as complexity and the number of different layouts increase? And when can an LLM actually reduce maintenance effort?
本文的目的并不是要证明大模型在任何情况下都更好。事实并非总是如此。一个更有趣的问题是：随着复杂性和不同布局数量的增加，传统的提取流水线在什么节点开始达到极限？以及大模型在何时真正能减少维护工作量？

### 1 – Step-by-Step Guide
### 1 – 分步指南

We rebuild both approaches step by step. First, we create two sample PDFs containing the same business information but using different layouts. Afterwards, we extract the data once with a traditional OCR and regex pipeline and once with an OCR and LLM pipeline. This allows us to compare both approaches under identical conditions.
我们将分步重建这两种方法。首先，我们创建两个包含相同业务信息但布局不同的示例 PDF。然后，分别使用传统的 OCR 加正则表达式流水线，以及 OCR 加大模型流水线提取数据。这使我们能够在相同的条件下比较这两种方法。

The traditional approach basically asks: “Can I find the exact pattern that I programmed?” The LLM-based approach instead asks: “Can I understand the meaning of this field in context?”
传统方法本质上是在问：“我能找到我预先编程的精确模式吗？”而基于大模型的方法则是在问：“我能理解这个字段在上下文中的含义吗？”

### Before We Start — Mise en Place
### 开始之前 — 准备工作

**pip vs. Anaconda**
In this guide, we use pip, Python’s standard package manager. This means we install all libraries directly through the command line using `pip install …`. pip is already included automatically when you install Python. If you know Python tutorials that work with Anaconda, that is simply another way to achieve the same goal (using `conda install …`).
**pip 与 Anaconda**
在本指南中，我们使用 Python 的标准包管理器 pip。这意味着我们通过命令行使用 `pip install …` 直接安装所有库。安装 Python 时，pip 会自动包含在内。如果你熟悉使用 Anaconda 的 Python 教程，那只是实现相同目标的另一种方式（使用 `conda install …`）。

**Create and activate a new virtual environment**
Create a new python environment with `python –m venv b2bdocumentextractor` (you can change the name) in a terminal and activate it with `b2bdocumentextractor\Scripts\activate`.
**创建并激活新的虚拟环境**
在终端中使用 `python –m venv b2bdocumentextractor`（你可以更改名称）创建一个新的 Python 环境，并使用 `b2bdocumentextractor\Scripts\activate` 激活它。

### Step 1 – Install Tesseract
### 第 1 步 – 安装 Tesseract

Tesseract is the OCR engine. It is the tool that actually reads text from images or scanned PDFs using OCR (Optical Character Recognition). pytesseract is only the Python bridge to Tesseract. This means: Our Python code can communicate with Tesseract through pytesseract, but the real text recognition is done by Tesseract itself. Without installing Tesseract first, pytesseract cannot work.
Tesseract 是 OCR 引擎。它是通过 OCR（光学字符识别）从图像或扫描版 PDF 中读取文本的工具。pytesseract 只是通往 Tesseract 的 Python 桥梁。这意味着：我们的 Python 代码可以通过 pytesseract 与 Tesseract 通信，但真正的文本识别是由 Tesseract 本身完成的。如果不先安装 Tesseract，pytesseract 将无法工作。

---

### Step 2 – Install Poppler
### 第 2 步 – 安装 Poppler

Next, we install pdf2image. This is our library for converting PDFs into images and it requires Poppler in the background. Poppler is an open-source PDF rendering library used to display PDF files.
接下来，我们安装 pdf2image。这是我们用于将 PDF 转换为图像的库，它需要在后台运行 Poppler。Poppler 是一个开源的 PDF 渲染库，用于显示 PDF 文件。

---

### Step 3 – Install Python Libraries
### 第 3 步 – 安装 Python 库

Now we install all Python libraries we need. Make sure you reactivate the Python environment beforehand:
现在我们安装所有需要的 Python 库。请确保事先重新激活了 Python 环境：

*   **pytesseract**: We install this library as the bridge between Python and Tesseract.
*   **pdf2image**: pytesseract is an OCR engine, which means it recognizes text from pixels in an image. It cannot read PDF structures directly. pdf2image therefore performs an intermediate step: It renders each PDF page as an image.
*   **pytesseract**：我们安装此库作为 Python 和 Tesseract 之间的桥梁。
*   **pdf2image**：pytesseract 是一个 OCR 引擎，这意味着它从图像的像素中识别文本。它无法直接读取 PDF 结构。因此，pdf2image 执行了一个中间步骤：它将每一页 PDF 渲染为图像。
