How to build initially robust code
如何从初始阶段构建健壮代码

The first section I’ll cover is how to build robust code initially. And the following section will cover how to verify the robustness of code and fix it after it’s been built. I view this as two separate problems, and I use separate techniques to solve them, which is why I divided it into two sections.
第一部分我将介绍如何从初始阶段构建健壮代码。接下来的部分将涵盖如何在代码构建完成后验证其健壮性并进行修复。我将这两者视为独立的问题，并使用不同的技术来解决，这就是我将其分为两部分的原因。

Active usage of plan mode
主动使用规划模式 (Plan Mode)

Plan mode is the first technique I’ll cover which I think is very important if you want to get the most out of coding agents. Using plan mode enables the coding agents to spend more time planning the implementation instead of just starting out with it right away. This typically improves the models ability to see the bigger picture and thus avoids bugs for example caused by updates to one component changing things in other components.
规划模式是我要介绍的第一项技术，我认为如果你想充分利用编程智能体，这一点非常重要。使用规划模式可以让智能体花更多时间规划实现方案，而不是直接开始编码。这通常能提升模型把握全局的能力，从而避免因更新某个组件而导致其他组件出现错误等问题。

Plan mode also asks clarifying questions so that any ambiguities are made clear. having coding agents ask new questions instead of you asking coding agents is an incredibly powerful feature I urge you to actively use more. You want to let the model do as much of the thinking as possible and only come back to you once it needs to clarify something or understand better what you want to implement. It’s way more powerful to have the LLMs ask you questions than you asking the LLMs questions. This, in most cases, leads to fewer bugs and a model being more efficient at implementing a solution.
规划模式还会提出澄清性问题，以便消除任何歧义。让编程智能体主动向你提问，而不是你不断询问智能体，这是一个极其强大的功能，我强烈建议你多加利用。你应该让模型尽可能多地进行思考，只有在需要澄清问题或更好地理解你的实现意图时，才来找你。让大语言模型向你提问，比你向模型提问要高效得多。在大多数情况下，这能减少错误，并使模型在实现解决方案时更加高效。

While of course plan mode initially takes more time since you have to plan with the agent and not start the implementation right away, it’s usually worth it in the long run because of the fewer bugs you experience and having to spend less time iterating with the agent after an implementation to ensure you get the exact implementation you desire.
虽然规划模式在初期确实需要更多时间，因为你需要与智能体共同规划而不能立即开始实现，但从长远来看，这通常是值得的，因为你会遇到更少的错误，并且在实现后无需花费大量时间与智能体反复迭代，从而确保获得你想要的精确实现。

Keeping skill files
维护技能文件 (Skill Files)

The second part is the MD files that you have in your repository. Over time, when you spend time coding in the repository, the number of markdown files should increase steadily, highlighting how agents should behave in the repository, previous bugs that have been reported and how they were fixed, and other issues that have arisen in the repository previously. This is incredibly important and useful for the coding agents because they often have enough context to be able to actively utilize this knowledge. And it makes them less prone to make incorrect decisions that they’ve made in the past.
第二部分是仓库中的 Markdown 文件。随着时间的推移，当你在这个仓库中进行编码时，Markdown 文件的数量应该稳步增加。这些文件应重点说明智能体在仓库中应如何表现、之前报告过的错误及其修复方法，以及仓库中曾出现的其他问题。这对编程智能体来说极其重要且有用，因为它们通常拥有足够的上下文来主动利用这些知识，这能减少它们重蹈覆辙、做出错误决策的可能性。

These Markdown files are typically made from mistakes that agents have made in previous sessions, and so having a lot of them helps them all make better decisions. To create these Markdown files, I urge you to make an agent generalize knowledge from a thread after every chat thread that you’ve had with your coding agent. This is probably the number one tip that makes coding with agents more effective.
这些 Markdown 文件通常源于智能体在之前会话中犯下的错误，因此拥有大量此类文件有助于它们做出更好的决策。为了创建这些文件，我强烈建议你在每次与编程智能体结束对话后，让它从对话线程中总结知识。这可能是让智能体编程更高效的第一秘诀。

Secondly, that every time you discover and fix a bug, you store a description of the problem and how it was solved in a markdown file. If you apply these concepts every time you are coding, you will develop an incredibly powerful knowledge base inside your repository, and your agents will definitely improve over time and become more and more effective and less prone to errors, and thus build more robust code.
其次，每当你发现并修复一个错误时，都要将问题的描述及其解决方法存储在一个 Markdown 文件中。如果你在每次编码时都应用这些概念，你将在仓库内建立一个极其强大的知识库。随着时间的推移，你的智能体肯定会不断进步，变得越来越高效，出错率也会降低，从而构建出更健壮的代码。
