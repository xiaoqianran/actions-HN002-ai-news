In this article, I’ll be discussing how to run a lot of coding agents in parallel effectively.
在本文中，我将探讨如何高效地并行运行大量编程智能体。

If you’re running coding agents sequentially and not in multiple runs in parallel, you’re losing out. One of the key benefits of coding agents is that you can start completing work in parallel, something that was never really possible before when working on software engineering tasks.
如果你还在按顺序运行编程智能体，而不是并行运行多个任务，那么你正错失良机。编程智能体的核心优势之一在于能够并行完成工作，这在以往的软件工程任务中是几乎不可能实现的。

However, when I start running a lot of parallel coding sessions, it’s often difficult to keep track. You need specialized techniques to easily keep an overview of all the sessions that you’re running, quickly catch up on conversations when coming back to them, and so on.
然而，当我开始运行大量并行编程会话时，往往很难跟踪进度。你需要专门的技巧来轻松掌握所有正在运行的会话，并在回到某个会话时快速跟上之前的对话进度。

### Why it’s hard to run coding agents in parallel
### 为什么并行运行编程智能体很困难

First, I want to cover why it is challenging to run coding agents in parallel. To a certain extent, this should be quite self-explanatory. Before LLMs, software engineers and programmers would naturally only work on one task at a time. The simple reason was that if you tried multi-tasking on software engineering tasks, you would just end up being less effective.
首先，我想谈谈为什么并行运行编程智能体具有挑战性。在某种程度上，这不言自明。在大型语言模型（LLM）出现之前，软件工程师和程序员自然只会一次处理一个任务。原因很简单：如果你尝试在软件工程任务中进行多任务处理，最终只会降低效率。

The reason is that software engineering tasks are often more complex and require your full attention. If you try to do other stuff at the same time, the performance on all tasks will likely suffer.
这是因为软件工程任务通常较为复杂，需要全神贯注。如果你试图同时处理其他事情，所有任务的执行效果都可能会大打折扣。

However, that game has changed quite a bit, as you don’t really write all of the code yourself anymore. At least in my opinion, you shouldn’t be writing all of the code yourself anymore, because coding agents should be writing the code for you. As a programmer, you should now act as a manager of coding agents, instead of writing code yourself.
然而，游戏规则已经发生了很大变化，因为你不再需要亲自编写所有代码了。至少在我看来，你不应该再亲自编写所有代码，因为编程智能体应该为你代劳。作为程序员，你现在应该扮演编程智能体管理者的角色，而不是亲自写代码。

However, if you’re a manager of coding agents, you naturally have to handle coding agents performing different tasks. Performing tasks in parallel by nature requires the tasks not to be connected to each other. This then again presents the same challenge as before, where you’re working on programming tasks in parallel, and you need to keep a lot of context in your working memory, and you need to be able to, for example:
然而，如果你是编程智能体的管理者，你自然需要处理执行不同任务的智能体。并行执行任务本质上要求这些任务之间互不干扰。这又带来了与之前相同的挑战：当你并行处理编程任务时，你需要在大脑中保持大量的上下文信息，并且需要能够做到例如：

*   Answer your coding agent whenever it asks you a question.
*   随时回答编程智能体提出的问题。
*   Test the coding agent implementation after it’s done. You need to know what to test, how to test it, and how to ensure the coding agent did what it was supposed to do.
*   在智能体完成工作后测试其实现。你需要知道测试什么、如何测试，以及如何确保智能体完成了预期的工作。

### How to effectively run many parallel coding agents
### 如何高效运行多个并行编程智能体

In this section, I’ll cover some specific techniques that I use and apply daily to effectively run a lot of parallel coding agents.
在本节中，我将介绍一些我每天使用并应用的技术，以高效地运行大量并行编程智能体。

A more effective technique you can apply to easily have a more comprehensive overview of your agents is to use an agent view. For example, the agent view available in Claude Code. A lot of different providers have different ways of presenting this. I know Warp, the terminal, has also recently introduced a new way to view a lot of different agents.
你可以应用一种更有效的技术来轻松获得智能体的全面概览，那就是使用“智能体视图”（Agent View）。例如，Claude Code 中提供的智能体视图。许多不同的提供商有不同的呈现方式。我知道终端工具 Warp 最近也引入了一种查看多个不同智能体的新方法。

The good thing about the agent view is that you don’t have to full-screen each conversation you have with an agent. It simply becomes a single line, which you can press enter on if you want more details on it. If not, it will simply be a task running in the background, and it will ask you for input whenever any of the Claude sessions are asking for input from you.
智能体视图的好处在于，你不必将与每个智能体的对话都全屏显示。它只是变成单行显示，如果你想了解更多细节，只需按回车键即可。否则，它就是一个在后台运行的任务，当任何 Claude 会话需要你输入时，它都会提示你。

I think this is a pretty effective way to work, since you can easily have a lot of different coding sessions running at once without it being confusing which agent is running, which agent needs input, and so on. You can activate the agents view in Claude Code with: `claude agents`
我认为这是一种非常高效的工作方式，因为你可以轻松地同时运行许多不同的编程会话，而不会混淆哪个智能体正在运行、哪个智能体需要输入等问题。你可以通过以下命令在 Claude Code 中激活智能体视图：`claude agents`

### Alert when the coding agent needs input
### 当编程智能体需要输入时发出提醒

Another important thing you can do is to be alerted whenever coding agents need input. Of course, if you use agent mode, you could have this, as it is clearly marked out for you whenever a coding agent needs input from you. However, if you don’t like the agent’s view or want to use something else, there are different options.
另一件重要的事情是，确保在编程智能体需要输入时能收到提醒。当然，如果你使用智能体模式，这通常是内置的，因为每当智能体需要你输入时，它都会有明确的标记。然而，如果你不喜欢智能体视图或想使用其他工具，还有其他选择。

In the image above, you see how I have different Claude Code sessions running in different terminal tabs. The tab includes a star between the title of the tab and the Claude Code icon if it needs input. This is a very simple way for me to see which terminal tabs I need to do something with and which tabs are just running in the background.
在上图中，你可以看到我在不同的终端标签页中运行着不同的 Claude Code 会话。如果某个标签页需要输入，标签标题和 Claude Code 图标之间会出现一颗星号。这对我来说是一种非常简单的方法，可以区分哪些终端标签页需要我处理，哪些只是在后台运行。

Another thing you can do is to also have an audio signal whenever an agent needs input from you. You can, for example, implement this by utilizing hooks in Claude Code, which are processes that run at certain points in time. A hook can, for example, be triggered every time Claude needs input from you, and you can connect this hook to an audio signal that plays, so it informs you whenever one of your coding agents needs input.
你还可以做的另一件事是，在智能体需要输入时设置音频信号提醒。例如，你可以通过利用 Claude Code 中的“钩子”（hooks）来实现这一点，这些钩子是在特定时间点运行的进程。例如，每当 Claude 需要你输入时，钩子就会被触发，你可以将此钩子连接到播放音频信号的程序上，这样每当有编程智能体需要输入时，你都会收到通知。
