# Most AI Agents Fail in Production Because They’re Built Backwards
# 大多数 AI Agent 在生产环境中失败，是因为它们的构建逻辑是倒置的

Good models don't save bad architecture, and most teams learn that the hard way.
优秀的模型无法挽救糟糕的架构，大多数团队都是在吃尽苦头后才明白这一点。

The first time I saw a multi-agent system seriously fail in production, it wasn’t dramatic. There was no crash. No error message. The system just kept running and producing outputs that looked reasonable until someone actually read them carefully enough to notice something was off.
我第一次看到多智能体（multi-agent）系统在生产环境中严重故障时，并没有发生什么戏剧性的场面。没有崩溃，也没有报错信息。系统只是在持续运行并输出看起来合理的结论，直到有人仔细阅读后才发现有些不对劲。

When we decided to look into it, it took us two days’ worth of debugging to figure out what was going on. Funny enough, the model wasn’t hallucinating, and the input-output tools were delivering the correct results. The problem, when we finally found it, was architectural. The model and the tools were set up correctly, but the idea was that reasoning would tie the whole thing together, which, as you would guess, obviously failed. Turns out reasoning does not do that sort of thing.
当我们决定深入调查时，花了整整两天时间调试才弄清楚发生了什么。有趣的是，模型并没有产生幻觉，输入输出工具也提供了正确的结果。当我们最终找到问题所在时，发现是架构问题。模型和工具的设置都是正确的，但我们原本寄希望于通过“推理”将整个流程串联起来，正如你所料，这显然失败了。事实证明，推理并不能起到这种作用。

That experience is what I keep coming back to when I think about why so many AI agents that work in demos don’t really survive real-world use. It’s not a capability problem. It’s an architectural one. And if you’ve read my previous piece here on TDS, *Why AI Engineers Are Moving Beyond LangChain to Native Agent Architectures*, the pattern should sound familiar: systems built top-down, from goal to tools to model, with the quiet assumption that intelligent behavior fills in the gaps. That assumption is what “built backwards” means. And it’s more common than most teams realize until something breaks.
每当我思考为什么那么多在演示中表现良好的 AI Agent 无法在现实世界中存活时，我总会想起那次经历。这不是能力问题，而是架构问题。如果你读过我之前在 TDS 上发表的文章《为什么 AI 工程师正在超越 LangChain 转向原生 Agent 架构》，你应该会对这种模式感到熟悉：系统自上而下构建，从目标到工具再到模型，并默认智能行为会自动填补中间的空白。这种假设就是所谓的“构建逻辑倒置”。在系统崩溃之前，大多数团队都没有意识到这一点。

### Agents Aren’t Entities. They’re Systems.
### Agent 不是实体，而是系统。

A production AI agent isn’t a single intelligent thing. Rather, there is a set of interacting pieces with different responsibilities, failure modes, and levels of observability. The LLM is one of those components, not the whole system. Just one piece of it. It may sound obvious when you say it out loud. But the “autonomous agent” framing that dominated 2023 and most of 2024 kept pulling engineers toward a different mental model: one entity, one reasoning loop, everything handled by the model. All you need is tools, a good system prompt, and a hope that everything will fall into place.
生产环境中的 AI Agent 并不是一个单一的智能体。相反，它是一组具有不同职责、故障模式和可观测性水平的交互组件。LLM 只是其中的一个组件，而不是整个系统。仅仅是其中的一部分。说出来可能觉得显而易见，但 2023 年和 2024 年大部分时间里主导的“自主智能体”框架，一直将工程师引向一种不同的思维模型：一个实体、一个推理循环，一切都由模型处理。你所需要的只是工具、一个好的系统提示词，以及对一切能顺利运行的祈祷。

In contrast, engineers who have shipped real AI-based products rarely describe their systems that way. What they actually describe sounds a lot more like distributed systems architecture. Not because they read a book about design patterns, but because they got burned enough times that they started putting structure more seriously in their workflow.
相比之下，那些真正交付过 AI 产品的工程师很少这样描述他们的系统。他们实际描述的更像是分布式系统架构。这并不是因为他们读过设计模式的书，而是因为他们吃过太多亏，开始更认真地在工作流中引入结构化设计。

Building top-down, starting from “what should this agent do” and working backwards into tools and prompts, is quick to get started. It’s also how you end up with a system where the model is responsible for too much, and nothing is individually debuggable. The architecture was decided by the goal, not by the engineering requirements. That’s the backwards part.
自上而下地构建——从“这个 Agent 应该做什么”开始，再反向推导工具和提示词——上手很快。但这也是导致系统模型负担过重、且没有任何部分可以单独调试的原因。架构是由目标决定的，而不是由工程需求决定的。这就是“倒置”的部分。

### So What Really Goes Into a Production System?
### 那么，生产系统到底需要什么？

The abstract version is easy to nod along to. Here’s what it actually looks like. Every production AI system I have seen that works cleanly has something like a decision layer, whether the team named it like this or not. It’s the part where the model lives and does its actual job. The instinct is to push everything into this layer: parsing requests, managing memory, handling retries, resolving tool failures. This is okay if you’re working in a Jupyter notebook. In production, under load, with real users, this becomes the part of your system where everything is everyone’s fault, and most times, nothing can be debugged.
抽象的理论很容易让人点头赞同，但实际情况是这样的：我所见过的每一个运行良好的生产级 AI 系统，都有一个类似“决策层”的东西，无论团队是否这样命名它。这是模型存在并执行其核心任务的地方。人们的直觉是把所有事情都塞进这一层：解析请求、管理内存、处理重试、解决工具故障。如果你是在 Jupyter Notebook 里工作，这没问题。但在生产环境、高负载且面对真实用户时，这会变成系统中最混乱的部分——所有问题都混在一起，大多数时候根本无法调试。

The decision layer should do one thing well, and that is deciding what to do next, given a certain context that is already prepared for it. That’s the whole job. Who prepares the context? Something else. Who acts on the decision? Also something else. That “something else” is the orchestration layer, and in most well-built systems, it’s genuinely just code: conditionals, asynchronous runners, retry handling, queue routing, maybe even a state machine depending on how involved the workflow is.
决策层应该只做好一件事：在已经准备好的上下文基础上，决定下一步做什么。这就是它的全部工作。谁来准备上下文？由其他组件负责。谁来执行决策？也由其他组件负责。那个“其他组件”就是编排层（orchestration layer）。在大多数构建良好的系统中，它本质上就是代码：条件判断、异步运行器、重试处理、队列路由，甚至根据工作流的复杂程度引入状态机。

Instead of expecting the model to do everything, treat it like just another component. Here, standard code does the heavy lifting with state and tools, so the LLM only has to worry about making the next decision.
不要指望模型做所有事情，把它当作另一个组件来对待。在这里，标准代码负责处理状态和工具等繁重工作，这样 LLM 只需要专注于做出下一个决策。

Many teams reach for frameworks here because bare orchestration code feels too simple, like surely there’s supposed to be more infrastructure. There usually isn’t. The less magic this layer contains, the faster you’ll find bugs when they appear. And they will appear.
许多团队在这里倾向于使用框架，因为纯粹的编排代码看起来太简单了，让人觉得似乎应该有更多的基础设施。但通常并没有。这一层包含的“魔法”越少，当 Bug 出现时，你定位问题的速度就越快。而且，Bug 迟早会出现。

From experience, I learned this the hard way on a project where the orchestration lived inside a framework’s execution model. Something was retrying tool calls in a way that was corrupting state downstream. We spent two days finding the issue. Two days for a bug that could have been resolved in no time at all if the retry logic had been three lines of Python I wrote myself.
根据经验，我曾在一个项目中吃过大亏，当时编排逻辑被封装在框架的执行模型内部。某种机制在重试工具调用时破坏了下游的状态。我们花了整整两天时间才找到问题。如果重试逻辑是我自己写的几行 Python 代码，这个问题本可以在瞬间解决。

This leads us to the tools and execution layer, where all communication happens. Now, the tools and execution layer is where things talk to the outside world. This layer usually has just one job, and that is to take a well-defined input and then produce a predictable output. But the failure I kept seeing, and kept repeating, honestly, was tools that tried to be helpful by doing more than one thing. A single function that calls an API, updates a cache, and does other things. In a setup like that, when it breaks, you don’t know where. Even when you try to replace the API, you’re untangling logic that shouldn’t have been tangled in the first place.
这就引出了工具和执行层，这是所有通信发生的地方。工具和执行层是系统与外部世界交互的接口。这一层通常只有一个任务：接收定义明确的输入，然后产生可预测的输出。但我不断看到（且自己也曾不断重复）的错误是：工具试图通过做多件事来表现得“乐于助人”。比如一个函数既调用 API，又更新缓存，还做其他事情。在这种设置下，一旦出错，你根本不知道问题出在哪。即使你想替换 API，也会发现自己陷入了本不该纠缠在一起的逻辑泥潭中。

Memory and state is where I’d push hardest, because it’s where most teams are most underprepared. Most teams think about memory as “what the model knows.” The more important question is what the system knows, and whether that knowledge is current.
关于内存和状态，我要强调一点，因为这是大多数团队准备最不足的地方。大多数团队认为内存就是“模型知道什么”。但更重要的问题是：系统知道什么，以及这些知识是否是最新的。
