# The Infrastructure Behind Making Local LLM Agents Actually Useful
# 让本地 LLM Agent 真正实用的基础设施

**Lessons from building a fast, reliable scientific agent with local open-weight models, vLLM, and long-context infrastructure**
**利用本地开源权重模型、vLLM 和长上下文基础设施构建快速、可靠的科学 Agent 的经验教训**

Hussen Mohammed Ibrahim | May 28, 2026

---

Running a language model locally sounds straightforward. Download the weights, start the server, and send requests. That works for a chatbot, but it doesn’t automatically work for an agent. In my case, I’ve been building an agent for automated single-cell RNA-seq analysis. The idea is that, given raw data, the agent can run the full pipeline on its own, deciding which tools to call, reading the results, and working through the analysis step by step.

在本地运行语言模型听起来很简单：下载权重、启动服务器、发送请求。这对于聊天机器人来说确实可行，但对于 Agent 来说却并非如此。以我为例，我一直在构建一个用于自动化单细胞 RNA-seq 分析的 Agent。其核心构想是：给定原始数据，Agent 可以独立运行整个流程，决定调用哪些工具、读取结果，并逐步完成分析。

You might ask why not just use something like Claude Code with a single-cell analysis Skill. The short answer is that for scientific workflows, that’s not quite enough. Skills are ultimately prompts and can thus be overridden or ignored. More importantly, scientific work requires reproducibility and provenance tracking: knowing exactly which parameters were used, which cells were filtered, which clustering resolution produced which result, etc. That record needs to be structured and persistent, not reconstructed from a conversation.

你可能会问，为什么不直接使用像 Claude Code 这样带有单细胞分析技能（Skill）的工具呢？简短的回答是，对于科学工作流来说，这还不够。技能本质上是提示词（Prompts），因此可能会被覆盖或忽略。更重要的是，科学工作需要可重复性和溯源追踪：必须确切知道使用了哪些参数、过滤了哪些细胞、哪种聚类分辨率产生了什么结果等。这些记录需要是结构化且持久的，而不是从对话中重新构建出来的。

For long-running sessions, you also need explicit world state management rather than relying on context compaction to preserve what matters. These are things you have to build deliberately. Building all of these on top of a local model also means you own the infrastructure, and that’s what I’m going to be focusing on here. The agent we built runs on institutional HPC hardware using recent open-weight models.

对于长时间运行的会话，你需要明确的“世界状态管理”（World State Management），而不是仅仅依赖上下文压缩来保留重要信息。这些都需要刻意去构建。在本地模型之上构建所有这些功能，也意味着你拥有了基础设施，而这正是我接下来要重点讨论的内容。我们构建的 Agent 运行在机构的高性能计算（HPC）硬件上，并使用了最新的开源权重模型。

It is easy to assume open-weight models are not strong enough for this kind of work. But that is becoming less true. Recent releases like Qwen3.6–27B and Gemma 4–31B are genuinely useful for structured, tool-driven workloads. And that’s one of the main reasons why local hosting makes sense here. Our agent also supports cloud APIs like Claude and GPT, but when you use those, all of the infrastructure I’m about to describe is invisible to you. Someone else has already solved it. When you host the model yourself, those problems become yours.

人们很容易认为开源权重模型不足以胜任此类工作，但事实正在改变。像 Qwen3.6–27B 和 Gemma 4–31B 等近期发布的模型，在处理结构化、工具驱动的工作负载时确实非常有用。这也是为什么本地部署在此场景下具有重要意义的主要原因之一。我们的 Agent 也支持 Claude 和 GPT 等云端 API，但当你使用它们时，我即将描述的所有基础设施对你来说都是不可见的——因为别人已经帮你解决了。而当你自行托管模型时，这些问题就成了你自己的挑战。

When I ran the model the first time, it worked in a narrow sense. The model would call tools, the tools would run, and the analysis would move forward. But it wasn’t really usable yet. A simple single-cell analysis could have 50–80 tool calls in a loop. Every call carried the same fixed baggage: the system prompt, the tool schemas, and the growing conversation history. For this agent, the system prompt and tool schemas alone were about 36k tokens. Before the model could decide anything, it first had to read tens of thousands of tokens of instructions and tool definitions. Then it had to do that again on the next iteration. And again on the one after that. Each iteration took 10 to 15 seconds. And a long session would eventually crash out with context overflow errors, taking all the in-memory analysis state with it.

当我第一次运行该模型时，它在狭义上是可用的：模型会调用工具，工具会运行，分析也能推进。但它还谈不上真正“好用”。一个简单的单细胞分析可能需要在循环中进行 50 到 80 次工具调用。每次调用都携带相同的“固定负担”：系统提示词、工具架构定义以及不断增长的对话历史。对于这个 Agent 来说，仅系统提示词和工具架构就占了约 3.6 万个 Token。在模型做出任何决定之前，它必须先读取数万个 Token 的指令和定义，然后在下一次迭代中再次读取，周而复始。每次迭代需要 10 到 15 秒。长时间的会话最终会因为上下文溢出错误而崩溃，导致内存中的所有分析状态丢失。

This article is about fixing both of those problems. The first part covers making inference faster through a set of compounding optimizations to the vLLM inference server. The second part covers keeping long sessions alive through better context management and a structured world state that survives trimming. I ran experiments on A100 and H100 GPUs to measure the impact of each change, and those are described below.

本文旨在解决这两个问题。第一部分涵盖了通过对 vLLM 推理服务器进行一系列复合优化来提高推理速度；第二部分涵盖了通过更好的上下文管理和在修剪后仍能存续的结构化世界状态，来维持长会话的稳定性。我在 A100 和 H100 GPU 上进行了实验以衡量每项改进的影响，具体内容如下。

### Part 1: Making Inference Fast
### 第一部分：加速推理

Before getting into the individual optimizations, it helps to understand what’s actually happening on each iteration of the agent loop. The diagram below shows a single iteration: the agent sends a request containing the system prompt, tool schemas, and the full conversation history to the model. The model reads all of it and decides which tools to call. The tool runs and returns a result, and that result gets appended to the history before the next iteration begins.

在深入探讨各项优化之前，了解 Agent 循环每次迭代中实际发生的事情很有帮助。下图展示了单次迭代的过程：Agent 向模型发送包含系统提示词、工具架构和完整对话历史的请求。模型读取所有内容并决定调用哪些工具。工具运行并返回结果，该结果在下一次迭代开始前被追加到历史记录中。

Two things are worth noting here. The fixed prefix, which is the system prompt plus tool schemas, is roughly 36k tokens and gets sent on every single call. And the conversation history grows with every iteration. By iteration 40, the model is no longer reading a short instruction. It’s reading a long analysis transcript with many tool calls, tool outputs, intermediate results, etc. Both of these things affect the performance of the agent.

这里有两点值得注意。首先，作为“固定前缀”的系统提示词和工具架构大约有 3.6 万个 Token，且每次调用都会被发送。其次，对话历史会随着每次迭代而增长。到了第 40 次迭代时，模型读取的不再是简短的指令，而是一份包含大量工具调用、输出和中间结果的长分析记录。这两者都会影响 Agent 的性能。

#### 1.1 CUDA Graphs: Reducing Hundreds of Instructions Per Token to One
#### 1.1 CUDA Graphs：将每个 Token 的数百条指令减少为一条

To understand this one, it helps to know what happens inside a GPU when it generates a single token. Generating a single token during the decode phase involves executing a sequence of GPU kernels in order: attention, feed-forward, normalization, and so on. Each of these kernel launchers has a small coordination cost on the CPU side. The CPU has to queue an instruction telling the GPU exactly which kernel to run, with which tensor shapes and memory pointers. For a 27-billion-parameter model we’ve been working with, this means hundreds of individual dispatches per token. Each one is small, but they add up.

要理解这一点，需要了解 GPU 在生成单个 Token 时内部发生了什么。在解码阶段生成单个 Token 涉及按顺序执行一系列 GPU 内核：注意力机制、前馈网络、归一化等。每个内核启动器在 CPU 端都有微小的协调开销。CPU 必须排队发送指令，告诉 GPU 确切运行哪个内核、使用哪些张量形状和内存指针。对于我们使用的 270 亿参数模型，这意味着每个 Token 需要数百次单独的调度。虽然每次调度都很小，但累积起来就很可观。

CUDA graphs eliminate this overhead. Before handling any real requests, vLLM can run a warmup pass where it records all the kernel dispatches for the decode step into one single replayable object. After that, generating each token is one instruction to the GPU instead of hundreds. The result is roughly 20–25% lower latency from this single change, with no change to the model itself.

CUDA Graphs 消除了这种开销。在处理任何实际请求之前，vLLM 可以运行一个预热过程，将解码步骤的所有内核调度记录到一个可重放的对象中。此后，生成每个 Token 只需向 GPU 发送一条指令，而不是数百条。仅此一项改动，在不改变模型本身的情况下，延迟就降低了约 20%–25%。
