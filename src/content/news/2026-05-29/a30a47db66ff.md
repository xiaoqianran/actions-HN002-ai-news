# DiffuJudge-AV: A Diffusion-Inspired Framework for Calibrated AV Video Evaluation

**DiffuJudge-AV: A Diffusion-Inspired Framework for Calibrated AV Video Evaluation**
DiffuJudge-AV：一种用于自动驾驶视频校准评估的扩散启发式框架

A diffusion-inspired framework for stress-testing and denoising LLM-as-a-Judge pipelines, applied to safety-critical driving video.
这是一个受扩散模型启发，用于对“大模型作为裁判”（LLM-as-a-Judge）流水线进行压力测试和去噪的框架，并应用于安全关键型驾驶视频评估。

***

**I Treated an LLM Judge Like a Noisy Sensor. It Changed Which Autonomous-Driving Evaluator I Would Ship.**
**我把大模型裁判当作一个噪声传感器，这改变了我对自动驾驶评估器的选择。**

There is a particular kind of result that looks impressive until you ask the wrong second question. In this project, that result was a Pearson correlation of 0.753 from a text-only Claude judge grading autonomous-driving visual-QA answers. At first glance, that looks like a usable evaluator. It tracks the gold scores, it produces rationales, it is a strong closed model. Good enough to triage model outputs, right?
有一种结果看起来非常惊人，直到你问出第二个关键问题。在本项目中，这个结果是仅基于文本的 Claude 模型在评估自动驾驶视觉问答（VQA）时，得出的 0.753 皮尔逊相关系数。乍一看，这似乎是一个可用的评估器。它能追踪标准分数，能生成理由，且是一个强大的闭源模型。用来筛选模型输出应该足够了吧？

Then I looked at quadratic-weighted Cohen’s κ. It was 0.057. That is the moment the project changed. The judge was rank-correlated with the gold labels, but it was not behaving like an ordinal safety evaluator. It had learned the safest-looking failure mode: compress almost everything toward the middle of the 1–5 scale. For ordinary benchmark reporting, that might pass unnoticed. For an autonomous-driving review pipeline that needs to flag bad answers before they gate a software release, it is dangerous.
然而，当我查看二次加权 Cohen’s κ 系数时，它仅为 0.057。那一刻，项目方向发生了改变。该裁判虽然与标准标签存在秩相关性，但其表现并不像一个序数安全评估器。它学会了最“安全”的失效模式：将几乎所有分数压缩到 1-5 分制的中间值。对于普通的基准测试报告，这可能不会被察觉；但对于需要在软件发布前拦截错误答案的自动驾驶审查流水线来说，这是极其危险的。

So I built DiffuJudge-AV, a small evaluation-of-evaluation framework for LLM/VLM judges on driving video. The idea is simple: treat a judge’s score as a noisy observation of a latent true rubric score, deliberately expose the judge to known sources of scoring bias, then denoise the resulting score distribution with a one-step Tweedie posterior mean and report calibrated uncertainty.
因此，我构建了 DiffuJudge-AV，这是一个针对驾驶视频领域 LLM/VLM 裁判的“评估之评估”（evaluation-of-evaluation）框架。其核心理念很简单：将裁判的分数视为对潜在真实评分的噪声观测，故意让裁判暴露在已知的评分偏差源中，然后利用一步 Tweedie 后验均值对所得的分数分布进行去噪，并报告校准后的不确定性。

Across 28,400 judge evaluations on Wayve’s LingoQA benchmark, the most interesting finding was not that a larger closed model won. It did not. The best judge in the experiment was Qwen2.5-VL-7B, an open 7B vision-language model.
在 Wayve 的 LingoQA 基准测试中，通过 28,400 次裁判评估，最有趣的发现并非“更大的闭源模型获胜”——事实并非如此。实验中表现最好的裁判是 Qwen2.5-VL-7B，一个开源的 7B 参数视觉语言模型。

It reached:
它达到了：
*   Pearson r = 0.857
*   Spearman ρ = 0.856
*   Quadratic-weighted Cohen’s κ = 0.837
*   MAE = 0.57
*   Fail-detection F1 = 0.712

Note: The LingoQA benchmark is released under a non-commercial license. The dataset creators at Wayve have granted permission for its use in this article. For this AV-style evaluation task, an open VLM was not just competitive. It was better on the metrics that actually matter.
注：LingoQA 基准测试以非商业许可发布。Wayve 的数据集创建者已授权本文使用该数据。对于这种自动驾驶风格的评估任务，开源 VLM 不仅具有竞争力，而且在真正关键的指标上表现更优。

***

**Why “evaluation of evaluation”?**
**为什么要进行“评估之评估”？**

When a model answers a question about a driving scene, the obvious evaluation question is: Did the model answer correctly? For example:
当模型回答关于驾驶场景的问题时，最直观的评估问题是：模型回答正确吗？例如：
*   Question: Are there any parked cars on the side of the road? (问题：路边有停放的车辆吗？)
*   Reference: Yes, there are two cars parked on the right. (参考答案：是的，右侧停了两辆车。)
*   Candidate answer (model under test): I don’t know. (候选答案（被测模型）：我不知道。)
*   Gold score: 1.13 (low). (标准分数：1.13（低分）。)

For a human, this is easy. Watch the clip, compare the answer to the scene, assign a score. At scale though, human evaluation becomes the bottleneck. Modern autonomy stacks generate more perception clips, scenario logs, counterfactual rollouts, and model outputs than any annotation team can score manually. So teams naturally reach for LLM-as-a-Judge or VLM-as-a-Judge: give a model the question, reference, candidate answer, rubric, and sometimes the frames, then ask it to score.
对人类来说，这很简单：观看视频片段，将答案与场景对比，然后打分。但在大规模场景下，人工评估成为了瓶颈。现代自动驾驶技术栈生成的感知片段、场景日志、反事实推演和模型输出的数量，远超任何标注团队的手动处理能力。因此，团队自然会转向“大模型作为裁判”：将问题、参考答案、候选答案、评分标准，有时还有视频帧交给模型，然后让它打分。

That creates a second-order problem: If the judge is a model, how do we know the judge is reliable? This is evaluation of evaluation (eval-of-eval). Instead of only asking whether the AV model is correct, we ask whether the evaluator itself is stable, calibrated, bias-resistant, and useful for downstream decisions.
这产生了一个二阶问题：如果裁判本身也是模型，我们如何确保裁判是可靠的？这就是“评估之评估”（eval-of-eval）。我们不再仅仅询问自动驾驶模型是否正确，而是询问评估器本身是否稳定、是否经过校准、是否抗偏差，以及对下游决策是否有用。

***

**The intuition: a judge score is a noisy sensor reading**
**直觉：裁判分数是一种噪声传感器读数**

An LLM judge score looks clean because it is a number: 1, 2, 3, 4, or 5. But that number can move for reasons that have nothing to do with the actual quality of the answer. Change the order of options. Paraphrase the rubric. Reorder criteria. Swap score labels from Arabic numerals to Roman. Resample exemplars. Change temperature. Shuffle the video frames you sample. The true answer quality did not change. The judge changed.
大模型裁判的分数看起来很整洁，因为它只是一个数字：1、2、3、4 或 5。但这个数字的变动可能与答案的实际质量毫无关系。改变选项顺序、改写评分标准、重排评估准则、将分数标签从阿拉伯数字换成罗马数字、重新采样示例、改变温度参数、打乱采样的视频帧顺序——答案的真实质量没变，但裁判变了。

That suggests a useful mental model: Treat the judge like a noisy sensor. There is a latent score $s_0$. The judge never observes it directly. Each prompt variant produces a noisy reading $\tilde{s}_t = s_0 + \epsilon_t, \quad t \in \{1, \ldots, 7\}$.
这提供了一个有用的思维模型：将裁判视为一个噪声传感器。存在一个潜在的真实分数 $s_0$，但裁判无法直接观测到它。每一个提示词变体都会产生一个噪声读数 $\tilde{s}_t = s_0 + \epsilon_t, \quad t \in \{1, \ldots, 7\}$。
