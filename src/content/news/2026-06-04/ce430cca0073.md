### 2. Why does padding exist at all? (a one-minute crash course)
### 2. 为什么会有填充？（一分钟速成课）

Skip this if you already know. For everyone else: A GPU loves rectangles. Specifically, it loves matrices where every row is the same length, because that lets it run thousands of identical math operations in parallel. This is what makes a GPU a GPU instead of an expensive paperweight.
如果你已经知道了，请跳过这一节。对于其他人：GPU 热爱矩形。具体来说，它喜欢每一行长度相同的矩阵，因为这让它能够并行运行数千个相同的数学运算。这就是让 GPU 成为 GPU 而不是昂贵镇纸的原因。

But text is not a rectangle. Text is a ragged mess. One tweet is 12 tokens. One legal contract is 4,000. If you want to feed 8 of them at once through a transformer, you have two options:
但文本不是矩形。文本是一团乱麻。一条推文是 12 个 token，一份法律合同是 4000 个。如果你想一次性将 8 个文本输入 Transformer，你有两个选择：

1. **Pad them all to the longest one with a <pad> token.** The math runs on rectangles, your code is simple, and your GPU now spends most of its time multiplying zero by another zero. (HuggingFace’s default.)
1. **用 <pad> token 将它们全部填充到最长序列的长度。** 数学运算在矩形上运行，代码简单，但你的 GPU 现在大部分时间都在做零乘以零的运算。（HuggingFace 的默认做法。）

2. **Concatenate them into one long 1-D ribbon and tell the attention kernel “hey, please don’t let token 12 talk to token 4001, they’re from different documents.”** This is variable-length attention, and it’s what FlashAttention-2’s flash_attn_varlen_func exists to do.
2. **将它们连接成一条长的一维带，并告诉注意力算子“嘿，请不要让第 12 个 token 与第 4001 个 token 通信，它们来自不同的文档。”** 这就是变长注意力机制，也是 FlashAttention-2 的 `flash_attn_varlen_func` 存在的原因。

Option 2 is obviously better, and a growing number of production inference stacks (vLLM, TensorRT-LLM, SGLang, FlashInfer, TGI internals) already use variants of it. But calling flash_attn_varlen_func is the easy part. The hard part — the part everyone keeps re-implementing in slightly different shapes — is organizing the ribbon: deciding which documents go in which batch, in what order, up to what total length, while keeping the GPU saturated and the host-side overhead invisible. WarpGroup-Backend is the annoying-organizer half of that equation, written in C++ so it can be fast about it.
选项 2 显然更好，越来越多的生产推理栈（vLLM、TensorRT-LLM、SGLang、FlashInfer、TGI 内部实现）已经在使用它的变体。但调用 `flash_attn_varlen_func` 只是简单部分。困难的部分——每个人都在以略微不同的形式重复实现的部分——是组织这条“带”：决定哪些文档进入哪个批次、以什么顺序、达到什么总长度，同时保持 GPU 满载并使主机端的开销不可见。WarpGroup-Backend 就是这个等式中那个“烦人的组织者”部分，用 C++ 编写以确保其速度。

### 3. The “just pack them” lightbulb (and why it’s harder than it sounds)
### 3. “直接打包”的灵感（以及为什么它比听起来更难）

The pitch is simple: take your queue of variable-length sequences and pack them into bins like a really intense moving-day grandma. Each bin holds up to N tokens total. You stuff as many small sequences as you can fit alongside each big one, then ship the whole bin to the GPU as one flat ribbon.
核心思路很简单：把你队列中的变长序列像一位极其认真的搬家老奶奶一样装进箱子里。每个箱子最多容纳 N 个 token。你尽可能多地塞入小序列来填补大序列旁边的空隙，然后将整个箱子作为一个平坦的“带”发送给 GPU。

This is a classic bin-packing problem, and the textbook solution is First-Fit Decreasing (FFD): Sort sequences from longest to shortest. For each sequence, walk through the open bins. Drop it in the first one that has room. If none have room, open a new bin.
这是一个经典的装箱问题，教科书式的解决方案是“首次适应递减算法”（FFD）：将序列从长到短排序。对于每个序列，遍历已打开的箱子。将其放入第一个有空间的箱子中。如果没有箱子有空间，则打开一个新箱子。

This is the same algorithm your brain uses when packing a suitcase: big rocks first, then pebbles fill the cracks. It’s not optimal, but it gets within ~22% of optimal in the worst case while being O(n log n) instead of taking until the heat death of the universe.
这和你打包行李箱时大脑使用的算法是一样的：先放大的，再用小的填补缝隙。它不是最优解，但在最坏情况下也能达到最优解的约 22% 以内，且时间复杂度为 O(n log n)，而不是等到宇宙热寂。

Now here are the three things that make this not a 10-line Python script:
现在，以下三点让这不仅仅是一个 10 行的 Python 脚本：

**Problem A: How big should the bin be?**
**问题 A：箱子应该多大？**

Easy answer: “as big as VRAM lets me.” Wrong answer: that’s not a number you can look up. The actual usable VRAM depends on:
简单回答：“显存允许的最大值。” 错误回答：这不是一个你可以直接查到的数字。实际可用的显存取决于：

* Model size (a 7B in bf16 eats ~14 GB before you’ve done anything).
* 模型大小（一个 bf16 格式的 7B 模型在做任何事之前就占用了约 14 GB）。
* The activation/KV-cache memory for this specific sequence length (which scales weirdly).
* 该特定序列长度的激活/KV 缓存内存（其缩放方式很奇怪）。
* PyTorch’s allocator fragmentation (an art, not a science).
* PyTorch 的分配器碎片（这是一门艺术，而非科学）。
* Whether you sneezed on the driver.
* 你是否对驱动程序“打了个喷嚏”（指不可控的外部干扰）。

You can’t compute this. You have to measure it. Phase 0 of WarpGroup is literally “the GPU’s job interview” — keep asking it to swallow larger and larger sequences until it OOMs, then back off 10% for safety.
你无法计算出这个值，你必须测量它。WarpGroup 的第 0 阶段实际上是“GPU 的面试”——不断要求它吞下越来越长的序列，直到它显存溢出（OOM），然后为了安全起见回退 10%。

**Problem B: GPUs are picky eaters (the 16-token thing)**
**问题 B：GPU 是挑食的（关于 16 个 token 的事）**

NVIDIA Tensor Cores execute their matrix multiply-accumulate (MMA) instructions on fixed tile shapes — for example m16n8k16 on Hopper, with the exact dimensions varying by datatype and architecture.
NVIDIA Tensor Core 在固定的瓦片（tile）形状上执行矩阵乘加（MMA）指令——例如 Hopper 架构上的 m16n8k16，具体维度因数据类型和架构而异。
