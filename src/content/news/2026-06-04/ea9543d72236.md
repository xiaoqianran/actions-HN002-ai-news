# What AI Agents Should Never Do on Their Own
# AI 智能体绝不应独自执行的任务

Agentic AI: What AI Agents Should Never Do on Their Own. How to set the rules that keep agents effective and out of trouble.
智能体 AI：AI 智能体绝不应独自执行的任务。如何设定规则，让智能体保持高效并避免麻烦。

All the content about agents focuses on what they can do. Autonomy gets framed as the goal: give them tools, give them access, let them run. The more freedom, the better the output. That framing is mostly accurate. I use agents daily. They’ve genuinely increased my output. I’m a believer!
关于智能体的所有内容都聚焦于它们“能做什么”。自主性被视为目标：给它们工具，给它们权限，让它们运行。自由度越高，产出就越好。这种说法大体上是准确的。我每天都在使用智能体，它们确实提高了我的产出。我是它们的忠实拥护者！

And I’ve also lost two hours of work through an agent that was doing exactly what I asked. I was working on a feature branch cleanup. The task description said “remove unused files and clean up the repo.” The agent interpreted “unused” broadly, deleted a config directory I hadn’t touched in months but still referenced from the deploy script, and kept going. I caught it during the diff review. The config wasn’t in version control. Two hours reconstructing it from memory and git history. The task was clear and the agent followed instructions, the only problem was that nothing told it where to stop.
但我也曾因为一个完全按照我要求执行任务的智能体而损失了两个小时的工作。当时我正在清理一个功能分支，任务描述写着“删除未使用的文件并清理仓库”。智能体对“未使用”的定义过于宽泛，删除了一个我几个月没动过但部署脚本仍在引用的配置目录，并且继续执行。我在差异审查（diff review）时发现了这一点。该配置不在版本控制中，我花了两个小时凭记忆和 git 历史记录才将其恢复。任务描述很清晰，智能体也遵循了指令，唯一的问题是没有任何规则告诉它在哪里停止。

Knowing which tasks to gate is part of running agents well. Give them full freedom on the wrong category and you’ll spend the afternoon undoing what took them thirty seconds.
知道哪些任务需要设置门槛，是高效使用智能体的关键。如果在错误的类别上给予它们完全的自由，你将不得不花一下午的时间去撤销它们仅用三十秒就造成的后果。

### What the agent should never touch alone
### 智能体绝不应独自触碰的任务

Some tasks are reversible. For example, a refactored function can be reverted or a new unit test can be removed. The cost of a mistake is low. Recovery cost varies by task. A refactored function takes seconds to revert; you just revert the commit, but a dropped production table might take your entire week, if recovery is even possible. The question before you run a task: can this be undone? If yes, let the agent move. If no, add a checkpoint before it runs.
有些任务是可逆的。例如，重构后的函数可以撤销，新的单元测试可以删除。犯错的代价很低。恢复成本因任务而异。重构函数只需几秒钟即可撤销；你只需回滚提交即可。但如果删除了生产环境的数据库表，可能需要你整整一周的时间来恢复，甚至根本无法恢复。在运行任务前请问自己：这可以撤销吗？如果可以，让智能体去执行；如果不可以，请在运行前添加一个检查点。

### The categories that should always require a human
### 始终需要人工介入的类别

Some categories require a human checkpoint regardless of how well-specified the task is. The risk of a mistake is too high, and the recovery cost too steep, to let an agent decide on its own.
无论任务描述多么详尽，有些类别始终需要人工检查点。犯错的风险太高，恢复成本太昂贵，不能让智能体自行决定。

**1. Destructive file operations**
`rm -rf`, `git clean -fd`, `git reset --hard`. These delete or discard work that may not be recoverable. An agent will run them if the task description implies cleanup. I’ve had one run `git clean -fd` in the middle of a refactor because the task said “clean up temporary files.” My uncommitted work was gone. There was no malfunction, as the agent did exactly what the words said. The safeguard is an explicit block list with a confirmation step, not trusting the agent to infer where “clean up” ends.
**1. 破坏性文件操作**
`rm -rf`、`git clean -fd`、`git reset --hard`。这些操作会删除或丢弃可能无法恢复的工作。如果任务描述暗示了“清理”，智能体就会执行这些命令。我曾遇到过一个智能体在重构过程中运行了 `git clean -fd`，因为任务要求“清理临时文件”。我未提交的工作就这样没了。这并非故障，因为智能体完全按照字面意思执行了指令。保障措施是建立一个明确的黑名单并设置确认步骤，而不是指望智能体去推断“清理”的边界在哪里。

**2. Database writes and migrations**
Any DELETE without a WHERE clause, any DROP or TRUNCATE, any schema migration touching production data. A typo in a WHERE clause can wipe a table. A migration that runs out of order can corrupt data that’s impossible to reconstruct. Always review before running.
**2. 数据库写入与迁移**
任何没有 WHERE 子句的 DELETE 操作、任何 DROP 或 TRUNCATE 操作，以及任何涉及生产数据的模式迁移。WHERE 子句中的一个拼写错误就可能抹掉整张表。顺序错误的迁移可能会破坏无法重建的数据。运行前务必进行审查。

**3. Cloud infrastructure**
`terraform apply`, `kubectl delete`, `aws iam *`, `gcloud iam *`. Infrastructure changes affect live systems and often other teams. Permissions changes are especially dangerous because the damage can be invisible until something fails.
**3. 云基础设施**
`terraform apply`、`kubectl delete`、`aws iam *`、`gcloud iam *`。基础设施变更会影响线上系统，通常还会波及其他团队。权限变更尤其危险，因为在系统崩溃之前，你可能根本察觉不到损害。

**4. Production deployments**
Any deployment to a production environment should go through a human review step, even if the code was agent-generated. CI/CD pipelines can run agent output automatically, and that’s fine. The decision to deploy to production is yours. You know what’s in flight, what incidents are open, what maintenance is scheduled. The agent doesn’t have any of that context, and it can’t ask for it mid-pipeline.
**4. 生产环境部署**
任何向生产环境的部署都应经过人工审查步骤，即使代码是由智能体生成的。CI/CD 流水线可以自动运行智能体的输出，这没问题。但决定是否部署到生产环境的权力在你手中。你知道哪些功能正在上线、哪些事故尚未解决、哪些维护计划正在进行。智能体没有任何这些背景信息，且无法在流水线运行中途询问。

**5. Auth and security logic**
Authentication flows, authorization rules, token handling, session management. Bugs here don’t show up in unit tests, they show up in incident reports, sometimes months later. An agent writing auth logic will produce something that looks correct and passes the happy path. The dangerous cases are the edge conditions: a token that doesn’t expire under a specific sequence of API calls, a route that bypasses middleware when a parameter is missing. Those are exactly what unit tests miss and what security review catches. Every auth change needs a human who’s specifically looking for those gaps, not one who’s satisfied the happy path is covered.
**5. 认证与安全逻辑**
认证流程、授权规则、令牌处理、会话管理。这里的漏洞不会出现在单元测试中，而是出现在事故报告里，有时甚至在几个月后才显现。智能体编写的认证逻辑看起来很正确，也能通过“快乐路径”（happy path）测试。危险在于边缘情况：例如在特定 API 调用序列下令牌不会过期，或者在缺少参数时绕过中间件的路由。这些正是单元测试会遗漏、而安全审查能发现的问题。每一次认证变更都需要一个专门寻找这些漏洞的人，而不是一个只满足于“快乐路径”被覆盖的人。

**6. Secrets, `.env` files, API keys**
An agent reading or writing credentials creates exposure risk. Keep this category off-limits by default and handle it manually.
**6. 密钥、`.env` 文件、API 密钥**
智能体读取或写入凭据会产生泄露风险。默认情况下将此类操作设为禁区，并手动处理。

`git push --force` sits in its own category because it rewrites history on the remote. Once pushed, other contributors’ local branches diverge. Recovery is painful and sometimes impossible. Humans should be careful with all of these commands too. Agents just make them easier to trigger by accident, buried inside a longer sequence of otherwise safe steps.
`git push --force` 属于特殊类别，因为它会重写远程仓库的历史。一旦推送，其他贡献者的本地分支就会产生分歧。恢复过程非常痛苦，有时甚至是不可能的。人类在执行这些命令时也应格外小心。智能体只是让这些命令更容易被意外触发，因为它们可能被隐藏在一长串看似安全的步骤之中。

### AGENTS.md: write the contract
### AGENTS.md：编写契约

Give agents specific structure from the start. An AGENTS.md file at the root of your repo tells the agent what the project is, how to run it, and what it’s not allowed to touch without asking. A vague AGENTS.md gets you an agent filling gaps with guesses. I learned this on a codebase that had no AGENTS.md at all. The task was “organize the project structure.” The agent moved files across directories based on naming conventions that made sense to it. Everything that referenced those paths broke. The task took the agent twenty minutes; the cleanup took me two hours. Three lines of scope constraints would have prevented it entirely.
从一开始就为智能体提供明确的结构。在仓库根目录下放置一个 `AGENTS.md` 文件，告诉智能体项目是什么、如何运行，以及在未询问的情况下禁止触碰什么。模糊的 `AGENTS.md` 只会让智能体靠猜测来填补空白。我是在一个完全没有 `AGENTS.md` 的代码库中吸取这一教训的。当时的任务是“整理项目结构”。智能体根据它认为合理的命名规范移动了文件，结果所有引用这些路径的地方全挂了。智能体花了二十分钟完成任务，而我花了两个小时进行清理。其实，只需三行范围约束就能完全避免这种情况。
