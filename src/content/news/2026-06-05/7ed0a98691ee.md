# Your AI Agent Craves Curation. Here’s the FADEMEM Memory Architecture That Delivers It.
# 你的 AI 智能体渴望精选：FADEMEM 记忆架构为你实现这一目标

You have explained your tech stack to your coding agent four times this month. You mentioned your preferred approach to a problem in January, and your agent has no idea it ever happened. You corrected a decision last week and the old version is still surfacing. You set up context at the start of every session because there is nowhere for it to go at the end. This is not a model problem, as GPT-4, Claude, and Gemini all have the same limitations. The model is stateless. They all have inbuilt memory, and still every session starts from zero unless you have the infrastructure to persist what matters and surface it at the right moment. That sophisticated memory infrastructure is what most developers do not have.

这个月你已经向你的编程智能体解释了四次你的技术栈。你在 1 月份提到了你处理问题的首选方法，但你的智能体完全不记得这件事。你上周纠正了一个决定，但旧版本仍然不断出现。你不得不在每次会话开始时重新设置上下文，因为会话结束时这些信息无处可存。这不是模型本身的问题，因为 GPT-4、Claude 和 Gemini 都存在同样的局限性。模型是无状态的。它们虽然都有内置记忆，但除非你拥有能够持久化重要信息并在关键时刻调用的基础设施，否则每个会话依然是从零开始。而这种复杂的记忆基础设施正是大多数开发者所缺乏的。

VEKTOR Slipstream v1.6.3 is a local-first memory SDK for AI agents. This release adds the layer most memory systems skip: not just storing what you tell it, but managing what should still be there months later: curation.

VEKTOR Slipstream v1.6.3 是一款面向 AI 智能体的“本地优先”记忆 SDK。此版本增加了大多数记忆系统所忽略的层级：不仅是存储你告诉它的内容，还包括管理几个月后哪些信息依然有效：即“精选（Curation）”。

### What you actually get
### 你将获得什么

Before the architecture: What changes for you as a developer embedding this SDK. Every AI memory system forces decisions you didn’t realise you were making. Where does your agent’s context actually lives, is it on your machine or on someone else’s server? Are you paying per token every time your agent understands a memory, or does that happen locally? When you connect your GitHub, your calendar, your files — where does all that data go, and who can see it? Most memory systems answer all four questions for you, quietly, in their terms of service. VEKTOR’s answer to all four is the same: your machine, your data, your rules.

在深入架构之前：作为嵌入此 SDK 的开发者，你会发生什么变化？每个 AI 记忆系统都会强迫你做出一些你并未意识到的决定。你的智能体上下文到底存储在哪里？是在你的机器上，还是在别人的服务器上？每次智能体理解一段记忆时，你是否都在按 Token 付费，还是在本地完成的？当你连接 GitHub、日历或文件时，所有这些数据去了哪里，谁能看到它们？大多数记忆系统会在服务条款中悄悄地为你回答这四个问题。而 VEKTOR 对这四个问题的回答始终如一：你的机器，你的数据，你的规则。

Memory lives in a single SQLite file you own. Embeddings run locally on CPU — no API calls, no per-token cost, no data leaving the process. MCP connectors spawn as local stdio processes; nothing is routed through an external service. There is no telemetry, no cloud sync, no account required. If you want to understand exactly what your agent knows about you, you open the database with any SQLite browser and read it. That is what local-first actually means.

记忆存储在你拥有的单个 SQLite 文件中。嵌入（Embeddings）在本地 CPU 上运行——没有 API 调用，没有按 Token 计算的成本，数据不会离开进程。MCP 连接器作为本地 stdio 进程启动；没有任何数据通过外部服务路由。没有遥测，没有云同步，也不需要账户。如果你想确切了解你的智能体对你的了解程度，只需用任何 SQLite 浏览器打开数据库即可查看。这就是“本地优先”的真正含义。

Your agent stops asking you to repeat yourself. Decisions, preferences, project context, and personal facts persist across sessions and surface when relevant without being re-explained. A context you registered in January is still there in June — if it is still relevant. If it is not, it has faded and stopped competing with what is actually current.

你的智能体不再要求你重复自己。决策、偏好、项目背景和个人事实会在会话间持久保存，并在相关时自动浮现，无需重新解释。你在 1 月份注册的上下文在 6 月份依然存在——前提是它仍然相关。如果不再相关，它会逐渐淡化，不再干扰当前的信息。

Your agent stops surfacing contradictions. When you update a fact, the old version does not linger as an equally valid memory. The conflict resolver determines which one wins based on source trust and recency, and the loser is quietly retired rather than deleted — preserved for audit but excluded from recall.

你的智能体不再出现前后矛盾。当你更新一个事实时，旧版本不会作为同样有效的记忆残留。冲突解决程序会根据来源的可信度和时效性来决定胜出者，失败者会被悄悄“退役”而非删除——它们被保留以备审计，但不会被再次调用。

Your agent’s memory stays a manageable size. Without active management, memory graphs grow indefinitely. Every new project adds nodes that never leave. v1.6.3 introduces per-source budgets, automatic decay, and cold storage, so the graph reflects what is currently relevant rather than everything that has ever been stored.

你的智能体记忆将保持在可控范围内。如果没有主动管理，记忆图谱会无限增长。每一个新项目都会增加永远不会消失的节点。v1.6.3 引入了基于来源的配额、自动衰减和冷存储，因此图谱反映的是当前相关的内容，而不是曾经存储过的所有内容。

You do not need a cloud backend. One SQLite file. Runs on a laptop. No API calls to a cloud host memory service, no extra costs for connectors. No data leaving your machine.

你不需要云后端。一个 SQLite 文件，在笔记本电脑上运行。无需向云端记忆服务发起 API 调用，连接器也没有额外成本。数据不会离开你的机器。

### The architecture: what is new in v1.6.3
### 架构：v1.6.3 的新特性

**Decay: memory that fades when it should**
**衰减：该淡化时就淡化的记忆**

The new vektor-decay.js implementation uses the FadeMem architecture from a February 2026 paper (https://arxiv.org/abs/2601.18642) by researchers at Alibaba and Peking University. To our current knowledge, at this time VEKTOR is one of the first production SDK implementations of this research. The core idea: memories age differently depending on whether you use them. Every memory is classified as Long-term Memory Layer (high importance, frequently recalled) or Short-term Memory Layer (lower importance, infrequently accessed). LML memories decay slowly—roughly an 11-day half-life at default settings. SML memories decay four times faster.

新的 vektor-decay.js 实现采用了阿里巴巴和北京大学研究人员在 2026 年 2 月发表的论文（https://arxiv.org/abs/2601.18642）中的 FadeMem 架构。据我们目前所知，VEKTOR 是首批将该研究成果应用于生产环境的 SDK 之一。其核心思想是：记忆会根据你的使用频率以不同速度老化。每条记忆都被归类为“长期记忆层”（高重要性，频繁调用）或“短期记忆层”（低重要性，较少访问）。在默认设置下，长期记忆的半衰期约为 11 天，而短期记忆的衰减速度快四倍。

What drives the tier assignment is not just what you set when you stored it. Importance recalculates as a weighted function of semantic relevance to your current goals, access frequency, and position in the causal graph. A memory you actually revisit weekly climbs. One you flagged as important and never touched again gradually drifts down.

决定分层的不仅仅是你存储时设定的内容。重要性会根据与你当前目标的语义相关性、访问频率以及在因果图中的位置，通过加权函数重新计算。你每周都会回顾的记忆重要性会上升；而你标记为重要但之后再未触及的记忆，其重要性会逐渐下降。

The FadeMem paper reports 45% storage reduction versus append-only systems at equivalent recall quality. Their ablation shows that removing the dual-layer architecture alone drops multi-hop reasoning F1 by 33.9%. Conflict resolution removal drops it by 22.4%. These are the components now live in VEKTOR’s REM cycle.

FadeMem 论文报告称，在同等召回质量下，其存储空间比仅追加（append-only）系统减少了 45%。消融实验表明，仅移除双层架构就会导致多跳推理 F1 分数下降 33.9%，移除冲突解决机制则会导致下降 22.4%。这些组件现在已在 VEKTOR 的 REM 循环中上线。

**Conflict resolution: memory that keeps itself consistent**
**冲突解决：保持自身一致性的记忆**

vektor-conflict.js compares every new memory against existing ones above a similarity threshold. When it finds overlap, it classifies the relationship across five outcomes: the new memory supersedes the old, both coexist as independently valid, the new is subsumed by something already known, the new is more general and absorbs the old, or it is a duplicate and nothing changes.

vektor-conflict.js 会将每条新记忆与相似度阈值以上的现有记忆进行比对。当发现重叠时，它会将关系归类为五种结果之一：新记忆取代旧记忆、两者作为独立有效信息共存、新记忆被已知信息涵盖、新记忆更具概括性并吸收旧记忆，或者它是重复信息，无需任何更改。

Trust determines who wins. The system maps source type and actor type to a trust score. A direct user note scores 1.0. An automated bot event scores 0.28. A low-trust source cannot overwrite a high-trust one regardless of recency — your CI pipeline cannot quietly overwrite a decision your team made.

信任度决定了胜出者。系统将来源类型和执行者类型映射为信任分数。直接的用户笔记分数为 1.0，自动化机器人事件分数为 0.28。无论时效性如何，低信任度来源都无法覆盖高信任度来源——这意味着你的 CI 流水线无法悄悄覆盖团队做出的决定。

The FadeMem paper measures 68.9% macro-averaged accuracy across three conflict types (contradiction, update, overlap). That is the baseline the production implementation is building toward.

FadeMem 论文在三种冲突类型（矛盾、更新、重叠）中测得 68.9% 的宏观平均准确率。这就是生产环境实现所追求的基准。

**Standing queries: memory that knows what you are working on**
**常驻查询：了解你当前工作的记忆**

vektor-standing.js synthesises your current priorities weekly from your top-importance recent memories. The output is a small set of embedded goal statements stored in the database. Every new memory that arrives is scored for relevance against these goals before being assigned a tier. A commit directly relevant to an active project gets a higher initial importance score than one with no connection to your current work.

vektor-standing.js 每周会根据你近期最重要的记忆综合出你当前的优先级。输出结果是一组存储在数据库中的嵌入式目标陈述。每条新记忆在被分配层级之前，都会根据这些目标进行相关性评分。与当前活跃项目直接相关的提交，其初始重要性得分会高于与你当前工作无关的提交。
