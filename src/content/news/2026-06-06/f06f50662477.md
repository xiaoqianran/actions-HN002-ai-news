# My AI Couldn’t See My Files — I Built a Zero-Dependency MCP Server
# 我的 AI 看不到我的文件——我构建了一个零依赖的 MCP 服务器

I built a pure Python MCP layer that runs over stdio for local clients and HTTP/SSE for concurrent connections. One flag. Zero dependencies. 5 clients. Under 50ms.
我构建了一个纯 Python 的 MCP 层，它可以通过 stdio 为本地客户端运行，也可以通过 HTTP/SSE 为并发连接运行。只需一个标志，零依赖，支持 5 个客户端，响应时间低于 50 毫秒。

I was refactoring a scraper. The functions had grown too long and the variable names made no sense anymore. Every time I wanted feedback on a file, I stopped, opened the chat, copied the whole thing in, and waited. Then went back to the editor, applied the change, opened the next file, and did it again. At some point I counted. Six files. Eleven pastes. Twenty minutes of switching before I wrote a single new line.
我当时正在重构一个爬虫程序。函数变得太长，变量名也变得毫无意义。每次我想对某个文件获取反馈时，我都得停下来，打开聊天窗口，把整个文件内容复制进去，然后等待。接着回到编辑器，应用更改，打开下一个文件，再重复一遍。有一次我数了一下：6 个文件，11 次粘贴。在写出新的一行代码之前，我花了 20 分钟在不同窗口间切换。

The obvious fix was to give the AI tool direct access to my project folder. That’s when I ran into MCP — the Model Context Protocol — which is exactly built for this. A server runs locally, exposes tools, and the AI client calls those tools directly instead of waiting for me to paste things.
最明显的解决方案是让 AI 工具直接访问我的项目文件夹。就在那时，我接触到了 MCP（模型上下文协议），它正是为此而生的。服务器在本地运行并公开工具，AI 客户端直接调用这些工具，而无需等待我手动粘贴内容。

So I looked at existing implementations. Most required FastAPI, uvicorn, LangChain, or the official MCP SDK. Before writing a single line of business logic I had five packages in my requirements file and a server I wasn’t confident would run on Windows without a fight.
于是我查看了现有的实现。大多数都需要 FastAPI、uvicorn、LangChain 或官方的 MCP SDK。在写下一行业务逻辑之前，我的需求文件中就已经有了五个包，而且我并不确定这个服务器在 Windows 上能否顺利运行。

I stepped back and read the actual MCP spec. The protocol is JSON-RPC 2.0 over a transport layer. One JSON object per line. Client sends, server responds. The spec defines exactly two transports: stdio for local single-client connections, and HTTP with Server-Sent Events for concurrent clients. That’s the whole protocol.
我退后一步，阅读了实际的 MCP 规范。该协议是基于传输层的 JSON-RPC 2.0。每行一个 JSON 对象。客户端发送，服务器响应。规范明确定义了两种传输方式：用于本地单客户端连接的 stdio，以及用于并发客户端的带有服务器发送事件（SSE）的 HTTP。这就是协议的全部内容。

I asked a different question: what does this actually need that Python’s standard library doesn’t already provide? sys.stdin, sys.stdout, http.server, threading, queue, pathlib, json. That’s it. Not a single pip install. This article is that implementation — both transports, a production security model, 50 tests, and the numbers from running it.
我提出了一个不同的问题：这到底需要什么 Python 标准库没有提供的功能？sys.stdin、sys.stdout、http.server、threading、queue、pathlib、json。就这些。不需要任何 pip 安装。本文介绍的就是这个实现——包含两种传输方式、生产级安全模型、50 个测试用例以及运行数据。

### TL;DR
### 简而言之

Most MCP implementations feel heavier than they should. The spec only defines two transports, stdio and HTTP/SSE, but in practice they are usually wrapped in frameworks and extra dependencies. I built both transports from scratch using only the Python standard library. It runs as a single file with one runtime flag. No installs, no setup. For local work, it uses stdio with a single client. When you need concurrency, it switches to HTTP/SSE and handles multiple clients without changing anything else. Under the hood, everything stays consistent. Same dispatcher, same tools, same security model.
大多数 MCP 实现感觉比预想的要臃肿。规范只定义了两种传输方式（stdio 和 HTTP/SSE），但在实践中，它们通常被框架和额外的依赖项所包裹。我仅使用 Python 标准库从零构建了这两种传输方式。它作为一个单文件运行，只需一个运行时标志。无需安装，无需配置。对于本地工作，它使用 stdio 连接单个客户端；当你需要并发时，它会切换到 HTTP/SSE，无需更改任何其他内容即可处理多个客户端。在底层，一切保持一致：相同的调度器、相同的工具、相同的安全模型。

### The Mistake That Shaped the Whole Design
### 塑造了整个设计的错误

Before the architecture, I want to tell you about the thing that nearly made me give up on the whole thing. Early in development I was testing the search tool. I pointed the server at C:\Users\Admin and ran it looking for Python files. The server started. The demo started running. Then it just kept running. Thirty seconds. A minute. Five minutes. I thought there was an infinite loop somewhere. I went back through the code three times. Everything looked correct. I killed the process and restarted. Same result. Ten minutes in I finally understood what was happening.
在谈论架构之前，我想告诉你那件差点让我放弃整个项目的事情。在开发初期，我正在测试搜索工具。我将服务器指向 C:\Users\Admin 并运行它以查找 Python 文件。服务器启动了，演示程序开始运行。然后它就一直运行下去。30 秒、1 分钟、5 分钟……我以为哪里有死循环。我把代码检查了三遍，一切看起来都很正确。我杀掉进程并重启，结果一样。十分钟后，我终于明白了发生了什么。

The search tool was using rglob() by default. I had pointed it at my entire user directory and it was scanning everything — virtual environments, AppData, every cached file on the machine. Tens of thousands of files, one at a time. I killed the process and changed one line:
搜索工具默认使用了 rglob()。我把它指向了我的整个用户目录，它正在扫描所有内容——虚拟环境、AppData、机器上的每一个缓存文件。成千上万个文件，一个接一个地扫描。我杀掉进程并修改了一行代码：

```python
# Before — recursive by default, scans everything
for match in target.rglob(pattern):
# After — shallow by default, opt-in for recursion
for match in target.glob(pattern):
```

And made recursive=False the default parameter. The client has to pass recursive=True explicitly. The server will never scan recursively on its own. That single change is why search completes in under 30ms on a real project folder today instead of running forever. And it became the rule I applied everywhere: no behavior that destroys performance should ever be the default.
我将 recursive=False 设为默认参数。客户端必须显式传递 recursive=True。服务器永远不会自行进行递归扫描。正是这一处改动，使得搜索在现在的真实项目文件夹中能在 30 毫秒内完成，而不是无限期运行。这也成为了我到处应用的准则：任何会破坏性能的行为都不应成为默认设置。

### What MCP Actually Is
### MCP 到底是什么

The Model Context Protocol is a standardised way for AI clients to call tools on external servers. It uses JSON-RPC 2.0 as its message format. In practice, this means AI clients like Claude or ChatGPT can directly access and reason over local files instead of relying on copy-paste.
模型上下文协议（MCP）是一种让 AI 客户端调用外部服务器工具的标准化方式。它使用 JSON-RPC 2.0 作为消息格式。在实践中，这意味着像 Claude 或 ChatGPT 这样的 AI 客户端可以直接访问并分析本地文件，而无需依赖复制粘贴。

The handshake has three phases. First the client initializes, then it asks what tools are available, then it starts calling them.
握手分为三个阶段。首先客户端进行初始化，然后询问有哪些可用工具，接着开始调用它们。

Everything after that is the transport carrying messages back and forth. The spec defines two transports. stdio runs over standard input and output — one JSON object per line, flushed immediately. HTTP/SSE runs requests over HTTP POST, with responses streamed back over a persistent Server-Sent Events connection.
此后的所有内容都是在来回传输消息。规范定义了两种传输方式。stdio 通过标准输入和输出运行——每行一个 JSON 对象，并立即刷新。HTTP/SSE 通过 HTTP POST 发送请求，响应则通过持久的服务器发送事件（SSE）连接流式传输回来。

Most implementations pick one. This one implements both, with the same dispatcher and the same four tools sitting behind each.
大多数实现只选择其中一种。而这个实现两者兼备，且背后拥有相同的调度器和相同的四个工具。
