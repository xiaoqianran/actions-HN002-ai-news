# Increase Recommendation Systems’ Precision with LLMs, Using Python
# 使用 Python 和大语言模型 (LLM) 提升推荐系统的精度

Large Language Models Increase Recommendation Systems’ Precision with LLMs, Using Python This is how LLMs are used today to increase precision in recommendation systems Piero Paialunga Jun 8, 2026 10 min read Share Photo by Amanda Jones on Unsplash
大语言模型 (LLM) 正在提升推荐系统的精度。本文将介绍当今如何利用 LLM 来提高推荐系统的准确性。作者：Piero Paialunga，2026年6月8日，阅读时长10分钟。图片来源：Unsplash 上的 Amanda Jones。

One of my favorite sayings in American culture is the following: “You can’t have your cake and eat it too.” I find this sentence extremely poetic but also very practical and useful. The message of this saying is straightforward: everything you accomplish is achieved through a tradeoff, as everything has a price. The philosophical discussion is out of scope for this article, but the practical consequences of these considerations are very much in line with data science and software engineering in general. Let me explain.
在美国文化中，我最喜欢的一句谚语是：“你不可能既拥有蛋糕又吃掉它（鱼与熊掌不可兼得）。”我觉得这句话既富有诗意，又非常实用。这句谚语传达的信息很简单：你所取得的每一项成就都是通过权衡取舍实现的，因为任何事物都有代价。哲学层面的讨论超出了本文的范围，但这些考量的实际后果与数据科学和软件工程领域高度契合。让我来解释一下。

In software engineering and data science, there is no such thing as the “perfect design” per se. The same algorithm that is fantastic for a given application fails miserably in others. Think of the computation versus memory tradeoffs in the following cases: It makes a lot of sense to precompute the distance between two cities and store them in a dataset, and it doesn’t make sense to compute them on the flight. This is because you expect the dataset to be reasonably low maintenance (cities don’t just move around often), and it would be stupid to compute the distance between New York and San Francisco every fraction of a second. [Case A]
在软件工程和数据科学中，本质上不存在所谓的“完美设计”。同一个算法在某个应用场景中表现出色，但在其他场景中却可能彻底失败。试想一下以下案例中计算与内存之间的权衡：预先计算两个城市之间的距离并将其存储在数据集中是非常合理的，而实时计算则毫无意义。这是因为你预期数据集的维护成本较低（城市不会频繁移动），如果每隔几分之一秒就计算一次纽约和旧金山之间的距离，那将是非常愚蠢的。[案例 A]

However, it would be equally stupid (and probably impossible) for a chatbot to memorize all the possible questions a human can ask and pull the answer to that question whenever it’s asked. This is because the nature of the problem is much more dynamic, and it requires an “on the fly” computation. [Case B]
然而，如果让聊天机器人记住人类可能提出的所有问题，并在被问及时提取答案，同样是愚蠢的（而且可能是不可能的）。这是因为问题的本质更加动态，需要“实时”计算。[案例 B]

In Case A, we are sacrificing memory and getting extremely quick computation. In Case B, we are spending more computational time, but we are not using any “query” memory. Can you get no computational time and no memory? Not really, because you can’t have your cake and eat it too 🙂 But let’s take a less obvious and more “trendy” example. Let’s talk about Large Language Models (LLMs).
在案例 A 中，我们牺牲了内存以获得极快的计算速度。在案例 B 中，我们花费了更多的计算时间，但没有使用任何“查询”内存。你能做到既不需要计算时间也不占用内存吗？实际上不行，因为鱼与熊掌不可兼得 🙂 但让我们看一个不那么明显且更“时髦”的例子。让我们谈谈大语言模型 (LLM)。

LLMs are the most powerful AI models we have, and they are trained on all the knowledge available to the world. They are also massive. They are actually so big that we rarely have them in-house, and we usually invoke them through APIs. However, API call = tokens = cost. Now imagine you want to use an intelligent system to pick the best restaurant for tonight. You would ask ChatGPT something like: “Can you provide me with a good Italian restaurant that is not super expensive but romantic and in a good location?”
LLM 是我们目前拥有的最强大的 AI 模型，它们接受了全球可用知识的训练。它们也非常庞大。实际上，它们大到我们很少能在内部部署，通常通过 API 调用。然而，API 调用意味着 Token，意味着成本。现在想象一下，你想使用一个智能系统来挑选今晚最好的餐厅。你会问 ChatGPT 类似这样的问题：“你能给我推荐一家不错的意大利餐厅吗？要求不太贵、浪漫且位置好。”

Now, imagine if the GPT model had to explore all the restaurants in the universe and decide if they are Italian, not expensive, in a good location, and close to your place. Best-case scenario: you would spend millions in tokens, and you’d already be in bed by the time the computation is executed. However, we also don’t want to completely give up all the juicy, natural-language interpretative, and information-retrieving power of the LLMs. The key is that, in order to use the LLM and get smart information, we can’t use the most intelligent part of the pipeline all the time (that would be like having your cake and eat it too).
现在，想象一下如果 GPT 模型必须探索宇宙中所有的餐厅，并判断它们是否是意大利菜、价格是否昂贵、位置是否优越以及是否离你很近。最好的情况是：你将花费数百万 Token，而且等到计算执行完毕时，你可能已经上床睡觉了。然而，我们也不想完全放弃 LLM 在自然语言理解和信息检索方面的强大能力。关键在于，为了利用 LLM 获取智能信息，我们不能一直使用流水线中最智能的部分（那就像是既想拥有蛋糕又想吃掉它一样）。

In this article, I am going to give you a recipe for these smart, LLM-improved recommendation systems, using the restaurant recommendation example we were doing as a use case. The input of this system will be the user’s description of their ideal restaurant in a specific city, and the output will be a set of recommended restaurants. Let’s get started!
在本文中，我将以餐厅推荐为例，为你提供一套构建这种智能、经 LLM 改进的推荐系统的方案。该系统的输入是用户对其所在城市理想餐厅的描述，输出则是一组推荐餐厅。让我们开始吧！

### 1. System Design
### 1. 系统设计

The cake saying we discussed is also known in engineering as the Accuracy-Scale-Time triangle:
我们讨论的关于蛋糕的谚语在工程学中也被称为“准确性-规模-时间”三角：

*   You can make something accurate and on a massive dataset, but it will be slow
*   You can make something accurate and fast, but it won’t scale well on a large dataset
*   You can make something fast and scale well, but it won’t be that accurate.
*   你可以让系统在海量数据集上保持准确，但速度会很慢；
*   你可以让系统准确且快速，但在大规模数据集上无法很好地扩展；
*   你可以让系统快速且易于扩展，但准确性不会太高。

Of course, we want our results to be ultimately accurate, so option 3 alone won’t cut it. However, we can refine option 3 with a more accurate model on top of the first one. In other words, Option 3 can give us a good list of candidates with a small computational time, and we can select the most accurate list of recommendations using a Large Language Model.
当然，我们希望最终结果尽可能准确，因此仅靠选项 3 是不够的。然而，我们可以在选项 3 的基础上，叠加一个更精确的模型来优化它。换句话说，选项 3 可以用较少的计算时间为我们提供一份不错的候选名单，然后我们可以利用大语言模型从中筛选出最准确的推荐列表。

In other words, the design looks like this:
换句话说，设计方案如下：

1.  A quick and simple search will find the top K closest restaurants (rule-based, high recall, low precision)
2.  A slow, very intelligent Large Language Model will help us choose, among the top K, the best based on the query. (AI-based, high precision)
1.  通过快速简单的搜索找到前 K 个最相关的餐厅（基于规则，高召回率，低精度）；
2.  利用一个缓慢但非常智能的大语言模型，在 K 个候选者中根据查询选择出最佳结果（基于 AI，高精度）。

By doing this, we are not wasting time and money on the slow LLM, but we are still getting their smartness by using them on a selected list of candidates. Enough yapping. Let’s start coding!
通过这种方式，我们既不会在缓慢的 LLM 上浪费时间和金钱，又能通过在精选的候选名单上使用它们来获得其智能优势。废话少说，让我们开始编码吧！

#### 2.2 Data Generation
#### 2.2 数据生成

Before we can recommend anything, we need something to recommend. In a real system, we would use a restaurant database in an S3 location. For this article, we generate a synthetic one so the whole thing is fully reproducible and free to run. This is the job of the RestaurantDataGenerator class inside datagenerator.py.
在推荐任何东西之前，我们需要有可推荐的内容。在实际系统中，我们会使用存储在 S3 中的餐厅数据库。为了本文演示，我们生成了一个合成数据集，以便整个过程完全可复现且免费运行。这是 `datagenerator.py` 中 `RestaurantDataGenerator` 类的任务。

It builds a reproducible table of ~10,000 restaurants scattered across eight cities (New York, San Francisco, Chicago, Austin, Seattle, Boston, Miami, and Denver). Each restaurant gets:
它构建了一个包含约 10,000 家餐厅的可复现表格，这些餐厅分布在八个城市（纽约、旧金山、芝加哥、奥斯汀、西雅图、波士顿、迈阿密和丹佛）。每家餐厅包含：

*   a randomly assembled name
*   a city and a latitude/longitude sampled around that city’s center (within ~13 km),
*   a cuisine style (Italian, Japanese, Mexican, Thai, French, …),
*   a dietary profile (omnivore/vegetarian/vegan)
*   an average score
*   a number of votes
*   a price range (10 / 100 / 1000)
*   随机生成的名称；
*   城市以及围绕该城市中心采样（约 13 公里内）的经纬度；
*   菜系风格（意大利菜、日本菜、墨西哥菜、泰国菜、法国菜等）；
*   饮食偏好（杂食/素食/纯素）；
*   平均评分；
*   投票数；
*   价格区间（10 / 100 / 1000）。
