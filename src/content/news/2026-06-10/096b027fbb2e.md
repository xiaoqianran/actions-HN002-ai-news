# Prefill Once, Fan Out: KV Snapshot Sharing for Multi-Agent LLM Pipelines

**Prefill Once, Fan Out: KV Snapshot Sharing for Multi-Agent LLM Pipelines**
一次预填充，多路分发：面向多智能体 LLM 流水线的 KV 快照共享

A systems-engineering approach to KV cache reuse that proves computing a shared prompt once beats N parallel prefills every time.
这是一种关于 KV 缓存重用的系统工程方法，它证明了“计算一次共享提示词”的效果始终优于“N 次并行预填充”。

A humorous-but-real tour of SwarmKV — KV-snapshot fan-out, copy-on-fork host buffers, and how to make a two-agent analytical pipeline ~1.95× faster (and the second branch’s activation latency 52× faster) by being mildly mean to llama.cpp.
这是一次关于 SwarmKV 的幽默而真实的探索——涵盖了 KV 快照分发、写时复制（copy-on-fork）主机缓冲区，以及如何通过对 llama.cpp 进行一些“小手段”，使双智能体分析流水线提速约 1.95 倍（并将第二个分支的激活延迟降低 52 倍）。

This is Part 1 of the “Production-Grade Agentic Inference” series. Each part removes one kind of redundant work from an agentic LLM pipeline. Part 1 (this post) kills redundant prefill. Part 2 tackles redundant waiting — how 50 micro-agents share one GPU through time-slicing. Part 3 keeps RAG retrieval on the GPU with a custom CUDA Top-K kernel. Part 4 persists agent state across hand-offs so the next agent never has the cold-start problem.
这是“生产级智能体推理”系列的第一部分。每一部分都旨在消除智能体 LLM 流水线中的一种冗余工作。第一部分（本文）旨在消除冗余预填充。第二部分将解决冗余等待问题——即 50 个微型智能体如何通过时间片轮转共享一个 GPU。第三部分将通过自定义 CUDA Top-K 内核将 RAG 检索保留在 GPU 上。第四部分将在任务交接时持久化智能体状态，从而彻底解决下一个智能体的冷启动问题。

### Key takeaways
### 核心要点

**The problem:** when several agents read the same document, a default serving stack makes each one rerun the exact same prefill. That redundant dense attention pass is pure waste.
**问题所在：** 当多个智能体读取同一文档时，默认的服务栈会强制每个智能体重新执行完全相同的预填充。这种冗余的密集注意力计算纯属浪费。

**The fix:** run prefill once, serialize the KV cache to a host buffer, memcpy it per branch, and restore it before decoding. “Compute once, fan out.”
**解决方案：** 只运行一次预填充，将 KV 缓存序列化到主机缓冲区，按分支进行内存拷贝（memcpy），并在解码前恢复它。即“计算一次，多路分发”。

**The receipts:** on a seven-year-old GTX 1080, a two-agent pipeline got 48.69% faster end to end (~1.95×) and the second agent’s activation latency dropped 98.09% (~52×), eliminating 8,685 ms of redundant compute.
**实测数据：** 在一块七年前的 GTX 1080 上，双智能体流水线的端到端速度提升了 48.69%（约 1.95 倍），第二个智能体的激活延迟降低了 98.09%（约 52 倍），消除了 8,685 毫秒的冗余计算。

**The kicker:** this is not a new algorithm. It is systems engineering — and it is the same “broadcast shared state once” decision a 5G cell tower has made every 80 ms since LTE.
**关键点：** 这不是什么新算法，而是系统工程——这与 5G 基站自 LTE 时代以来每 80 毫秒执行一次的“广播共享状态”决策如出一辙。

### TL;DR:
### 简而言之：

Standard LLM serving makes every analytical agent re-prefill the same shared document. Your GPU dutifully re-executes billions of redundant prefix-prefill multiplications. The same bytes. The same weights. The same quantization. All to recalculate a state it already finished calculating four seconds ago.
标准的 LLM 服务方式会让每个分析型智能体都重新预填充同一个共享文档。你的 GPU 会忠实地重复执行数十亿次冗余的前缀预填充乘法。同样的字节、同样的权重、同样的量化，所有这些仅仅是为了重新计算一个四秒前就已经计算完成的状态。

SwarmKV runs the prefill once, serializes the resulting KV state to a host buffer via `llama_state_get_data`, memcpys that buffer into a per-branch allocation, and lets each branch restore the snapshot with `llama_state_seq_set_data` before decoding from where the document left off.
SwarmKV 只运行一次预填充，通过 `llama_state_get_data` 将生成的 KV 状态序列化到主机缓冲区，将该缓冲区内存拷贝到每个分支的分配空间中，并允许每个分支在解码前通过 `llama_state_seq_set_data` 恢复快照，从而从文档中断处继续执行。

Yes, it is a real round-trip—serialize, copy, restore, but because redundant prefill compute scales quadratically while the KV state transfer scales linearly, moving the data across a constrained Pascal memory bus is still vastly cheaper than recalculating the attention matrices from scratch.
没错，这确实涉及序列化、拷贝、恢复的往返过程，但由于冗余的预填充计算呈二次方增长，而 KV 状态传输呈线性增长，因此在受限的 Pascal 内存总线上移动数据，其成本远低于从头开始重新计算注意力矩阵。

That is reflected on the result on a seven-year-old GTX 1080: 48.69% end-to-end speedup on a two-agent pipeline, 98.09% reduction in branch-2 activation latency (~52×), 8,685 ms of redundant dense compute eliminated, zero new transformer tricks. Just the systems-engineering position that “compute once, fan out” beats “compute N times, hope nobody notices.”
这反映在七年前的 GTX 1080 的测试结果上：双智能体流水线端到端提速 48.69%，第二个分支的激活延迟降低 98.09%（约 52 倍），消除了 8,685 毫秒的冗余密集计算，且未使用任何新的 Transformer 技巧。这仅仅体现了系统工程的原则：“计算一次，多路分发”远胜于“计算 N 次，祈祷没人发现”。

### 1. A confession: most of your second agent’s “work” is a rerun
### 1. 坦白：你的第二个智能体大部分“工作”都是在重跑

If you have ever pointed two analytical agents at the same document through vanilla llama.cpp, here is what really happens (with a little bit of intentional dramatization):
如果你曾经通过原生 llama.cpp 让两个分析型智能体读取同一个文档，以下是实际发生的情况（带有一点戏剧性夸张）：

**You:** “Please provide an overview of this 3,500-token spec, and separately, list its license obligations.”
**你：** “请提供这份 3,500 token 规格说明书的概述，并分别列出其许可义务。”

**llama.cpp (Agent 1):** “Sure. Loading model. Prefilling document. Decoding answer.”
**llama.cpp (智能体 1)：** “好的。正在加载模型。正在预填充文档。正在解码答案。”

*GPU spends 4,346 ms on dense attention*
*GPU 花费 4,346 毫秒进行密集注意力计算*

**llama.cpp (Agent 1):** “Done. Here’s a 6-token answer.”
**llama.cpp (智能体 1)：** “完成。这是 6 个 token 的答案。”

**You:** “Great. Now Agent 2.”
**你：** “很好。现在轮到智能体 2。”

**llama.cpp (Agent 2):** “Sure. Loading model. Prefilling document — “
**llama.cpp (智能体 2)：** “好的。正在加载模型。正在预填充文档——”

**You:** “Wait, you literally just did that.”
**你：** “等等，你刚才明明已经做过了。”

**llama.cpp (Agent 2):** “I’m an independent llama_context. I have no memory of Agent 1. I have no memory of anything. I am a beautiful, stateless newborn.” 🫡
**llama.cpp (智能体 2)：** “我是一个独立的 llama_context。我不记得智能体 1，我什么都不记得。我是一个美丽、无状态的新生儿。” 🫡

*GPU spends another 4,339 ms on bit-for-bit identical attention math.*
*GPU 又花费 4,339 毫秒进行完全相同的注意力数学运算。*

Your GPU thermal sensor: gets a good workout;
你的 GPU 温度传感器：得到了充分的锻炼；
Your AWS bill: develops a sense of humor;
你的 AWS 账单：开始变得“幽默”起来；
Your second agent’s TTFT: 4.3 seconds before it can answer a 4-token question.
你的第二个智能体的首字延迟（TTFT）：在回答一个 4 token 的问题前需要等待 4.3 秒。

That is the joke. That is the dirty secret of every “agentic” pipeline that spreads out from a shared document. Each branch starts from a blank slate and rebuilds the same KV cache the previous branch just finished building. The deeper the document, the worse the tax.
这就是笑话所在。这也是每一个从共享文档分发出的“智能体”流水线背后的肮脏秘密。每个分支都从零开始，重建上一个分支刚刚构建好的同一个 KV 缓存。文档越长，这种代价就越沉重。

At 3,500 tokens on a Pascal GPU, the vast majority of the second agent’s perceived latency is not the answer — it is reading the document again.
在 Pascal 架构的 GPU 上处理 3,500 token 时，第二个智能体感知到的绝大部分延迟并非来自回答本身，而是来自对文档的重复阅读。

SwarmKV is what happens when you decide the second reading is optional and you would rather write 1,500 lines of C++ than let each agent build the same KV cache over and over again.
SwarmKV 的出现，是因为你决定第二次阅读是可选的，并且你宁愿写 1,500 行 C++ 代码，也不愿让每个智能体一遍又一遍地构建相同的 KV 缓存。

Now imagine, the toy demo in this repo is about two agents over a summary/license-check. The real shape of the workload it is built for is N specialized evaluators over one dense technical document.
现在想象一下，本仓库中的演示示例是关于两个智能体进行摘要/许可检查。它所构建的真实工作负载形态是 N 个专业评估器针对一份密集的技术文档进行处理。

Picture an AI patent and prior-art pipeline: one 50,000-token technical specification at the root, and fifty concurrent branches evaluating novelty, mapping claims, retrieving prior art, checking freedom-to-operate, assessing ethical compliance, and translating into jurisdiction-specific language.
想象一个 AI 专利和现有技术检索流水线：根节点是一份 50,000 token 的技术说明书，五十个并发分支分别评估新颖性、映射权利要求、检索现有技术、检查自由实施权、评估伦理合规性，并翻译成特定司法管辖区的语言。

The baseline cost of that pipeline on a default serving stack is fifty full prefills of the same spec. The SwarmKV cost is one prefill plus fifty memcpys. That asymmetry is intentionally designed.
在默认服务栈上，该流水线的基准成本是五十次对同一说明书的完整预填充。而 SwarmKV 的成本是一次预填充加上五十次内存拷贝。这种不对称性是刻意设计的。
