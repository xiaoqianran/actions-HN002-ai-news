# You Probably Don’t Need an Agent Framework
# 你可能根本不需要代理框架

Large Language Models You Probably Don’t Need an Agent Framework. Most LLM applications need a clear workflow, not an autonomous agent. Here's how to build one in plain Python.
大型语言模型：你可能根本不需要代理框架。大多数大模型（LLM）应用需要的是清晰的工作流，而不是自主代理。以下是如何使用原生 Python 构建工作流的方法。

You want to build an LLM application. Ok, the first thought that comes to your mind is: let’s build a powerful agent! But immediately you ask yourself: which agent framework should I use? Should it be CrewAI? LangGraph? Microsoft Agent Framework? Or something else? You then open a few documentation pages, compare examples, and try to decide which framework best suits your problem at hand. A couple of hours passed by, you feel overwhelmed, and you haven’t even started coding.
你想构建一个 LLM 应用。好吧，你脑海中浮现的第一个念头是：让我们构建一个强大的代理（Agent）！但紧接着你会问自己：我该用哪个代理框架？是 CrewAI？LangGraph？微软的 Agent Framework？还是别的什么？于是你打开几个文档页面，对比示例，试图决定哪个框架最适合手头的问题。几个小时过去了，你感到不知所措，甚至还没开始写代码。

But wait a second: Do you really need an agent framework here? In fact, do you even need to build an LLM agent? In the past two years, I have built quite a few LLM applications across different domains. One lesson I learned is that for many useful LLM applications, what reliably works in the end is not really an autonomous agent. It’s workflow. And for building workflows, you might not even need any frameworks in the first place.
但等一下：你真的需要代理框架吗？事实上，你真的需要构建一个 LLM 代理吗？在过去两年里，我在不同领域构建了不少 LLM 应用。我学到的一个教训是：对于许多实用的 LLM 应用来说，最终能可靠运行的并不是自主代理，而是工作流。而且在构建工作流时，你甚至根本不需要任何框架。

In this post, I’ll show you how to prototype an LLM workflow using plain Python, local functions, structured outputs, and the OpenAI Responses API (the same pattern also applies to other LLM providers). We’ll get hands-on and solve an anomaly explanation problem. My goal here is not to argue that frameworks are bad. They are clearly useful in various scenarios, and we’ll discuss that at the end of this post. My goal is to show that for many applications, a clear workflow may be the first abstraction you actually need.
在这篇文章中，我将展示如何使用原生 Python、本地函数、结构化输出和 OpenAI Responses API 来原型化一个 LLM 工作流（同样的模式也适用于其他 LLM 提供商）。我们将通过实践来解决一个异常解释问题。我的目的并不是说框架不好，它们在各种场景下显然很有用，我们会在文末讨论这一点。我的目标是证明，对于许多应用而言，清晰的工作流可能是你真正需要的第一个抽象层。

### 1. Try Workflows Before Jumping to Agents
### 1. 在转向代理之前，先尝试工作流

An LLM agent usually refers to an autonomous system that can decide for itself what to do next. You give it a goal and a set of tools. It will then plan, act, observe the result, and continue iterating. Instead of following a fixed path, the agent dynamically chooses the next step based on the current situation. For open-ended problems, this is powerful. But many real-world problems are not that open-ended. In many cases, we do know roughly how the problem should be solved. But if we know that already, why ask the model to reinvent the wheel?
LLM 代理通常指能够自主决定下一步做什么的系统。你给它一个目标和一套工具，它就会进行规划、行动、观察结果并持续迭代。代理不是遵循固定的路径，而是根据当前情况动态选择下一步。对于开放式问题，这非常强大。但许多现实世界的问题并没有那么开放。在很多情况下，我们大致知道问题应该如何解决。既然已经知道了，为什么还要让模型去重复造轮子呢？

This is where workflows come in. In an LLM workflow, we developers define the main steps, and the LLM is used inside selected steps as a reasoning engine. This is still an LLM-powered application. The difference is that the LLM is not treated as the whole system. It is treated only as a decision node inside a larger process.
这就是工作流的用武之地。在 LLM 工作流中，我们开发者定义主要步骤，而 LLM 仅在选定的步骤中作为推理引擎使用。这仍然是一个由 LLM 驱动的应用，区别在于 LLM 不再被视为整个系统，而仅仅被视为更大流程中的一个决策节点。

So, what are the benefits of adopting an LLM workflow? The following points are important in my opinion: A workflow is transparent. You can easily examine, as each step has a well-defined role and a clear input and output contract. A workflow is modular, meaning that you can change one step without rewriting the entire application. Most importantly, a workflow has deterministic control flow. The LLM’s reasoning and decisions can still vary inside a step, but the overall path is owned by code. That alone removes a lot of uncertainty when we are trying to build something reliable. I know, this sounds less exciting than building a fully autonomous agent. But your goal is to deliver an effective solution. If boring stuff works, then we should happily use it.
那么，采用 LLM 工作流有什么好处呢？我认为以下几点很重要：工作流是透明的。你可以轻松检查，因为每个步骤都有明确定义的角色以及清晰的输入和输出契约。工作流是模块化的，这意味着你可以更改其中一个步骤而无需重写整个应用。最重要的是，工作流具有确定性的控制流。LLM 在步骤内部的推理和决策可能会有所不同，但整体路径是由代码控制的。当我们试图构建可靠的系统时，仅这一点就消除了大量的不确定性。我知道，这听起来不如构建一个完全自主的代理那么令人兴奋，但你的目标是提供有效的解决方案。如果枯燥的方法有效，我们就应该乐于使用它。

### 2. What We Actually Need to Design
### 2. 我们真正需要设计的是什么

In my experience, there are four key ingredients: Control flow, Role instructions (system prompts), Prompt builders, Structured output. Let me unpack each of them.
根据我的经验，有四个关键要素：控制流、角色指令（系统提示词）、提示词构建器、结构化输出。让我逐一拆解。

#### 2.1 Control flow
#### 2.1 控制流

Control flow defines how the application moves from input to output. A useful way to think about control flow is to view it as a graph. In this graph, we have nodes and edges: Nodes: Each node represents one step in the application. It can be a deterministic processing step performed by the code (e.g., loading data, calling a local function). It can also be an LLM step, where an LLM is employed to make a decision, extract information, or write an explanation. Edges: Each edge represents how information moves from one step to the next. Just like nodes, edges also have different types. For example, the edge can be static, i.e., always calling a predefined next step after processing the current step; or it can be conditional, for instance, if the LLM in the current steps says more evidence is needed, the edge links to a local tool node; if the LLM believes enough evidence has been gathered, the edge points to the final explanation. A key thing: in a workflow, code owns this graph. LLMs are bound to specific nodes rather than running freely. I’d start here as it forces you to first think through what the application actually has to do, which tasks should be handled by deterministic code, which go to LLM, where the workflow branches, and when it should stop.
控制流定义了应用如何从输入流向输出。思考控制流的一个有效方法是将其视为一个图。在这个图中，我们有节点和边：节点：每个节点代表应用中的一个步骤。它可以是代码执行的确定性处理步骤（例如加载数据、调用本地函数），也可以是 LLM 步骤，即利用 LLM 进行决策、提取信息或撰写解释。边：每条边代表信息如何从一个步骤移动到下一个步骤。就像节点一样，边也有不同的类型。例如，边可以是静态的，即在处理完当前步骤后总是调用预定义的下一步；也可以是条件性的，例如，如果当前步骤中的 LLM 表示需要更多证据，则边连接到本地工具节点；如果 LLM 认为已收集到足够证据，则边指向最终解释。关键点在于：在工作流中，代码拥有这个图。LLM 被绑定在特定的节点上，而不是自由运行。我建议从这里开始，因为它会迫使你首先思考应用到底需要做什么，哪些任务应该由确定性代码处理，哪些交给 LLM，工作流在哪里分支，以及何时应该停止。

#### 2.2 Role Instructions
#### 2.2 角色指令

A workflow typically uses LLMs in several roles, and each role needs its own instruction (system prompts). A role instruction defines how the LLM should behave inside one specific node. It typically tells the LLM its persona, what tasks it’s performing, what it should pay attend to, and what to avoid. Any domain-specific rules should also be specified here.
工作流通常会在多个角色中使用 LLM，每个角色都需要自己的指令（系统提示词）。角色指令定义了 LLM 在特定节点内应如何表现。它通常会告诉 LLM 它的角色设定、正在执行的任务、应该注意什么以及应该避免什么。任何特定领域的规则也应在此处指定。

#### 2.3 Prompt Builders
#### 2.3 提示词构建器

While role instructions define how an LLM should behave, prompt builders decide what the LLM can see. Prompt builders assemble the context for an LLM call based on the objective of this LLM and the current workflow state. In practice, it’s just a function: it takes in the dynamic values from the current workflow, optionally preprocesses them, and feeds them into a prompt template. The output is the final prompt payload sent to the LLM. Prompt builders are where we control the context window. The context must be tailored and sufficient to support LLM’s reasoning.
如果说角色指令定义了 LLM 的行为方式，那么提示词构建器则决定了 LLM 能看到什么。提示词构建器根据 LLM 的目标和当前工作流状态来组装 LLM 调用所需的上下文。在实践中，它就是一个函数：接收来自当前工作流的动态值，选择性地进行预处理，然后将其填入提示词模板。输出结果就是发送给 LLM 的最终提示词载荷。提示词构建器是我们控制上下文窗口的地方。上下文必须经过裁剪，且足以支持 LLM 的推理。
