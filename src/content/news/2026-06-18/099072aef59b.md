A user types one string. “What is the maximum coverage amount? Don’t confuse it with the deductible, they’re often listed together.” The parser turns it into a row of typed columns: a topic, an expected answer shape (an amount), a scope hint (this contract), a negative cue (not the deductible) routed to the generation brief, and a layout hint (often listed together) that retrieval can use. Each piece becomes its own column on `question_df`.
用户输入一个字符串：“最高承保金额是多少？别把它和免赔额搞混了，它们通常列在一起。”解析器将其转换为一行具有类型定义的列：主题、预期的答案形态（金额）、范围提示（本合同）、路由到生成摘要的负面提示（非免赔额），以及检索可以利用的布局提示（通常列在一起）。每一部分都成为 `question_df` 中的独立列。

### 1. The five field families the parser fills
### 1. 解析器填充的五个字段系列

A question is more than its words. It also tells you what shape the answer should take, where to look in the document, whether it’s compound or too vague to act on. The parser captures each of these and writes it as a column on `question_df`. Read the rest of this article as a menu of what’s available, not as a checklist.
问题不仅仅是文字。它还告诉系统答案应呈现的形态、在文档中的查找位置、问题是否为复合问题，或者是否因过于模糊而无法处理。解析器会捕获这些信息，并将其作为 `question_df` 中的列写入。请将本文的后续内容视为一份功能菜单，而非必须逐项执行的清单。

**Keywords:** Tokens to feed retrieval. Several sources combine: explicit (the user named them), direct (extracted from the question), LLM rewrites, an expert concept dictionary, and high-signal regex anchors like L131-1.
**关键词：** 用于驱动检索的标记。结合了多种来源：显式（用户指定的）、直接（从问题中提取的）、LLM 重写的、专家概念词典，以及像 L131-1 这样高信号的正则表达式锚点。

**Answer shape and answer type:** Two orthogonal axes: the expected cardinality (single, listing, table, tree, nested_json) and the value type (text, amount, date, iban, address, …).
**答案形态与答案类型：** 两个正交轴：预期的基数（单个、列表、表格、树状、嵌套 JSON）和值类型（文本、金额、日期、IBAN、地址等）。

**Scope:** Where in the document to look: a page, a chapter, a section, a layout (table / image), a date range, a jurisdiction.
**范围：** 在文档中的查找位置：页面、章节、部分、布局（表格/图像）、日期范围、司法管辖区。

**Decomposition:** Sub-questions when the question is compound.
**分解：** 当问题为复合问题时的子问题。

**Clarification:** A short follow-up question when the input is too vague to act on.
**澄清：** 当输入过于模糊而无法处理时，提出的简短后续问题。

### 1.1 Keywords
### 1.1 关键词

Retrieval needs words to search the document with. The parser picks them out of the question and hands them over. The user’s wording almost never matches the document’s wording on the first try, so the parser collects from several sources at once.
检索需要词汇来搜索文档。解析器从问题中提取这些词汇并将其传递出去。用户的措辞几乎不可能在第一次尝试时就与文档中的措辞完全匹配，因此解析器会同时从多个来源进行收集。

Here is the minimal schema we’ll grow as the section goes:
以下是我们将在本节中逐步扩展的最小架构：

```python
class ParsedQuestion(BaseModel):
    original_question: str
    keywords: list[str]
```

For “What is the maximum coverage amount?”, the parser produces:
对于“最高承保金额是多少？”，解析器生成：

```python
ParsedQuestion(
    original_question="What is the maximum coverage amount?",
    keywords=["maximum", "coverage", "amount"],
)
```

Keywords inherit whatever typos the question carries. Pull tokens straight from “How does multi-head atention compare to self-atention?” and retrieval searches for `atention`, a string the document never contains. Zero hits, the system returns nothing, the user concludes the topic isn’t covered. The fix is a cheap pre-step that runs before keyword extraction: one LLM call that corrects typos and grammar without changing meaning, so the keywords come out clean.
关键词会继承问题中携带的任何拼写错误。如果直接从“How does multi-head atention compare to self-atention?”中提取标记，检索系统会搜索 `atention`，而文档中从未包含该字符串。结果是零命中，系统返回空，用户会认为该主题未被覆盖。解决方法是在关键词提取前运行一个低成本的预处理步骤：调用一次 LLM，在不改变含义的情况下纠正拼写和语法错误，从而使关键词变得准确。

```python
def correct_spelling(question: str) -> str:
    """Fix typos and grammar without changing meaning."""
    prompt = f"""Fix any typos and grammar mistakes in the question below. 
    Do not change the meaning. Do not add or remove information. 
    Return only the corrected question, nothing else. 
    Question: {question}"""
    resp = client.responses.create(model="gpt-4.1-mini", input=prompt)
    return resp.output_text.strip()
```

In production, this is cached (the same question typed by multiple users gets corrected once) and skipped when the input is clean.
在生产环境中，此步骤会被缓存（多个用户输入的相同问题只需纠正一次），并且在输入本身就很规范时会被跳过。

Let users name the keywords themselves. Some users (analysts, paralegals, anyone fluent in the document’s vocabulary) already know exactly which terms they want matched. A UI hint, “List exact terms to search for, separated by commas”, opens the highest-precision retrieval path the system has. The user’s tokens go in verbatim: weight 1.0, source direct, no LLM, no synonym expansion (unless they opt in).
让用户自己指定关键词。一些用户（分析师、律师助理、任何熟悉文档词汇的人）非常清楚他们想要匹配哪些术语。UI 提示“列出要搜索的确切术语，用逗号分隔”可以开启系统最高精度的检索路径。用户的标记将按原样输入：权重 1.0，来源为直接输入，不经过 LLM，不进行同义词扩展（除非用户选择开启）。

For “Please find ‘force majeure’, ‘rescission’, ‘event of default’ in this contract”, the parser pulls the three quoted phrases as-is. Faster, cheaper, more accurate than any LLM rewrite when the user can name the terms.
对于“请在合同中查找‘不可抗力’、‘解除’、‘违约事件’”，解析器会按原样提取这三个带引号的短语。当用户能够明确指出术语时，这种方式比任何 LLM 重写都更快、更便宜、更准确。
