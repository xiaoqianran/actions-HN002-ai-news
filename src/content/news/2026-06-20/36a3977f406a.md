我原以为这是一个定时任务（Scheduling）的问题，但结果发现，首先要解决的其实是可移植性（Portability）问题。

这个系列中的每一篇文章都记录了我实际构建的项目、我所做的决策、遇到的故障以及从中获得的经验。第一篇文章是我为期 12 个月的自学路线图，我在其中规划了如何完成这次转型。第二篇文章记录了我作为一个完全的初学者，如何从零开始使用 GitHub API 构建我的第一个 ETL 流水线。在第三篇中，我通过添加 SQLite 存储、幂等性处理和 Google Drive 持久化，将该流水线变得更具生产可用性，而这一切都是在 Google Colab 中完成的。

### The First Wall: My Pipeline Lived in Colab
### 第一堵墙：我的流水线“寄居”在 Colab 中

Before I even got to scheduling, I wanted to understand what it would actually take to run my pipeline automatically. So I looked at my code properly for the first time with that question in mind. Here’s what the load section looked like:
在开始设置调度之前，我想先弄清楚自动运行流水线到底需要什么。于是，我带着这个问题第一次认真审视了我的代码。以下是数据加载（load）部分的代码：

`conn = sqlite3.connect('/content/drive/MyDrive/github_repos.db')`

That path, `/content/drive/MyDrive/`, only exists inside Google Colab. It’s the mounted Google Drive path that Colab gives you when you connect your Drive to a notebook. Outside Colab, that path doesn’t exist. If any scheduler tried to run this script, it would crash right there.
这个路径 `/content/drive/MyDrive/` 仅存在于 Google Colab 内部。这是当你将 Google Drive 连接到笔记本时，Colab 为你挂载的路径。在 Colab 之外，这个路径是不存在的。如果任何调度器尝试运行这个脚本，它会立刻报错崩溃。

The interesting thing is that my code had no `google.colab` imports. No Colab-specific libraries. Just one hardcoded path that I had been typing without really thinking about it. That path was the dependency, not the code. This was the first thing I didn’t expect. I thought the challenge would be learning a scheduling tool. Instead, the first lesson was that my environment was part of my pipeline, and I hadn’t noticed.
有趣的是，我的代码中并没有导入 `google.colab`，也没有使用任何 Colab 特有的库。仅仅是一个我随手写下、从未深思的硬编码路径。这个路径成了依赖项，而不是代码本身。这是我没想到的第一点。我原以为挑战在于学习调度工具，但第一课却是：我的运行环境其实是流水线的一部分，而我之前竟然没意识到。

The fix was simple. Instead of hardcoding the Colab path, I made the database path configurable through an environment variable:
解决方法很简单。我不再硬编码 Colab 路径，而是通过环境变量使数据库路径可配置：

```python
import os
DB_PATH = os.environ.get('DB_PATH', 'github_repos.db')
conn = sqlite3.connect(DB_PATH)
```

Now the script uses whatever path is set in the environment. If nothing is set, it falls back to creating a local `github_repos.db` file in the same folder. One change, and the pipeline was no longer tied to Colab.
现在，脚本会使用环境变量中设置的任何路径。如果没有设置，它会默认在同一文件夹下创建一个本地的 `github_repos.db` 文件。只需这一处修改，流水线就不再绑定于 Colab 了。

### Running It Outside Colab for the First Time
### 第一次在 Colab 之外运行

Before setting up any scheduler, I wanted to confirm the script actually worked on its own. So I saved it as `pipeline.py`, created a `requirements.txt` with the two libraries it needs (`requests`, `pandas`), and ran it from my terminal. It printed: `Pipeline complete. Duplicates handled.` And a file called `github_repos.db` appeared in my folder.
在设置任何调度器之前，我想确认脚本是否能独立运行。于是我将其保存为 `pipeline.py`，创建了一个包含所需两个库（`requests`, `pandas`）的 `requirements.txt`，并在终端中运行了它。程序输出了：`Pipeline complete. Duplicates handled.`，并且文件夹中出现了一个名为 `github_repos.db` 的文件。

The same pipeline I had been running in Colab was now running as a plain Python script, anywhere. That felt like a bigger deal than I expected. Not because the change was complex, it wasn’t. But because I realized I had been thinking of my pipeline as a notebook, when what I actually had was a script that happened to live inside one.
我之前在 Colab 中运行的同一个流水线，现在可以作为普通的 Python 脚本在任何地方运行了。这比我预想的意义重大得多。并不是因为修改过程有多复杂（其实并不复杂），而是因为我意识到，我一直把流水线当成一个“笔记本（Notebook）”，而实际上我拥有的只是一个恰好寄存在笔记本里的脚本。

### Choosing a Scheduling Tool
### 选择调度工具

At this point I had a standalone script. Now I needed something to run it on a schedule. I looked at a few options.
现在我拥有了一个独立的脚本，接下来需要一个工具来定时运行它。我研究了几个选项。

*   **APScheduler** lets you define schedules inside your Python code, which works while a session is running but stops the moment you close your terminal. That’s not really scheduling, that’s just a loop.
    **APScheduler** 允许你在 Python 代码中定义调度，它在会话运行时有效，但一旦关闭终端就会停止。那其实不算调度，只是一个循环。
*   **Airflow** is the industry standard for orchestrating pipelines, but it requires running a server, a metadata database, and a web interface. That’s a lot of infrastructure for where I am right now.
    **Airflow** 是流水线编排的行业标准，但它需要运行服务器、元数据库和 Web 界面。对于我目前的阶段来说，这需要维护的基础设施太庞大了。
*   **GitHub Actions** sat in the middle. It’s free, it runs on GitHub’s servers, the schedule is defined in code, and it doesn’t require me to maintain any infrastructure. The tradeoff is that it’s designed for CI/CD workflows, not pipeline orchestration, so it has limits around complex dependencies and monitoring. But for a pipeline at my stage, it’s a practical choice.
    **GitHub Actions** 处于中间位置。它是免费的，运行在 GitHub 的服务器上，调度逻辑定义在代码中，且不需要我维护任何基础设施。代价是它专为 CI/CD 工作流设计，而非流水线编排，因此在处理复杂依赖和监控方面存在局限。但对于我目前阶段的流水线来说，这是一个务实的选择。

I also want to be honest: tools like Airflow exist for a reason. When a pipeline grows, when you have dependencies between tasks, when you need visibility into what ran and what failed, you need proper orchestration. GitHub Actions is not that. But it’s a good first step, and understanding why it’s limited is part of learning what those more serious tools are actually solving.
我也想坦诚一点：像 Airflow 这样的工具存在是有原因的。当流水线规模扩大、任务之间存在依赖关系，或者你需要清晰地了解哪些任务运行成功、哪些失败时，你就需要专业的编排工具。GitHub Actions 并非为此而生。但它是一个很好的起点，理解它的局限性，也是学习那些更专业的工具究竟解决了什么问题的过程。

### Setting Up GitHub Actions
### 设置 GitHub Actions

GitHub Actions works through workflow files, which are YAML files you place in a specific folder in your repository. The folder structure looks like this:
GitHub Actions 通过工作流文件（Workflow files）运行，这些文件是放置在仓库特定文件夹中的 YAML 文件。文件夹结构如下：

```text
github-etl/
├── .github/
│   └── workflows/
│       └── schedule.yml
├── pipeline.py
└── requirements.txt
```

Here’s the full workflow file I created:
这是我创建的完整工作流文件：

```yaml
name: Run ETL Pipeline
on:
  schedule:
    - cron: '0 9 * * *'
  workflow_dispatch:
jobs:
  run-pipeline:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install dependencies
        run: pip install -r requirements.txt
      - name: Run pipeline
        run: python pipeline.py
```

Let me walk through what each part is doing. `cron: '0 9 * * *'` is the actual schedule. Cron is a time-based job scheduling format that’s been around in Unix systems.
让我来解释一下每一部分的作用。`cron: '0 9 * * *'` 是实际的调度时间。Cron 是一种基于时间的作业调度格式，在 Unix 系统中已经存在很久了。
