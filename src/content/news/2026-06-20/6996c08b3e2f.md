# Building a Custom GStreamer Plugin for NVIDIA DeepStream
# 为 NVIDIA DeepStream 构建自定义 GStreamer 插件

NVIDIA DeepStream gives you a production-ready pipeline for multi-stream video analytics: hardware-accelerated decoding, tracking, on-screen display, and message brokering, all wired through GStreamer. For standard detection models exported to TensorRT, nvinfer handles everything. However the common case has limits. Vision-language models, custom post-processing, rotated bounding boxes, or the need to hot-swap models at runtime, these are places where nvinfer‘s assumptions break down.
NVIDIA DeepStream 为多流视频分析提供了一个生产就绪的流水线：通过 GStreamer 串联起硬件加速解码、追踪、屏幕显示和消息代理等功能。对于导出为 TensorRT 的标准检测模型，`nvinfer` 可以处理一切。然而，常见场景往往存在局限性。视觉语言模型、自定义后处理、旋转边界框，或者需要在运行时热插拔模型，这些都是 `nvinfer` 的预设逻辑失效的场景。

Sometimes you have a mature PyTorch inference stack your team has carefully tuned, and you want DeepStream to call that rather than reimplementing it in a config file. It’s worth noting that for YOLO-family models specifically, DeepStream-Yolo by Marcos Luciano has already done excellent work implementing custom postprocessing in C++. If C++ is on the table, start there. This article takes a different angle: achieving the same result entirely in Python, using a custom GStreamer plugin with pyservicemaker without sacrificing throughput.
有时，你的团队已经拥有了一套精心调优的成熟 PyTorch 推理栈，你希望 DeepStream 直接调用它，而不是在配置文件中重新实现。值得注意的是，对于 YOLO 系列模型，Marcos Luciano 开发的 DeepStream-Yolo 已经在 C++ 自定义后处理方面做得非常出色。如果可以使用 C++，请优先考虑该方案。本文将采用不同的视角：完全使用 Python，通过 `pyservicemaker` 构建自定义 GStreamer 插件，在不牺牲吞吐量的前提下实现相同的结果。

The key insight that makes this possible: downstream elements like nvtracker, nvdsosd, and nvmsgconv don’t care which element produced detection metadata. Write to DeepStream’s metadata structure correctly and the rest of the ecosystem works as if nvinfer was never in the picture.
实现这一点的关键洞察在于：下游组件（如 `nvtracker`、`nvdsosd` 和 `nvmsgconv`）并不关心检测元数据是由哪个组件生成的。只要正确写入 DeepStream 的元数据结构，整个生态系统的其余部分就会像 `nvinfer` 从未出现过一样正常工作。

### DeepStream Metadata
### DeepStream 元数据

Every buffer flowing through a DeepStream pipeline carries more than pixel data. From the moment frames pass through nvstreammux, each GstBuffer has an NvDsBatchMeta structure attached to it. The hierarchy is straightforward and can be found in the official documentation.
流经 DeepStream 流水线的每个缓冲区携带的信息远不止像素数据。从帧通过 `nvstreammux` 的那一刻起，每个 `GstBuffer` 都会附带一个 `NvDsBatchMeta` 结构。其层级结构非常直观，可以在官方文档中找到。

NvDsBatchMeta describes the whole batch. Each NvDsFrameMeta corresponds to one source stream and carries frame-level information like the source ID and frame number. Each NvDsObjectMeta represents a single detection, meaning that when our plugin writes detections, we’ll write an NvDsObjectMeta for each one. The critical thing to understand is that none of this is owned by nvinfer. It’s a shared data contract.
`NvDsBatchMeta` 描述了整个批次。每个 `NvDsFrameMeta` 对应一个源流，并携带帧级信息，如源 ID 和帧编号。每个 `NvDsObjectMeta` 代表一个检测结果，这意味着当我们的插件写入检测结果时，我们将为每一个检测结果写入一个 `NvDsObjectMeta`。需要理解的关键点是，这些数据并不由 `nvinfer` 独有，而是一种共享的数据契约。

Any GStreamer element in the pipeline can read from it, write to it, or both: nvtracker reads object bounding boxes and writes tracking IDs. nvdsosd reads boxes and labels to draw overlays. nvmsgconv reads the whole structure to produce message payloads. Our custom plugin will simply write detections into this structure the same way nvinfer would and everything downstream picks them up without modification.
流水线中的任何 GStreamer 组件都可以读取、写入或同时执行这两项操作：`nvtracker` 读取对象边界框并写入追踪 ID；`nvdsosd` 读取框和标签以绘制覆盖层；`nvmsgconv` 读取整个结构以生成消息负载。我们的自定义插件只需像 `nvinfer` 那样将检测结果写入此结构，下游的所有组件即可直接获取，无需任何修改。

One important constraint worth understanding before we write any code: NvDsObjectMeta instances cannot be constructed directly from Python. Attempting to instantiate the class raises a `No constructor defined!` error at runtime. The reason is architectural. DeepStream manages its metadata objects through memory pools, pre-allocated blocks that get recycled across frames to avoid the overhead of repeated heap allocation and deallocation in a high-throughput pipeline.
在编写代码之前，有一个重要的约束需要理解：`NvDsObjectMeta` 实例不能直接从 Python 中构造。尝试实例化该类会在运行时引发 `No constructor defined!` 错误。其原因是架构性的。DeepStream 通过内存池管理其元数据对象，这些预分配的块在帧之间循环使用，以避免在高吞吐量流水线中重复进行堆分配和释放带来的开销。

The Python bindings expose access to those pools, but deliberately don’t expose a Python-side constructor, because creating an NvDsObjectMeta outside the pool would bypass the lifecycle management that keeps DeepStream’s memory usage predictable. The correct way to get one is to ask the batch for it: `batch_meta.acquire_object_meta()`, which hands you a pre-allocated instance from the pool. When the frame is done, DeepStream returns it to the pool automatically.
Python 绑定提供了对这些内存池的访问权限，但特意没有公开 Python 端的构造函数，因为在内存池之外创建 `NvDsObjectMeta` 会绕过生命周期管理，而该管理机制保证了 DeepStream 内存使用的可预测性。获取实例的正确方法是向批次请求：`batch_meta.acquire_object_meta()`，它会从池中为你提供一个预分配的实例。当帧处理完成后，DeepStream 会自动将其归还给内存池。

### The Python Bridge: pyservicemaker
### Python 桥接：pyservicemaker

To interact with DeepStream’s metadata from Python, we’ll use pyservicemaker, NVIDIA’s current, supported Python SDK for DeepStream. The official documentation covers the basics of pipelines and flows, but stops short of showing how to write and attach metadata from a custom inference element. That’s the gap this article fills.
为了从 Python 与 DeepStream 的元数据进行交互，我们将使用 `pyservicemaker`，这是 NVIDIA 目前支持的 DeepStream Python SDK。官方文档涵盖了流水线和数据流的基础知识，但并未展示如何从自定义推理组件中编写和附加元数据。这正是本文要填补的空白。

The key abstraction is `BatchMetadataOperator`. Subclassing it and implementing `handle_metadata(batch_meta)` gives you access to the full `NvDsBatchMeta` for every buffer flowing through the pipeline. From there, iterating frames is just as simple as using `batch_meta.frame_items` and attaching a detection object.
核心抽象是 `BatchMetadataOperator`。通过继承它并实现 `handle_metadata(batch_meta)`，你可以访问流经流水线的每个缓冲区的完整 `NvDsBatchMeta`。在此基础上，遍历帧就像使用 `batch_meta.frame_items` 并附加检测对象一样简单。

### A Discoverable Python GStreamer Plugin
### 可被发现的 Python GStreamer 插件

GStreamer discovers plugins at runtime by scanning directories listed in `GST_PLUGIN_PATH`. For Python plugins specifically, it looks inside a `python/` subdirectory within each of those paths. That means your plugin is just a `.py` file dropped in the right place, no compilation, no CMake, no shared library.
GStreamer 在运行时通过扫描 `GST_PLUGIN_PATH` 中列出的目录来发现插件。对于 Python 插件，它会专门查找这些路径下的 `python/` 子目录。这意味着你的插件只需是一个放置在正确位置的 `.py` 文件，无需编译，无需 CMake，也无需共享库。

The tradeoff is that the registration pattern is strict and getting it wrong produces silent failures that are genuinely hard to debug.
其代价是注册模式非常严格，一旦出错就会产生难以调试的静默失败。

```bash
$GST_PLUGIN_PATH/
└── python/
    └── gstexampleplugin.py # your plugin
```

Set `GST_PLUGIN_PATH` to point at the parent directory and GStreamer will find `python/gstexampleplugin.py` automatically on the next pipeline run.
将 `GST_PLUGIN_PATH` 指向父目录，GStreamer 就会在下一次运行流水线时自动找到 `python/gstexampleplugin.py`。
