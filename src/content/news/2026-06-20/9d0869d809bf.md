### What is Python’s New Just-In-Time (JIT) compiler?
### 什么是 Python 的全新即时（JIT）编译器？

To understand the 3.14 JIT, we need to know how Python traditionally runs. Standard Python (CPython) is an interpreted language. When you run a script, your code is compiled into bytecode, which is a set of instructions that the CPython virtual machine executes. The JIT changes this flow. Instead of simply interpreting bytecode line-by-line, the JIT monitors which parts of your code are executed most frequently (the “hot” paths). When a function or loop is deemed “hot,” the JIT translates the bytecode into native machine code (instructions the CPU understands). Then, the next time the code is invoked, no interpretation is required. Instead, it just runs as it is. This can be a great time-saver, as we’ll see later on.
要理解 3.14 的 JIT，我们需要了解 Python 传统的运行方式。标准 Python (CPython) 是一种解释型语言。当你运行脚本时，代码会被编译成字节码，这是一组由 CPython 虚拟机执行的指令。JIT 改变了这一流程。JIT 不再只是逐行解释字节码，而是会监控代码中哪些部分执行最频繁（即“热点”路径）。当某个函数或循环被判定为“热点”时，JIT 会将字节码转换为原生机器码（CPU 能理解的指令）。这样，下次调用该代码时，就不再需要解释，而是直接运行。正如我们稍后将看到的，这可以极大地节省时间。

### How the JIT fits into CPython
### JIT 如何融入 CPython

The Python 3.14 JIT is not a total rewrite. It is designed as an opt-in component that works alongside the existing interpreter. It uses a technique called “copy-and-patch,” which allows the JIT to be lightweight and portable across different CPU architectures without requiring a massive, complex compiler backend like LLVM.
Python 3.14 的 JIT 并非完全重写。它被设计为一个可选组件，与现有的解释器协同工作。它使用了一种称为“复制与修补”（copy-and-patch）的技术，这使得 JIT 能够保持轻量级，并可在不同的 CPU 架构间移植，而无需像 LLVM 那样庞大且复杂的编译器后端。

### What Changed in Python 3.14?
### Python 3.14 有什么变化？

Python 3.13 had a basic, experimental JIT, but it was disabled by default. If you wanted to test it, you had to clone the CPython source tree and compile it with specific experimental flags such as `--enable-experimental-jit`. With Python 3.14, everything changed. It offered the JIT in the official .msi (Windows) and .pkg (macOS) installers. It also meant that you no longer needed a C compiler on your machine to experience JIT benefits. While still “experimental,” the inclusion in official binaries signals that the core team believes the JIT is stable enough for broad community testing.
Python 3.13 拥有一个基础的实验性 JIT，但默认是禁用的。如果你想测试它，必须克隆 CPython 源码树并使用特定的实验性标志（如 `--enable-experimental-jit`）进行编译。而在 Python 3.14 中，一切都变了。它在官方的 .msi (Windows) 和 .pkg (macOS) 安装程序中提供了 JIT。这也意味着你不再需要在机器上安装 C 编译器就能体验 JIT 的优势。尽管它仍处于“实验性”阶段，但被纳入官方二进制文件表明核心团队认为 JIT 已经足够稳定，可以进行广泛的社区测试。

### Enabling the JIT
### 启用 JIT

By default, the JIT is disabled. This is a safety measure; because it is experimental, the Python Steering Council wants to ensure that users don’t face unexpected regressions in stability or memory usage without explicitly choosing to. To activate the JIT, you use an environment variable. This tells the CPython runtime to initialise the JIT engine upon startup.
默认情况下，JIT 是禁用的。这是一种安全措施；由于它处于实验阶段，Python 指导委员会希望确保用户在未明确选择的情况下，不会遇到稳定性或内存使用方面的意外回归。要激活 JIT，你需要使用环境变量。这会告诉 CPython 运行时在启动时初始化 JIT 引擎。

On Windows (PowerShell):
在 Windows (PowerShell) 上：
`$env:PYTHON_JIT=1 python my_script.py`

On macOS/Linux (Bash/Zsh):
在 macOS/Linux (Bash/Zsh) 上：
`PYTHON_JIT=1 python my_script.py`

Once enabled, CPython doesn’t JIT-compile everything immediately. It uses a tiering system. Basically, it tries to run code as cheaply as possible first, and only spends compilation/optimisation effort on the parts that prove to be hot.
一旦启用，CPython 不会立即对所有内容进行 JIT 编译。它使用分层系统。基本上，它会先尝试以尽可能低的成本运行代码，只有在代码被证明是“热点”时，才会投入编译/优化工作。

*   **Tier 0:** Standard interpretation. (第 0 层：标准解释)
*   **Tier 1:** Specialised bytecode (introduced in 3.11). (第 1 层：专用字节码，3.11 引入)
*   **Tier 2 (The JIT):** Machine code generation for the most frequently used paths. (第 2 层（JIT）：为最频繁使用的路径生成机器码)

### Measuring the Impact of the JIT
### 衡量 JIT 的影响

When testing a JIT, you can’t simply use the `time.time()` around a function. JITs require a warm-up period. The first few iterations of a loop might be slower than normal as the JIT profiles the code, but subsequent iterations can be significantly faster.
在测试 JIT 时，不能简单地在函数周围使用 `time.time()`。JIT 需要预热期。循环的前几次迭代可能会比平时慢，因为 JIT 正在对代码进行分析，但随后的迭代速度可能会显著加快。
