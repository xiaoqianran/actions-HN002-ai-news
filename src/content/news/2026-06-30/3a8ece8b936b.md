# Prompt Engineering Fails Quietly — Prompt Regression Is Why
# 提示工程的静默失效——这就是“提示词回归”的原因

A practical framework to detect hidden prompt regressions before they silently break production systems.
一个在提示词静默破坏生产系统之前，检测隐藏的“提示词回归”的实用框架。

***

**TL;DR**
**摘要**

Prompts are not static config files. Every instruction you add changes the behaviour of every query type the prompt already handles. Most teams catch prompt failures through user reports, not tests. This article builds the test suite. The suite runs 40 golden queries across four prompt versions, validates outputs with four deterministic checks, and detects the False Improvement pattern, where overall accuracy rises while a critical category collapses. v4, the “best” prompt at 67.5% overall accuracy, triggered FALSE IMPROVEMENT DETECTED due to a 66.7% collapse in negation classification. Zero external dependencies. Pure Python. Runs in under two seconds.
提示词不是静态配置文件。你添加的每一条指令都会改变提示词已处理的所有查询类型的行为。大多数团队是通过用户反馈而非测试来发现提示词失效的。本文构建了一个测试套件。该套件在四个提示词版本上运行 40 个黄金查询，通过四个确定性检查验证输出，并检测“虚假改进”模式——即整体准确率上升，但关键类别却出现崩溃。v4 版本虽然以 67.5% 的整体准确率被认为是“最佳”提示词，却触发了“检测到虚假改进”警报，原因是其否定分类能力下降了 66.7%。零外部依赖，纯 Python 编写，运行时间不到两秒。

***

My RAG query layer was working fine. Then I added document routing for PDFs and policies, and the prompt ballooned from six instructions to fourteen. I spot-tested a few cases, everything looked right, and I shipped it. Three weeks later, I was tracking down a support issue where negation queries (stuff like “Which products are not covered under warranty?”) were being misclassified as standard policy lookups instead of negation checks. The weird part was that I hadn’t touched the classification logic or the routing code. The only thing that changed was the system prompt.
我的 RAG（检索增强生成）查询层原本运行良好。后来我为 PDF 和政策文档添加了文档路由功能，提示词从 6 条指令激增至 14 条。我抽查了几个案例，一切看起来都很正常，于是我发布了更新。三周后，我在排查一个支持工单时发现，否定查询（例如“哪些产品不在保修范围内？”）被错误地分类为标准政策查询，而不是否定检查。奇怪的是，我根本没有动过分类逻辑或路由代码，唯一改变的就是系统提示词。

That’s when I understood the problem. I was treating my prompt like a static config file. It isn’t. A prompt is a stochastic API, and every time you add instructions to it, you are changing the API contract for every query type it handles, not just the ones you were thinking about.
就在那时，我明白了问题所在。我一直把提示词当作静态配置文件，但它并不是。提示词是一个随机 API，每当你向其中添加指令时，你实际上是在改变它所处理的所有查询类型的 API 契约，而不仅仅是你所关注的那部分。

The software engineering world has a name for what I didn’t have: a regression test suite. The idea is simple. Before any change ships, you run the tests. If something that was passing is now failing, you do not ship. I had nothing like that for prompts. Most teams don’t.
软件工程界对我所缺失的东西有一个专门的术语：回归测试套件。这个概念很简单：在发布任何更改之前，先运行测试。如果原本通过的测试现在失败了，就不要发布。我当时没有为提示词做任何类似的工作，大多数团队也没有。

This mirrors the core idea behind Test-Driven Development (Beck [5]): define expected behavior before making changes. The discipline forces you to define correct behavior before you touch the code. Applied to prompts, this means defining valid classification logic for each category before adding a new instruction. Without these definitions, you have no way to detect when a change breaks something you weren’t even thinking about.
这反映了测试驱动开发（TDD，Beck [5]）背后的核心思想：在进行更改之前定义预期行为。这种规范迫使你在触碰代码之前先定义正确的行为。应用到提示词上，这意味着在添加新指令之前，必须为每个类别定义有效的分类逻辑。如果没有这些定义，你就无法检测到更改何时破坏了你未曾考虑到的功能。

The hidden cost problem exists in ML systems as well. Sculley et al. [4] documented how undeclared dependencies and unstable data interfaces accumulate as technical debt in production ML pipelines. A prompt that silently alters behavior across categories without detection is this exact class of problem. The interface looks stable from the outside, but the behavior has drifted underneath.
隐藏成本问题在机器学习系统中同样存在。Sculley 等人 [4] 记录了未声明的依赖项和不稳定的数据接口如何在生产级机器学习流水线中积累为技术债。一个在未被检测的情况下静默改变跨类别行为的提示词，正是这类问题。从外部看接口似乎很稳定，但底层的行为已经发生了偏移。

All numbers below are from real runs of this system on Python 3.12, Windows 11, CPU only. The code is at: https://github.com/Emmimal/prompt-regression-suite
以下所有数据均来自该系统在 Python 3.12、Windows 11、仅 CPU 环境下的实际运行结果。代码地址：https://github.com/Emmimal/prompt-regression-suite

### The Setup
### 设置

The regression suite tests four prompt versions against 40 golden queries across six intent categories, built on top of a RAG intent classification system [1]. The four versions reflect a real iteration sequence from the RAG intent classification system I built for this article. Every single change was made for a legitimate reason, and every single one introduced a hidden problem.
该回归测试套件基于一个 RAG 意图分类系统 [1]，针对六个意图类别下的 40 个黄金查询测试了四个提示词版本。这四个版本反映了我为本文构建的 RAG 意图分类系统的真实迭代序列。每一次更改都是出于正当理由，但每一次更改也都引入了隐藏的问题。

v1 is the baseline. It handles clean intent classification with minimal instructions and zero reasoning steps. There is just one rule about keeping things concise and another about the JSON output format.
v1 是基准版本。它通过最少的指令和零推理步骤处理简单的意图分类。它只有一条关于保持简洁的规则和一条关于 JSON 输出格式的规则。

v2 adds chain-of-thought reasoning. I brought this in because multi-hop queries like checking a response time for an enterprise plan with a P1 ticket after hours were getting misclassified. Chain-of-thought has been shown to significantly improve performance on complex reasoning tasks [2], and it did fix that specific problem. The mistake was applying it globally. The v2 prompt now tells the model to “be concise” in one rule, while demanding it “explain your reasoning step by step” in another. Those two rules contradict each other on every simple query the system touches.
v2 添加了思维链推理。我引入它是为了解决多跳查询（例如在非工作时间检查企业计划的 P1 工单响应时间）被错误分类的问题。思维链已被证明能显著提高复杂推理任务的性能 [2]，它确实解决了那个特定问题。错误在于将其全局应用了。v2 提示词现在一方面要求模型“保持简洁”，另一方面又要求它“逐步解释你的推理过程”。这两条规则在系统处理的每一个简单查询上都相互矛盾。

v3 adds document routing. The new instructions tell the model to check for tabular, policy, and PDF signals before it classifies intent. One line in particular completely broke negation handling: “Prioritize document routing before intent classification.” Negation queries like “Which regions are excluded from the express shipping policy?” contain policy keywords, so under v3, the model resolves the document type before it ever touches intent. The negation check never even fires.
v3 添加了文档路由。新指令要求模型在分类意图之前先检查表格、政策和 PDF 信号。其中一行指令彻底破坏了否定处理：“在意图分类之前优先进行文档路由。”像“哪些地区不在快递政策范围内？”这样的否定查询包含政策关键词，因此在 v3 下，模型在触及意图分类之前就先解析了文档类型。否定检查甚至从未被触发。

v4 combines both changes, and this is what became the production prompt. The total instruction surface area roughly tripled, and the latent conflicts from v2 and v3 are now compounding.
v4 结合了上述两项更改，这就是最终的生产环境提示词。指令总数大约增加了两倍，v2 和 v3 中潜在的冲突现在叠加在一起了。

### The Golden Set
### 黄金数据集

The 40 queries are distributed across six categories.
40 个查询分布在六个类别中。

| Category | Failure Mode Targeted |
| :--- | :--- |
| simple_intent | overreasoning_noise |
| comparison | missing_comparative_anchor |
| aggregation | numeric_scope_collapse |
| negation | instruction_conflict |
| multi_hop | benefits_from_cot |
| edge_ambiguous | false_confidence |
| **TOTAL** | **40** |

| 类别 | 针对的失效模式 |
| :--- | :--- |
| simple_intent (简单意图) | overreasoning_noise (过度推理噪声) |
| comparison (比较) | missing_comparative_anchor (缺失比较锚点) |
| aggregation (聚合) | numeric_scope_collapse (数值范围崩溃) |
| negation (否定) | instruction_conflict (指令冲突) |
| multi_hop (多跳) | benefits_from_cot (受益于思维链) |
| edge_ambiguous (边缘模糊) | false_confidence (虚假置信度) |
| **总计** | **40** |

Each query was chosen to expose a specific failure mode, not to be a general representation. Take the comparison category, for instance. It is a known failure in this system because comparison queries require a comparative anchor that the current prompt architecture simply does not resolve. I am not hiding that in this benchmark, and you will see the [KNOWN FAILURE] annotation in every single diff report.
选择每个查询都是为了暴露特定的失效模式，而不是为了作为通用代表。以“比较”类别为例，这是该系统中的一个已知故障，因为比较查询需要一个当前的提示词架构根本无法解析的比较锚点。我并没有在基准测试中掩盖这一点，你会在每一份差异报告中看到 [KNOWN FAILURE]（已知故障）的标注。

Instead of checking against a hardcoded reference answer, each query carries a validation signature: a set of deterministic constraints.
每个查询不再是与硬编码的参考答案进行比对，而是携带一个验证签名：一组确定性的约束条件。
