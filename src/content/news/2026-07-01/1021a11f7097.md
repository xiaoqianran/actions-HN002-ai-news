# Stop Choosing Between Local and Cloud LLMs: A Field Guide to Hybrid Patterns
# 不必再纠结于本地还是云端大模型：混合架构实战指南

A hands-on walkthrough of a hybrid local-cloud workflow using Gemma 4 and GPT-5.4, with reasoning and structured outputs.

Today, when people build LLM applications, two deployment choices are commonly seen: either we go fully cloud, i.e., sending everything to a cloud LLM API, or we go fully local, i.e., run everything with an open model served locally.
如今，在构建大模型应用时，常见的部署选择有两种：要么完全依赖云端，即将所有数据发送至云端大模型 API；要么完全本地化，即所有任务均由本地运行的开源模型处理。

Cloud LLMs reason better, but we send our sensitive context outside. Local models keep that context private, but limited by the local compute, they can struggle when tasks get complex. Naturally, we’d like to ask: Can we enjoy the reasoning capability of the cloud, while still keeping private context local? A hybrid local-cloud pattern might achieve that.

### 1. When Does A Hybrid Local-Cloud LLM Pattern Make Sense?
### 1. 何时适合采用本地-云端混合大模型架构？

When people talk about local-cloud hybrid LLM applications, they immediately think of privacy. That’s certainly an important consideration. But privacy is not the only reason to go hybrid. Based on my experience, I find it useful to reason about this hybrid pattern from the lens of a three-axis coordinate system:
当人们谈论本地-云端混合大模型应用时，首先想到的往往是隐私。这固然是一个重要的考量，但隐私并非采用混合架构的唯一理由。根据我的经验，从三轴坐标系的视角来思考这种混合模式非常有效：

*   **Direction**, which answers “who acts first?”. It can be local-first or cloud-first.
*   **方向（Direction）**：回答“谁先行动？”的问题。可以是本地优先或云端优先。
*   **Trigger**, which answers “when is the cloud used?”. In some application scenarios, the cloud is always called. At other times, the cloud is only invoked conditionally.
*   **触发（Trigger）**：回答“何时调用云端？”的问题。在某些场景下，云端会被持续调用；而在另一些场景下，云端仅在特定条件下被触发。
*   **Purpose**, which answers “why split the workflow?”. As we mentioned previously, privacy is a major motivation. But other factors can be cost & latency, as well as trust & reliability.
*   **目的（Purpose）**：回答“为什么要拆分工作流？”的问题。如前所述，隐私是主要动机，但成本、延迟、信任度及可靠性也是重要因素。

With these three axes, we can place many real-world hybrid workflows in the same coordinate map. Although this is far from a perfect taxonomy, I believe it makes the design choices intuitively understandable.
通过这三个轴，我们可以将许多现实中的混合工作流置于同一个坐标系中。虽然这远非完美的分类法，但我相信它能让设计选择变得直观易懂。

#### Pattern 1: Sanitize-and-Solve
#### 模式 1：脱敏与求解 (Sanitize-and-Solve)

This pattern is local-first (axis 1), always triggers cloud LLM (axis 2), and has privacy-preserving in mind (axis 3). The local model consumes the unstructured, messy private context, but converts it into an abstract problem, which later gets sent to the cloud LLM. The cloud model only sees the abstract problem and solves it.
该模式为本地优先（轴 1），始终触发云端大模型（轴 2），且以隐私保护为核心（轴 3）。本地模型处理非结构化、杂乱的私有上下文，将其转化为抽象问题，随后发送给云端大模型。云端模型仅能看到抽象问题并进行求解。

#### Pattern 2: Plan-then-Ground
#### 模式 2：规划与落地 (Plan-then-Ground)

This pattern sits in the opposite direction. It is cloud-first (axis 1), the cloud model is always triggered (axis 2), and still privacy-preserving (axis 3). Here, the cloud model is responsible for producing a generic plan based on an abstract goal. Then, the plan is sent to the local model, who plays the role of the actual executor.
该模式方向相反。它是云端优先（轴 1），始终触发云端模型（轴 2），且同样具备隐私保护功能（轴 3）。云端模型负责根据抽象目标生成通用计划，随后将计划发送给本地模型，由本地模型担任实际执行者。

#### Pattern 3: Escalate-on-Hard
#### 模式 3：遇难升级 (Escalate-on-Hard)

For many applications, it’s not necessary to call the cloud model for every request. The local model handles the easy majority. The cloud model is called only when the local model cannot sufficiently or reliably address the request. This pattern is local-first (axis 1), conditional triggering (axis 2), and often motivated by cost and latency (axis 3).
对于许多应用而言，无需对每个请求都调用云端模型。本地模型可以处理绝大多数简单任务。仅当本地模型无法充分或可靠地处理请求时，才会调用云端模型。该模式为本地优先（轴 1），条件触发（轴 2），通常出于成本和延迟的考量（轴 3）。

#### Pattern 4: Draft-then-Refine
#### 模式 4：草拟与精修 (Draft-then-Refine)

In this pattern, the local model produces an immediate draft. Optionally, the cloud model can work in the background to provide a more careful answer. If the cloud answer is better, the application replaces or enriches the initial response. This is local-first (axis 1), optional triggering (axis 2), and motivated by cost and latency (axis 3).
在此模式下，本地模型生成即时草稿。云端模型可在后台运行，提供更严谨的回答。如果云端回答更优，应用将替换或丰富本地模型的初始响应。该模式为本地优先（轴 1），可选触发（轴 2），同样出于成本和延迟的考量（轴 3）。

#### Pattern 5: Cross-Check
#### 模式 5：交叉验证 (Cross-Check)

Here, the local and cloud models are treated as equal. They may act like independent reviewers: one proposes an answer, the other checks it; or both answer the same question, and their agreement becomes the signal for downstream processing. This pattern flows in both directions (axis 1), cloud is always on (axis 2), and it’s driven by trust & reliability (axis 3).
在此模式中，本地和云端模型被视为对等。它们像独立的评审员：一个提出答案，另一个进行核对；或者两者同时回答同一问题，它们的一致性将成为后续处理的信号。该模式双向流动（轴 1），云端始终开启（轴 2），主要由信任与可靠性驱动（轴 3）。

### 2. Case Study: Should I Run the Dishwasher Now or Later?
### 2. 案例研究：洗碗机现在开还是稍后开？

In this section, we move from abstract concepts to a concrete case study. Specifically, we’ll look at a smart-home scheduling problem.
在本节中，我们将从抽象概念转向具体案例研究，探讨一个智能家居调度问题。

#### 2.1 Case Setup
#### 2.1 案例背景

Suppose the user asks the assistant: "Should the dishwasher run now or later?" The assistant needs to solve a scheduling problem. It knows the current time is 18:30 and has access to household memory, device facts, and tariff data.
假设用户问助手：“洗碗机现在开还是稍后开？”助手需要解决一个调度问题。它知道当前时间是 18:30，并能访问家庭记忆、设备信息和电价数据。
