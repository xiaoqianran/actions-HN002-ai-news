# What Can We Do When Memory Becomes the New Bottleneck in Data Engineering?
# 当内存成为数据工程的新瓶颈时，我们该怎么办？

Data Engineering: How Pandas chunking, Dask, and Polars help process millions of records when adding more compute isn't an option.
数据工程：当无法增加计算资源时，Pandas 分块处理、Dask 和 Polars 如何帮助处理数百万条记录。

In the AI era, memory has undoubtedly become a critical resource. As the demand for memory and storage infrastructure driven by the AI boom has reached historical highs, companies like Micron Technology and Sandisk have attracted unprecedented attention and increased product prices by wielding strong pricing power.
在人工智能时代，内存无疑已成为关键资源。随着人工智能热潮推动对内存和存储基础设施的需求达到历史新高，美光科技（Micron Technology）和闪迪（Sandisk）等公司凭借强大的定价权，吸引了前所未有的关注并提高了产品价格。

But this is not good news to companies that build data-intensive applications, rely on high-capacity storage for AI training, implement large-scale analytics, or operate on tight margins in cloud services. For data engineers, this isn’t just market news. Instead, it’s a daily constraint. When RAM and flash become more and more expensive, the old reflex of “add more capacity” no longer works. Budgets don’t scale with data volume and cloud bills are under scrutiny.
但这对于那些构建数据密集型应用、依赖大容量存储进行 AI 训练、实施大规模分析或在云服务中维持微薄利润的公司来说，并不是好消息。对于数据工程师而言，这不仅仅是市场新闻，更是日常的制约因素。当内存和闪存变得越来越昂贵时，“增加容量”的旧习惯已不再奏效。预算不会随着数据量的增加而增加，云账单也正受到严格审查。

As data engineers, what can we do when our dataset doubles but the cluster doesn’t? We must be creative. In this article, I’ll start from a real-world ETL challenge that requires to complete the data transformation of over 6 million social media posts with mixed-types data fields within limited computing capacity. I’ll walk through some solutions — including both classic and cutting-edge ones — to keep your ETL pipeline running without a hardware or cloud upgrade.
作为数据工程师，当数据集翻倍而集群规模不变时，我们该怎么办？我们必须发挥创造力。在本文中，我将从一个真实的 ETL 挑战出发，该挑战要求在有限的计算能力下，完成超过 600 万条包含混合数据类型字段的社交媒体帖子的数据转换。我将介绍一些解决方案（包括经典方案和前沿方案），以确保您的 ETL 流水线在无需硬件或云升级的情况下持续运行。

### The Problem: A 6.2 Million-Row Dataset That Wouldn’t Fit in Memory
### 问题：无法装入内存的 620 万行数据集

The story starts from a new ETL pipeline that you’re going to create. The raw data are 6.2 million posts from a social media platform. The dataset is extracted from the social media API and turns to over 200 columns after JSON flattening, and the most problematic part is: a large number of the data fields are mixed data types.
故事始于您即将创建的一个新的 ETL 流水线。原始数据来自某社交媒体平台的 620 万条帖子。该数据集从社交媒体 API 提取，在 JSON 扁平化后扩展为 200 多个列，最棘手的部分是：大量数据字段存在混合数据类型。

Because PySpark requires a consistent schema and the social media API schema changes from time to time, you consider using Pandas to tackle the mixed-types columns. Unlike PySpark, Pandas stores these columns as object by default and does not require every row to conform to the same schema.
由于 PySpark 要求模式（Schema）一致，而社交媒体 API 的模式经常变动，您考虑使用 Pandas 来处理这些混合类型列。与 PySpark 不同，Pandas 默认将这些列存储为对象（object），并不要求每一行都符合相同的模式。

Simple and straightforward. However, when you tried to run this code, the execution process was terminated because the memory usage exceeded available resources. The job failed.
代码简单直接。然而，当您尝试运行这段代码时，执行过程因内存使用超过可用资源而被终止。任务失败了。

### A Classic Solution: Reducing Peak Memory with Chunk-Based Processing
### 经典解决方案：通过基于分块的处理降低峰值内存

The bottleneck is the 6.2 million rows. The dataset size is approximately 30GB, which is larger than standard cloud worker memory instance. It exceeds the available worker memory during intermediate dataframe transformations. So instead of performing the data type conversions for the entire column, which forces Pandas to materialize large temporary objects in memory, the chunking technique divides each column by chunks. For this case, it’s suitable to set the chunking size as 250,000. So Pandas only needs to proceed 250k rows at one time, then release the memory and move to the next chunk.
瓶颈在于这 620 万行数据。数据集大小约为 30GB，超过了标准云工作节点（worker）的内存实例。在中间数据帧转换过程中，它超出了可用工作内存。因此，与其对整列执行数据类型转换（这会迫使 Pandas 在内存中实例化大型临时对象），不如采用分块技术将每一列划分为多个块。对于本例，将分块大小设置为 250,000 是合适的。这样 Pandas 每次只需处理 25 万行，然后释放内存并移动到下一个块。

After peak memory becomes much smaller, the data transformation completes successfully and the pipeline becomes stable. However, the running time become much longer. This is not surprising, as chunking technique is fundamentally a trade-off. It trades execution speed for pipeline reliability.
在峰值内存大幅降低后，数据转换成功完成，流水线也变得稳定。然而，运行时间变得长得多。这并不奇怪，因为分块技术本质上是一种权衡：它以牺牲执行速度为代价换取了流水线的可靠性。

### From Manual Chunking to Automated Parallel Execution
### 从手动分块到自动并行执行

Besides manual chunking in Pandas, Dask automatically partitions a DataFrame into multiple smaller partitions and resolves the memory crash during data transformation. But its internal execution mechanics is different from chunking. When I set chunk_size in Pandas, it reads one chunk, runs my code on it, drops it from RAM, and then moves to the next. It uses one CPU core at a time, so for the cloud services which provide multiple CPU cores, it didn’t make full use of the capacities. Additionally, I have to manually write a loop to aggregate the results, which makes the code complex and lengthy.
除了 Pandas 中的手动分块外，Dask 可以自动将 DataFrame 分割成多个较小的分区，从而解决数据转换过程中的内存崩溃问题。但其内部执行机制与分块不同。当我在 Pandas 中设置 chunk_size 时，它读取一个块，运行代码，将其从内存中删除，然后移动到下一个。它一次只使用一个 CPU 核心，因此对于提供多个 CPU 核心的云服务来说，它没有充分利用计算能力。此外，我必须手动编写循环来聚合结果，这使得代码复杂且冗长。
