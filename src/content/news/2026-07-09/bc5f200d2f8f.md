# The Threshold Is a Price, Not a Percentage
# 阈值是一个价格，而不是一个百分比

**Agentic AI: The Threshold Is a Price, Not a Percentage**
**智能体 AI：阈值是一个价格，而不是一个百分比**

**How to decide when an AI agent should act on its own by using cost asymmetry instead of a fixed confidence cutoff**
**如何利用成本不对称性而非固定的置信度阈值，来决定 AI 智能体何时应自主行动**

Most agent configs have a line like `ESCALATION_THRESHOLD = 0.90`. Above it, the agent acts. Below it, a human gets pinged. It’s simple. It’s tunable. It lets everyone feel like autonomy is under control. But it’s not the right solution. The fix isn’t a better number. The fix is noticing that the escalation threshold was never a percentage in the first place. It’s a price.
大多数智能体的配置中都有一行类似 `ESCALATION_THRESHOLD = 0.90` 的代码。高于此值，智能体就执行；低于此值，就通知人工。这很简单，也很容易调整，让每个人都觉得自主权在掌控之中。但这不是正确的解决方案。解决办法不是找一个更好的数字，而是意识到升级阈值从一开始就不是一个百分比，而是一个价格。

### Start with the wrong question
### 从错误的问题开始

Most teams ask a capability question. Can the agent write the SQL? Can it issue the refund? If yes, let it run. But being able to do something and deciding to do it alone are different issues. The second is really about the decision itself: What does a mistake cost, and what does it cost to involve a human instead? That’s the real choice. If the agent acts, you risk the cost of a mistake. If you escalate, you pay for a human’s time, whether the agent was right or not. Put numbers on it and the comparison is almost embarrassingly simple.
大多数团队问的是能力问题：智能体能写 SQL 吗？能处理退款吗？如果能，就让它运行。但“能够做某事”和“决定独自做某事”是两码事。后者实际上关乎决策本身：犯错的代价是什么？引入人工的代价又是什么？这才是真正的选择。如果智能体行动，你就要承担犯错的风险；如果你升级处理，无论智能体是否正确，你都要支付人工成本。把这些数字列出来，对比结果简单得令人尴尬。

One assumption first: the human gets escalated tickets right. We’ll relax that later. For now, act alone when:
`(1 - p) * cost_of_error < cost_of_escalation`
where `p` is the probability the agent is correct. Rearranged, escalate whenever:
`p < 1 - (cost_of_escalation / cost_of_error)`
首先做一个假设：人工处理升级后的工单总是正确的。我们稍后再放宽这个假设。目前，当满足以下条件时，应自主行动：
`(1 - p) * 错误成本 < 升级成本`
其中 `p` 是智能体正确的概率。重新排列后，当满足以下条件时应升级：
`p < 1 - (升级成本 / 错误成本)`

That value on the right is the real threshold. Notice what it depends on. Not the model. Not a policy from a workshop. Just a ratio between two costs. If errors are cheap, the threshold is lower and the agent can act more often. If errors are expensive, the threshold is higher, and even a confident agent should ask for help. A fixed cutoff assumes that ratio is the same for every decision the agent will ever face. That’s rare. This approach is based on Chow’s rejection rule from 1970, which is the foundation of today’s learning-to-defer research. Old ideas age well when they’re right.
右边的那个值才是真正的阈值。注意它取决于什么：不是模型，也不是研讨会制定的政策，仅仅是两个成本之间的比率。如果错误代价低，阈值就低，智能体可以更频繁地行动；如果错误代价高，阈值就高，即使是很有把握的智能体也应该寻求帮助。固定的截止点假设智能体面临的每一个决策其比率都是相同的，但这很少见。这种方法基于 1970 年的 Chow 拒绝规则，它是当今“学习推迟”（learning-to-defer）研究的基础。正确的旧思想往往历久弥新。

### Two tickets
### 两个工单

The easiest way to show it: support-ticket triage. An agent reads incoming tickets and either resolves them or routes them to a person. The numbers here are invented for the argument, but they hang together.
展示这一点的最简单方法是：支持工单分流。智能体读取收到的工单，要么解决它们，要么将它们路由给人工。这里的数字是为了论证而虚构的，但它们在逻辑上是自洽的。

A routine refund request. Small amount, clear policy. If the agent botches it, the cleanup is a follow-up ticket and a goodwill credit. Call it £15. Escalating burns about three minutes of a specialist’s time. Call that £4. The threshold comes out at 1 minus 4/15. About 0.73. So if the agent is 90% confident, it should clearly act. Escalating would spend £4 to avoid only £1.50 of expected error. When you say it out loud, no one would make that trade, but a fixed cutoff does it automatically all day.
一个常规退款请求：金额小，政策明确。如果智能体搞砸了，后续处理就是一张跟进工单和一笔商誉补偿，算作 15 英镑。升级处理会消耗专家约三分钟的时间，算作 4 英镑。阈值计算结果为 1 减去 4/15，约为 0.73。因此，如果智能体有 90% 的把握，它显然应该行动。升级处理会花费 4 英镑，却只避免了 1.5 英镑的预期错误成本。说出来没人会做这种交易，但固定的截止点却整天自动执行这种操作。

Now a possible account takeover. A customer reports a password change they didn’t make. If the agent treats this as routine, the downside isn’t a credit. It’s fraud loss, remediation, regulatory exposure, a customer who never comes back. Put a conservative £2,000 on it. Escalation still costs £4. The new threshold is 1 minus 4/2000, or 0.998. In practice, this means you should always escalate. At 90% confidence, letting the agent act alone risks £200 in expected cost, compared to just £4 to escalate. It’s fifty times cheaper to hand it over. Same agent. Same 90%. Opposite correct answers. The 90% rule says act in both cases, and in the second one it quietly burns £196 per ticket to save someone three minutes.
现在看一个可能的账户接管案例：客户报告密码被修改，但并非本人操作。如果智能体将其视为常规处理，后果就不是补偿金那么简单了，而是欺诈损失、补救措施、监管风险以及客户流失。保守估计损失为 2,000 英镑。升级成本仍为 4 英镑。新的阈值是 1 减去 4/2000，即 0.998。实际上，这意味着你应该总是升级。在 90% 的置信度下，让智能体自主行动的预期风险成本为 200 英镑，而升级仅需 4 英镑。移交处理要便宜 50 倍。同样的智能体，同样的 90% 置信度，却有截然不同的正确决策。90% 的规则在两种情况下都要求行动，而在第二种情况下，它为了节省某人三分钟的时间，悄悄浪费了 196 英镑。

*Figure 2: Same 90% confidence, opposite correct decisions.*
*图 2：同样的 90% 置信度，却有截然不同的正确决策。*

That observation usually lands in conversation. One honest wrinkle. The example assumes the agent knows which ticket it’s holding. In real triage, that’s the uncertainty. If the agent is 90% sure a ticket is a routine refund, part of the remaining 10% could be a takeover in disguise. So weight the error cost across what the ticket might be, not what it probably is. When in doubt, price it as the worst plausible class. Cheap-looking tickets are exactly where expensive mistakes hide.
这一观察结果通常会引发讨论。有一个现实的难点：这个例子假设智能体知道它处理的是什么工单。在实际分流中，这正是不确定性所在。如果智能体有 90% 的把握确定工单是常规退款，那么剩余 10% 的可能性中可能隐藏着账户接管。因此，要根据工单“可能是什么”而不是“大概是什么”来权衡错误成本。有疑问时，按最坏的合理类别定价。看起来廉价的工单往往隐藏着昂贵的错误。

### The catch
### 陷阱

All of that arithmetic leans on one assumption. When the agent says 90%, it’s right about 90% of the time. That’s calibration, and it’s not the same thing as confidence. Confidence is a number the model emits. Calibration is whether the number means anything. Feed that into the rule and watch it break. If stated 0.95 really means 0.70, your expected error cost is six times what the arithmetic claims. The threshold you priced so carefully is fiction. You’re not managing risk anymore. You’re laundering it through a number that looks like a probability but isn’t one.
所有这些算术都依赖于一个假设：当智能体说 90% 时，它确实有 90% 的概率是正确的。这就是校准（calibration），它与置信度（confidence）不是一回事。置信度是模型输出的一个数字，而校准是指这个数字是否有实际意义。将其代入规则，你会发现它失效了。如果声称的 0.95 实际上只意味着 0.70，那么你的预期错误成本就是算术计算出的六倍。你精心定价的阈值纯属虚构。你不再是在管理风险，而是在通过一个看起来像概率但实际上并非概率的数字来掩盖风险。

*Figure 3: An overconfident agent. Stated confidence runs ahead of realised accuracy.*
*图 3：一个过度自信的智能体。声称的置信度高于实际准确率。*

So before any of this, check the number. Log every decision with its stated confidence. Bucket by confidence band, per decision class. Compare stated against realised accuracy in each bucket. If the agent says 90% and hits 78%, you’ve found your correction: map one to the other before anything touches the threshold. Isotonic regression or Platt scaling will do the fitting. Two details bite here. Agents are often worst calibrated on the rare, high-stakes class, precisely because it’s rare. That’s the class carrying your extreme threshold, so a global average hides the failure that matters. And the map drifts as ticket mix shifts, so re-measure it on a schedule. The technique matters less than the habit. Until you’ve measured the gap, no threshold has earned your trust.
因此，在做这一切之前，先检查数字。记录每一个决策及其声称的置信度。按置信度区间和决策类别进行分组。比较每个分组中声称的准确率与实际准确率。如果智能体声称 90% 但实际只有 78%，你就找到了修正方法：在触及阈值之前，将两者进行映射。保序回归（Isotonic regression）或普拉特缩放（Platt scaling）可以完成拟合。这里有两个细节需要注意：智能体在罕见的高风险类别上往往校准得最差，恰恰因为它们罕见。而这正是承载你极端阈值的类别，所以全局平均值掩盖了关键的失败。此外，随着工单组合的变化，映射关系会发生偏移，因此要定期重新测量。技术本身不如习惯重要。在你测量出差距之前，没有任何阈值值得你的信任。

### The rule in code
### 代码中的规则

Here it is as a teaching toy. A few lines to make the structure visible, not something I’ve built or shipped.
以下是作为教学演示的代码。几行代码让结构变得清晰，这并非我构建或发布的产品。

```python
def should_act_autonomously(stated_confidence: float, cost_of_error: float, cost_of_escalation: float, calibrate) -> bool:
    """
    Teaching toy: act alone or escalate, by expected cost.
    `calibrate` maps stated confidence to a measured probability of being correct, built from logged data.
    教学演示：根据预期成本决定自主行动还是升级。
    `calibrate` 将声称的置信度映射为基于日志数据测量的正确概率。
    """
    measured_p = calibrate(stated_confidence)
    
    # Expected cost of acting alone: (1 - p) * cost_of_error
    # 自主行动的预期成本：(1 - p) * 错误成本
    expected_error_cost = (1 - measured_p) * cost_of_error
    
    # Act if the cost of error is less than the cost of escalating
    # 如果错误成本小于升级成本，则行动
    return expected_error_cost < cost_of_escalation
```
