# Long Context Isn’t Free — I Built a Safe Prompt-Pruning Layer That Makes LLM Systems Work
# 大语言模型的长上下文并非免费——我构建了一个安全的提示词剪枝层，让 LLM 系统更高效

Your prompt doesn't need to remember everything. It needs to remember what still matters.
你的提示词不需要记住所有事情，它只需要记住那些仍然重要的内容。

Emmimal P Alexander | Jul 11, 2026 | 22 min read
Emmimal P Alexander | 2026年7月11日 | 22分钟阅读

In several long-running LLM systems I’ve worked on, conversation state tends to grow quickly over time. It’s common to resend large portions of the history on each turn—including older tool outputs, repeated RAG retrievals, and context that’s no longer relevant. As this accumulates, prompts can become significantly larger, which may increase inference cost and latency, and in some cases affect reasoning performance.
在我参与的几个长期运行的 LLM 系统中，对话状态往往会随时间迅速增长。在每一轮对话中重复发送大量历史记录是很常见的现象——包括旧的工具输出、重复的 RAG 检索结果以及不再相关的上下文。随着这些内容的堆积，提示词会变得异常庞大，这不仅增加了推理成本和延迟，在某些情况下还会影响推理性能。

I built a deterministic pipeline that prunes this redundant state before the prompt ever reaches the model. The version I implemented avoids LLM calls, embeddings, and external dependencies. Relying strictly on standard library components ensures every pruning decision remains fully deterministic and reproducible.
我构建了一个确定性的流水线，在提示词到达模型之前就剪除这些冗余状态。我实现的这个版本避免了 LLM 调用、嵌入（embeddings）和外部依赖。完全依赖标准库组件确保了每一个剪枝决策都是完全确定且可复现的。

State tracking executes in three distinct passes: Expired Context Elimination, Duplicate Context Elimination, and Dependency Restoration. The third pass is what helps make the first two safer in practice. It ensures that nothing a later message depends on is accidentally removed.
状态跟踪通过三个不同的步骤执行：过期上下文消除、重复上下文消除以及依赖恢复。第三步是使前两步在实践中更安全的关键，它确保了后续消息所依赖的任何内容都不会被意外删除。

While building it, I ran into two bugs that changed the design. My first benchmark corpus used a fixed number of duplicates and stale tool calls, which made reduction percentages shrink as conversations grew. That didn’t reflect what I would expect from real-world behavior. My dependency restoration logic also went completely untested at first because my synthetic data never created a case where a required message was actually removed. Both issues are covered here, along with how fixing them changed the results.
在构建过程中，我遇到了两个改变了设计的 Bug。我的第一个基准测试语料库使用了固定数量的重复项和陈旧的工具调用，这导致随着对话增长，缩减比例反而下降了。这并不符合我对真实世界行为的预期。我的依赖恢复逻辑起初也完全未经测试，因为我的合成数据从未产生过“必需消息被意外删除”的情况。本文涵盖了这两个问题，以及修复它们后结果的变化。

After correcting the pipeline, I benchmarked it across three workloads: plain chat, a RAG assistant, and a tool-heavy agent. Each was tested at five conversation sizes, for a total of 15 configurations on two different machines. Across these runs, all labeled required facts were preserved. The system also reached a stable fixed point after a single pass, which means pruning an already pruned prompt produces no further changes.
在修正流水线后，我在三种工作负载上进行了基准测试：普通聊天、RAG 助手和工具密集型智能体。每种工作负载都在五种对话规模下进行了测试，在两台不同的机器上总共进行了 15 种配置的测试。在这些运行中，所有标记为必需的事实都被保留了下来。该系统还在单次处理后达到了稳定的不动点，这意味着对已经剪枝过的提示词再次剪枝不会产生任何进一步的变化。

Token reduction depends on the workload. It is about 2 to 4 percent for plain chat, 27 to 32 percent for a RAG assistant, and 33 to 34 percent for a tool-heavy agent. Even at 2,000 turns and 131,000 tokens, preprocessing stayed under 50 milliseconds. Full code, all 35 tests, and the raw terminal output are included below so you can run the pipeline yourself.
Token 的缩减比例取决于工作负载。普通聊天约为 2% 到 4%，RAG 助手为 27% 到 32%，工具密集型智能体为 33% 到 34%。即使在 2,000 轮对话和 131,000 个 Token 的情况下，预处理时间也保持在 50 毫秒以下。完整的代码、全部 35 个测试用例以及原始终端输出均附在下方，以便您可以亲自运行该流水线。

### Every prompt I ship keeps getting heavier
### 我发送的每一个提示词都变得越来越沉重

I kept seeing the exact same pattern pop up across every long-running agent I built. A conversation starts out perfectly clean. Fifty turns later, it’s an absolute mess. By that fiftieth turn, the prompt payload you’re shipping out on every single request includes the system prompt, the entire chat history, four tool outputs (two of which are completely stale because the tool ran twice), six retrieved chunks (three are near-duplicates because the user circled back to an old topic), a SQL result from twenty turns ago that’s just sitting there, and a single user preference stated once and never brought up again.
在我构建的每一个长期运行的智能体中，我不断看到完全相同的模式出现。对话开始时非常整洁，五十轮之后就变得一团糟。到了第五十轮，你在每次请求中发送的提示词负载包括：系统提示词、完整的聊天历史、四个工具输出（其中两个因为工具运行了两次而完全陈旧）、六个检索到的片段（其中三个几乎是重复的，因为用户回到了旧话题）、二十轮前的一个 SQL 结果（它就那样躺在那里），以及用户曾经提到过一次但再也没提起过的偏好设置。

None of this stuff is technically wrong. Every single piece made sense the exact moment it was injected. The real issue is that nothing ever gets cleared out. The prompt turns into an append-only log of every historical event, and you’re dumping the whole thing onto the model, every single turn, indefinitely.
从技术上讲，这些内容都没有错。每一条信息在注入的那一刻都是有意义的。真正的问题在于，没有任何东西被清理掉。提示词变成了一个记录所有历史事件的“仅追加”日志，而你却在每一轮对话中无休止地将整个日志倾倒给模型。

That’s not just a storage problem. Reasoning performance measurably degrades as input length grows, even when the added content is irrelevant to the task [1], and models are worse at using information buried in the middle of a long context than information near the edges [2].
这不仅仅是一个存储问题。随着输入长度的增加，推理性能会明显下降，即使添加的内容与任务无关 [1]；而且模型在使用长上下文中间埋藏的信息时，表现远不如使用靠近边缘的信息 [2]。

The immediate knee-jerk reaction when you see this bloating is just basic truncation. Slice the last N messages and drop everything else. It’s literally one line of code. The catch is it silently breaks your conversation chains in ways you won’t even realize until it blows up on a real user.
当你看到这种膨胀时，最直接的本能反应就是简单的截断：切掉最后 N 条消息，丢弃其余所有内容。这确实只需要一行代码。但问题在于，它会以一种你无法察觉的方式悄悄破坏对话链，直到在真实用户身上出现故障时你才会意识到。

*Turn 3: User: My preferred output format is CSV.*
*Turn 4: Assistant: Got it.*
*...*
*Turn 47: User: Export the results.*
*第 3 轮：用户：我偏好的输出格式是 CSV。*
*第 4 轮：助手：明白了。*
*...*
*第 47 轮：用户：导出结果。*

If your window is capped at the last 20 turns, turn 3 disappears by turn 47. The assistant loses the context that the user asked for a CSV. That data was not stale; it was a hard dependency, and simple positional truncation cannot differentiate between old, expendable context and old context that a later turn still relies on. That is the exact design constraint this project addresses. Any mechanism that removes redundant state must distinguish between those two categories. Recency alone is insufficient.
如果你的窗口限制在最后 20 轮，那么到第 47 轮时，第 3 轮的内容就会消失。助手会丢失用户要求 CSV 格式的上下文。那条数据并非陈旧信息，而是一个硬依赖。简单的位置截断无法区分“旧的、可丢弃的上下文”和“后续轮次仍依赖的旧上下文”。这正是本项目旨在解决的设计约束。任何删除冗余状态的机制都必须区分这两类信息。仅靠“最近性”是不够的。

### Borrowing an idea from operating systems
### 借鉴操作系统的思想

Here is the reframe I built the rest of this around: an operating system is constantly deciding which pages stay resident in RAM and which get evicted. A long-running LLM conversation has the exact same problem, except nothing plays the role of the memory manager. Context just accumulates forever because no process owns the job of deciding what stops earning its place in the prompt. The Prompt Pruner in this article is that missing piece.
这是我构建后续内容的核心思路：操作系统不断决定哪些页面保留在内存中，哪些被换出。长期运行的 LLM 对话面临着完全相同的问题，只是没有任何东西扮演“内存管理器”的角色。上下文会永远堆积，因为没有进程负责决定哪些内容不再值得留在提示词中。本文中的“提示词剪枝器”（Prompt Pruner）就是那个缺失的组件。

The multi-pass contextual pruning architecture utilized to eliminate redundant token overhead and resolve structural dependencies prior to LLM compilation.
多轮上下文剪枝架构，用于在 LLM 编译前消除冗余 Token 开销并解决结构性依赖。

Every piece of conversation state (a user turn, an assistant turn, a tool output, a retrieved chunk) is a Message object with a role, a turn number, and a bit of bookkeeping metadata:
每一段对话状态（用户轮次、助手轮次、工具输出、检索片段）都是一个 Message 对象，包含角色、轮次编号以及一些簿记元数据：

```python
@dataclass
class Message:
    id: str
    role: str
    content: str
    turn: int
    tool_call_key: Optional[str] = None
    expires_after_turn: Optional[int] = None
    defines_keys: list = field(default_factory=list)
```

Three passes run over that list in order. Each one is a pure function: messages in, a filtered list out.
三个步骤按顺序对该列表进行处理。每一个步骤都是一个纯函数：输入消息，输出过滤后的列表。
