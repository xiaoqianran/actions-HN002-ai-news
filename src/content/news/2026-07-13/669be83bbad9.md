### 1. Summary for busy engineers
### 1. 忙碌工程师摘要

5.0 productizes client reliability using a classic state pattern behind a stable facade:
5.0 版本通过稳定的外观模式（Facade）背后的经典状态模式，实现了客户端可靠性的产品化：

| Mode | Config reads | Mutations / hooks | Notes |
| :--- | :--- | :--- | :--- |
| **Ready** | Connected to hub | Live in-memory tree | Full Production happy path |
| **Offline** | Disconnected but LKG available | Last Known Good (read-only) | No-op / ignored | Survives hub blips without inventing values |
| **Safe** | Fail-closed | Empty / null-safe | No-op | Diagnostic dumps must not overwrite LKG |

| 模式 | 配置读取 | 变更/钩子 | 备注 |
| :--- | :--- | :--- | :--- |
| **Ready** | 连接至中心 | 实时内存树 | 完整的生产环境正常路径 |
| **Offline** | 断开连接但 LKG 可用 | 最后已知良好状态 (只读) | 无操作/忽略 | 在中心闪断时无需虚构值即可存活 |
| **Safe** | 故障关闭 | 空/空安全 | 无操作 | 诊断转储不得覆盖 LKG |

Public entry remains: `Kiponos kiponos = Kiponos.createForCurrentTeam();`
公共入口保持不变：`Kiponos kiponos = Kiponos.createForCurrentTeam();`

You do not receive mode instances as the API surface. Modes switch internally. Query with:
你不会直接接收模式实例作为 API 接口。模式在内部切换。请使用以下方法查询：
`kiponos.getCurrentMode();`
`kiponos.isReadyMode();`
`kiponos.isOfflineMode();`
`kiponos.isSafeMode();`

### 3. Architecture (state pattern)
### 3. 架构（状态模式）

```text
Application code
│
▼ Kiponos / KiponosBase ◄── stable facade (one reference for app lifetime)
│
▼ volatile SdkState
├── ReadyMode*    → live WebSocket + full Folder ops
├── OfflineMode*  → LKG reads only
└── SafeMode*     → fail-closed + safe diagnostic dump
```

Design rule: never return Ready/Offline/Safe objects to callers. Returning a mode freezes the application on that mode forever. Boot modules stay thin adapters (ReadyModeBoot3, connection controllers) over sdk-common cores.
设计原则：永远不要将 Ready/Offline/Safe 对象返回给调用者。返回模式会将应用程序永久锁定在该模式下。Boot 模块保持为 sdk-common 核心之上的轻量级适配器（ReadyModeBoot3，连接控制器）。

### 4. Folder API (the surface you code against)
### 4. Folder API（你编码所使用的接口）

**Root:** `KiponosFolder root = kiponos.getRootFolder();`
**根目录：** `KiponosFolder root = kiponos.getRootFolder();`

**Navigation:**
`KiponosFolder server = root.path("server", "http");`
// or stepwise:
`KiponosFolder s = root.folder("server").folder("http");`
**导航：**
`KiponosFolder server = root.path("server", "http");`
// 或分步：
`KiponosFolder s = root.folder("server").folder("http");`

**Useful members (conceptual contract):**
`path(String... folders)`, `folder(String folderName)`, `folderOrCreate(String folderName)`, `createFolder / renameFolder / deleteFolder / hasFolder`, `getFolderName / getFolders`, `get(String key)`, `get(String key, String def)`, `getInt(String key)`, `set(String key, String value)`, `hasKey(String keyName)`
**常用成员（概念契约）：**
`path(String... folders)`, `folder(String folderName)`, `folderOrCreate(String folderName)`, `createFolder / renameFolder / deleteFolder / hasFolder`, `getFolderName / getFolders`, `get(String key)`, `get(String key, String def)`, `getInt(String key)`, `set(String key, String value)`, `hasKey(String keyName)`

**Recommended reads:**
```java
// Prefer typed getters on the SDK surface
int port = kiponos.getRootFolder()
    .path("server", "http")
    .getInt("port");

String region = kiponos.getRootFolder()
    .path("deploy")
    .get("region", "eu-west-1");
```
**推荐读取方式：**
```java
// 优先在 SDK 表面使用类型化获取器
int port = kiponos.getRootFolder()
    .path("server", "http")
    .getInt("port");

String region = kiponos.getRootFolder()
    .path("deploy")
    .get("region", "eu-west-1");
```

Avoid `Integer.parseInt(folder.get("port"))` for normal integer keys — use `getInt`.
对于普通的整数键，请避免使用 `Integer.parseInt(folder.get("port"))` —— 请使用 `getInt`。

**Writes (Ready mode):**
`kiponos.getRootFolder().path("feature-flags").set("checkout-v2", "true");`
In Offline/safe modes, mutations are intentionally no-ops (do not assume silent remote write).
**写入（Ready 模式）：**
`kiponos.getRootFolder().path("feature-flags").set("checkout-v2", "true");`
在 Offline/Safe 模式下，变更操作被有意设计为无操作（不要假设会进行静默远程写入）。

### 5. Last Known Good (LKG) Intent
### 5. 最后已知良好状态 (LKG) 意图

When the hub is unreachable, apps should continue with a known good snapshot, not hang and not invent random defaults.
当中心不可达时，应用应继续使用已知的良好快照，而不是挂起或虚构随机的默认值。

**Behavior sketch (5.0):**
| Event | Behavior |
| :--- | :--- |
| Live tree healthy | Ready serves live data; LKG can be refreshed from good state |
| Disconnect / unreachable | Prefer Offline with LKG reads |
| No trustworthy backup | Safe fail-closed |
| Safe diagnostic dump | Writes safe-dump-* style artifacts — never overwrites LKG |

**行为概览 (5.0)：**
| 事件 | 行为 |
| :--- | :--- |
| 实时树健康 | Ready 提供实时数据；LKG 可从良好状态刷新 |
| 断开连接/不可达 | 优先使用 Offline 并读取 LKG |
| 无可信备份 | Safe 故障关闭 |
| 安全诊断转储 | 写入 safe-dump-* 风格的制品 — 绝不覆盖 LKG |

LKG resolution conceptually prefers: `live memory → backup → on-disk LKG file` under the SDK dump location.
LKG 解析在概念上优先顺序为：`实时内存 → 备份 → SDK 转储位置下的磁盘 LKG 文件`。

**App guidance:**
*   Treat Offline as read-only truth of last good.
*   Do not assume writes succeed without Ready.
*   Use mode queries if UI or probes must show “degraded but serving LKG”.
**应用指南：**
*   将 Offline 视为最后良好状态的只读真理。
*   不要假设在非 Ready 状态下写入会成功。
*   如果 UI 或探针必须显示“降级但正在提供 LKG 服务”，请使用模式查询。

### 6. Lifecycle
### 6. 生命周期

```java
Kiponos kiponos = Kiponos.createForCurrentTeam();
try {
    // work
} finally {
    kiponos.disconnect();
}
```
**Notes:** Always disconnect on orderly shutdown. Intentional disconnect can remain in Offline rather than surprising flip back to Ready (verify against your mode transition policy). Reconnect recovery continues to improve.
**注意：** 在有序关闭时务必调用 disconnect。主动断开连接可以保持在 Offline 状态，而不是意外地跳回 Ready（请根据你的模式转换策略进行验证）。重连恢复功能在持续改进中。

### 7. Upgrade from 4.4.x
### 7. 从 4.4.x 升级

| Step | Action |
| :--- | :--- |
| 1 | Change dependency version to 5.0.0.260710 |
| 2 | Rebuild / run tests against staging hub |
| 3 | Replace `parseInt(get(...))` with `getInt` where applicable |
| 4 | Re-test disconnect / reconnect paths (unit + one live smoke) |
| 5 | Optional: surface `getCurrentMode()` in health/admin endpoints |
| 6 | Confirm no code assumed “throw and hang” as the only offline strategy |

| 步骤 | 操作 |
| :--- | :--- |
| 1 | 将依赖版本更改为 5.0.0.260710 |
| 2 | 针对预发布中心重新构建/运行测试 |
| 3 | 在适用处将 `parseInt(get(...))` 替换为 `getInt` |
| 4 | 重新测试断开/重连路径（单元测试 + 一次实时冒烟测试） |
| 5 | 可选：在健康/管理端点中暴露 `getCurrentMode()` |
| 6 | 确认没有代码将“抛出异常并挂起”作为唯一的离线策略 |

Breaking risk: low for pure read apps. Behavioral change is improved offline safety; re-validate anything that depended on exceptions during disconnect.
破坏性风险：对于纯读取应用较低。行为变化在于提高了离线安全性；请重新验证任何依赖于断开连接期间抛出异常的逻辑。

### 8. Example: Spring-friendly service skeleton
### 8. 示例：Spring 友好的服务骨架

```java
import io.kiponos.sdk.Kiponos;
import io.kiponos.sdk.configs.KiponosFolder;
import org.springframework.stereotype.Component;
import javax.annotation.PreDestroy;

@Component
public class CheckoutLimits {
    private final Kiponos kiponos;
    private final KiponosFolder limits;

    public CheckoutLimits() {
        this.kiponos = Kiponos.createForCurrentTeam();
        this.limits = kiponos.getRootFolder().path("checkout", "limits");
    }

    public int maxCartItems() { return limits.getInt("max-cart-items"); }
    public String currency() { return limits.get("currency", "USD"); }

    @PreDestroy
    public void shutdown() { kiponos.disconnect(); }
}
```
Dashboard changes to `checkout/limits/*` propagate live while Ready; Offline continues serving LKG values.
对 `checkout/limits/*` 的仪表板更改在 Ready 状态下实时传播；Offline 状态下继续提供 LKG 值。
