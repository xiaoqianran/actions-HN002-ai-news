“It runs on my own GPU, so it’s basically free.” I’ve said that, you’ve probably said that, and it’s the kind of claim that sounds true until someone measures it. So I measured it. The box is “ardi” — one openSUSE machine with a single RTX 3090 (24 GB). On it I ran a controlled benchmark: three local models served by ollama, each driven under an identical fixed workload, while a dashboard priced every run by the real GPU energy it burned.
“它是在我自己的 GPU 上运行的，所以基本上是免费的。”我说过这句话，你可能也说过。这种说法听起来很有道理，直到有人真正去测量它。所以我进行了测量。测试设备名为“ardi”——一台运行 openSUSE 的机器，配备单张 RTX 3090 (24 GB) 显卡。我在上面运行了一个受控基准测试：通过 ollama 提供三个本地模型，每个模型在完全相同的工作负载下运行，同时通过仪表盘根据 GPU 实际消耗的电能对每次运行进行定价。

Not a thermal-design-power estimate, not a back-of-envelope guess — power-sampled from nvidia-smi every 10 seconds and integrated over each run’s exact start→end window, at my real day/night electricity tariff: 0.30 BGN day / 0.18 BGN night (Bulgarian lev — my actual utility rate; converted to EUR throughout this piece at the fixed ECB peg, 1 BGN ≈ €0.5113). The result is one number per model: euros per million output tokens. Five of eight models I measured came in cheaper than a hosted cloud API. Three didn’t — and it’s not the three you’d guess from parameter count alone.
这不是基于热设计功耗（TDP）的估算，也不是拍脑袋的猜测——而是每 10 秒从 nvidia-smi 采样一次功率，并根据每次运行的精确起止时间进行积分计算。电费采用我实际的昼夜电价：白天 0.30 保加利亚列弗 (BGN) / 夜间 0.18 BGN（这是我实际的公用事业费率；本文统一按欧洲央行固定汇率折算为欧元，1 BGN ≈ 0.5113 欧元）。最终结果是每个模型对应一个数值：每百万输出 Token 的欧元成本。在我测量的八个模型中，有五个比托管的云端 API 更便宜。有三个则不然——而且这三个模型并不是你仅凭参数规模就能猜到的。

The claim I wanted to test: The folk wisdom is that local inference is the cheap option — you bought the GPU, the marginal token is free, the cloud is the thing with the meter running. Every part of that deserves a number, so here’s the one I went after: the marginal energy cost of generating a token locally, model by model, and how it stacks up against what a hosted “Flash”-class API charges for the same token. To make that a measurement and not an argument, I needed three things held constant: the same workload for every model, the same hardware, and a meter I trusted reading the GPU during each run. The third one is the hard part at home — which is where the instrument comes in.
我想验证的观点是：坊间普遍认为本地推理是更便宜的选择——你已经买了 GPU，边际 Token 成本为零，而云端服务才是那个“计费器一直在跑”的东西。这种观点的每一个部分都值得用数据来量化，所以我追求的目标是：逐个模型计算本地生成 Token 的边际能源成本，并将其与托管的“Flash”级 API 收取的同等 Token 费用进行对比。为了让这成为一次测量而非争论，我需要保持三点不变：每个模型的工作负载相同、硬件相同，以及一个我在每次运行期间都能信任的 GPU 功率监测工具。第三点在家庭环境中很难实现——这正是本文所用工具的用武之地。

The method: Three models, picked to span the size range I actually keep on the box: gemma3:1b (tiny), gemma4:26b (Google’s Gemma 4–25.8B params under the hood; ollama just rounds it to “26b” in the tag), and gemma3:27b (the older big one). All three are Q4_K_M-quantized GGUF weights pulled through ollama, so the comparison is like-for-like on quantization. Each got the same fixed workload — a loop of 256-token generations cycling through five fixed prompts — sustained for ~4 minutes (240 s) so the GPU reached a steady state rather than being judged on a cold first call.
方法：选择了三个模型，涵盖了我实际在机器上运行的规模范围：gemma3:1b（微型）、gemma4:26b（底层为 Google Gemma 4–25.8B 参数；ollama 在标签中将其四舍五入为“26b”）以及 gemma3:27b（较旧的大型模型）。这三个模型均采用 Q4_K_M 量化的 GGUF 权重，并通过 ollama 拉取，因此在量化方面具有可比性。每个模型都执行相同的固定工作负载——循环生成 256 个 Token，轮换五个固定提示词——持续约 4 分钟（240 秒），以确保 GPU 达到稳态，而不是仅凭冷启动的第一次调用来评判。

While each model ran, I tracked it as a priced experiment in HomeLab Monitor — my own dashboard, open source, MIT, one container. It samples GPU power from nvidia-smi every 10 s, and for a tracked run it integrates power over the run’s start→end window into kWh, then multiplies by whichever tariff was in effect (day vs night) — in the currency I configured it with, BGN. That converts “what did this model cost?” from a guess into a measured figure; I then convert to EUR for readability. Then the arithmetic that turns a 4-minute run into a comparable rate: € per 1M output tokens = (run_cost_BGN × 0.5113) ÷ (output_tokens / 1,000,000)
在每个模型运行期间，我使用 HomeLab Monitor 对其进行跟踪，将其作为一项定价实验。这是我自己的仪表盘，开源（MIT 协议），仅需一个容器。它每 10 秒从 nvidia-smi 采样一次 GPU 功率，并将运行期间的功率积分转换为 kWh，然后乘以当时生效的电价（白天或夜间），单位是我配置的货币 BGN。这使得“这个模型花了多少钱？”从猜测变成了实测数据；为了便于阅读，我随后将其转换为欧元。接着通过算术将 4 分钟的运行成本转化为可比费率：每百万输出 Token 的欧元成本 = (运行成本 BGN × 0.5113) ÷ (输出 Token 总数 / 1,000,000)。

以下三个模型的生成基准测试是完全可复现的。以下是如何在你自己的机器上搭建该环境的方法。

Step 1 - bring up the monitor (one container)
步骤 1 - 启动监控器（一个容器）

On the machine with the GPU:
在配备 GPU 的机器上：

```bash
COMPOSE_URL=https://raw.githubusercontent.com/SikamikanikoBG/homelab-monitor/main/docker-compose.yml
curl -fsSLO "$COMPOSE_URL"
docker compose up -d
```

Open http://<hub>:9800. The host it runs on shows up immediately — CPU, RAM, disks, and (if it sees one) the GPU with per-model VRAM and live power draw. No config. This is the meter. Check Settings for the currency and tariff it’s configured with before you trust any € figure it shows you.
打开 http://<hub>:9800。运行该服务的宿主机将立即显示——包括 CPU、内存、磁盘，以及（如果检测到）带有各模型显存占用和实时功耗的 GPU 信息。无需配置。这就是测量工具。在信任它显示的任何欧元数值之前，请先检查“设置”中配置的货币和电价。

Step 2 - mint an ingest key
步骤 2 - 生成接入密钥

The benchmark script needs to push runs into the hub, so it needs a key. Settings → Integrations → Create key. The key (an hlm_… string) is shown once — copy it then, you won’t see it again. Keep it in an env var; don’t paste it into the script.
基准测试脚本需要将运行数据推送到中心，因此需要一个密钥。进入“设置”→“集成”→“创建密钥”。密钥（以 hlm_ 开头的字符串）仅显示一次——请立即复制，之后将无法再次查看。将其保存在环境变量中；不要直接粘贴到脚本里。

```bash
export HOMELAB_KEY="hlm_xxxxxxxxxxxxxxxxxxxx"
```

The same panel has a Download client link for the Python file you’ll use in Step 3, and a copy-paste quickstart — so this one screen covers the next two steps.
同一个面板中还有一个用于下载步骤 3 所需 Python 文件的链接，以及一个可直接复制的快速入门指南——因此这一个页面就涵盖了接下来的两个步骤。

Step 3 - grab the tiny client
步骤 3 - 获取轻量级客户端

The hub serves its own Python client. It’s stdlib-only — nothing to pip install:
该中心提供其专属的 Python 客户端。它仅依赖标准库，无需使用 pip 安装：

```bash
CLIENT_URL=http://<hub>:9800/static/homelab_run.py
curl -O "$CLIENT_URL"
```

That one file is the whole tracking API. The pattern is deliberately small: configure once, then wrap any block of work in a run context manager and log metrics inside it.
这一个文件就是完整的跟踪 API。其设计模式刻意保持简洁：配置一次，然后将任何工作块包裹在运行上下文管理器中，并在其中记录指标。

homelab.configure(url="http://<hub>:9800", key="hlm_...")

with homelab.run("llm-cost-gemma3-1b", params={"model": "gemma3:1b", "engine": "ollama", "gpu": "RTX 3090 24GB", "num_predict": 256}) as r:
    # ... do the work ...
    r.log_metric("tokens_per_sec", tps, step=i)

When the with block opens, the hub marks a run start; when it closes, it marks the end and prices the GPU energy burned in between. The window is the measurement.
当 with 代码块打开时，中心会标记运行开始；当它关闭时，会标记结束并计算期间消耗的 GPU 电能成本。这个时间窗口就是测量范围。
