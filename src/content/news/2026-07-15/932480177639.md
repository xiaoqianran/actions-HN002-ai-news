# What Context Does a Coding Agent Actually Need to Act?
## 编程智能体在执行任务时究竟需要什么上下文？

**Abstract:** A modern coding agent can hold an entire repository in its context window. Most of its reading is wasted -- and the interesting question is not how much context an agent can use, but what it actually needs. We study that question at the moment it matters most: when the agent must edit code. Separating finding the work site from acting on it, we hold localization fixed with an oracle, vary only how the code is represented, and score context against real issue resolution on SWE-bench Verified.

**摘要：** 现代编程智能体可以在其上下文窗口中容纳整个代码库。然而，它读取的大部分内容都是浪费的——真正有趣的问题不是智能体能使用多少上下文，而是它实际上需要什么。我们在最关键的时刻研究了这个问题：当智能体必须编辑代码时。我们将“定位工作点”与“执行操作”分离开来，通过预言机（Oracle）固定定位，仅改变代码的表示方式，并在 SWE-bench Verified 基准测试上根据实际问题的解决情况对上下文进行评分。

The answer is starkly minimal. The signal lives in the code being edited itself: natural-language summaries of it answer almost none of the behavioral questions that the source answers ($4/45$ vs. $27/45$, held-out repositories, independent judge), and the gap belongs to the representation, not the summarizer -- a frontier model's summaries score exactly as poorly as a 3B model's. The surrounding context hardly matters either: across every multi-file instance in Verified, under a protocol frozen before any data, rendering a file's remainder as UML skeletons and signatures resolves no more issues than deleting that remainder outright ($N=70$, exact McNemar $p=0.75$). That was our registered hypothesis, and it failed.

答案极其简洁。有效信号存在于被编辑的代码本身：代码的自然语言摘要几乎无法回答源代码所能回答的行为问题（4/45 对比 27/45，基于留存代码库及独立评判），这种差距源于表示方式而非摘要生成器本身——前沿模型的摘要得分与 3B 参数模型一样糟糕。周围的上下文也几乎无关紧要：在 Verified 基准测试的所有多文件实例中，在预先设定的协议下，将文件的其余部分渲染为 UML 骨架和签名，其解决问题的效果并不比直接删除这些部分更好（N=70，精确 McNemar p=0.75）。这正是我们预先注册的假设，但它失败了。

Compressed context, meanwhile, matches whole files at a third of the tokens: a resolved issue costs $19$K context tokens, not $94$K. The instrument also yielded a finding the field should keep: temperature-0 API inference flips $\sim9\%$ of per-instance outcomes between byte-identical runs. That is a noise floor under every small effect reported on this benchmark, including ours. We release the instrument -- gold-validated environments, per-instance proof that every reference edit is expressible from every arm's context, deterministic patch construction, and pre-registered hypotheses whose nulls we publish.

与此同时，压缩后的上下文在仅使用三分之一 Token 的情况下，效果与完整文件相当：解决一个问题仅需 1.9 万个上下文 Token，而非 9.4 万个。该研究工具还产生了一个值得业界关注的发现：在温度参数为 0 的 API 推理中，字节完全相同的运行在约 9% 的实例中会产生不同的结果。这是该基准测试中报告的每一个微小效应下的噪声基准，包括我们自己的研究。我们发布了该研究工具，包括经过黄金标准验证的环境、证明每个参考编辑均可从各实验组上下文中表达的实例证据、确定性的补丁构建过程，以及我们已发布零假设的预注册假设。
