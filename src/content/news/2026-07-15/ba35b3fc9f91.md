在我上一篇关于结构化输出的文章中，我们讨论了从大模型（LLM）获取机器可读响应的三种主要方法：JSON 模式、函数调用（Function Calling）以及 OpenAI 的结构化输出（Structured Outputs）。如果你还没读过那篇文章，建议在阅读本文前先快速浏览一下，因为本文的内容将直接建立在之前的基础之上。

In particular, we need to parse it, validate the data types, access the fields, handle any unexpected values, and so on. And that’s where Pydantic comes in.

今天，我们将更进一步，探讨一种能够改变结构化输出实际体验的技术——Pydantic。更具体地说，虽然 OpenAI 的结构化输出功能保证了模型返回符合模式（Schema）的有效 JSON，但在 Python 端，我们仍然需要对这些 JSON 进行处理。特别是，我们需要解析它、验证数据类型、访问字段、处理意外值等等。而这正是 Pydantic 发挥作用的地方。

### What about Pydantic?
### 什么是 Pydantic？

Pydantic is a Python library for data validation using type annotations. This means that it lets you define the shape and types of your data as a Python class, and then validates that any data you pass in actually conforms to that description. If it doesn’t, Pydantic raises a clear, descriptive error rather than letting bad data silently propagate through your system.

Pydantic 是一个利用类型注解进行数据验证的 Python 库。这意味着你可以将数据的结构和类型定义为一个 Python 类，然后它会验证你传入的任何数据是否真正符合该描述。如果数据不符合，Pydantic 会抛出一个清晰、描述性的错误，而不是让错误的数据在你的系统中悄无声息地传播。

Here’s the simplest possible Pydantic model:
以下是一个最简单的 Pydantic 模型示例：

```python
from pydantic import BaseModel

class PersonInfo(BaseModel):
    name: str
    age: int
    city: str
```

In this way, we now have a schema that enforces that name and city are always going to be a string, whereas age is always an integer. If someone tries to create a PersonInfo with age="thirty-two", Pydantic will catch it immediately and tell us exactly what went wrong.

通过这种方式，我们定义了一个模式，强制要求 `name` 和 `city` 必须是字符串，而 `age` 必须是整数。如果有人尝试用 `age="thirty-two"` 来创建一个 `PersonInfo` 对象，Pydantic 会立即捕获该错误并准确告知我们哪里出了问题。

### But isn’t this just what we were already doing with Function Calling and Structured Outputs?
### 这难道不是我们已经通过函数调用和结构化输出在做的事情吗？

Yes, but with a very important difference. On one side, with JSON Mode or Function Calling, the model may or may not return a response that matches the schema we had in mind. Even when it does get the schema right, it still returns a plain string of JSON, leaving us to manually parse fields, cast types, and validate values on the Python side.

是的，但有一个非常重要的区别。一方面，使用 JSON 模式或函数调用时，模型返回的响应可能符合也可能不符合我们预期的模式。即使模型正确生成了模式，它返回的仍然是一个纯 JSON 字符串，我们仍需在 Python 端手动解析字段、转换类型并验证数值。

On the other side, Structured Outputs improves on this by guaranteeing that the returned JSON always conforms to our defined schema, thanks to constrained decoding at the model level. Nonetheless, it is still just a string of JSON that we need to handle ourselves in Python.

另一方面，结构化输出通过在模型层面进行约束解码，保证了返回的 JSON 始终符合我们定义的模式，从而改进了这一点。尽管如此，它仍然只是一个我们需要在 Python 中自行处理的 JSON 字符串。

With Pydantic, we define our schema as a Python class, and the integration with OpenAI’s API means we get back a proper Python object, not a dictionary, with all fields typed and validated automatically. The JSON Schema that the API needs is generated from our Pydantic model behind the scenes. We never have to write it ourselves.

使用 Pydantic，我们将模式定义为 Python 类，而与 OpenAI API 的集成意味着我们得到的是一个真正的 Python 对象（而不是字典），所有字段都已自动完成类型标注和验证。API 所需的 JSON Schema 是在后台根据我们的 Pydantic 模型自动生成的，我们完全无需手动编写。

In other words, Pydantic is the schema definition layer that sits between our Python code and OpenAI’s API, making the whole structured output experience cleaner, safer, and more maintainable.

换句话说，Pydantic 是位于我们的 Python 代码和 OpenAI API 之间的模式定义层，它使整个结构化输出的体验变得更简洁、更安全、更易于维护。

### Comparison: Before vs. After Pydantic
### 对比：使用 Pydantic 前后

**Before (Manual parsing):**
**之前（手动解析）：**

```python
# ... (omitted code for brevity)
# parse manually — we get back a plain dictionary
result = json.loads(response.choices[0].message.tool_calls[0].function.arguments)
print(result["name"]) # "Maria" — but what if the key is missing? KeyError.
print(result["age"])  # might be "32" instead of 32 — type not guaranteed
```

**After (With Pydantic):**
**之后（使用 Pydantic）：**

```python
# ... (omitted code for brevity)
# we get back a proper Python object, fully typed and validated
result = response.choices[0].message.parsed
print(result.name) # "Maria" — always a string
print(result.age)  # 32 — always an integer, never a string
```

Obviously, the Pydantic version is shorter, more readable, and safer. Notice how we pass our Pydantic model to `response_format`, and the OpenAI SDK, which handles generating the JSON Schema, just makes the API call with `strict: True`, and parses the response back into a proper `PersonInfo` object.

显然，Pydantic 版本更短、更易读且更安全。请注意，我们将 Pydantic 模型传递给 `response_format`，OpenAI SDK 会负责生成 JSON Schema，以 `strict: True` 发起 API 调用，并将响应解析回一个标准的 `PersonInfo` 对象。

In this way, we get type safety, validation, and dot-notation access to fields. On top of this, also notice the `.parse()` method, instead of the usual `.create()`. This is because `.parse()` is a method on the OpenAI SDK specifically designed to work with Pydantic models, handling the full structured output pipeline end to end.

通过这种方式，我们获得了类型安全、数据验证以及通过点号（dot-notation）访问字段的能力。此外，请注意这里使用的是 `.parse()` 方法，而不是通常的 `.create()`。这是因为 `.parse()` 是 OpenAI SDK 中专门为配合 Pydantic 模型而设计的方法，它处理了结构化输出的整个端到端流程。
