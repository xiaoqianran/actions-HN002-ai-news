When engineers talk about context engineering today, they usually mean retrieving the right context from a document: chunk selection, hybrid search, reranking, TOC-aware retrieval. That is one half of the story. The other half is that the question itself is context the LLM will see, and the question deserves the same treatment as the retrieved passage.

图片来源：Steve A Johnson, via Pexels。当今工程师谈论“上下文工程”时，通常指的是从文档中检索正确的上下文：分块选择、混合搜索、重排序、基于目录的检索。这只是故事的一半。另一半是：问题本身也是大模型（LLM）将要看到的上下文，它理应获得与检索到的段落同等的对待。

Take an insurance analyst asking: “What is the maximum coverage amount? Don’t confuse it with the deductible, they’re often listed together.” One string, four signals (a topic, a negative cue, an expected shape, a structural hint), each meant for a different downstream step. Handed verbatim to top-k cosine, none of them goes where it should: the embedding pulls in deductible-bearing lines, generation picks the first plausible number, the trace shows nothing about why. That is not a retrieval failure and not a generation failure. It is a context-engineering failure on the question side: the right typed pieces were never assembled in the first place.

以一位保险分析师的提问为例：“最高承保金额是多少？别把它和免赔额搞混了，它们经常列在一起。”这是一个字符串，却包含了四个信号（主题、负面提示、预期格式、结构提示），每个信号都旨在服务于不同的下游步骤。如果直接将其原封不动地交给 top-k 余弦相似度检索，结果往往不尽如人意：嵌入模型会抓取包含“免赔额”的行，生成模型会挑选第一个看似合理的数字，而追踪记录却无法解释原因。这既不是检索失败，也不是生成失败，而是问题端的上下文工程失败：因为从一开始，就没有构建出正确的类型化片段。

What follows names those pieces, one strategy at a time, and shows what each one is for on the receiving side. ### 1. The parser both reads context and writes context
### 1. 解析器既读取上下文，也写入上下文

Every brick in the pipeline can be read as a typed-context writer. Parsing writes line_df / page_df. Retrieval writes a filtered subset plus an audit. Generation reads them and produces the answer. Question parsing is the interesting case. It is the only brick that is both a consumer of context (it makes an LLM call itself) and a writer of context (its output typed row feeds three downstream calls).

流水线中的每一个基石都可以被视为一个“类型化上下文写入器”。解析过程写入 line_df / page_df；检索过程写入过滤后的子集及审计信息；生成过程读取这些信息并产生答案。问题解析是一个有趣的特例。它是唯一一个既是上下文消费者（它自身会发起 LLM 调用），又是上下文写入器（其输出的类型化行会馈送给三个下游调用）的基石。

Read as a consumer, the parser’s own LLM call is context-engineered by the same rules any other typed LLM call in the pipeline follows. Its context window is assembled from four typed slots: A fixed PARSE_QUESTION_SYSTEM_PROMPT at module level, cached across every question in a session, never rewritten per call. The DocContext compact JSON: one hundred tokens of doc-level facts (doc_type, n_pages, typical_fields) so the parser knows if it is looking at a two-page product sheet or a two-hundred-page policy. A stable few-shot block of (question, ParsedQuestion) pairs (mise en place, one bowl per shape). The raw user question: the only per-call payload. Nothing else goes into that window. No verbatim document text, no retrieval output, no memory. That constraint is deliberate. The parser sits before retrieval, so retrieval output does not exist yet. No memory belongs in a call that has to be reproducible. The context engineering here is minimal by design.

作为消费者，解析器自身的 LLM 调用遵循流水线中任何其他类型化 LLM 调用所遵循的规则进行上下文工程。其上下文窗口由四个类型化槽位组成：模块级别的固定 PARSE_QUESTION_SYSTEM_PROMPT（在会话中的每个问题间缓存，不会针对每次调用重写）；DocContext 紧凑 JSON（包含 100 个 token 的文档级事实，如文档类型、页数、典型字段，以便解析器判断是在处理两页的产品说明书还是两百页的保单）；稳定的少样本（few-shot）块，包含 (问题, ParsedQuestion) 对；以及原始用户问题（这是唯一的单次调用负载）。窗口中不包含其他任何内容：没有文档原文，没有检索输出，没有记忆。这种约束是刻意为之的。解析器位于检索之前，因此检索输出尚不存在。对于必须可复现的调用，不应包含任何记忆。这里的上下文工程设计为最小化。

Read as a writer, the same brick writes one row of question_df plus three derived pieces the assembly stage will thread into three separate downstream calls. Each downstream receiver already knows the field names and reads them directly. No free-form re-parsing on the way in.

作为写入器，同一个基石会写入一行 question_df，以及三个派生片段，组装阶段会将这些片段串联到三个独立的下游调用中。每个下游接收者都已经预知字段名称并直接读取它们，无需在输入时进行自由格式的重新解析。

### 2. Four strategies, four typed pieces
### 2. 四种策略，四个类型化片段

LangChain’s context-engineering taxonomy lists four canonical strategies: write, select, compress, isolate. Read as a writer, the question-parsing brick embodies each of them in a different piece. That is the mapping worth remembering:

### 2.1 Write: ParsedQuestion as a contract
### 2.1 写入：作为契约的 ParsedQuestion

Most RAG pipelines summarise the user question into a rewritten string that keeps the flavour and loses the structure. The context-engineering equivalent is the opposite: write out a typed row with named fields, so every downstream call gets the same shape and can address fields by name. ParsedQuestion is that row. What matters here is not the field list (see Article 6B for it), but that the field names are the contract. A downstream test can assert “if pages_hint is present, the retrieval brief must include it”. A schema linter can flag a field that no downstream call reads.

大多数 RAG 流水线将用户问题总结为一个重写后的字符串，保留了语意却丢失了结构。上下文工程的做法恰恰相反：写出一个带有命名字段的类型化行，这样每个下游调用都能获得相同的格式，并能通过名称访问字段。ParsedQuestion 就是这样一行数据。这里重要的不是字段列表（详见文章 6B），而是字段名称即契约。下游测试可以断言“如果存在 pages_hint，检索摘要必须包含它”。模式检查器（schema linter）可以标记出没有任何下游调用读取的字段。
