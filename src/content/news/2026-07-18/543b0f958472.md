# One RAG Pipeline, Four Very Different PDFs: Same Four Bricks, Every Answer Typed and Cited
# 一个 RAG 流水线，四份截然不同的 PDF：同样的四个模块，实现类型化且带引用的回答

Enterprise Document Intelligence [Vol.1 #9B] – One call wires the four upgraded bricks together, run on a paper, a NIST standard, and a report with a broken TOC.
企业文档智能 [第1卷 #9B] —— 一次调用将四个升级后的模块串联起来，并在论文、NIST 标准以及目录损坏的报告上进行测试。

In the previous article (A production RAG pipeline for PDFs: relational parsing, TOC retrieval, typed answers) we upgraded each of the four bricks: document parsing, question parsing, retrieval, and generation, and wired them into one clean, linear pipeline. A PDF goes in, and a typed, cited answer comes out. That was one paper and one question.
在上一篇文章（《用于 PDF 的生产级 RAG 流水线：关系解析、目录检索、类型化回答》）中，我们升级了四个模块中的每一个：文档解析、问题解析、检索和生成，并将它们串联成一个简洁、线性的流水线。输入一份 PDF，输出一个类型化且带有引用的回答。那只是针对一篇论文和一个问题的测试。

The real test is what happens when you run the same pipeline, without changing a line, on documents that look nothing alike. So that is what we do here: we point the one pipeline at four very different PDF — a research paper, a NIST standard, another paper, and a report whose table of contents is broken, and see how it holds up on each.
真正的考验在于：当你在外观迥异的文档上运行同一条流水线且不修改任何代码时，会发生什么？因此，我们在这里进行了尝试：我们将这条流水线指向四份截然不同的 PDF——一篇研究论文、一份 NIST 标准、另一篇论文，以及一份目录损坏的报告，并观察它在每份文档上的表现。

This article is the second of two parts on the upgraded pipeline in Part III of Enterprise Document Intelligence, a series that builds an enterprise RAG system from four bricks: document parsing, question parsing, retrieval, and generation. The first part, Upgrading a baseline RAG, brick by brick, upgraded each brick on its own. This part wires the four upgraded bricks into a single call and runs it end to end on real documents.
本文是“企业文档智能”系列第三部分中关于升级版流水线的两篇系列文章之二。该系列旨在通过四个模块构建企业级 RAG 系统：文档解析、问题解析、检索和生成。第一部分《逐个模块升级基准 RAG》分别对每个模块进行了升级。本部分则将这四个升级后的模块串联成一个单一调用，并在真实文档上进行端到端运行。

📓 The runnable notebook for this article is on GitHub: doc-intel/notebooks-vol1. It runs the single pdf_qa call on all four documents, prints the typed answer and the per-brick audit trail for each, and lets you drop in your own PDF and watch the same pipeline handle it unchanged.
📓 本文的可运行 Notebook 已上传至 GitHub：doc-intel/notebooks-vol1。它会对所有四份文档运行单一的 `pdf_qa` 调用，打印出类型化的回答以及每个模块的审计追踪，并允许你放入自己的 PDF，观察同一流水线在不作修改的情况下如何处理它。

Part A upgraded each brick on its own: parsing now returns a relational set with a TOC and a typed parsing_summary; question parsing turns the noisy user input into a structured brief; retrieval reads the document’s own TOC through a small LLM and merges those pages with the keyword hits; generation returns a typed answer with one citable span per item plus four context-quality indicators. Built in isolation, the four still have to run as one.
A 部分分别升级了每个模块：解析现在返回一个包含目录和类型化 `parsing_summary` 的关系集；问题解析将嘈杂的用户输入转化为结构化摘要；检索通过小型 LLM 读取文档自身的目录，并将这些页面与关键词命中结果合并；生成返回一个类型化的回答，其中每个条目都有一个可引用的片段，外加四个上下文质量指标。虽然这四个模块是独立构建的，但它们最终必须作为一个整体运行。

This part wires them into a single function, pdf_qa, and reads the result. One call takes a PDF and a question and returns the typed answer plus the full audit trail from question to citation. Two side channels appear only once the bricks run together: parsing_summary feeding both LLM bricks, and a feedback signal from generation back to retrieval.
本部分将它们串联成一个单一函数 `pdf_qa` 并读取结果。一次调用接收一份 PDF 和一个问题，并返回类型化的回答以及从问题到引用的完整审计追踪。当模块协同运行时，会出现两个侧信道：`parsing_summary` 同时为两个 LLM 模块提供信息，以及从生成模块反馈回检索模块的信号。

The running paper stays the same public 15-page arXiv submission, Attention Is All You Need, with the same two-typo question, “What are the options for positional encoding?”. Then the assembled pipeline is run on three more documents that stress different bricks: the NIST Cybersecurity Framework 2.0, the original Retrieval-Augmented Generation paper, and the World Bank Commodity Markets Outlook (April 2024), whose PDF ships a broken table of contents.
测试所用的论文依然是那篇公开的 15 页 arXiv 论文《Attention Is All You Need》，并使用同样带有两个拼写错误的问题：“What are the options for positional encoding?”。随后，组装好的流水线在另外三份文档上运行，这些文档分别侧重于测试不同的模块：NIST 网络安全框架 2.0、原始的 RAG 论文，以及世界银行的《大宗商品市场展望》（2024 年 4 月版），其 PDF 的目录是损坏的。（来源和许可信息列在文末。）

### 1. The pipeline end to end
### 1. 流水线端到端

Article 1 (minimal RAG)’s diagram showed four boxes in a row with one input pill per gap. Enough to introduce the bricks, not enough to capture what the upgraded pipeline actually carries. The version below adds the two side channels that the production code threads.
文章 1（最小化 RAG）的图表展示了四个并排的方框，每个间隙有一个输入胶囊。这足以介绍模块，但不足以体现升级后流水线实际承载的内容。下方的版本增加了生产代码中串联的两个侧信道。

Read the diagram in three passes. The four bricks on the main row. Same names as Article 1 (minimal RAG). Each brick consumes the typed object the previous one emitted (line_df out of parsing, ParsedQuestion out of question parsing, anchor + context out of retrieval) and the contract is the Pydantic schema, not a plain dict. The four bricks stay independent: none of them imports any other.
请分三步阅读该图表。主行上的四个模块与文章 1（最小化 RAG）中的名称相同。每个模块消耗前一个模块输出的类型化对象（解析输出 `line_df`，问题解析输出 `ParsedQuestion`，检索输出 `anchor + context`），且契约是 Pydantic 模式，而非普通的字典。这四个模块保持独立：它们互不引用。

The first side channel: parsing_summary feeding question parsing. Question parsing in Article 1 (minimal RAG) saw only the raw text of the question. In the upgraded pipeline it also receives a compact projection of the doc-level synthesis the parsing brick computed (doc_type, n_pages, typical_fields, summary). The same question “what is the name?” parses very differently on a CV (doc_type=resume, typical_fields=[name, email, ...]) and on a 200-page annual report.
第一个侧信道：`parsing_summary` 为问题解析提供信息。文章 1 中的问题解析仅能看到问题的原始文本。在升级后的流水线中，它还会接收解析模块计算出的文档级综合信息的紧凑投影（文档类型、页数、典型字段、摘要）。同样的问题“名字是什么？”，在简历（文档类型=简历，典型字段=[姓名, 邮箱, ...]）和 200 页的年度报告中，解析结果截然不同。

The channel is dashed because it travels through PromptContext in the user content of the LLM call, not through a kwarg of the brick API. The brick API stays `parse_question(question, *, context=PromptContext(...))` regardless of what doc_type is.
该信道用虚线表示，因为它通过 LLM 调用中用户内容的 `PromptContext` 传输，而不是通过模块 API 的关键字参数。无论文档类型如何，模块 API 始终保持为 `parse_question(question, *, context=PromptContext(...))`。

The second side channel: the feedback loop. Generation does not always succeed in one shot. The LLM may flag “the retrieved context covered only part of the answer” (complete_answer_found=False) or “the document uses ‘excess’ where the question said ‘deductible’” (llm_discovered_keywords). The orchestrator reads those signals and routes back, typically to retrieval with an expanded keyword list, sometimes to question parsing for a query rewrite.
第二个侧信道：反馈循环。生成并不总是一次成功。LLM 可能会标记“检索到的上下文仅涵盖了部分答案”（`complete_answer_found=False`）或“文档中使用的是‘excess’，而问题中提到的是‘deductible’”（`llm_discovered_keywords`）。编排器读取这些信号并进行路由，通常是带着扩展的关键词列表返回检索模块，有时则返回问题解析模块进行查询重写。

Article 13 (the workflow pipeline) walks the loop in detail. This article runs the single-pass version, which is what the typical short paper or compliance document needs.
文章 13（工作流流水线）详细介绍了该循环。本文运行的是单次通过版本，这正是典型的短篇论文或合规文档所需要的。
