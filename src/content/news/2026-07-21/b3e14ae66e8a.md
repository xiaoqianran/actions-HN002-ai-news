# Minimal Git CI using hooks

使用 Git hooks 实现极简 CI

I have a personal server where I am hosting my git repos. Setting up a new git repo is easy: I just run `ssh server git init --bare repo.[1]` That's it. Cloning my newly created repo is as simple as `git clone server:repo`.

我有一台个人服务器用于托管 Git 仓库。创建新仓库很简单：只需运行 `ssh server git init --bare repo.[1]` 仅此而已。克隆新建的仓库同样简单：`git clone server:repo`。

For some projects I've needed a CI, and I looked around for some existing solutions. The CI systems I found are convoluted, yaml-ridden, slow, messy or hard to self-host.

在某些项目中我需要 CI，因此我四处寻找现有解决方案。我发现 CI 系统往往复杂、充满 YAML 配置、缓慢、混乱或难以自托管。

Luckily, I do not need many "modern" CI features like complete isolation for my builds or secret management. I just need something that runs some tests, builds a project or moves some files.

幸运的是，我不需要许多“现代”CI 功能，例如构建的完全隔离或密钥管理。我只需要能运行测试、构建项目或移动文件的东西。

What I have settled on is a simple CI using git hooks. Specifically, I added a remote post-receive hook, which is a hook that runs whenever I push to a project. It is just a shell-script that I add to the hooks directory in a bare git repository.

我最终选择了一种使用 git hooks 的简单 CI。具体来说，我添加了一个远程 post-receive 钩子，该钩子在我推送代码到项目时运行。它只是一个添加到裸 Git 仓库 hooks 目录中的 shell 脚本。

The thing to watch out for with post-receive hooks is if the script fails, the code you push will be rejected. Also, if the script is slow, pushing will take a while. You generally don't want either.

使用 post-receive 钩子需要注意：如果脚本失败，你推送的代码将被拒绝。此外，如果脚本运行缓慢，推送过程会耗时较长。你通常不希望发生这两种情况。

I solved this by using `nq`, a minimal job queue, to queue up my jobs in the background instead.

我通过使用 `nq`（一个极简任务队列）将作业放入后台队列来解决此问题。

The result is quite sleek: my post-receive hook just executes `nq`, and I can access the logs by running `ssh server nqtail -a`. The CI runs instantly, and is very easy to set up, so I am happy with the result.

结果非常简洁：我的 post-receive 钩子只需执行 `nq`，我可以通过运行 `ssh server nqtail -a` 查看日志。CI 立即运行，且设置非常简单，因此我对结果很满意。

如果你想自己设置，我创建了一个简短教程。

There are many ways you could expand the base I provide in the tutorial: you could sandbox the builds (for example with landdown), use podman to isolate builds from the environment, or use sops to manage secrets.

你可以用许多方式扩展教程中提供的基础：可以对构建进行沙盒隔离（例如使用 landdown），使用 podman 将构建与环境隔离，或使用 sops 管理密钥。

For bazaar-style development, receiving git patches via email should be enough. You can also set up cathedral-style development using git-shell or git http-backend.

对于集市式开发，通过电子邮件接收 Git 补丁应该足够了。你还可以使用 git-shell 或 git http-backend 设置大教堂式开发。
