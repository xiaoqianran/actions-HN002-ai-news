# Annoying and alarming things about OpenCode  
# 关于 OpenCode 的恼人之处与危险警报

---

If you don’t know what OpenCode is, imagine a boot stamping on a human face forever. The boot is made of TypeScript and the face is everything we have learned about security and systems software since the invention of the electronic computer in the 1940s. The creators describe it as an AI coding agent. As far as I can tell it’s the most popular open-source coding agent, and it currently has 161k stars on GitHub. I’ve tried out OpenCode with a local LLM. My conclusion is that OpenCode is clown-car turboslop with a security posture of “let me bend over for you daddy”. Everyone using it should stop using it. There are two parts to this post: annoying things and alarming things. The second part is longer. I wrote this post with reference to source code from OpenCode git version baef5cd4. I don’t consider anything in this post to be a security disclosure. OpenCode is fundamentally a web-stack tool for piping llm | bash, and all the issues I describe are in the “pipe” part. The ways it fails are fascinating in the fractal nature of the poor decision making, but the outcome was foregone. I tried to keep discussion of LLM use separate from whether everyone using LLMs should have their machines trivially exploited or accidentally wiped. There is a post-script with some brief thoughts on local LLMs.

如果你不知道 OpenCode 是什么，想象一只靴子永远踩在一张人脸上。这只靴子由 TypeScript 制成，而那张脸是从 1940 年代电子计算机发明以来我们所学的所有关于安全与系统软件的知识。其创建者将其描述为一个 AI 编程代理。据我所知，它是最流行的开源编程代理，目前在 GitHub 上有 16.1 万颗星。我曾用本地 LLM 测试过 OpenCode。我的结论是：OpenCode 是“小丑车 turboslop（混乱产物）”，其安全姿态如同“让我为你弯腰，爸爸”。所有使用它的人都应该停止使用。本文分为两部分：恼人之处与危险警报。第二部分更长。本文参考了 OpenCode 的 git 版本 baef5cd4 的源代码。我不认为本文任何内容属于安全披露。OpenCode 本质上是一个用于 `llm | bash` 管道的 Web 栈工具，我描述的所有问题都出在“管道”部分。它的失败方式因其决策的“分形”性质而令人着迷，但结果早已注定。我试图将 LLM 的使用讨论与“使用 LLM 的人是否应让机器被轻易入侵或意外 wipe”分开。文末附有关于本地 LLM 的简短思考。

**Annoying Things**  
**恼人之处**

Let’s put security to one side for a moment and examine how OpenCode fails as a tool even when it’s not causing you to get your shit popped. There is a kind of Bethesda Effect with OpenCode where it’s impossible to tell what is a bug and what is by design, so I stuck with a description of “annoying”.

暂且把安全放在一边，看看 OpenCode 即使没有导致你的系统被攻破，作为一个工具是如何失败的。OpenCode 有一种“Bethesda 效应”——你无法区分什么是 bug，什么是设计如此，所以我统一用“恼人”来描述。

---

**Prompt Cache Misses**  
**提示缓存未命中**

Most local LLM servers use some variant of the OpenAI /v1/chat/completions API. The idea is: You POST a JSON blob with the entire conversation so far. You get back a stream of SSE events which add up to the response. The upload cost over a session is quadratic, and download is amplified by wrapping tiny deltas in JSON with repeated metadata. Tool calls use the elusive “double JSON encoding” so they can be serialised as multiple JSON-encoded deltas that reassemble into more JSON. The setup has one benefit, which is the server is stateless. As usual, the way you make stateless things fast is: state. The server caches evaluations. When it receives a request, it: Finds the longest matching cached prefix. Evaluates from end of prefix to end of the last posted message (“prefill”). Generates new tokens until it encounters an end-of-sequence token.

大多数本地 LLM 服务器使用 OpenAI `/v1/chat/completions` API 的某种变体。其思路是：你 POST 一个包含至今全部对话的 JSON 数据块。你返回一个 SSE 事件流，这些事件加起来就是响应。整个会话的上传成本是二次方的，而下载成本因将微小增量用重复元数据的 JSON 包裹而被放大。工具调用使用难以捉摸的“双重 JSON 编码”，以便它们能被序列化为多个 JSON 编码的增量，这些增量重新组装成更多 JSON。这种设置有一个好处：服务器是无状态的。像往常一样，让无状态事物变快的方法是：状态。服务器会缓存评估结果。当它收到请求时：找到最长的匹配缓存前缀。从前缀末尾评估到最后发布的消息（“预填充”）。生成新令牌直到遇到序列结束令牌。

I used Qwen3.6-27B dense on an M4 Max, which has decent memory bandwidth (~0.5 TB/s, high for a CPU SoC, low for a GPU). Token generation is usable but it is extremely compute-bound in prefill. If my server can’t find a good matching prefix for the request prompt when I’m deep into the window then I might have to wait 10 minutes of max GPU usage for it to start generating a response. That’s fine, because this should happen rarely. Should.

我在 M4 Max 上使用了 Qwen3.6-27B 密集模型，其内存带宽不错（约 0.5 TB/s，对 CPU SoC 来说很高，对 GPU 来说很低）。令牌生成可用，但预填充阶段极其受计算限制。如果我的服务器在窗口深入时找不到请求提示的良好匹配前缀，我可能得等待 10 分钟的最大 GPU 使用后才能开始生成响应。这没关系，因为这种情况应该很少发生。应该是。

Here are some of the ways OpenCode missed the memo on this one:

以下是 OpenCode 在这方面忽视最佳实践的一些方式：

It globs your filesystem and re-reads AGENTS.md (injected in turn-0 system prompt) on every SSE turn. If you put a quick note in AGENTS.md to be read in the next session, you immediately force a full re-evaluation.

它在每次 SSE 回合中都全局扫描你的文件系统并重新读取 AGENTS.md（在第 0 轮系统提示中注入）。如果你在 AGENTS.md 中放一个快速笔记供下一会话读取，你会立即强制一次完全重新评估。

It prunes context from tool calls on every agent → user transition, invalidating a large part of the prefix. Pruning just discards tool call results more than a fixed distance const PRUNE_PROTECT = 40_000 behind the write head. In the best case you’re taking a 40k context miss, which is equivalent to reading a full-length novel per two or three turns.

它在每次 代理 → 用户 转换时都从工具调用中修剪上下文，使大部分前缀失效。修剪只是丢弃超过固定距离 `const PRUNE_PROTECT = 40_000`（在写头之后）的工具调用结果。在最好的情况下，你承受一个 40k 的上下文未命中，相当于每两三轮读一遍完整小说。

Agent → user transition includes interruption, so if you need to pull the clanker out of a rabbit hole and re-steer it, OpenCode immediately trashes the prompt cache and makes you wait for a response.

代理 → 用户 转换包括中断，所以如果你需要把“clanker（笨拙的代理）”从兔子洞里拉出来并重新引导，OpenCode 会立即销毁提示缓存，让你等待响应。

Personal favourite: it puts the current date in the turn-0 system prompt and re-evaluates every SSE turn. If you’re using OpenCode at midnight you get a full prompt cache miss. I think this tries to remediate the clanker tendency to believe the current date is the date of its training cutoff, and refuse to believe newer things exist. It’s still funny.

我最喜欢的：它把当前日期放在第 0 轮系统提示中，并在每次 SSE 回合都重新评估。如果你在午夜使用 OpenCode，你会得到一个完整的提示缓存未命中。我认为这是试图纠正“clanker”倾向于认为当前日期是其训练截止日期、拒绝相信新事物存在的毛病。这仍然很可笑。

These are the prompt cache misses that fit solely in this category. There are many more; I’ll call them out as we go.

这些是仅属于此类的提示缓存未命中。还有更多；我们继续时会指出它们。

**Pruning**  
**修剪**

I mentioned pruning in the previous section. The prompt cache misses aren’t worth it so I disabled it. The other glaring issue is the lack of protection for early reads. It might not be obvious how completely broken this is, so let’s work through an example.

我在上一节提到了修剪。提示缓存未命中不值得，所以我禁用了它。另一个刺眼的问题是缺乏对早期读取的保护。这可能不明显它有多完全 broken（破损），所以让我们看一个例子。

假设你开始一个新会话，告诉你的“clanker”先读一个规范或实现计划，然后写一些代码：

The spec is read into context. The clanker goes and reads related code, very likely putting it over the fixed 40k pruning threshold. The clanker is ready to implement, but either immediately dives down a dumb rabbit hole, or sits in chain-of-thought dithering about something that is actually very simple or already well-specified. You interrupt the clanker to re-steer it. The interruption causes the entire spec to be deleted from the context window. The clanker writes code without being able to refer to the spec.

规范被读入上下文。“clanker”去读取相关代码，很可能使其超过固定的 40k 修剪阈值。“clanker”准备好实现，但要么立即陷入一个愚蠢的兔子洞，要么在链式思考中犹豫不决，纠结于实际上非常简单或已明确定义的事情。你中断“clanker”以重新引导它。中断导致整个规范从上下文窗口中被删除。“clanker”在无法参考规范的情况下写代码。

Pruning applies equally to all results of all tools except for skill, which is never pruned.

修剪平等地应用于所有工具的结果，除了“skill”（技能），它从不被修剪。

**Compaction**  
**压缩**

Want to sit for 10 minutes while the LLM server prefills the entire session with a new prompt prefixed to it, just to turn it into 5 bullet points that go at the top of a new session? Me neither. I get what they are going for, but I’ve not seen it work well. Neither compaction nor pruning is implemented well, and they interact poorly. If you want to summarise a session then the summarisation prompt should be injected at the end to avoid prefilling the entire session from scratch. The best method I’ve found is just an explicit handoff by telling the clanker to write out notes. It’s ugly but it works better than OpenCode’s compaction mechanism, and creates an on-disk artefact that I can edit or reuse in multiple sessions.

想坐等 10 分钟，让 LLM 服务器用新提示前缀预填充整个会话，只是为了把它变成新会话开头的 5 个要点吗？我也不想。我明白他们的目标，但我没见过它工作得好。压缩和修剪都实现得不好，而且它们交互很差。如果你想总结一个会话，那么总结提示应该在最后注入，以避免从头预填充整个会话。我发现的最好方法只是通过告诉“clanker”写出笔记来进行明确交接。这很丑陋，但比 OpenCode 的压缩机制工作得更好，并创建了一个磁盘产物，我可以编辑或在多个会话中重用。
