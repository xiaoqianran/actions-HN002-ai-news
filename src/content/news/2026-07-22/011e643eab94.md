# AI Didn't Kill Software Engineering. It Made It More Important Than Ever.

# AI 没有杀死软件工程。它让软件工程变得比以往任何时候都更重要。

---

For years we asked the wrong question: "Will AI replace developers?" Today we know that wasn't the right question. The right question is: What becomes valuable when writing code is no longer expensive? This changes software engineering completely. For decades, methodologies like Waterfall, Scrum, XP, DDD, and Clean Architecture were born in a world where writing code was expensive. Documentation aged quickly because rewriting it was costly. Specifications were abandoned because implementing them consumed weeks. Then AI arrived. For the first time in history, producing code became almost free. Value migrated. Code became cheap. Judgment did not.

多年来，我们问错了问题：“AI 会取代开发者吗？”如今我们知道，那不是正确的问题。正确的问题是：当编写代码不再昂贵时，什么变得有价值？这彻底改变了软件工程。几十年来，像瀑布模型、Scrum、极限编程、领域驱动设计和整洁架构等方法论诞生于编写代码成本高昂的世界。文档很快过时，因为重写成本高昂。需求规格被放弃，因为实现它们需要数周时间。然后 AI 出现了。历史上第一次，生产代码变得几乎免费。价值发生了迁移。代码变得便宜。判断力没有。

---

Today any LLM produces hundreds of lines of code in seconds. But it doesn't decide: which problem to solve; what business rules exist; which exceptions matter; which architectural compromises should remain for the next five years. Those remain human responsibilities. The bottleneck shifted. Before, we wrote code. Now we write decisions. Specifications. Architectures. Acceptance criteria. Reviews. The competitive advantage stopped being typing speed. It became clarity of thought.

如今，任何大语言模型都能在几秒钟内生成数百行代码。但它无法决定：解决什么问题；存在哪些业务规则；哪些异常重要；哪些架构权衡应在未来五年内保持。这些仍然是人类的职责。瓶颈发生了转移。以前，我们编写代码。现在我们编写决策。需求规格。架构。验收标准。评审。竞争优势不再是打字速度。而是思维的清晰度。

---

Why "vibe coding" doesn't scale. Conversations are a terrible source of truth. Each prompt increases the context. Each correction adds more tokens. Each interaction forces the AI to reconstruct its intention. At some point, it stops reasoning about the system and starts reasoning about the conversation. This is the true hidden cost of vibe coding.

为什么“氛围编码”无法扩展。对话是糟糕的真相来源。每个提示都会增加上下文。每次修正都会添加更多令牌。每次交互都迫使 AI 重建其意图。在某个时刻，它停止对系统进行推理，转而开始对对话进行推理。这就是氛围编码真正的隐藏成本。

---

Specifications become the project's center. It was this perception that gave birth to Spec Driven Development. The idea is simple: The conversation stops being the project's memory. The specification does. Each feature is born from a spec, evolves into a plan, transforms into tasks, and only then is implemented. The result is predictability. Less rework. Fewer tokens. Fewer ambiguities. [https://books.kodel.com.br/pt-br/livros/sdd/](https://books.kodel.com.br/pt-br/livros/sdd/)

需求规格成为项目的中心。正是这种认知催生了“需求驱动开发”。这个想法很简单：对话不再是项目的记忆。需求规格才是。每个功能都源于一份规格说明，演变为计划，转化为任务，然后才被实现。结果是可预测性。更少的返工。更少的令牌。更少的歧义。[https://books.kodel.com.br/pt-br/livros/sdd/](https://books.kodel.com.br/pt-br/livros/sdd/)

---

But specifications don't solve architecture. Even good specifications fail when the project is chaotic. Architecture exists to answer only one question: Where does this belong? That question gave birth to the second book: FOCUS Architecture. Instead of dozens of layers, the proposal reduces responsibilities to four well-defined pieces: View, Orchestrator, Use Cases, Repository. The goal was never to create a new architecture. It was to reduce the cost of change. In the AI era, that also means reducing context, token consumption, and the risk of unexpected changes. [https://books.kodel.com.br/pt-br/livros/focus/](https://books.kodel.com.br/pt-br/livros/focus/)

但需求规格不能解决架构问题。即使好的需求规格在项目混乱时也会失败。架构的存在只是为了回答一个问题：它属于哪里？这个问题催生了第二本书：《FOCUS 架构》。该提案不是用数十个层，而是将职责简化为四个定义明确的部分：视图、协调器、用例、存储库。目标从来不是创造一种新架构。而是降低变更成本。在 AI 时代，这也意味着减少上下文、令牌消耗和意外变更的风险。[https://books.kodel.com.br/pt-br/livros/focus/](https://books.kodel.com.br/pt-br/livros/focus/)

---

Standing on the shoulders of giants. These books don't reinvent engineering. They synthesize decades of ideas from Martin Fowler, Kent Beck, Eric Evans, Bertrand Meyer, Robert C. Martin, Andrew Hunt, David Thomas, and many others, reinterpreted for a reality where AI agents write much of the code. Two books. One goal. Spec Driven Development organizes thinking. FOCUS Architecture organizes code. One answers what to build. The other answers where to build. Together, they help human teams and AI agents produce predictable, readable, and sustainable software.

站在巨人的肩膀上。这些书没有重新发明工程。它们综合了马丁·福勒、肯特·贝克、埃里克·埃文斯、伯特兰·迈耶、罗伯特·C·马丁、安德鲁·亨特、大卫·托马斯等人数十年的思想，并重新诠释以适应 AI 代理编写大部分代码的现实。两本书。一个目标。需求驱动开发组织思维。FOCUS 架构组织代码。一个回答构建什么。另一个回答在哪里构建。两者共同帮助人类团队和 AI 代理生产可预测、可读且可持续的软件。

---

Final considerations. AI made code cheap. But it made engineering even more valuable. The future doesn't belong to teams with the best AI. It belongs to teams that think better.

最终考量。AI 让代码变得便宜。但它让工程变得更有价值。未来不属于拥有最好 AI 的团队。它属于思考更好的团队。
