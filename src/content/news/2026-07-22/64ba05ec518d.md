# The Silent Vector Contamination Bug: Why Your Concurrent Embeddings Might Be Lying to You  
# 静默向量污染漏洞：为何你的并发嵌入可能正在欺骗你  

---

**TL;DR:** If you run concurrent inference (e.g., via OpenVINO AsyncInferQueue or custom threading) for text/code embeddings, your tests might show 0 exceptions and 0 errors, while silently returning embeddings belonging to other inputs in the batch. Here is how we caught a subtle race condition using a cosine-similarity contamination test.  
**摘要：** 如果你为文本/代码嵌入运行并发推理（例如通过 OpenVINO AsyncInferQueue 或自定义线程），你的测试可能显示 0 异常和 0 错误，却静默返回属于批次中其他输入的嵌入向量。以下是我们如何通过余弦相似度污染测试捕获一个微妙的竞争条件。

## The Setup  
## 环境配置  

We use OpenVINO with an INT8 quantized E5-small model for in-process code embedding. To maximize throughput on multi-core CPUs, we set up an asynchronous inference queue (AsyncInferQueue) with jobs=4. In standard unit testing, everything looked pristine:  
我们使用 OpenVINO 和 INT8 量化的 E5-small 模型进行进程内代码嵌入。为了在多核 CPU 上最大化吞吐量，我们设置了作业数=4 的异步推理队列（AsyncInferQueue）。在标准单元测试中，一切看起来完美无瑕：  

- All infer jobs completed with exit code 0  
  所有推理作业均以退出码 0 完成  
- No None values or zero-filled tensors were returned  
  未返回任何 None 值或零填充张量  
- Latency and throughput were great (~37 chunks/sec on Ryzen 5600)  
  延迟和吞吐量表现优异（在 Ryzen 5600 上约 37 个代码块/秒）  

However, during end-to-end RAG retrieval tests, we noticed weird semantic anomalies: searching for authentication logic would occasionally return chunks related to database migrations or UI components with unreasonably high confidence.  
然而，在端到端 RAG 检索测试中，我们注意到奇怪的语义异常：搜索认证逻辑时，偶尔会返回与数据库迁移或 UI 组件相关的代码块，且置信度异常高。

## The Bug: Silent Contamination  
## 漏洞：静默污染  

The root cause was a subtle race condition in callback/userdata mapping inside the async wrapper. Because the inputs were processed concurrently across multiple execution streams, a shared user-data context wasn't strictly isolated per inference request.  
根本原因在于异步包装器内部回调/userdata 映射中的一个微妙竞争条件。由于输入在多个执行流中并发处理，共享的用户数据上下文未按每个推理请求严格隔离。  

When Request A (auth.py) and Request B (payment.py) were scheduled back-to-back:  
当请求 A（auth.py）和请求 B（payment.py）连续调度时：  

- Both requests succeeded without throwing exceptions  
  两个请求均成功完成，未抛出异常  
- The output tensor for Request A was mapped to the metadata/chunk wrapper of Request B  
  请求 A 的输出张量被映射到请求 B 的元数据/代码块包装器  
- The resulting vector was syntactically valid and non-zero, but it represented the wrong text input  
  生成的向量在语法上有效且非零，但代表了错误的文本输入  
- Standard assertion tests like `assert output_vector is not None` or `assert output_vector.shape == (384,)` passed 100% of the time.  
  标准断言测试（如 `assert output_vector is not None` 或 `assert output_vector.shape == (384,)` 100% 通过。  

The pipeline was silently corrupting the vector store.  
该流水线正在静默破坏向量存储。

## The Code  
## 代码  

**Before fix:** shared results dict across concurrent calls  
**修复前：** 跨并发调用共享结果字典  

```python
self._ov_results = {}
def _callback(request, userdata):
    # BUG: userdata is a global index (0, 1, 2, ...)
    # Two concurrent calls reuse the same indices!
    self._ov_results[userdata] = request.get_tensor().data
```

The problem: `userdata` was a simple integer counter (0, 1, 2, ...) that reset between `embed_batch` calls. When two calls overlapped, they wrote to the same dictionary keys.  
问题：`userdata` 是一个简单的整数计数器（0, 1, 2, ...），在 `embed_batch` 调用之间会重置。当两个调用重叠时，它们会写入相同的字典键。

## The Solution: Cosine Contamination Testing  
## 解决方案：余弦污染测试  

To catch this reliably in CI, we wrote an explicit cross-contamination test designed for concurrent embedding queues.  
为了在 CI 中可靠捕获此问题，我们编写了一个专为并发嵌入队列设计的显式交叉污染测试。  

### The Test Logic  
### 测试逻辑  

```python
import asyncio
import numpy as np
import pytest

def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """Calculate cosine similarity between two vectors."""
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)))

@pytest.mark.asyncio
async def test_async_embedder_no_cross_contamination(embedder):
    """Verify that concurrent embedding doesn't cross-contaminate vectors."""
    # 1. Semantically distinct inputs
    samples = {
        "auth": "def authenticate_user(username, password_hash): return verify_jwt(token)",
        "sql": "SELECT u.id, u.email FROM users u JOIN orders o ON u.id = o.user_id WHERE o.status = 'active'",
        "html": "<div class='flex-container'><span id='user-profile'>Profile View</span></div>",
        "rust": "pub fn allocate_buffer(size: usize) -> Result<Vec<u8>, MemoryError> { Vec::with_capacity(size) }"
    }

    # 2. Sequential baseline (ground truth)
    # IMPORTANT: baseline must use clean synchronous infer(), NOT AsyncInferQueue
    # even with jobs=1, to avoid contamination in the baseline itself
    baseline_vectors = {}
    for key, text in samples.items():
        baseline_vectors[key] = await embedder.embed_single_sync(text)

    # 3. High-concurrency stress test with randomized queue order
    async_tasks = []
    keys_order = list(samples.keys()) * 10  # 40 concurrent requests
    np.random.shuffle(keys_order)
    for key in keys_order:
        async_tasks.append(embedder.embed_async(samples[key]))
    async_results = await asyncio.gather(*async_tasks)

    # 4. Verify identity & cross-isolation via Cosine Similarity
    for expected_key, async_vec in zip(keys_order, async_results):
        # Self-similarity against ground truth must be ~1.0
        self_sim = cosine_similarity(async_vec, baseline_vectors[expected_key])
        assert self_sim > 0.98, (
            f"Contamination detected! Vector for '{expected_key}' drifted "
            f"(sim={self_sim:.2f}, expected >0.98)"
        )
        # Cross-similarity against distinct inputs must remain low
        for other_key, other_vec in baseline_vectors.items():
            if other_key != expected_key:
                cross_sim = cosine_similarity(async_vec, other_vec)
                assert cross_sim < 0.6, (
                    f"Cross-talk detected between '{expected_key}' and "
                    f"'{other_key}' (sim={cross_sim:.2f}, expected <0.6)"
                )
```

## The Fix  
## 修复方案  

The fix was simple once we understood the problem:  
一旦理解问题，修复方案很简单：  

**After fix:** isolated results per call  
**修复后：** 每次调用隔离结果  

```python
self._ov_results = {}
def _callback(request, userdata):
    # FIX: userdata is now (index, local_results_dict)
    # Each embed_batch call creates its own dict
    index, local_dict = userdata
    local_dict[index] = request.get_tensor().data
```

Since OpenVINO's Python API allows passing any Python object as `userdata` (unlike C++ where it's typically `void*`), we can bundle the index and a call-specific dictionary together. Each `embed_batch` call creates its own isolated dictionary. The callback writes to the call-specific dict, not a shared global. No locks needed — complete isolation by design.  
由于 OpenVINO 的 Python API 允许将任何 Python 对象作为 `userdata` 传递（与 C++ 中通常为 `void*` 不同），我们可以将索引和调用特定字典捆绑在一起。每个 `embed_batch` 调用创建自己的隔离字典。回调写入调用特定字典，而非共享全局字典。无需锁——设计上实现完全隔离。

## Key Takeaways  
## 关键要点  

- **0 exceptions ≠ Correctness.** Silent data corruption doesn't throw errors.  
  **0 异常 ≠ 正确性。** 静默数据 corruption 不会抛出错误。  
- **Valid shape ≠ Valid embedding.** A (384,) tensor with non-zero floats can represent the wrong input.  
  **有效形状 ≠ 有效嵌入。** 一个具有非零浮点数的 (384,) 张量可能代表错误的输入。  
- **Write cross-contamination tests.** If you use async inference queues or multi-threading for vector generation, verify that Vector X actually belongs to Input X under concurrent load.  
  **编写交叉污染测试。** 如果你使用异步推理队列或多线程生成向量，请验证在并发负载下向量 X 实际属于输入 X。  
- **Cosine similarity is your friend.** A simple similarity check between concurrent outputs and a sequential baseline can catch subtle mapping errors that all other tests miss.  
  **余弦相似度是你的朋友。** 并发输出与顺序基线之间的简单相似度检查，可以捕获所有其他测试都遗漏的微妙映射错误。
