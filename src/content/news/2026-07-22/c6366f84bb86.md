**Title: Thoughts On Integers (2023)**  
**标题：关于整数类型的思考 (2023)**

---

In this post I’ll lay out my views on integer types, and how I’ve come to those views. Hopefully you’ll disagree initially and be convinced by the end!  
在这篇文章中，我将阐述我对整数类型的看法，以及我是如何形成这些观点的。希望你们一开始会持不同意见，但读到最后会被说服！

---

The status quo  
**现状**

Let’s begin with what I’d characterize as the status quo in popular programming languages today. But first, some caveats: I won’t mention scripting languages throughout this article for obvious reasons. Moreover, I’m leaving Java out of this section because it has some weirdness surrounding integers (int vs Integer and no distinct unsigned types).  
让我们从当今流行编程语言的现状说起。但首先有一些说明：出于明显原因，本文不会提及脚本语言。此外，我将Java排除在本节之外，因为它在整数方面有些奇怪（int与Integer的区别，以及没有独立的无符号类型）。

We’ll start with a classic systems programming language, C. In addition to the traditional C integer types int, char, short, long, long long, etc, C99 added fixed size types with names like int8_t, uint64_t, and so on. On modern 64-bit platforms int is always signed and 32 bits wide. Additionally, C has a confusing array of machine-dependent types such as ptrdiff_t and size_t. In general, subscripts and sizes are of type size_t, but C’s implicit conversions mean you’ll often see int or something else used instead. When signed integers overflow in C, they invoke undefined behavior, meaning the compiler assumes the overflow never occurs. Overflow for unsigned integers, on the other hand, is defined to wrap.  
我们从经典的系统编程语言C开始。除了传统的C整数类型int、char、short、long、long long等，C99还添加了固定大小的类型，如int8_t、uint64_t等。在现代64位平台上，int总是有符号的且为32位宽。此外，C还有一堆令人困惑的机器相关类型，如ptrdiff_t和size_t。通常，下标和大小使用size_t类型，但C的隐式转换意味着你经常会看到使用int或其他类型。在C中，有符号整数溢出会引发未定义行为，这意味着编译器假设溢出永远不会发生。而无符号整数的溢出则被定义为回绕（wrap）。

Next up, C#, which I’d characterize as a typical “application programming” language that’s used when low level control isn’t necessary. C#’s types are named similarly to C’s: int, short, long and byte. int is 32 bits wide. Unsigned versions are obtained by prefixing the type with u. Subscripts and sizes use a plain int, and by default integers wrap on overflow.  
接下来是C#，我将其描述为一种典型的“应用程序编程”语言，在不需要底层控制时使用。C#的类型命名与C类似：int、short、long和byte。int是32位宽。无符号版本通过在类型前加u获得。下标和大小使用普通的int，且默认情况下整数在溢出时回绕。

Go is a contemporary alternative to languages in the same domain as C#. Here we see the beginning of a trend: rather than using unintuitive C-style integer type names, Go just appends the bitwidth of each type: int8, uint64, etc. Another interesting difference crops up in the size of int and uint: they match the size of an address on the target platform, rather than being hardcoded to a particular width. Again, sizes and subscripts use int. Go defines integer overflow to wrap.  
Go是C#同类领域的现代替代语言。这里我们看到一个趋势的开始：Go不使用难以理解的C风格整数类型名，而是直接在类型后附加位宽：int8、uint64等。另一个有趣的差异出现在int和uint的大小上：它们与目标平台的地址大小匹配，而不是硬编码为特定宽度。同样，大小和下标使用int。Go将整数溢出定义为回绕。

Swift is yet another recent language in this “compiled statically-typed non-low-level commercial software” niche, and its integer types are similar to Go in several ways. Type names are identical to Go except for capitalization: Int8, UInt64, etc. Int and UInt have the same sizing behavior as Go, and subscripts & sizes both use Int. Curiously, Swift defaults to panicking on integer overflow, even in release builds.  
Swift是“编译型静态类型非底层商业软件”领域的另一种新语言，其整数类型在多个方面与Go相似。类型名称与Go相同，只是首字母大写：Int8、UInt64等。Int和UInt具有与Go相同的大小行为，下标和大小都使用Int。奇怪的是，Swift默认在整数溢出时触发panic，即使在发布版本中也是如此。

Rust: a better approach  
**Rust：一种更好的方法**

Did you notice a commonality in all the examples I listed above? All those languages have some kind of default int type. They require extra typing to use unsigned types instead of signed ones and explicitly-sized types instead of the default-sized int. I’d say this encourages the use of int over other types. After all: would you rather stare at a screenful of ints or uint32_ts? This goes beyond aesthetics, too. It’s so easy to take the lazy way out and type int without thinking, rather than scrutinize the use-case to identify the best type for the job.  
你是否注意到我上面列举的所有例子都有一个共同点？所有这些语言都有某种默认的int类型。要使用无符号类型而非有符号类型，或使用显式指定大小的类型而非默认大小的int，都需要额外的输入。我认为这鼓励了使用int而非其他类型。毕竟：你更愿意看到满屏的int还是uint32_ts？这还不止是美观问题。很容易偷懒，不假思索地输入int，而不是仔细分析用例以确定最适合的类型。

Rust eliminates each of these points: all integer types have an explicit size, and all types are prefixed with either i or u for signed and unsigned respectively. There’s no bias towards a certain size or signedness. You’re forced to consider the circumstances and make a judgement on what type is most appropriate to the situation. Maybe memory is conserved as integers are narrowed, and bugs are prevented as – to invoke a dogma oft-cited by the Rust community – invalid values (negative numbers) become unrepresentable.  
Rust消除了以上所有问题：所有整数类型都有明确的大小，且所有类型分别以i（有符号）或u（无符号）为前缀。没有对特定大小或符号的偏向。你必须考虑具体情况，判断哪种类型最合适。也许通过缩小整数可以节省内存，并且可以防止bug——借用Rust社区常引用的一条信条——无效值（负数）变得无法表示。

| name   | signed? | size in bits |
|--------|---------|--------------|
| u8     | no      | 8            |
| u16    | no      | 16           |
| u32    | no      | 32           |
| u64    | no      | 64           |
| usize  | no      | address-sized|
| i8     | yes     | 8            |
| i16    | yes     | 16           |
| i32    | yes     | 32           |
| i64    | yes     | 64           |
| isize  | yes     | address-sized|

One thing you might notice about this table is the presence of usize and isize. usize is used for subscripts and sizes, just like size_t in C. Contrary to C, Rust doesn’t have implicit integer casting, so you really do have to use usize for your array indexes (unless you want to drown in as-casts). Using usize rather than a 32-bit int means arrays can have more than two billion elements, and using an unsigned type means negative array indexes can never occur.  
你可能注意到这张表中存在usize和isize。usize用于下标和大小，就像C中的size_t。与C相反，Rust没有隐式整数转换，因此你确实必须对数组索引使用usize（除非你想淹没在as转换中）。使用usize而非32位int意味着数组可以拥有超过二十亿个元素，而且使用无符号类型意味着负数组索引永远不会出现。

Rust panics on overflow in debug builds to catch bugs, but uses wrapping in release builds for performance. Note the lack of UB on overflow.  
Rust在调试版本中遇到溢出会panic以捕获bug，但在发布版本中使用回绕以提高性能。注意溢出时没有未定义行为（UB）。

There is a caveat to what I said about Rust not being biased to a particular integer type:  
关于Rust不对特定整数类型有偏向的说法，有一个注意事项：

```rust
fn demo() {
    let x = 1; // x: i32
}
```

Rust defaults to i32 when it has no type information for an integer literal, so there is an (albeit minor) bias to signed 32-bit integers. In my experience of Rust, though, I found I used signed integers so rarely that I’d almost advocate for making unsigned integers the default. If Rust didn’t put unsigned and signed integers on equal footing I’d likely have never reached this conclusion.  
当Rust对整数字面量没有类型信息时，默认使用i32，因此存在（尽管是轻微的）对32位有符号整数的偏向。然而，根据我在Rust中的经验，我发现我几乎不使用有符号整数，以至于我几乎主张将无符号整数设为默认。如果Rust没有将无符号和有符号整数置于平等地位，我可能永远不会得出这个结论。

To me, all of these changes seem like obvious improvements.  
对我来说，所有这些变化似乎都是显而易见的改进。

A Rust programmer tries other languages  
**一个Rust程序员尝试其他语言**

When I started experimenting with C#, Swift and Go I quickly became frustrated. With my attitude of picking the right integer type for the task, I attempted to use uint for array indexes everywhere instead of the signed int these languages default to. It was impractical due to the amount of casting required, so I gave up in frustration. This just made me more certain: why can’t every language have integer types like Rust?  
当我开始尝试C#、Swift和Go时，我很快就感到沮丧。抱着为任务选择正确整数类型的态度，我试图在所有地方使用uint作为数组索引，而不是这些语言默认的有符号int。由于需要大量的类型转换，这很不切实际，所以我沮丧地放弃了。这反而让我更加确信：为什么不是每种语言都有像Rust这样的整数类型？

Why does C make signed integer overflow UB, anyway?  
**为什么C要使有符号整数溢出成为未定义行为？**

Originally the reason was differences in hardware behavior, but today the argument you’ll hear most often is performance. But don’t most CPUs perform wrapping when an arithmetic operation overflows, anyway? Why not just define overflow to wrap (at no extra cost, since it comes with the CPU “for free”) and be done with it?  
最初的原因是硬件行为的差异，但今天你听到的最常见的论点是性能。但大多数CPU在算术运算溢出时不是都会执行回绕吗？为什么不直接将溢出定义为回绕（没有额外成本，因为这是CPU“免费”提供的）并就此了事呢？

I watched a talk by Chandler Carruth on the topic of UB, in which he includes an enlightening example. (We’ll assume we’re running on an architecture with 64-bit addresses.¹)  
我看了Chandler Carruth关于未定义行为（UB）的演讲，其中他举了一个启发性的例子。（我们假设运行在具有64位地址的架构上。¹）

```c
bool mainGtU(int32_t i1, int32_t i2, uint8_t *block) {
    uint8_t c1, c2;
    c1 = block[i1];
    c2 = block[i2];
    return c1 > c2;
}
```
