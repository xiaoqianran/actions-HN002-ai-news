# GitHub suddenly rejected my SSH key (the fix was a .pub file?!)
# GitHub突然拒绝了我的SSH密钥（修复方法竟是一个.pub文件？）

Today, git pull stopped working on my main laptop. Permission denied (publickey), out of nowhere. I had changed nothing, the key was still registered on GitHub, and pulling the same repository from another laptop (with another key) worked fine.
今天，`git pull` 在我主力笔记本上突然失效了。毫无征兆地出现“Permission denied (publickey)”。我什么都没改，密钥仍注册在GitHub上，而且用另一台笔记本（配了另一把密钥）拉取同一个仓库完全正常。

The rabbit hole I’ll spare you the full troubleshooting session¹. The key was mathematically sound (`openssl rsa -check` said RSA key ok), the signature algorithm was the modern `rsa-sha2-512`, my `~/.ssh/config` was clean, and GitHub’s status page was all green. Still: Permission denied.
我就不详述整个排错过程了¹。密钥在数学上完全有效（`openssl rsa -check` 显示 RSA key ok），签名算法是现代的 `rsa-sha2-512`，我的 `~/.ssh/config` 干净无配置，GitHub 状态页面也全绿。但依然：权限被拒绝。

The fix turned out to be embarrassingly small. My `~/.ssh/github_rsa` has lived without a companion `.pub` file since I re-installed my laptop. Generating one made authentication work again:
修复方法出奇地简单。我的 `~/.ssh/github_rsa` 从重装笔记本后就一直缺少对应的 `.pub` 文件。生成一个后，认证立刻恢复正常：
```bash
$> ssh-keygen -y -f ~/.ssh/github_rsa > ~/.ssh/github_rsa.pub
```

I did not believe it either, so I tested it six times with the `.pub` file and six times without. Twelve out of twelve: no `.pub` – rejected, `.pub` – accepted.
我也不信，于是测试了六次带 `.pub` 文件，六次不带。十二次结果一致：无 `.pub` 被拒绝，有 `.pub` 则通过。

Why on earth would that matter? Because the `.pub` file changes which authentication flow OpenSSH uses. With a `.pub` file present, the client first probes (offers the public key, waits for the server’s OK) and only then signs. With only a private key, OpenSSH skips the probe and sends a fully signed authentication request directly.
这到底为什么重要？因为 `.pub` 文件会改变 OpenSSH 使用的认证流程。存在 `.pub` 文件时，客户端先探测（提供公钥，等待服务器许可），然后才签名。若只有私钥，OpenSSH 会跳过探测，直接发送已完全签名的认证请求。

Both flows are perfectly legal according to RFC 4252, and a stock `sshd` accepts both. GitHub, as of today, apparently does not.
根据 RFC 4252，两种流程都完全合法，标准 `sshd` 两者都接受。但截至今天，GitHub 显然不接受后者。

Did something change on GitHub’s side? My money says yes. The server banner in my debug logs reads `6a2c000`, where GitHub’s SSH 前端 historically identified itself as `babeld-<hash>`. That smells like new server software – one that rejects direct-signed publickey requests.
是 GitHub 那边出了什么变化吗？我打赌是的。我调试日志中的服务器标识显示为 `6a2c000`，而 GitHub 的 SSH 前端历史上一直标识为 `babeld-<hash>`。这闻起来像新服务器软件——一种会拒绝直接签名公钥请求的软件。

It would explain perfectly why everything worked in the morning and nothing worked an hour later, on a machine where nothing changed.
这完美解释了为何早上一切正常，一小时后什么都没改的机器就全部失效。

Has anyone else hit this? Do you know what `6a2c000` is? Let me know – and in the meantime: make sure your private keys have their `.pub` siblings.
还有谁遇到这问题？你知道 `6a2c000` 是什么吗？告诉我——同时记得：确保你的私钥都有对应的 `.pub` 文件。
