**The simple stuff: good reads, writes and schemas**  
基础部分：良好的读取、写入和模式设计  

**Writing a good schema**  
编写良好的模式  

After you’re deployed, schemas are by far the hardest to change moving forward, so it’s worth spending some time on them.  
部署后，模式是未来最难更改的部分，因此值得花时间。  

I’d recommend building your schema iteratively: start with a rough approximation for your tables and primary keys, then write some queries on those tables based on your application needs.  
我建议迭代构建模式：从表和主键的粗略近似开始，然后根据应用需求编写一些查询。  

You can approximate this with some questions: Is this a high-read and/or high-write table? What are the most common filters on reads? Which columns am I updating the most?  
您可以通过一些问题来近似：这是高读取和/或高写入的表吗？读取时最常用的过滤器是什么？我最常更新哪些列？  

If you want to be more formal about it, you can look into database normalization into 1NF/2NF/3NF, but I’ve found normal forms to sometimes be at odds with query efficiency and ease of use, which is critical when you’re moving fast—sometimes it’s just easier to dump data into a jsonb column.  
如果您想更正式一些，可以了解数据库规范化到 1NF/2NF/3NF，但我发现范式有时与查询效率和使用便捷性相冲突，而这对快速推进至关重要——有时只需将数据转储到 jsonb 列中更简单。  

My rules of thumb for schemas are: Use identity columns (auto-incrementing integers, slightly more performant than bigserial) or built-in UUIDs for primary keys Always use timestamptz Always use primary keys Use foreign keys with cascading deletes for low-volume tables, particularly where database consistency and correctness are important. Careful at higher volume.  
我的模式经验法则：对主键使用标识列（自增整数，比 bigserial 性能稍好）或内置 UUID；始终使用 timestamptz；始终使用主键；对低容量表使用具有级联删除的外键，特别是在数据库一致性和正确性重要的地方。高容量时需谨慎。  

**Writing good read queries**  
编写高效的读取查询  

Let’s start with SELECT queries. A useful—albeit slightly inaccurate—mental model for fast selects is: under the hood, Postgres is either going to find a single row in a table very quickly, or it’s going to read every single row in your table using something called a sequential scan 😞.  
让我们从 SELECT 查询开始。一个有用但略微不准确的快速选择心理模型是：在底层，Postgres 要么非常快速地找到表中的单行，要么使用称为顺序扫描的东西读取表中的每一行 😞。  

It’s going to find a single row very quickly when you filter by: An explicit index A unique constraint (just a special case of index) A primary key (these are automatically indexed in Postgres)  
当您通过以下方式过滤时，它会非常快速地找到单行：显式索引、唯一约束（只是索引的特殊情况）、主键（这些在 Postgres 中自动索引）。  

Indexes by default use a btree implementation. It’s most helpful to think of indexes as just another table in Postgres, with data stored in a specific format which is optimized for lookups (more on this later).  
索引默认使用 B 树实现。将索引视为 Postgres 中的另一张表最有帮助，数据以特定格式存储，针对查找进行了优化（稍后详述）。  

These trees are great because finding a single row happens in approximately log(n) time, where n is the number of rows in the table—in other words, really fast.  
这些树很棒，因为查找单行大约在 log(n) 时间内完成，其中 n 是表中的行数——换句话说，非常快。  

When Postgres can’t use an index, it’ll use something called a sequential scan, or seq scan. Seq scans are much slower than index lookups, but modern databases are so fast at loading rows into memory that you probably won’t even notice at first: seq scans on tables with less than 20k rows are pretty much instant.  
当 Postgres 无法使用索引时，它将使用称为顺序扫描或 seq scan 的东西。顺序扫描比索引查找慢得多，但现代数据库将行加载到内存中的速度非常快，以至于您最初可能甚至不会注意到：对少于 20k 行的表进行顺序扫描几乎是瞬时的。  

**Writing performant joins**  
执行高效的连接  

For inner joins, there’s rarely an argument for not using primary keys as the inner join; it usually speaks to a schema design or normalization problem.  
对于内连接，几乎不需要争论不使用主键作为内连接；这通常表明模式设计或规范化问题。  

Treat ON clauses with the same respect as a WHERE clause—the same principles apply. Use an index.  
以与 WHERE 子句相同的尊重对待 ON 子句——相同原则适用。使用索引。

**Writing good write queries**  
编写良好的写入查询  

The premise of successful writes is: Keep transactions short. Don’t go querying an external service in the middle of a transaction unless you have a really good reason to.  
成功写入的前提是：保持事务简短。除非有充分理由，否则不要在事务中间查询外部服务。  

Be careful of the rows you’re locking for writing; in other words, only lock what you need. Every time you update a row, you’re taking out a lock on that row for a short period of time until the transaction commits.  
注意您锁定用于写入的行；换句话说，只锁定需要的行。每次更新行时，您都会在该行上获取一个短时间的锁，直到事务提交。  

As your system gets busier, you’re going to start noticing the impact of locks more. In particular, you might try to create an index at some point in the future with a simple CREATE INDEX concurrently.  
随着系统变得更繁忙，您将开始更多地注意到锁的影响。特别是，您将来可能会尝试使用简单的 CREATE INDEX CONCURRENTLY 创建索引。
