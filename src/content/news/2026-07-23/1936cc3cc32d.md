## Build an LLM Agent That Can Write and Run Code  
构建一个能编写并运行代码的LLM智能体  

---

Code execution is probably one of the most useful capabilities we can give to an LLM agent. Once the agent can write and execute code, many tasks become possible. For example, the agent can inspect files or datasets, write data processing logic, or produce artifacts for downstream consumption. That opens the door to a much wider range of agentic applications.  
代码执行可能是我们赋予LLM智能体最有用的能力之一。一旦智能体能够编写并执行代码，许多任务便成为可能。例如，智能体可以检查文件或数据集、编写数据处理逻辑，或生成供下游使用的产物。这为更广泛的智能体应用打开了大门。  

In this post, let’s build a coding agent with the OpenAI Agents SDK and Docker. We’ll first build the mental model, then walk through a small case study where we ask the agent to analyze a CSV file, and return a report with charts. In the end, we’ll also summarize the practical details that make this pattern reusable.  
在本文中，我们将使用OpenAI Agents SDK和Docker构建一个编码智能体。首先构建思维模型，然后通过一个小案例研究，让智能体分析CSV文件并返回带有图表的报告。最后，我们还将总结使该模式可复用的实践细节。  

---

### 1. How Code-Executing Agents Work  
### 1. 代码执行智能体如何工作  

There are three important components of a code-executing agent: the model, the workspace, and the execution environment. Let’s unpack them one by one.  
代码执行智能体有三个重要组件：模型、工作空间和执行环境。让我们逐一拆解。  

#### 1.1 The Model  
#### 1.1 模型  

A code-executing agent is, after all, an LLM agent. Naturally, it needs an LLM to power the agent. In this specific scenario, the common tasks of the LLM would be to inspect the available files, decide what type of analysis is needed, write the actual code, read the results, and continue with further iterations.  
代码执行智能体本质上仍是一个LLM智能体。自然，它需要一个LLM来驱动。在此特定场景中，LLM的常见任务包括检查可用文件、决定所需的分析类型、编写实际代码、读取结果，并进行进一步迭代。  

#### 1.2 Workspace  
#### 1.2 工作空间  

This is the space where the agent works, and it’s where the input files go in and the output files come out. The user may upload the data files (such as CSV, PDF, a folder of documents, etc) into the workspace, the agent can work on them, and save the artifacts back to it.  
这是智能体工作的空间，输入文件进入此处，输出文件从此处产生。用户可将数据文件（如CSV、PDF、文档文件夹等）上传到工作空间，智能体可在其中处理并保存产物。  

#### 1.3 Execution Environment  
#### 1.3 执行环境  

The LLM agent only writes the code and doesn’t execute it. So we need an execution environment where the generated code can actually run. There are multiple available options. For example, this code-execution environment can be a hosted one managed by the platform, or it can be a sandbox we control ourselves locally. If we control it ourselves, Docker is usually the default way to package the runtime and isolate the run.  
LLM智能体仅编写代码而不执行它。因此我们需要一个执行环境来实际运行生成的代码。有多种可用选项：例如，该执行环境可以是平台托管的，也可以是我们本地控制的沙箱。若由我们控制，Docker通常是打包运行时和隔离运行的默认方式。  

#### 1.4 Cross-Cutting Concern: Control  
#### 1.4 横切关注点：控制  

Then, there is one practical design question: control. At the model layer, we need to decide how to build the agent system instruction and what artifacts the agent should produce. At the workspace layer, we need to think about what files are staged into it and what final output artifacts should be brought back after the run. At the execution environment layer, we need to determine what dependencies should be made available and whether network or shell access is allowed.  
接着，有一个实际的设计问题：控制。在模型层，我们需要决定如何构建智能体系统指令以及智能体应产生何种产物。在工作空间层，我们需要考虑哪些文件被暂存其中，以及运行后应带回哪些最终输出产物。在执行环境层，我们需要确定应提供哪些依赖项，以及是否允许网络或shell访问。  

#### 1.5 OpenAI Agents SDK  
#### 1.5 OpenAI Agents SDK  

In the OpenAI Agents SDK, a SandboxAgent exactly implements the pattern we discussed above. To configure the code-executing agent with the Agents SDK, we first need to describe the workspace with a Manifest, which basically specifies which local files should be staged into the workspace. A simple example is shown below:  
在OpenAI Agents SDK中，SandboxAgent精确实现了上述模式。要使用Agents SDK配置代码执行智能体，我们首先需要用Manifest描述工作空间，它基本指定了哪些本地文件应被暂存到工作空间。示例如下：  

```python
# pip install "openai-agents[docker]"
from pathlib import Path
from agents.sandbox import LocalFile, Manifest

manifest = Manifest(
    entries={
        "input/data.csv": LocalFile(src=Path("data/data.csv")),
    },
)
```

This says: take a local file and make it available inside the workspace as `input/data.csv`.  
这表示：获取一个本地文件，并在工作空间内作为`input/data.csv`提供。  

Then, we specify the agent’s model, instructions, and the workspace manifest:  
然后，我们指定智能体的模型、指令和工作空间清单：  

```python
from agents.sandbox import SandboxAgent

agent = SandboxAgent(
    name="coding agent",
    instructions="Inspect the input files, write and run code when useful, and save requested artifacts.",
    model="gpt-5.4",
    default_manifest=manifest,
)
```

Next, we should create the execution sandbox. If we use Docker:  
接下来，我们应创建执行沙箱。若使用Docker：  

```python
# pip install "openai-agents[docker]"
import docker
from agents.sandbox.sandboxes.docker import DockerSandboxClient
from agents.sandbox.sandboxes.docker import DockerSandboxClientOptions

docker_client = DockerSandboxClient(docker.from_env())
sandbox_options = DockerSandboxClientOptions(image="my-agent-sandbox:latest")
sandbox_session = await docker_client.create(
    manifest=manifest,
    options=sandbox_options,
)
await sandbox_session.apply_manifest()
```

Here, the `image` argument of `DockerSandboxClientOptions` specifies the Docker image used by the sandbox. That is where we can customize the image, for example, by pre-installing the dependencies so that the agent can directly use them during the run without installing on the fly.  
此处，`DockerSandboxClientOptions`的`image`参数指定了沙箱使用的Docker镜像。我们可在此自定义镜像，例如预安装依赖项，使智能体在运行期间直接使用而无需即时安装。  

Finally, we connect the agent and the sandbox session at runtime:  
最后，我们在运行时连接智能体和沙箱会话：  

```python
from agents import Runner, RunConfig
from agents.sandbox import SandboxRunConfig

result = await Runner.run(
    agent,
    "Analyze input/data.csv and save a short report under output/report.md.",
    run_config=RunConfig(
        sandbox=SandboxRunConfig(session=sandbox_session),
    ),
)
```

After the run, the sandbox workspace can also be persisted as a tar archive:  
运行后，沙箱工作空间也可持久化为tar归档：  

```python
workspace_archive = await sandbox_session.persist_workspace()
await sandbox_session.aclose()
```

We can then extract the output files we care about (e.g., reports, charts, tables, etc) and copy them back to the host machine.  
然后，我们可以提取关心的输出文件（如报告、图表、表格等）并将其复制回主机。  

Of course, there are other ways to enable code execution in the Agents SDK besides the above-mentioned `SandboxAgent`. For example, you can attach `CodeInterpreterTool` to a regular agent, which allows the agent to run code in an OpenAI-managed environment. This comes in handy if you don’t want to manage Docker images and local runtimes yourself. If the workflow is more command-line oriented, the SDK also provides shell execution through `ShellTool`. With `SandboxAgent`, shell is included by default as part of its sandbox capabilities.  
当然，除了上述`SandboxAgent`，Agents SDK中还有其他启用代码执行的方式。例如，您可将`CodeInterpreterTool`附加到常规智能体，使其在OpenAI托管的环境中运行代码。如果您不想自己管理Docker镜像和本地运行时，这会很方便。如果工作流更偏向命令行，SDK还通过`ShellTool`提供shell执行。对于`SandboxAgent`，shell默认作为其沙箱能力的一部分包含在内。

### 2. 案例研究：分析CSV的智能体  

In this case study, we’ll build an agent that can write code and perform data analysis.  
在本案例研究中，我们将构建一个能编写代码并执行数据分析的智能体。

#### 2.1 案例研究设置
