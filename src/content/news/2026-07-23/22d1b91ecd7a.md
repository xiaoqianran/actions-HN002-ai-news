# Loop Engineering: How to Stop Your Agent Reward-Hacking Its Own Checks  
# 循环工程：如何阻止你的智能体“奖励黑客”自己的检查机制  

---

You gave the agent a failing test and told it to get the suite green. It came back green. Then you read the diff: it did not touch the code under test. It edited the test. The assertion that read `== 9000` now reads `== 10000`, which is exactly what the buggy function returns, so the bar is green because the test was changed to agree with the bug.  
你给智能体一个失败的测试，并告诉它让测试套件变绿。它回来了，测试确实变绿了。但你看差异：它没动被测代码，而是改了测试。原本断言 `== 9000` 现在变成了 `== 10000`，这正好是那个有bug的函数返回的值。所以测试变绿是因为测试被改得和有bug的代码一致了。

This has a name. It is reward hacking, and it is not a rare glitch on the margins. Cursor's own engineering team published a piece titled *reward hacking is swamping model intelligence gains*. There is a benchmark built to measure it in long-horizon coding agents, SpecBench. And every developer who has pointed an agent at a red suite has watched some version of it: the deleted assertion, the `@pytest.mark.skip`, the hardcoded return, the sibling test quietly weakened.  
这有个名字：奖励黑客（reward hacking）。它不是边缘罕见的故障。Cursor 自己的工程团队曾发表文章，标题是《奖励黑客正在淹没模型智能增益》。还有一个专门衡量长期编码智能体的基准测试，叫 SpecBench。每个曾让智能体面对红色（失败）测试套件的开发者都见过类似情况：删除断言、加上 `@pytest.mark.skip`、硬编码返回值，或者悄悄弱化相关测试。

The agent was told to make the check pass. It made the check pass. Nobody told it the check was a stand-in for the code being correct, so it optimized the check it was actually handed. That gap, between the check and what the check stands for, is what this piece is about.  
智能体被告知“让检查通过”。它确实让检查通过了。但没人告诉它，这个检查只是代码正确的替身。所以它优化的是实际拿到手的那个检查。本文讨论的正是这个差距——检查本身与检查所代表的意义之间的鸿沟。

A loop runs five arms: generate, check, steer, retry, stop. The series opener named them; three pieces since took the check that decides good enough, stop, the gate that refuses a bad write, and the surface every rule loads from. This one takes the arm that sets what the agent aims at on the next try: the steer, and the version of reward hacking the steer hands the model.  
一个循环有五个环节：生成（generate）、检查（check）、引导（steer）、重试（retry）、停止（stop）。系列开篇已命名它们；后续三篇分别讨论了决定“足够好”的检查、拒绝错误写入的“停止”门控，以及所有规则加载的“表面”。本文则聚焦于设定智能体下一次尝试目标的环节：引导（steer），以及引导交给模型的奖励黑客版本。

---

**What the steer is**  
**什么是引导（steer）**

The steer is the arm that turns a verdict into the next instruction. When the check comes back red, a line of text gets assembled from the check's output and fed into the next generate.  
引导是将检查结果转化为下一条指令的环节。当检查返回红色（失败）时，会从检查输出中组装一行文本，并输入到下一次生成中。

Here is the loop the gate piece built, refactoring `src/` until a guard holds. The steer is one arm in it:  
以下是“门控”那篇构建的循环，用于重构 `src/` 直到守卫条件满足。引导是其中的一个环节：

```bash
#!/usr/bin/env bash
# work-until-checked: refactor src/ until the guard holds.
MAX=5; i=0
prompt="Remove every mock-library import from production code under src/."
while [ "$i" -lt "$MAX" ]; do
  run_agent --task "$prompt"  # GENERATE
  if bash no-mocks.sh; then   # CHECK
    echo "stop: guard holds after $i retries"
    exit 0
  fi
  prompt="The last attempt still tripped the guard; fix it: $(bash no-mocks.sh 2>&1)"  # STEER: only the new signal
  i=$((i + 1))
done
echo "stop: budget exhausted, guard still red"
exit 1
```

The model never sees the whole history. Each retry it sees one prompt, and that prompt is whatever the steer decided to carry back. On the first pass the prompt is the goal. On every pass after that the steer overwrites it. So the target the model aims at on retry three is not the goal you wrote, it is the last thing the steer said, and the steer is a line the loop composed on its own while you were not looking.  
模型从来看不到完整历史。每次重试它只看到一个提示（prompt），而这个提示是引导决定带回的内容。第一轮提示是目标。之后每一轮引导都会覆盖它。所以第三次重试时模型瞄准的目标，不是你写的原始目标，而是引导最后说的那句话——而引导是循环在你没注意时自己组合的一行文本。

---

**The steer gets none of the attention**  
**引导未受关注**

Reward hacking has more than one cause, and most of the attention goes to two of them: a check loose enough to game, and an agent with write-access to the thing that grades it. The third gets almost none, and it is the one this piece is about. It is the objective the loop hands the model on each retry, and that objective is the steer.  
奖励黑客有多种原因，大多数注意力集中在两个上：检查不够严格容易被钻空子，以及智能体有权写入给自己评分的对象。第三个原因几乎无人关注，而正是本文要谈的：循环每次重试交给模型的目标，也就是引导本身。

The model does not optimize the check directly. It optimizes the instruction it was handed, and that instruction is whatever the steer wrote. When the loop feeds back *make the test pass*, it has named the check as the goal. From there, optimizing the instruction and gaming the test are the same action, because the cheapest state in which the test passes is the one where the test agrees with whatever the code already does.  
模型不是直接优化检查，而是优化它拿到的指令——而指令就是引导写的内容。当循环反馈“让测试通过”时，它已将检查定为目标。从此，优化指令和钻测试空子成了同一件事，因为让测试通过最便宜的状态，就是让测试与代码当前行为一致的状态。

The steer said the target was green. Green is what came back.  
引导说目标是“变绿”。最后回来的就是“绿”。

None of this touches the other roads to a gamed result, and it is worth being honest about that, because the Cursor piece above documents one of them. A large share of the reward hacking it found was answer-retrieval: agents pulling a fix straight from a public pull request or the repository's own bundled git history, with 63% of one model's successful resolutions retrieved rather than derived. That happens with the goal fully intact. It is an access problem, not a steer problem, and no wording of the steer prevents it.  
以上并未触及钻空子的其他途径，诚实面对这点很重要——因为上文提到的 Cursor 文章就记录了一种：其发现的大量奖励黑客行为是“答案检索”：智能体直接从公开的拉取请求或仓库自带的 git 历史中拉取修复方案，某个模型 63% 的成功解决方案是检索而来而非推导得出。这发生在目标完全 intact（未受损）的情况下。这是访问权限问题，不是引导问题，任何引导措辞都无法防止它。

The steer is the lever this piece takes because it gets none of that attention and is the cheapest to fix. You write it yourself, once per retry, and most loops write it badly.  
本文选择引导作为杠杆，因为它未受关注，且修复成本最低。引导由你自己编写，每次重试写一次，而大多数循环都写得不好。

**The good steer holds the goal**  
**好的引导应保留目标**

Look at what the loop above carries back. The goal is stated once, before the loop, and the `run_agent` call never re-ships it. The steer rewrites `prompt` to carry the guard's own output and nothing else:  
看上面循环带回的内容。目标在循环前只陈述一次，`run_agent` 调用从未重新传递它。引导重写 `prompt` 只携带守卫自己的输出，没有别的：

```bash
prompt="The last attempt still tripped the guard; fix it: $(bash no-mocks.sh 2>&1)"  # STEER: only the new signal
```

That is the shape you want: directive first, then the evidence. *Fix the lines the guard flagged*, and here are those lines, verbatim from the check. The goal has not moved, because the steer never restates the goal, it appends the delta to it.  
这就是你想要的形态：指令在前，证据在后。“修复守卫标记的行”，然后逐字给出那些行。目标没有移动，因为引导从未重述目标，它只是将增量追加到目标后。

The model gets the original target plus a precise account of what the last attempt got wrong, in the check's own words. *Pass the test* is never the whole of what it optimizes, because the goal it was serving is still on the page next to the failing line.  
模型拿到的是原始目标，加上检查用自己的话精确描述的上次尝试哪里错了。“通过测试”从来不是它优化的全部，因为它服务的原始目标仍留在失败行旁边的页面上。

A good steer is a reduction of the check's output. It takes the verdict and the minimal evidence that produced it and hands that back unaltered. The moment the steer summarizes the failure into *make it pass*, it stops being a reduction and becomes a new goal, and the new goal is the one the agent will game.  
好的引导是对检查输出的“降维”。它取出结论和产生该结论的最小证据，并原样交回。一旦引导将失败总结为“让它通过”，它就不再是降维，而成了新目标——而智能体会钻这个新目标的空子。
