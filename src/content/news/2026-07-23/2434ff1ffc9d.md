# A Prisma Schema Change Has Three Lifecycles  
# Prisma 架构变更的三个生命周期  

One valid PostgreSQL row refused to appear on my Next.js page. The relationship was correct, the SQL join returned the row, and the repository compiled. At runtime, however, Prisma returned an undefined `jobOpportunity` relation.  
有一条有效的 PostgreSQL 行拒绝出现在我的 Next.js 页面上。关系是正确的，SQL 连接返回了该行，代码库也编译成功。但在运行时，Prisma 返回了一个未定义的 `jobOpportunity` 关联。  

The stale layer was not the database—I had changed the Prisma schema while the development server was running. That change involved three independent lifecycles: `prisma migrate` updates the database. `prisma generate` updates the TypeScript client. Restarting Next.js replaces the stale generated client held by the running process.  
过时的层并非数据库——我是在开发服务器运行时更改了 Prisma 架构。该变更涉及三个独立的生命周期：`prisma migrate` 更新数据库；`prisma generate` 更新 TypeScript 客户端；重启 Next.js 会替换运行进程持有的过时生成客户端。  

The first two steps were complete. The third was not.  
前两步已完成。第三步没有。  

Coming from PHP and CodeIgniter, this is an important mental-model shift. A request-scoped PHP application normally loads changed code on the next request. A long-running Node.js process can keep generated code and development caches alive.  
从 PHP 和 CodeIgniter 转过来，这是一个重要的思维模式转变。请求范围的 PHP 应用程序通常会在下一个请求时加载更改的代码。而长期运行的 Node.js 进程可能使生成的代码和开发缓存保持活动状态。  

Prisma provides valuable type safety, but it does not remove complexity—it relocates it.  
Prisma 提供了宝贵的类型安全性，但它并未消除复杂性——它只是转移了复杂性。  

My new debugging rule is simple: Application edit → trust hot reload. Prisma model change → migrate, generate, restart. Suspicious stale behavior → clear the framework cache and restart. Before rewriting a correct query, identify which layer is stale.  
我的新调试规则很简单：应用程序编辑 → 信任热重载。Prisma 模型变更 → 迁移、生成、重启。可疑的过时行为 → 清除框架缓存并重启。在重写正确的查询之前，先确定哪一层已过时。
