# Unicode Variation Selector-15 and some of my tears  
# Unicode 变体选择器-15 与我的几滴眼泪  

July 13, 2026  
2026年7月13日  

From my phone, I noticed that my website was displaying footnote backlink glyphs as emoji in my RSS reader. It had not always been this way. I opened the same page from a desktop browser and saw what I expected to see: a text-like pictograph, in my article body’s font face. No emoji to be found. I checked the HTML document source:  
在我手机上，我注意到我的网站在 RSS 阅读器中把脚注反向链接符号显示成了表情符号。但以前并非如此。我用桌面浏览器打开同一页面，看到了预期的样子：一个类似文本的象形文字，使用文章正文字体。没有表情符号。我检查了 HTML 文档源码：  

```html
<a href="#fnref:1" class="reversefootnote" role="doc-backlink"> &#8617; </a>
```  

The RSS feed XML contained the same. So, that can be presented as either text or an emoji? &x8617; is an HTML representation of the Unicode codepoint U+21A9, Leftwards Arrow with Hook. This led me to many relevant threads, pastes, and blog posts¹ in which others had observed similar issues with Leftwards Arrow with Hook and other, similar navigation-decoration characters. We were all reacting to U+21A9 as rendered without a Unicode standardized variation sequence specified. So it’s up to whatever program is rendering U+21A9 to decide what to do. Which version, the text or the emoji, should U+21A9 render.  
RSS 订阅源的 XML 中也包含相同内容。所以，这个符号既可以显示为文本，也可以显示为表情符号？`&#x8617;` 是 Unicode 码点 U+21A9（左钩箭头）的 HTML 表示。这让我找到了许多相关讨论、粘贴内容和博客文章¹，其中其他人也观察到左钩箭头及其他类似导航装饰字符存在类似问题。我们都在应对 U+21A9 在没有指定 Unicode 标准化变体序列情况下的渲染结果。因此，完全由渲染 U+21A9 的程序决定如何显示。U+21A9 应该渲染为文本版本还是表情符号版本？  

I found the correct solution and some less correct solutions to this issue. Before I describe them, I want to explore a slightly larger problem introduced by variation selectors.  
我找到了这个问题的正确解决方案和一些不太正确的方案。在描述它们之前，我想探讨一下变体选择器带来的一个稍大的问题。  

### The downstream effects  
### 下游影响  

For meta-ish content characters like backlink glyphs, the way forward is clear for people who are technical: fix and/or configure the thing that builds your web pages. Or complain in the direction of the people who can fix and/or configure the thing.  
对于像反向链接符号这类元内容字符，技术人员的前进方向很明确：修复和/或配置生成网页的工具。或者向能修复和/或配置该工具的人投诉。  

For many of the other characters, it’s more complicated. Many of these characters are used in body content to be expressive. Rendering the text presentation instead of the emoji presentation, at worst, could be a mistranslation.  
对于其他许多字符，情况更复杂。这些字符常被用于正文内容中以表达情感。在最坏情况下，渲染文本形式而非表情符号形式可能导致误译。  

The technical standard (see the emoji-data links, and specifically emoji-data.txt) lists over 1,000 Unicode codepoints that have both text and emoji presentations.  
技术标准（参见 emoji-data 链接，特别是 emoji-data.txt）列出了超过 1000 个同时具有文本和表情符号形式的 Unicode 码点。

**表1** 1000 多个同时具有文本和表情符号形式的 Unicode 码点的小样本。“字形”列可能因您阅读本文的应用程序而异。我希望 +U+FE0E 和 +U+FE0F 列对所有人渲染一致。  

There is no obvious, cut-and-dried user experience for writers here. Pictographs and emoji are regular text as far as most programs are concerned, and variation selectors remain invisible unless you go looking. Did the author mean to present an umbrella ⛱︎ or an umbrella ⛱️ ? We don’t know what their writing tool presented to them during editing. They don’t know that we don’t see what they think we see when we’re reading the thing.  
这对作者而言没有明显、直截了当的用户体验。在大多数程序看来，象形文字和表情符号就是普通文本，而变体选择器除非特意查看，否则不可见。作者是想展示雨伞 ⛱︎ 还是雨伞 ⛱️ ？我们不知道他们在编辑时写作工具展示的是什么。他们也不知道我们阅读时看到的并非他们以为我们看到的内容。  

In a perfect world, all software knows about and inserts variation selectors for the user. In reality, we know that software is bad. I found documentation that indicates that the Unicode Technical Committee is aware that variation selectors are complicated², and maybe weren’t the best strategy³. But what’s done is done and can’t be undone when it comes to the Unicode Standard.  
在理想世界中，所有软件都了解并为用户插入变体选择器。现实中，我们知道软件很糟糕。我找到的文档表明，Unicode 技术委员会意识到变体选择器很复杂²，可能并非最佳策略³。但木已成舟，Unicode 标准无法更改。  

### The most correct solution  
### 最正确的解决方案  

You can specify, in your source document, which version you want rendered. You do this by combining the variation selector codepoint with the preceding codepoint you want to render. In my case, that’s U+FE0E, Variation Selector-15.  
您可以在源文档中指定希望渲染的版本。方法是将变体选择器码点与您要渲染的前置码点结合。在我的例子中，就是 U+FE0E，变体选择器-15。  

Using my source document’s backlink HTML markup as an example:  
以我源文档中的反向链接 HTML 标记为例：  

```html
<a href="#fnref:1" class="reversefootnote" role="doc-backlink"> &#8617;&#xFE0E; </a>
```  

And, if I wanted, I could do the opposite: force the emoji presentation, using U+FE0F, Variation Selector-16.  
而且，如果我愿意，可以反过来：使用 U+FE0F（变体选择器-16）强制使用表情符号形式。  

** ↩︎ Fig. 1** The pictogram via U+21A9 U+FE0E.  
** ↩︎ 图1** 通过 U+21A9 U+FE0E 显示的象形图。  

** ↩ ↩️ Fig. 2** The emoji via U+21A9 alone, next to the emoji rendered via U+21A9 U+FE0F, where U+FE0F, Variation Selector-16 coerces the glyph to use its full-colour emoji variant.  
** ↩ ↩️ 图2** 仅用 U+21A9 显示的表情符号，旁边是通过 U+21A9 U+FE0F 渲染的表情符号，其中 U+FE0F（变体选择器-16）强制字形使用其全彩色表情符号变体。  

** Fig. 3** Here's the image of the pictogram version and the iOS emoji version as rendered via U+21A9 alone. As I've already mentioned, which variant you see sans-selector can differ per application.  
** 图3** 这是仅通过 U+21A9 渲染的象形图版本和 iOS 表情符号版本的图像。如前所述，不带选择器时您看到的变体可能因应用程序而异。  

So, I can fix this issue for myself. I’m fortunate that I build the software that generates this website; and I’m fortunate that the library I use to generate HTML from my Markdown source documents (Kramdown) has a configurable footnote backlink glyph.  
所以，我可以为自己解决这个问题。我很幸运，因为我构建了生成此网站的软件；也很幸运，因为我用于从 Markdown 源文档生成 HTML 的库（Kramdown）具有可配置的脚注反向链接符号。  

There are authors using the Mistune library (for example, via the Bear blogging platform)⁴, or perhaps some other niche content rendering library, that don’t have the same luxury as of this writing.  
有些作者使用 Mistune 库（例如，通过 Bear 博客平台）⁴，或可能使用其他小众内容渲染库，截至本文撰写时，他们尚无此便利。  

### Some less reliable solutions  
### 一些不太可靠的解决方案  

If you’re in a position where you don’t control the software that builds HTML from your source documents, and, for some reason, it’s not possible to insert a variation selector character where you want it, there are some other things you can try. (At least for applications that use CSS for document presentation, like web browsers.)  
如果您无法控制从源文档构建 HTML 的软件，且由于某种原因无法在所需位置插入变体选择器字符，您可以尝试其他方法。（至少对于使用 CSS 呈现文档的应用程序，如网页浏览器。）  

**font-variant-emoji**  
**font-variant-emoji**  

The `font-variant-emoji` CSS property is not currently supported in Safari. Hopefully it will be soon. It allows you to specify the preferred presentation of unicode characters for whichever DOM elements you like:  
`font-variant-emoji` CSS 属性目前 Safari 不支持。希望很快会支持。它允许您为任意 DOM 元素指定 Unicode 字符的首选呈现形式：  

```html
<article style="font-variant-emoji: text;">
  <span>Umbrella &#x26F1;</span>, or
  <span style="font-variant-emoji: emoji;">umbrella &#x26F1;</span>.
</article>
```  

But I don’t think this can ever be a foolproof solution. Even wide web browser support doesn’t fully account for web content being presented by non–web browser applications like feed readers, and so on.  
但我认为这永远不可能是一个万无一失的解决方案。即使广泛的网页浏览器支持，也无法完全涵盖由非网页浏览器应用程序（如订阅阅读器等）呈现的网页内容。  

**font-family: monospace**  
**font-family: monospace**  

If you want an even less reliable solution, setting the `font-family` CSS property to the value `monospace` sometimes works:  
如果您想要一个更不可靠的解决方案，将 `font-family` CSS 属性设置为 `monospace` 有时有效：  

```html
<article>
  <span style="font-family: monospace;">Umbrella &#x26F1;</span>, or
  <span>umbrella &#x26F1;</span>.
</article>
```
