# MergeForge: Resolve Git Conflicts in VS Code or Cursor Like in JetBrains  
**MergeForge：在 VS Code 或 Cursor 中实现 JetBrains 风格的 Git 冲突解决**

---

Tired of squinting at VS Code’s stacked merge editor? MergeForge brings a JetBrains-style three-pane conflict resolver to VS Code and Cursor — and pairs it with an AI assistant that actually reads your repository before it suggests a fix.  
厌倦了在 VS Code 堆叠式合并编辑器中费力分辨？MergeForge 将 JetBrains 风格的三窗格冲突解决器带到了 VS Code 和 Cursor——并配备了一个真正在提出修复建议前会阅读你代码库的 AI 助手。

---

The problem  
**问题所在**

We’ve all been there. You’re halfway through a rebase. Git stops. Twelve files are conflicted. You open one in VS Code… and get that familiar stacked layout: Incoming, Current, and a result pane that somehow still feels like a puzzle with half the pieces missing.  
我们都经历过这种情况：你正在 halfway through a rebase（变基中途），Git 突然停止。十二个文件发生冲突。你在 VS Code 中打开其中一个……然后看到熟悉的堆叠布局：传入（Incoming）、当前（Current）和一个结果窗格——它仍然感觉像少了 half the pieces（一半拼图）的谜题。

If you ever used WebStorm or IntelliJ, you know how good merge tools can feel: Your side on the left, Their side on the right, The result in the middle, Gutter arrows that just… work.  
如果你用过 WebStorm 或 IntelliJ，就知道合并工具能有多流畅：你的版本在左，他们的版本在右，结果在中间，边栏箭头操作起来行云流水。

In VS Code land, that flow never quite arrived. You click Accept Current, Accept Incoming, Accept Both, and hope nothing important got flattened. Word-level diffs? Authorship? A clear “who wrote this chunk?” signal? Often missing when you need them most.  
但在 VS Code 生态中，这种流畅感从未真正到来。你点击“接受当前”、“接受传入”、“接受两者”，然后祈祷没有重要内容被覆盖。词级差异？作者归属？明确的“这块代码谁写的？”提示？这些在你最需要时往往付之阙如。

And when AI entered the chat, a lot of tools treated conflicts like isolated text blobs: “Here are the <<<<<<< markers. Good luck.” But real merges need context. What was the branch trying to do? What does the surrounding file look like? Who touched this last? Without that, “AI resolve” is just confident guessing.  
而当 AI 加入战场后，许多工具将冲突视为孤立的文本块：“这是 <<<<<<< 标记。祝你好运。”但真正的合并需要上下文：这个分支想做什么？周围文件是什么样子？最后修改的人是谁？缺乏这些，“AI 解决”只是自信的猜测。

I wanted the JetBrains merge experience — inside VS Code and Cursor — with an assistant that behaves more like a careful teammate than a slot machine. So I built it.  
我渴望在 VS Code 和 Cursor 中获得 JetBrains 的合并体验——搭配一个更像谨慎队友而非老虎机的助手。于是我构建了它。

The solution: MergeForge  
**解决方案：MergeForge**

MergeForge is an open-source VS Code / Cursor extension that turns conflicted files into a proper three-pane visual merge.  
MergeForge 是一个开源 VS Code/Cursor 扩展，它将冲突文件转换为标准的三窗格可视化合并。

Layout: Left | Center | Right  
布局：左 | 中 | 右  
Yours (local) | Result (editable, seeded from the merge base) | Theirs (incoming)  
你的版本（本地）| 结果（可编辑，基于合并基点生成）| 他们的版本（传入）

Panes scroll together. Chunks connect with bands. Gutter controls let you accept, ignore, or blend sides without fighting the UI. When you’re done, Apply writes the result and stages it with git.  
窗格同步滚动。代码块通过连接带关联。边栏控件让你无需与 UI 搏斗即可接受、忽略或混合两侧。完成后，“应用”会将结果写入并暂存到 git。

If you prefer Cursor, you’re covered too. The editor works the same; for AI features you plug in your own provider key (Cursor doesn’t expose its built-in models to extensions).  
如果你偏爱 Cursor，同样支持。编辑器功能相同；AI 功能需插入你自己的提供商密钥（Cursor 不向扩展开放其内置模型）。

Install it from the marketplace or Open VSX:  
从市场或 Open VSX 安装：

- **VS Code Marketplace**  
  https://marketplace.visualstudio.com/items?itemName=byte-forge.merge-forge
- **Open VSX (Cursor / VSCodium-friendly)**  
  https://open-vsx.org/extension/byte-forge/merge-forge

Or search “MergeForge” in the Extensions view. Then, on a conflicted file:  
或在扩展视图中搜索“MergeForge”。然后在冲突文件上：

- Use the editor title button, or **Resolve in Merge Forge** above a conflict  
  使用编辑器标题按钮，或冲突上方的 **在 Merge Forge 中解决**
- Right-click in Explorer / Source Control  
  在资源管理器/源代码管理中右键点击
- Run **Merge Forge: Resolve Conflicts in File…** from the Command Palette  
  从命令面板运行 **Merge Forge: 解决文件中的冲突…**

Key features  
**核心功能**

- **JetBrains-style three-pane merge editor**  
  **JetBrains 风格三窗格合并编辑器**  
  Your version, the editable result, and theirs — side by side, scroll-synced, with » / « accept, × ignore, connector bands, word-level diffs, and full Cmd/Ctrl+Z undo.  
  你的版本、可编辑结果和他们的版本——并排显示、滚动同步，配备 »/« 接受、× 忽略、连接带、词级差异和完整的 Cmd/Ctrl+Z 撤销。

- **Handy extras:**  
  **便捷附加功能：**
  - **Apply All Non-Conflicting Changes** (the wand) clears the easy stuff so you only decide the real conflicts  
    **应用所有非冲突更改**（魔杖图标）清除简单部分，让你只处理真实冲突
  - **Magic Resolve** keeps both sides when both simply added lines — and refuses to guess on genuine rewrites  
    **魔法解决**在双方仅添加行时保留两侧——并在真正重写时拒绝猜测
  - **F7 / Shift+F7** to jump changes; **Alt+← / Alt+→** to accept left/right of the current chunk  
    **F7/Shift+F7** 跳转更改；**Alt+←/Alt+→** 接受当前块的左侧/右侧
  - Line endings, BOMs, and trailing newlines round-trip cleanly; binaries and huge files get honest “nope” messages instead of broken panes.  
    行尾、BOM 和尾随换行符往返无损；二进制和大文件会得到诚实的“无法处理”提示，而非破碎窗格。

AI that reads your repo, not just the conflict markers  
**读取整个代码库的 AI，而非仅冲突标记**

Explain conflicts, resolve them into the result pane, or **Fix all** (safe mechanical merges first, model only on the red chunks). The model can use read-only tools — open files, search code, inspect history, look up symbols — and you see live ⚙ activity so nothing is a black box.  
解释冲突、将解决方案写入结果窗格，或**全部修复**（先进行安全机械合并，模型仅处理红色块）。模型可使用只读工具——打开文件、搜索代码、检查历史、查找符号——你会看到实时的 ⚙ 活动，没有任何黑箱操作。

Works with:  
支持：

- **GitHub Copilot** in VS Code (Language Model API) out of the box  
  VS Code 中的 **GitHub Copilot**（语言模型 API）开箱即用
- Or your own key for **Anthropic, OpenAI, DeepSeek, Kimi/Moonshot**, or any OpenAI-compatible endpoint (OpenRouter, local Ollama, proxies)  
  或使用你自己的密钥接入 **Anthropic、OpenAI、DeepSeek、Kimi/Moonshot** 或任何 OpenAI 兼容端点（OpenRouter、本地 Ollama、代理）
- Keys live in VS Code secret storage. Nothing hits git until you hit Apply. Unhappy with an AI edit? Undo that chunk like any other change.  
  密钥存储在 VS Code 保密存储中。在你点击“应用”前，任何操作都不会触及 git。对 AI 编辑不满意？像撤销其他更改一样撤销该块。

---

Authorship chips, history timeline, and a real merge workflow  
**作者标识块、历史时间线与真实合并流程**

Every conflict side can show who last shaped it (GitHub avatars when available, offline-safe initials otherwise). Flip to a two-lane history timeline of the commits that produced the merge.  
每个冲突侧都可显示最后修改者（有 GitHub 头像时显示头像，否则显示离线安全的姓名首字母）。切换到产生合并的提交的双通道历史时间线。

Around the editor: a Conflicts dialog with progress, status-bar merge cluster, next-file loop after Apply, crash-safety snapshots, and correct left/right mapping during rebases and cherry-picks.  
编辑器周边：带进度的冲突对话框、状态栏合并集群、应用后下一个文件循环、崩溃安全快照，以及在变基和拣选期间正确的左右映射。

Built with care (and tests)  
**精心构建（并附带测试）**

MergeForge is TypeScript end to end:  
MergeForge 全程使用 TypeScript：

- **VS Code Extension API** for commands, SCM, secret storage, and Copilot LM integration  
  **VS Code 扩展 API** 用于命令、SCM、保密存储和 Copilot 语言模型集成
- **Monaco Editor** inside the webview for the three panes  
  webview 内使用 **Monaco Editor** 实现三窗格
- **node-diff3** + a custom chunking engine checked against real `git merge-file` output  
  **node-diff3** + 自定义分块引擎（对照真实 `git merge-file` 输出验证）
- **Vercel AI SDK** for multi-provider streaming and tool use  
  **Vercel AI SDK** 用于多提供商流式传输和工具调用
- **esbuild** for the extension + webview bundles  
  **esbuild** 用于扩展和 webview 打包
- **Vitest** for unit, git-parity, and end-to-end tests  
  **Vitest** 用于单元、git 一致性及端到端测试

The correctness bar matters: where git merges cleanly, MergeForge aims to match git byte for byte. There’s also a replay eval for the AI pipeline so prompt changes can be regression-checked against real historical merges.  
正确性至关重要：在 git 干净合并的地方，MergeForge 旨在逐字节匹配 git。此外还有 AI 管道的回放评估，以便针对真实历史合并检查提示词更改的回归。

```
git clone https://github.com/ognjenmarcheta/merge-forge.git
cd merge-forge

pnpm test
pnpm run watch # then F5 to launch the Extension Development Host
```
