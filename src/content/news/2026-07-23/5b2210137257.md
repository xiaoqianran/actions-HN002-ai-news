## Unranked, systemd, crawls  
## 无排名查询、systemd迁移与爬虫加速  

Posted: 2026-07-22 There’s been some changes to Marginalia Search: Migrated to systemd from docker, New unranked query endpoint, Wide domains got their own index for faster crawls.  
发布日期：2026-07-22 Marginalia Search 进行了一些更新：从 Docker 迁移至 systemd、新增无排名查询端点、宽泛域名获得独立索引以加速爬取。  

In brief, this has removed a lot of operational headaches, reduced expensive queries leaving more computational power for the rest, and cut crawl times in half.  
简而言之，这消除了许多运营难题，减少了昂贵查询，为其他任务释放了更多计算能力，并将爬取时间减半。  

Let’s tackle them in some order.  
让我们按顺序讨论它们。  

We have docker at home  
我们家里有 Docker  

The system has been migrated off docker compose, and onto bare systemd in production. This has been successful, and solved a number of issues.  
系统已从 docker compose 迁移至生产环境中的原生 systemd。这一迁移成功解决了多个问题。  

There are a few constraints that demand a non-standard deployment for Marginalia.  
有几个限制因素要求 Marginalia 采用非标准部署。  

NUMA (non-uniform memory architecture).  
NUMA（非统一内存架构）。  

The production server has two CPUs, each with their own RAM bank, and while they can reach over into each other’s memory, this comes at a cost.  
生产服务器有两颗 CPU，每颗拥有独立的内存库，虽然它们可以访问彼此的内存，但这会带来性能成本。  

In many scenarios this isn’t a big deal, but for indexes and databases, generally bottlenecked by RAM bandwidth, this is far from ideal.  
在许多场景下这并非大问题，但对于通常受内存带宽限制的索引和数据库而言，这远非理想状态。  

Process lifecycles.  
进程生命周期。  

A search engine is anything but stateless, some processes are fairly heavy and run for weeks, some services are very slow to start, some services need to restart relatively often.  
搜索引擎绝非无状态：有些进程相当庞大且运行数周，有些服务启动缓慢，有些服务需要相对频繁地重启。  

This all but demands that the search engine is cut up into different pieces.  
这几乎要求搜索引擎被拆分为不同的组件。  

Not necessarily microservices, but at least services.  
不一定是微服务，但至少是服务化。  

IP addresses.  
IP 地址。  

The crawler (and crawler-adjacent processes) run off about a dozen public IP addresses, from the same host. This is done via ipvlan using network namespaces.  
爬虫（及相关进程）从同一主机上的约十二个公网 IP 地址运行。这是通过使用网络命名空间的 ipvlan 实现的。  

Linux has many advanced capabilities for giving processes their own virtual network stack, which can be wired together with virtual switches and virtual patch cables.  
Linux 拥有许多高级功能，可为进程提供独立的虚拟网络栈，并可通过虚拟交换机和虚拟跳线连接。  

There are to my knowledge no tools for managing this I would consider good, it’s all either too low level, or abstracting away key functionality, or suffering from multiple sources of truths leading to jank when they inevitably differ.  
据我所知，没有我认为好用的管理工具：它们要么过于底层，要么抽象了关键功能，要么因多源真相导致 inevitably 差异时出现故障。  

Docker can deal with all of these constraints, but at the expense of considerable jank.  
Docker 可以处理所有这些限制，但代价是相当多的故障。  

Docker’s networking model in particular, while it supports ipvlan, doesn’t map onto it particularly well, if you have a finite subnet you pretty much have to have spare free IPs to be able to reliably restart services.  
尤其是 Docker 的网络模型：虽然支持 ipvlan，但映射效果不佳；如果子网有限，几乎必须预留空闲 IP 才能可靠重启服务。  

It further doesn’t let you set firewall rules for the ipvlan interface, which means that anything you put there better not bind on the public IP, or its ass is going to be hanging out in full view.  
此外，它不允许为 ipvlan 接口设置防火墙规则，这意味着放在那里的任何服务最好不要绑定到公网 IP，否则其“屁股”将完全暴露。  

This is in no way a limitation of ipvlans, but purely a docker problem.  
这绝非 ipvlan 的限制，纯粹是 Docker 的问题。  

This is made worse by the fact that there is inadequate tooling for letting the processes in the container know which network interface is public, so you have to kinda guess based on dowsing rods, RFC 1918, and looking at tea leaves.  
更糟糕的是，缺乏足够工具让容器内进程知道哪个网络接口是公网的，因此你不得不 kinda 基于探矿棒、RFC 1918 和看茶叶来猜测。  

This is no way to live.  
这不是过日子之道。  

There is no magic in docker, it’s just cgroups and namespaces all the way down, and you can fully replicate what it’s doing yourself, if you really want to torment yourself using syscalls or shellscripts, or if you want a middle ground using systemd.  
Docker 没有魔法，它只是层层嵌套的 cgroups 和命名空间；如果你真想用系统调用或 shell 脚本折磨自己，或想用 systemd 折中，完全可以自己复现其功能。  

If docker’s problem is that there is a mismatch between the docker model of the system and the system itself, systemd represents a more raw approach, where a lot more of the configuration is in your hands, while still offering many capabilities you’d want for a more serious deployment, such as health checks, automatic restarts that can be configured.  
如果 Docker 的问题在于其系统模型与实际系统不匹配，那么 systemd 代表了一种更原始的方法：更多配置掌握在你手中，同时仍提供严肃部署所需的能力，如健康检查、可配置的自动重启。  

Setting up a million .service:s is pretty tedious, but drop-ins help.  
设置一百万个 .service 文件相当繁琐，但附加配置（drop-ins）有帮助。  

There isn’t a ton to say here, other than after some fiddling, it works very well, and feels a lot more stable than docker ever did. Cutting JIB out of builds means builds now take 2-3 seconds for the most part.  
这里没什么可多说的，除了经过一些调试后，它运行良好，感觉比 Docker 稳定得多。从构建中移除 JIB 意味着构建现在大多只需 2-3 秒。  

Deployments are faster, there’s less jank in service discovery because “containers” retain the same internal and external IP, the whole operation feels considerably less floaty. Good upgrade.  
部署更快，服务发现中的故障更少，因为“容器”保持相同的内部和外部 IP，整个操作感觉更稳固。良好的升级。  

There are many opinions about systemd, and I think for most desktop-type systems, it’s incredibly overengineered and kind of a pain to work with. Though for something like this, its design is well motivated, and it really shines.  
关于 systemd 有很多观点，我认为对于大多数桌面系统，它过度设计且难以使用。但对于此类场景，其设计动机充分，真正大放异彩。  

Unranked Queries  
无排名查询  

The index now allows unranked query execution!  
索引现在支持无排名查询执行！  

The query execution pipeline used in the search engine is fairly computationally expensive, and does things that aren’t strictly necessary in many queries.  
搜索引擎使用的查询执行管道计算成本相当高，并在许多查询中执行非严格必要操作。  

If you’re for example just looking for backlinks, a pure intersection of the terms ‘site:foo.com links:bar.com’ is enough, there’s nothing to rank anything there, but things working the way they did, threads were allocated to ranking none the less, term positions were attempted to be retrieved, all a bunch of done that needed not to be done.  
例如，如果只是查找反向链接，纯交集（如 ‘site:foo.com links:bar.com’）就足够了，无需排名；但按原有方式，线程仍被分配用于排名，尝试检索词条位置，做了许多不必要的操作。  

So I added a path for unranked queries that just do dumb term intersections and a bare minimum of anything else.  
因此，我为无排名查询添加了一条路径，仅执行简单的词条交集和最少其他操作。  

The bonus of this is that with some fiddling and the construction of a cursor that can map to multiple index partitions, unlike the ranked path, unranked queries can be exhaustively retrieved with minimal additional work.  
额外好处是，通过一些调试和构建可映射到多个索引分区的游标，无排名查询无需太多额外工作即可完全检索，而排名路径则不然。

为了跟踪跨多个分区的位置以实现完全检索，我思考了一番，设计了一个如下所示的游标，可传递给客户端：  

28mbshfptkj.6ijoop7xty.43nivb3bn1.817ucldxy2x.90.12ria2fmp7a.32cydk0dei2t.73wrhi4igjr.53lb1o4iof7  
28mbshfptkj.6ijoop7xty.43nivb3bn1.817ucldxy2x.90.12ria2fmp7a.32cydk0dei2t.73wrhi4igjr.53lb1o4iof7  

Each period-delimited part starts with a single character mapping to an index partition, and then a string of alphanumerical values that’s a base 36-encoded document id, the document id to resume from on that partition.  
每个以句点分隔的部分以单个字符开头，映射到一个索引分区，随后是一串字母数字值，为 36 进制编码的文档 ID，即在该分区恢复的文档 ID。  

It’s short enough to pass on a query string or along an API query, which is really all that matters.  
它足够短，可通过查询字符串或 API 查询传递，这才是关键。  

Implementing this allows up to half of the query load to go to the new endpoint, though many of these queries are driven by bots and scrapers traversing the /site viewer so the exact percentage varies quite significantly.  
实现这一点后，多达一半的查询负载可转向新端点，尽管其中许多查询由遍历 /site 查看器的机器人和爬虫驱动，因此确切比例变化很大。  

The execution time of the new unranked queries usually sits in the ~5ms range, which is at least an order of magnitude below a full query. A big part of the win is that the unranked queries are single threaded, which frees up many threads from the execution pool to do more meaningful work.  
新无排名查询的执行时间通常在 ~5ms 范围内，至少比完整查询低一个数量级。主要优势在于无排名查询是单线程的，从而释放执行池中的许多线程去做更有意义的工作。
