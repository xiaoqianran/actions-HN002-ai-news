# RefluXFS: A Linux Kernel Local Privilege Escalation to Root in XFS (CVE-2026-64600)
# RefluXFS：Linux 内核 XFS 文件系统中的本地权限提升至 Root 漏洞 (CVE-2026-64600)

**Executive summary**
**执行摘要**

Qualys Threat Research Unit (TRU) identified CVE-2026-64600, a race condition in the Linux kernel’s XFS filesystem copy-on-write path. An attacker with an ordinary local account can exploit this race condition to overwrite protected files on disk and gain host root privileges on affected systems, including deployments running SELinux in Enforcing mode.
Qualys 威胁研究单位 (TRU) 识别出 CVE-2026-64600，这是 Linux 内核 XFS 文件系统写时复制路径中的一个竞争条件。拥有普通本地账户的攻击者可以利用此竞争条件覆盖磁盘上的受保护文件，并在受影响的系统上获得主机 root 权限，包括以强制模式运行 SELinux 的部署环境。

This discovery emerged from a structured research initiative between Qualys and Anthropic, where we integrated Claude Mythos Preview into our manual audit workflow to accelerate our research while maintaining strict human oversight.
这一发现源于 Qualys 与 Anthropic 之间的一项结构化研究计划，我们将 Claude Mythos Preview 集成到手动审计工作流程中，以加速研究，同时保持严格的人工监督。

Using this vulnerability, a process running as an ordinary, unprivileged user can trigger the flaw and gain the ability to overwrite any readable file on an XFS volume at the block layer. This primitive converts directly into host root privileges.
利用此漏洞，以普通、无特权用户身份运行的进程可以触发该缺陷，并获得在块层覆盖 XFS 卷上任何可读文件的能力。这一原始能力可直接转换为主机 root 权限。

Exploitation is highly reliable and leaves no kernel log output.
利用非常可靠，且不会留下内核日志输出。

The vulnerability affects any Linux distribution that ships an XFS root filesystem with reflink enabled. This includes default installations of major enterprise platforms such as RHEL, Oracle Linux, Amazon Linux, and Fedora.
该漏洞影响任何默认启用 reflink 并搭载 XFS 根文件系统的 Linux 发行版。这包括主要企业平台的默认安装，如 RHEL、Oracle Linux、Amazon Linux 和 Fedora。

This vulnerability has been present since kernel version 4.11 (circa 2017) and, based on analysis with Qualys CyberSecurity Asset Management, potentially impacts over 16.4 million systems worldwide, requiring immediate attention due to its high exploitability and lack of kernel logging.
自内核版本 4.11（约 2017 年）起，该漏洞就已存在。根据 Qualys 网络安全资产管理的分析，它可能影响全球超过 1640 万个系统，因其高可利用性和缺乏内核日志记录而需要立即关注。

Immediate kernel patching is recommended to neutralize this vulnerability.
建议立即修补内核以消除此漏洞。

Exploitation succeeds consistently under standard hardening settings, and the on-disk modification survives a system reboot.
在标准加固设置下，利用始终成功，且磁盘上的修改在系统重启后依然存在。

Vendor-fixed kernels are now available and being backported to enterprise distributions.
供应商已发布修复内核，并正在向后移植到企业发行版。

Organizations should prioritize patching exposed and multi-tenant systems and ensure a reboot to verify the update.
组织应优先修补暴露的和多租户系统，并确保重启以验证更新。

As of now, there are reliable or practical mitigations or temporary configuration changes available.
截至目前，没有可靠或实用的缓解措施或临时配置更改可用。

We rate this as an emergency priority because exploitation could begin from ordinary local privileges. The vulnerability is present in standard enterprise kernel builds, and a successful exploitation provides host root. The exploitation works under common kernel hardening settings, and fixed kernels are available.
我们将其评为紧急优先级，因为利用可从普通本地权限开始。该漏洞存在于标准企业内核构建中，成功利用可提供主机 root。利用在常见内核加固设置下有效，且修复内核已可用。

RefluXFS 的重点不在于 AI 能做什么，而在于如何负责任地使用它。功能强大的模型已广泛可用一段时间；限制因素是方向，而非能力。教训并非 AI 可以被用于软件以生成漏洞利用；而是 AI 与专家判断相结合，可以帮助防御者在漏洞被滥用之前发现并修复严重缺陷。

**How does RefluXFS work?**
**RefluXFS 如何工作？**

The vulnerability resides in the XFS filesystem’s copy-on-write (COW) mechanism for reflink copies. When a file with reflink enabled is duplicated, the kernel creates a new inode that initially points to the same data blocks as the original. Writes to either file trigger a COW operation to allocate new blocks.
该漏洞位于 XFS 文件系统针对 reflink 副本的写时复制 (COW) 机制中。当复制一个启用了 reflink 的文件时，内核会创建一个新的 inode，它最初指向与原始文件相同的数据块。对任一文件的写入都会触发 COW 操作以分配新块。

RefluXFS exploits a race condition between the block allocation and the update of the file’s block map. By racing to write to the original file while the COW operation is in progress for the duplicate, an attacker can cause the kernel to allocate a new block for the duplicate but then incorrectly update the original file’s block map to point to this newly allocated block.
RefluXFS 利用块分配与文件块映射更新之间的竞争条件。通过在副本的 COW 操作进行时竞相写入原始文件，攻击者可以导致内核为副本分配一个新块，但随后错误地更新原始文件的块映射以指向这个新分配的块。

This results in the original file’s on-disk data being overwritten with the attacker’s controlled data from the duplicate, all while bypassing all filesystem permission checks because the operation occurs at the block layer, not the inode layer.
这导致原始文件的磁盘数据被来自副本的攻击者控制数据覆盖，同时由于操作发生在块层而非 inode 层，因此绕过了所有文件系统权限检查。

**Why Existing Mitigations Don’t Stop It**
**为何现有缓解措施无法阻止它**

*   **SELinux/AppArmor:** These are MAC (Mandatory Access Control) frameworks that operate at the inode (file) level. The exploit works at the block device layer, below the filesystem, rendering them ineffective.
    **SELinux/AppArmor：** 这些是在 inode（文件）层操作的 MAC（强制访问控制）框架。漏洞利用在块设备层、文件系统下方工作，使其失效。
*   **File Capabilities/DAC:** Standard Unix discretionary access controls (file permissions, ownership) are also bypassed because the kernel’s block layer operation does not re-check them during the race window.
    **文件能力/DAC：** 标准的 Unix 自主访问控制（文件权限、所有权）也被绕过，因为内核块层操作在竞争窗口期间不会重新检查它们。
*   **Kernel Page Protection (e.g., PAN, UAO):** These protect against accessing kernel memory from user space. This exploit does not access kernel memory directly; it manipulates kernel data structures (block maps) via legitimate, raced system calls.
    **内核页保护（例如 PAN, UAO）：** 这些用于防止从用户空间访问内核内存。此漏洞利用不直接访问内核内存；它通过合法的、存在竞争的系统调用操纵内核数据结构（块映射）。
*   **Filesystem Mount Options (e.g., `nosuid`, `nodev`):** These affect how files are *interpreted* (e.g., executing SUID binaries). They do not prevent the raw overwriting of file *data* on disk.
    **文件系统挂载选项（例如 `nosuid`, `nodev`）：** 这些影响文件如何被*解释*（例如执行 SUID 二进制文件）。它们不能防止磁盘上文件*数据*的原始覆盖。

**Technical Details of the RefluXFS vulnerability:**
**RefluXFS 漏洞的技术细节：**

The specific race is between `xfs_reflink_remap_blocks()` (which updates the destination file’s block map) and the `xfs_iomap_write_allocate()` path (which allocates a new block for a write to the source file during COW). The kernel fails to hold the appropriate lock (`ip->i_iolock`/`ip->i_mmaplock`) across the entire sequence of allocating a block and updating the destination’s block map, allowing the source file’s write to interleave and corrupt the map update.
具体的竞争发生在 `xfs_reflink_remap_blocks()`（更新目标文件的块映射）和 `xfs_iomap_write_allocate()` 路径（在 COW 期间为源文件的写入分配新块）之间。内核未能在整个分配块和更新目标块映射的序列中持有适当的锁（`ip->i_iolock`/`ip->i_mmaplock`），允许源文件的写入交错并破坏映射更新。

**Conclusion**
**结论**

RefluXFS (CVE-2026-64600) is a severe, easily exploitable local privilege escalation vulnerability in the Linux kernel’s XFS filesystem. It bypasses all common security mechanisms, leaves no traces, and provides full root compromise on a vast number of enterprise systems.
RefluXFS (CVE-2026-64600) 是 Linux 内核 XFS 文件系统中一个严重、易于利用的本地权限提升漏洞。它绕过了所有常见安全机制，不留痕迹，并在大量企业系统上提供完全的 root 入侵。

The vulnerability is trivially exploitable by any local user and has been present in the kernel since 2017.
该漏洞可被任何本地用户轻易利用，且自 2017 年以来一直存在于内核中。

    **重启：** 需要系统重启以加载修补后的内核。
