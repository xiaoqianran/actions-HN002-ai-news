**Title: Unity's Path to CoreCLR: What the Mono Cutover Means for Your Studio**  
**标题：Unity 转向 CoreCLR 之路：Mono 退役对您工作室的影响**

---

Unity is replacing its scripting runtime. Not tweaking it, replacing it. The Mono runtime that has sat under every line of C# you have written in Unity for years is being retired in favour of Microsoft's CoreCLR. It is the most significant change to Unity's foundation in over a decade, and it is no longer a distant roadmap item: the Unity 6.7 public alpha is out now, with CoreCLR arriving as an experimental option. Most of the coverage treats this as good news wrapped in a version number. It is good news. But if you run a real project, the interesting questions are the practical ones: when does this actually reach me, what changes underneath my game, and what is going to break. Here is that read, from the perspective of a studio that plans and runs Unity upgrades for a living.  
Unity 正在更换其脚本运行时。不是小修小补，而是彻底替换。多年来支撑着您在 Unity 中编写的每一行 C# 代码的 Mono 运行时即将退役，取而代之的是微软的 CoreCLR。这是 Unity 基础架构十多年来最重大的变革，且它已不再是遥远的路线图项目：Unity 6.7 公开测试版现已发布，CoreCLR 作为实验性选项首次亮相。大多数报道将此视为包裹在版本号中的好消息。这确实是好消息。但如果您在管理实际项目，真正有趣的是那些实际问题：这何时能真正影响到我？我的游戏底层发生了什么变化？什么会出问题？以下是从一个以规划和执行 Unity 升级为专业的工作室视角的解读。

---

What CoreCLR Is, and Why Unity Is Doing It  
**CoreCLR 是什么，以及 Unity 为何要这样做**

Unity has been running a heavily customised fork of Mono for years. That custom fork is the reason Unity has always trailed the wider .NET ecosystem: while the rest of the C# world moved to modern runtimes, garbage collectors, and language features, Unity developers looked on from behind a runtime that could not easily keep pace. CoreCLR is Microsoft's modern, open-source .NET runtime, the same one that powers current .NET. Moving to it does three things at once: it gives Unity a far more capable runtime and garbage collector, it unlocks modern C# and the current .NET library ecosystem, and it dramatically improves iteration time, the write-save-wait-for-domain-reload loop that quietly eats hours of every Unity developer's week. Unity 6.8 is expected to target .NET 10 and C# 14. To make room for it, Unity has done something telling: it paused new work on animation and world-building workflows specifically to concentrate engineering on this migration and on architectural stability. That is a company choosing foundations over features, which is the right call, and a sign of how big this change is.  
多年来，Unity 一直运行着一个深度定制的 Mono 分支。这个定制分支是 Unity 始终落后于更广泛的 .NET 生态系统的原因：当 C# 世界的其他部分迁移到现代运行时、垃圾回收器和语言特性时，Unity 开发者却困在一个难以跟上步伐的运行时之后。CoreCLR 是微软现代、开源的 .NET 运行时，也是当前 .NET 所使用的基础。迁移到它同时带来三件事：它赋予 Unity 一个能力更强的运行时和垃圾回收器，它解锁了现代 C# 和当前的 .NET 库生态系统，并显著改善了迭代时间——那个悄悄吞噬每位 Unity 开发者每周数小时的“编写-保存-等待域重载”循环。Unity 6.8 预计将面向 .NET 10 和 C# 14。为了给它腾出空间，Unity 做了一个有意义的举动： specifically 暂停了动画和世界构建工作流的新开发，以便将工程力量集中在这项迁移和架构稳定性上。这是一家公司选择基础而非功能，这是正确的决定，也标志着这一变化的规模之大。

The Timeline That Actually Matters  
**真正重要的时间线**

The migration lands across two releases, and the distinction between them decides what you should do.  
迁移跨越两个版本，它们之间的区别决定了您应该采取的行动。

| Release | What happens | Your posture |
| :--- | :--- | :--- |
| **6.7** (alpha now, LTS targeted late 2026) | CoreCLR arrives as an experimental, opt-in scripting backend, desktop first, sitting alongside Mono and IL2CPP. | Test in a branch. Do not ship on it. |
| **6.8** (targeted late 2026) | Mono is removed entirely. CoreCLR becomes the foundation of the C# layer. No Mono option remains. | This is the real cutover. Plan for it now. |

| 版本 | 发生什么 | 您的姿态 |
| :--- | :--- | :--- |
| **6.7** (当前为 alpha，LTS 目标为 2026 年末) | CoreCLR 作为实验性、可选的脚本后端首次亮相，优先支持桌面平台，与 Mono 和 IL2CPP 并存。 | 在分支中测试。不要用于正式发布。 |
| **6.8** (目标为 2026 年末) | Mono 被完全移除。CoreCLR 成为 C# 层的基础。不再有 Mono 选项。 | 这是真正的切换点。现在就为此规划。 |

Two things worth internalising. First, Unity has been explicit that 6.8 aims for holistic performance parity first, not a blanket speed-up. Some things will be faster, some slower, and the larger gains come over time as the runtime settles. Treat promises of instant performance with appropriate scepticism. Second, because 6.7 is where you can first put hands on CoreCLR, it is the release to experiment with, even though 6.8 is where the decision becomes unavoidable.  
有两件事值得内化。首先，Unity 已明确表示 6.8 的首要目标是实现整体性能 parity（对等），而非全面的速度提升。有些方面会变快，有些会变慢，更大的收益将随着运行时稳定而随时间显现。对“即时性能提升”的承诺应持适当的怀疑态度。其次，因为 6.7 是您能首次亲手接触 CoreCLR 的版本，所以它是进行实验的发布版，尽管 6.8 才是决策无法回避的时刻。

What Changes Under the Hood  
**底层的变化**

The garbage collector is the headline. Mono uses the Boehm GC, which is conservative and non-moving: it cannot always tell precisely what is a live object, and it never relocates memory. CoreCLR brings a precise, moving, generational GC. It knows exactly which objects are live, and it can compact memory. In practice that means less collection overhead and fewer of the stalls that show up as frame hitches. For anyone who has spent time chasing GC spikes in a Unity profiler, this is the change that matters most.  
垃圾回收器是头条新闻。Mono 使用 Boehm GC，它是保守且非移动的：它无法总是精确判断哪些对象是存活的，也从不 relocated 内存。CoreCLR 带来了精确的、移动的、分代的 GC。它能精确知道哪些对象是存活的，并且可以压缩内存。实际上，这意味着更少的回收开销和更少表现为帧卡顿的停顿。对于任何曾在 Unity 分析器中追逐 GC 峰值的人来说，这是最重要的变化。

The memory allocator changes too. Unity is integrating MiMalloc, Microsoft's high-performance allocator, which handles the locking problems that hurt multithreaded allocation. If your project makes serious use of the C# Job System, that code gets more stable and more predictable under load essentially for free, just from the engine update.  
内存分配器也变了。Unity 正在集成 MiMalloc，这是微软的高性能分配器，它解决了损害多线程分配性能的锁问题。如果您的项目 serious 使用 C# 作业系统，那么这段代码在负载下会变得更稳定、更可预测，基本上只需通过引擎更新即可免费获得。

The C# and serialisation story improves. Beyond the language and library modernisation, Unity 6.7 adds native serialisation for `Dictionary<TKey, TValue>`, something the community has wanted for years and worked around with `ISerializationCallbackReceiver` gymnastics. Unity is also removing the old and insecure `BinaryFormatter`, which is a security improvement but a real migration item if your project or a dependency still uses it.  
C# 和序列化方面也得到改进。除了语言和库的现代化，Unity 6.7 为 `Dictionary<TKey, TValue>` 添加了原生序列化，这是社区多年来一直想要并曾通过 `ISerializationCallbackReceiver` 的复杂操作来绕过的功能。Unity 还移除了过时且不安全的 `BinaryFormatter`，这改进了安全性，但如果您的项目或某个依赖项仍在使用它，则是一个真正的迁移项。

Iteration gets faster. This is the day-to-day win that every developer on the team feels: compile and domain-reload times come down. On a large project, that is hours back per developer per week.  
迭代速度变快。这是团队中每位开发者都能感受到的日常胜利：编译和域重载时间缩短了。在大型项目中，这意味着每位开发者每周能节省数小时。

The Nuance for Mobile and Console: IL2CPP Is Not Going Away  
**移动和主机的细微差别：IL2CPP 不会消失**

This is the part a lot of the excitement skips, and it is the part that matters most if you ship where we ship. CoreCLR is a JIT runtime. It is aimed first at platforms that permit JIT compilation: desktop and the editor. But iOS, the consoles, and other restricted platforms require ahead-of-time compilation, and that is exactly what IL2CPP does. Unity has been clear that it has no plans to retire IL2CPP. So the real change is not "Mono to CoreCLR" across the board. It is that the old choice of Mono versus IL2CPP becomes a new choice of CoreCLR versus IL2CPP, with CoreCLR replacing Mono as the JIT option and IL2CPP continuing as the AOT option.  
这是许多兴奋情绪所忽略的部分，也是如果您在我们所谈论的平台上发布产品时最重要的部分。CoreCLR 是一个 JIT 运行时。它首先面向允许 JIT 编译的平台：桌面和编辑器。但 iOS、主机和其他受限平台需要提前编译，而这正是 IL2CPP 所做的。Unity 已明确表示没有退役 IL2CPP 的计划。因此，真正的变化不是全面地从“Mono 到 CoreCLR”。而是旧的 Mono 与 IL2CPP 之间的选择变成了新的 CoreCLR 与 IL2CPP 之间的选择，CoreCLR 作为 JIT 选项取代 Mono，而 IL2CPP 继续作为 AOT 选项。

The honest implication for a mobile team: your production iOS and console builds will very likely still go out through IL2CPP for the foreseeable future. That means the immediate, guaranteed win from this migration for a mobile studio is faster iteration and a modern .NET foundation across the team, rather than an automatic reduction in on-device GC stalls on your shipped AOT build. The deeper runtime and GC gains land where CoreCLR actually runs. Knowing which improvements apply to which of your targets is the difference between planning this well and being surprised by it.  
对于移动团队来说，诚实的含义是：在可预见的未来，您的生产环境 iOS 和主机构建很可能仍将通过 IL2CPP 发布。这意味着对于移动工作室，这次迁移带来的即时、确定的收益是更快的迭代速度和团队拥有一个现代的 .NET 基础，而不是您已发布的 AOT 构建中设备上 GC 停顿的自动减少。更深层的运行时和 GC 收益只在 CoreCLR 实际运行的地方落地。了解哪些改进适用于您的哪些目标平台，是妥善规划此事与事后感到惊讶之间的区别。
