## Key Takeaways  
## 关键要点  

**What this post is:** a step-by-step tour of a from-scratch CUDA inference runtime for Qwen2.5-Coder-7B-Instruct on NVIDIA H100 (`sm_90`) — every hot path annotated for why, not just what.  
**本文内容：** 在NVIDIA H100（`sm_90`）上为Qwen2.5-Coder-7B-Instruct构建的从零开始的CUDA推理运行时的分步详解——每个热点路径都标注了原因，而不仅仅是功能。  

**What the runtime does today:** greedy graph-tier token match passes on H100 at ≤4096 context, steady-state decode lands around ~16.7 ms/token (~60 tok/s), and first-token latency is ~128 ms for a 512-token prompt.  
**当前运行时性能：** 在H100上，上下文长度≤4096时，贪图层级令牌匹配通过；稳定状态解码约16.7毫秒/令牌（~60令牌/秒）；512令牌提示的首令牌延迟约128毫秒。  

**The one number that made everything else make sense:** eager decode was ~119 ms/token. Wrapping decode in a captured CUDA graph collapsed that to ~17 ms/token. A 7× win — same kernels, just less driver.  
**让一切清晰的关键数字：** 急切解码约119毫秒/令牌。将解码封装在捕获的CUDA图中后，降至约17毫秒/令牌。7倍提升——相同内核，仅减少驱动开销。  

**Three bugs did most of the teaching:** `__syncthreads()` inside a warp-specialized branch is undefined behaviour (and corrupts softmax at KV page tails), CUDA graphs are not optional, and `prmt.b32` beat `__dp4a` on Hopper for gate/up/down GEMV shapes — the INT8-activation path was tried and removed.  
**三大错误贡献了大部分经验：** 在线程束专业化分支内的`__syncthreads()`是未定义行为（并破坏KV页面尾部的softmax）；CUDA图不是可选项；在Hopper上，对于gate/up/down GEMV形状，`prmt.b32`优于`__dp4a`——INT8激活路径被尝试后移除。  

**llama.cpp is the reference implementation, not the rival:** at commit b3040 with Q4_K_M on the same GPU class, the same pp512/tg128 protocol posts ~43 ms TTFT and ~4.95 ms/token decode. It is the yardstick to check whether your own build is behaving reasonably.  
**llama.cpp是参考实现，而非竞争对手：** 在相同GPU类别上，使用Q4_K_M和提交b3040，相同pp512/tg128协议的首令牌延迟约43毫秒，解码约4.95毫秒/令牌。它是检查自己构建是否合理的标尺。

## 1. Why build your own LLM runtime at all?  
## 1. 为什么要自己构建LLM运行时？  

**Most production LLM inference already routes through a very well-tested path: GGUF → llama.cpp → done. It works, it is fast, it has an army of contributors, and it is the right tool for shipping. Let me be clear right now: nothing in this post is trying to challenge that, and I recommend you keep using it in production, as I do.**  
**大多数生产级LLM推理已通过经过充分测试的路径：GGUF → llama.cpp → 完成。它有效、快速、拥有大量贡献者，是部署的正确工具。我现在明确说明：本文没有任何内容试图挑战这一点，我建议你在生产环境中继续使用它，正如我所做。**  

**Ownership of the decode stack is where the value is. If you ever need to change the quantization format, add a custom sampler, plug in a novel attention variant, or ship on a new architecture, you cannot really do it from inside a black box. You need to be able to open every kernel and know what will break — and what has already broken and been fixed.**  
**对解码栈的掌控才是价值所在。如果你需要更改量化格式、添加自定义采样器、插入新颖的注意力变体，或在新架构上部署，你无法真正在黑盒内完成。你需要能打开每个内核，知道什么会破坏——以及什么已经破坏并被修复。**  

**I think, at some point of time, we have already or we will ask ourselves the following question: Can I write decode myself, understand every launch, own every barrier, and still get the correct tokens out? The short answer is yes.**  
**我认为，在某个时刻，我们已经或将会问自己以下问题：我能自己编写解码、理解每次启动、掌控每个屏障，并仍能输出正确令牌吗？简短答案是：可以。**  

**If you have ever been curious about how a real decoder is actually glued together — packed weight files, INT4 dequant, paged KV, warp specialization, CUDA graphs, TMA — but you have been too intimidated to read all fifty-thousand lines of GGML to find out, this repo (and this post) is the same journey compressed into roughly two dozen CUDA/C++ source files with dense comments. Every hot path in the code carries the why. Every barrier says which threads arrive on it and which threads must not. Every kernel says which shape it was tuned for and which shape it silently regressed on. It is the version of the codebase I wish had existed when I started.**  
**如果你曾好奇真实解码器如何组装——打包权重文件、INT4反量化、分页KV、线程束专业化、CUDA图、TMA——但被GGML的五万行代码吓退，此仓库（及本文）就是将同一旅程压缩到约二十几个带有密集注释的CUDA/C++源文件。代码中每个热点路径都承载着原因。每个屏障说明哪些线程到达、哪些线程不能到达。每个内核说明它针对哪种形状调优、哪种形状悄然退化。这是我 wish 在起步时就存在的代码库版本。**  

**Some of the annotations exist because I had to talk myself through fixing the code. Others exist because I had to talk myself out of a bad idea I had already implemented. This is the second kind of comment, in the wild:**  
**部分注释存在是因为我不得不说服自己修复代码。其他注释存在是因为我不得不说服自己放弃已实现的坏主意。这是野外第二种评论：**  

`// prmt.b32 path exercises Hopper integer permute pipes — baseline before fused scale broadcast.`  
`// prmt.b32路径测试Hopper整数置换管道——融合缩放广播前的基线。`  

`// ValueShuffle lane map is frozen to match offline packer — runtime cannot reinterpret nibbles.`  
`// ValueShuffle通道映射冻结以匹配离线打包器——运行时无法重新解释半字节。`  

**That comment is the epitaph of a wasted afternoon.**  
**那条注释是虚度下午的墓志铭。**

**在深入内部之前，先诚实地展示运行时的当前状态。测量基于相同H100级别，配置锁定在benchmark_e2e协议（batch=1, pp512/tg128, `--warmup 3 --prompt-tokens 512 --gen-tokens 128`）：**

**If you are rebuilding this stack yourself, the llama.cpp column is a reference implementation for the same protocol — pinned at commit b3040 with the standard offload flags (`-ngl 99 -c 8192 -b 512 -cb`) and the official Qwen/Qwen2.5-Coder-7B-Instruct-GGUF weights. Same GPU class, same model, same greedy decode, different amount of tuning behind them. It is the yardstick to check that your own build is behaving reasonably, not a rival to chase this early.**  
**如果你自己重建此栈，llama.cpp列是相同协议的参考实现——锁定在提交b3040，使用标准卸载标志（`-ngl 99 -c 8192 -b 512 -cb`）和官方Qwen/Qwen2.5-Coder-7B-Instruct-GGUF权重。相同GPU类别、相同模型、相同贪心解码，但背后调优量不同。它是检查自己构建是否合理的标尺，而非早期追逐的竞争对手。**  

**For the CUDA-flavored readers: the single biggest lever inside this stack was CUDA graph decode. Eager per-token decode on the same kernels came in around 119 ms/token. Wrapping steady-state decode in a captured graph and replaying it dropped that to ~17 ms/token. Same kernels, just submitted differently. Whole story on that in Section 6.**  
**对于CUDA爱好者：此栈中最大的杠杆是CUDA图解码。相同内核上的急切每令牌解码约119毫秒/令牌。将稳定状态解码封装在捕获图中并重放，降至约17毫秒/令牌。相同内核，仅提交方式不同。完整故事见第6节。**

## 3. The stack in one page  
## 3. 一页看懂栈结构  

**If you close this tab after this section you will still have gotten the shape. Every input token walks the following path, once per generated token, on the H100:**  
**即使在本节后关闭此标签，你仍能了解整体结构。每个输入令牌在H100上按以下路径行走，每生成一个令牌一次：**  

1. **Weight load & dequant** — INT4 packed → `__dp4a` or `prmt.b32` → FP16/ BF16 accumulator.  
   **权重加载与反量化** — INT4打包 → `__dp4a`或`prmt.b32` → FP16/BF16累加器。  
2. **Attention KV fetch** — paged KV cache lookup, TMA async copy into shared memory.  
   **注意力KV获取** — 分页KV缓存查找，TMA异步复制到共享内存。  
3. **Q/K/V compute** — Q from current token, K/V from cache. Matmul with dequant'd weights.  
   **Q/K/V计算** — 当前令牌的Q，缓存中的K/V。与反量化权重进行矩阵乘。  
4. **Softmax** — online softmax over KV cache (up to 4096 tokens).  
   **Softmax** — 对KV缓存（最多4096令牌）的在线softmax。  
5. **Output projection** — final matmul to vocab size, greedy sample.  
   **输出投影** — 最终矩阵乘至词汇表大小，贪心采样。  

**All of this is wrapped in a captured CUDA graph for steady-state decode. The first token (prompt processing) runs eager, then the graph is captured and replayed for all subsequent tokens.**  
**所有这些在稳定状态解码中封装在捕获的CUDA图中。第一个令牌（提示处理）运行急切，然后捕获图并重放用于所有后续令牌。**
