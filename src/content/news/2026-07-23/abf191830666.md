# PyPI releases now reject new files after 14 days  
# PyPI 现拒绝向超过14天的旧版本发布新文件

Releases now reject new files after 14 days The Python Package Index (PyPI) now rejects new files being uploaded to releases that are older than 14 days. This restriction was put in place to prevent old and long-stable releases from being poisoned in case publishing tokens or workflows of PyPI projects were compromised.  
PyPI 现拒绝向超过14天的旧版本发布新文件。Python 包索引（PyPI）现会拒绝向超过14天的旧版本发布上传新文件。此限制旨在防止在 PyPI 项目的发布令牌或工作流遭泄露时，长期稳定的旧版本被投毒。

As far as we are aware this has not yet been abused, but there is no technical reason beyond that attackers weren't aware it was possible. Users should not yet rely on this behavior as there are no defined semantics for "releases no longer accepting new files" or APIs available to confirm the state of the release. Instead, these semantics will be defined once "Upload 2.0 API" and "Staged Previews" have been standardized by PEP 694.  
据我们所知，此漏洞尚未被滥用，但技术上并非不可行，只因攻击者此前未意识到该可能性。用户目前不应依赖此行为，因为“版本不再接受新文件”的语义尚未定义，也没有可用 API 来确认版本状态。相反，这些语义将在“上传 2.0 API”和“暂存预览”通过 PEP 694 标准化后定义。

The discussion of this behavior began during PEP 740 (Digital Attestations) back in January 2024. The discussion was restarted in March 2026 after the popular packages LiteLLM and Telnyx were compromised. These packages were compromised due to a "mutable reference" in these projects' usage of the Trivy GitHub Action.  
该行为的讨论始于 2024 年 1 月的 PEP 740（数字证明）。2026 年 3 月，在流行包 LiteLLM 和 Telnyx 遭入侵后，讨论重启。这些包因项目中 Trivy GitHub Action 的“可变引用”而遭入侵。

Originally the discussion stalled due to some projects depending on this behavior to add support for new Python versions to already-published releases. To quantify how disruptive this change would be to existing workflows, the PyPI database was queried for projects that have published new files to old releases (bucketed by number of days since the release).  
最初讨论陷入停滞，因部分项目依赖此行为为已发布的旧版本添加对新 Python 版本的支持。为量化此变更对现有工作流的破坏性，PyPI 数据库被查询以统计向旧版本发布新文件的项目（按发布后天数分组）。

Later, specifically cp314 wheels were queried for the top 15,000 packages, revealing that only 56 projects of 15,000 had published a 3.14-compatible wheel more than 14 days after a release was available.  
随后，针对前 15,000 个包查询了 cp314 轮子，发现仅 56 个项目在版本发布超过 14 天后发布了 3.14 兼容轮子。

This topic was brought to the Packaging Summit at PyCon US 2026 by PyPI Safety & Security Engineer, Mike Fiedler. The rough consensus of the discussion was that the summit attendees thought it was "acceptable to require users to bump to the next version" to support new Python versions.  
此议题由 PyPI 安全工程师 Mike Fiedler 在 PyCon US 2026 的打包峰会提出。讨论的大致共识是，与会者认为“要求用户升级到下一版本以支持新 Python 版本”是可接受的。

With the data and consensus in hand, Seth moved forward with a patch to reject new files on old releases which was merged July 8th, 2026. This change will protect Python users and reduce the amount of "cleanup" work associated with project compromises for PyPI admins.  
手握数据与共识，Seth 推进了拒绝向旧版本发布新文件的补丁，该补丁于 2026 年 7 月 8 日合并。此变更将保护 Python 用户，并减少 PyPI 管理员在项目遭入侵时所需的“清理”工作。

This restriction also means that compromises don't put releases into an indeterminate and confusing state of both "compromised" and "not compromised", where only a subset of files could be poisoned with malware.  
此限制还意味着入侵不会使版本陷入“已入侵”与“未入侵”并存的不确定且混乱状态——即仅部分文件可能被植入恶意软件。

Stay tuned for the future where Upload 2.0 API provides semantics for releases that are "closed" instead of "open".  
敬请期待未来“上传 2.0 API”为“关闭”而非“开放”的版本提供语义定义。
