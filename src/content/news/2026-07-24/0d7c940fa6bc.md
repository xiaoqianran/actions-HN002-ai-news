# Migrating a live x402 service from V1 to V2 on Base mainnet  
# 将 Base 主网上运行的实时 x402 服务从 V1 迁移到 V2 实战指南  

x402 V1 → V2 Migration Guide Field notes from migrating a live, payment-gated API from x402 V1 to V2 on Base mainnet — the undocumented gotchas, working code from a service in production, and a checklist at the bottom. Written from x402ai, which settles real USDC on Base via the Coinbase CDP facilitator. Every code sample here is running in production, not reconstructed from the spec. Last verified: July 2026 · @x402/* 2.19.0 · @coinbase/x402 2.1.0  
x402 V1 → V2 迁移指南：来自将 Base 主网上一个实时支付门控 API 从 x402 V1 迁移到 V2 的实战笔记——包括未文档化的陷阱、生产环境中运行的真实代码，以及底部的检查清单。本文基于 x402ai 的实践，该服务通过 Coinbase CDP 协调器在 Base 上结算真实 USDC。此处的每个代码示例均在生产环境中运行，而非根据规范重构。最后验证时间：2026 年 7 月 · @x402/* 2.19.0 · @coinbase/x402 2.1.0  

In a hurry? → Migration checklist  
赶时间？→ 迁移检查清单  

Why you're probably being forced into this  
你为何可能被迫进行迁移  

If you built an x402 service before mid-2026, you're on V1. You may not have noticed, because V1 services keep serving 402s and keep settling payments. Nothing crashes. The migration pressure comes from two directions instead:  
如果你在 2026 年年中之前构建了 x402 服务，那么你正在使用 V1。你可能没有注意到这一点，因为 V1 服务会持续返回 402 状态码并持续结算付款。服务不会崩溃。迁移压力来自两个方向：  

Discovery rejects you. Submitting a V1 origin to x402scan returns, in effect, "x402 v1 response detected — migrate to v2 spec", plus a second complaint about the absence of a discovery document. Since x402scan is where autonomous agents actually find services, a V1 service is invisible to the thing that generates traffic. Agents are already failing against you silently.  
发现服务被拒绝。向 x402scan 提交 V1 源地址会返回类似“检测到 x402 v1 响应——请迁移到 v2 规范”的信息，并附带第二条关于缺少发现文档的投诉。由于 x402scan 是自主代理实际查找服务的地方，V1 服务对流量生成器不可见。代理已在对你的请求中静默失败。  

Check your logs for invalid_payload errors you never explained. Those are V2-capable clients attempting payment against V1 middleware and bouncing. You've been losing real requests, and the error message doesn't say why.  
检查日志中那些无法解释的 `invalid_payload` 错误。这些是 V2 兼容客户端尝试向 V1 中间件付款但被拒绝的情况。你一直在丢失真实请求，而错误信息并未说明原因。  

That second one is the part I'd flag hardest. I had those errors in my logs for weeks and filed them mentally as noise.  
后者是我认为最需要警惕的部分。这些错误在我的日志中存在数周，而我最初将其视为噪音。  

What actually changed  
实际发生了哪些变化  

Five things, in rough order of how much time they'll cost you:  
有五项变化，大致按它们将耗费你的时间排序：  

The package line moved. x402-hono is frozen at 1.2.0. The live line is the scoped @x402/* namespace.  
包路径已变更。`x402-hono` 已冻结在 1.2.0 版本。当前活跃版本位于作用域命名空间 `@x402/*` 下。  

The challenge moved from the body to a header. A V2 402 has an empty JSON body and carries the challenge base64-encoded in a payment-required header.  
挑战信息已从请求体移至请求头。V2 的 402 响应具有空的 JSON 请求体，并将 base64 编码的挑战信息携带在 `payment-required` 请求头中。  

Network identifiers are CAIP-2 now. eip155:8453 rather than the old string form.  
网络标识符现为 CAIP-2 格式。使用 `eip155:8453` 而非旧的字符串形式。  

CDP facilitator auth is supported natively. If you were reaching into SDK internals to mint JWTs, you can delete that.  
CDP 协调器认证现获原生支持。如果你曾深入 SDK 内部以铸造 JWT，现在可以删除那段代码。  

X-Forwarded-Proto is ignored. If you terminate TLS at a tunnel or proxy, your resource URLs will advertise as http:// unless you set them explicitly.  
`X-Forwarded-Proto` 请求头现被忽略。如果你在隧道或代理处终止 TLS，你的资源 URL 将 advertised 为 `http://`，除非你显式设置它们。  

The package landscape  
包生态格局  

This tripped me up before I wrote a line of code, because searching for x402 packages surfaces the frozen ones first.  
这在我编写任何代码之前就绊倒了我，因为搜索 x402 包时，冻结的版本会优先显示。  

Frozen (V1):  
冻结版本 (V1)：  
- x402-hono @ 1.2.0 — final release, do not build on it.  
- `x402-hono @ 1.2.0` —— 最终版本，请勿基于它构建。  

Live (V2):  
活跃版本 (V2)：  
- the scoped namespace — @x402/hono (or your framework's equivalent)  
- 作用域命名空间 —— `@x402/hono`（或你框架的等效包）  
- @x402/core  
- @x402/evm  
- @x402/extensions  
- @coinbase/x402 — for CDP facilitator config  
- `@coinbase/x402` —— 用于 CDP 协调器配置  

At time of writing the @x402/* packages were at 2.19.0 and @coinbase/x402 at 2.1.0. Check current versions; this line moves.  
截至撰写时，`@x402/*` 包版本为 2.19.0，`@coinbase/x402` 为 2.1.0。请检查当前版本；此版本线会变动。  

Install gotcha  
安装陷阱  

npm install @x402/hono@2 @x402/core@2 @x402/evm@2 @coinbase/x402@2 --legacy-peer-deps  
`npm install @x402/hono@2 @x402/core@2 @x402/evm@2 @coinbase/x402@2 --legacy-peer-deps`  

You will likely need --legacy-peer-deps. The optional paywall peer dependency wants React 19; if your project is on React 18 (or has any other React in the tree), npm refuses the install outright. The paywall is optional and unrelated to server-side payment gating, so the flag is safe here — but know why you're using it rather than reflexively adding it.  
你很可能需要 `--legacy-peer-deps` 标志。可选的付费墙对等依赖要求 React 19；如果你的项目使用 React 18（或依赖树中存在任何其他 React），npm 会直接拒绝安装。付费墙是可选的，与服务器端付费门控无关，因此在此处使用该标志是安全的——但请了解你使用它的原因，而非盲目添加。  

The server rewrite  
服务器端重写  

Before (V1): middleware took a routes object and a facilitator, with auth headers you assembled yourself.  
之前 (V1)：中间件接收一个路由对象和一个协调器，以及你自行组装的认证请求头。  

After (V2): you build a resource server, register a scheme against a network, then hand that to the middleware.  
之后 (V2)：你构建一个资源服务器，针对网络注册一个方案，然后将其传递给中间件。  

Watch the import subpaths  
注意导入子路径  

This cost me time before a single line of logic ran. The V2 packages use subpath exports, and the obvious top-level import is not the one you want:  
这在我运行任何逻辑代码之前就耗费了时间。V2 包使用子路径导出，而显而易见顶层导入并非你所需：  

```javascript
import { paymentMiddleware } from '@x402/hono'
import { x402ResourceServer, HTTPFacilitatorClient } from '@x402/core/server'
import { createFacilitatorConfig } from '@coinbase/x402'
import { ExactEvmScheme } from '@x402/evm/exact/server'
```

@x402/core/server, not @x402/core. @x402/evm/exact/server, not @x402/evm.  
应从 `@x402/core/server` 导入，而非 `@x402/core`。应从 `@x402/evm/exact/server` 导入，而非 `@x402/evm`。  

Building the handler  
构建处理器  

```javascript
const facilitatorClient = new HTTPFacilitatorClient(
  createFacilitatorConfig(
    process.env.CDP_API_KEY_NAME,
    process.env.CDP_API_KEY_PRIVATE_KEY
  )
)
const resourceServer = new x402ResourceServer(facilitatorClient)
  .register('eip155:8453', new ExactEvmScheme())
const routes = {
  'POST /api/embed': {
    accepts: {
      scheme: 'exact',
      price: prices['/api/embed'],
      network: 'eip155:8453',
      payTo: process.env.WALLET_ADDRESS,
    },
    resource: new URL('/api/embed', baseUrl).toString(),
    description: 'Text embeddings (768-dim) via nomic-embed-text. POST {input: string}.',
    mimeType: 'application/json',
  },
  // ...one entry per paid route
}
return paymentMiddleware(routes, resourceServer)
```

register() is fluent — it returns the resource server, so it chains off the constructor.  
`register()` 是流畅接口——它返回资源服务器实例，因此可以链式调用构造函数。  

Route config shape changed  
路由配置形状已变更  

V1 nested description under a config key — and silently ignored a top-level description, which is a fun thing to discover after wondering why your route descriptions never showed up anywhere.  
V1 将 `description` 嵌套在 `config` 键下——并静默忽略了顶层 `description`，这在你疑惑为何路由描述从未显示在任何地方后才被发现，真是有趣的发现。  

V2 keys routes by METHOD /path, takes accepts as a single object rather than an array, and puts description and mimeType at the top level as siblings of accepts.  
V2 以 `METHOD /path` 作为路由键，将 `accepts` 作为单个对象而非数组，并将 `description` 和 `mimeType` 置于顶层作为 `accepts` 的兄弟属性。  

Deleting the JWT hack  
删除 JWT 硬编码方案  

If you were on CDP's facilitator under V1, there's a decent chance you did what I did: imported generateJwt through @coinbase/cdp-sdk's internal _cjs path via createRequire, then hand-built auth headers on every request. That was never supported. It worked, but it depended on the internal module layout of someone else's package.  
如果你在 V1 下使用 CDP 协调器，你很可能和我一样做过：通过 `createRequire` 从 `@coinbase/cdp-sdk` 的内部 `_cjs` 路径导入 `generateJwt`，然后在每个请求上手动构建认证请求头。这从未被官方支持。它虽然能用，但依赖于他人包的内部模块结构。  

createFacilitatorConfig(apiKeyId, apiKeySecret) from @coinbase/x402 is the supported replacement.  
`@coinbase/x402` 中的 `createFacilitatorConfig(apiKeyId, apiKeySecret)` 是受支持的替代方案。  

The entire hack — the createRequire shim, the internal import, the createAuthHeaders function — deletes cleanly.  
整个硬编码方案——`createRequire` 填充物、内部导入、`createAuthHeaders` 函数——可以干净利落地删除。  

This was the single most satisfying part of the migration and the strongest argument for doing it even if discovery weren't forcing your hand.  
这是迁移中最令人满意的部分，也是即使发现服务未强制要求也值得进行迁移的最强理由。
