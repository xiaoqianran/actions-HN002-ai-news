**Title: Why Adding More AI Agents Made Our System Slower**  
**标题：为什么增加AI代理反而让系统变慢？**

You’re building a product around LLMs. You’re working on a previously impossible challenge that is now possible to tackle using LLMs. You deploy the first few agents into your system, and everything seems fine. Customers are happy. The responses aren’t the fastest, LLMs tend to be quite slow, but it’s manageable. As you start deploying more and more agents, however, you begin noticing strange anomalies. Slowly but steadily, latency starts increasing. At first, you assume it’s the LLM providers. Maybe the new OpenAI model is slower. Maybe Anthropic is under heavy load. LLMs are inherently slow anyway, right?  
你正在围绕LLM构建产品。你正在处理一个以前不可能解决的挑战，而现在使用LLM已成为可能。你部署了前几个代理到系统中，一切似乎都很好。客户很满意。响应速度不是最快的，LLM通常相当慢，但尚可接受。然而，随着你部署越来越多的代理，你开始注意到奇怪的异常现象。延迟缓慢但稳定地增加。起初，你认为是LLM提供商的问题。也许是新的OpenAI模型更慢了。也许是Anthropic负载过重。反正LLM本质上就很慢，对吧？

Then you start seeing timeout errors. When you dive into the logs and metrics, something doesn’t add up. OpenAI claims the requests are completing very quickly, yet your agent is taking an extremely long time to respond. “OpenAI must be lying to us!” Then comes the painful realization. OpenAI wasn’t the problem. Our code was.  
然后你开始看到超时错误。当你深入查看日志和指标时，发现有些不对劲。OpenAI声称请求完成得很快，但你的代理响应却耗时极长。“OpenAI一定在骗我们！”接着，你痛苦地意识到：OpenAI不是问题所在。我们的代码才是。

At Planck, we went through exactly that transition. As our agent ecosystem grew, so did our response times. This is the story of how we discovered the real bottleneck and how we solved it. If you’re starting to use LLM agents in production and wondering how they will scale, if you’re an AI engineer, or if you simply enjoy challenging asynchronous system design problems, I invite you to follow along.  
在Planck，我们经历了完全相同的转变。随着代理生态系统的增长，我们的响应时间也增加了。这就是我们如何发现真正瓶颈并解决它的故事。如果你刚开始在生产环境中使用LLM代理，并想知道它们如何扩展；如果你是AI工程师；或者你只是喜欢挑战异步系统设计问题，我邀请你继续阅读。

**The Setup**  
**设置**

At Planck, we had several LLM agents running in production. They were all served by a single service, meaning that a single request, whether via HTTP or a worker queue, would trigger all the agents simultaneously. Each agent would then make dozens of calls to its own sub-agents.  
在Planck，我们有多个LLM代理在生产环境中运行。它们都由单个服务提供支持，这意味着无论是通过HTTP还是工作队列，单个请求都会同时触发所有代理。然后每个代理会向其子代理发起数十次调用。

Here’s a simplified version of our code in Python:  
以下是我们用Python简化后的代码：

```python
import asyncio
import aiohttp

NUM_AGENTS = 5  # 单个请求触发的代理数量
CALLS_PER_AGENT = 30  # 每个代理发起的子代理LLM调用次数

async def call_sub_agent(session: aiohttp.ClientSession) -> dict:
    """单个子代理步骤 = 一次LLM调用（I/O密集型）。"""
    payload = {
        "model": "your-llm-model",
        "messages": [{"role": "user", "content": "some fake prompt"}],
    }
    async with session.post("llm_url", json=payload) as resp:
        return await resp.json()

async def run_agent(session: aiohttp.ClientSession, agent_id: int) -> list[dict]:
    """一个代理同时向数十个子代理LLM调用扩散。"""
    return await asyncio.gather(
        *(call_sub_agent(session) for _ in range(CALLS_PER_AGENT))
    )

async def handle_request() -> list[list[dict]]:
    """单个请求同时触发每个代理。"""
    async with aiohttp.ClientSession() as session:
        return await asyncio.gather(
            *(run_agent(session, agent_id) for agent_id in range(NUM_AGENTS))
        )
```

At first glance, this architecture looks ideal. Almost all the work is I/O bound, so using Python’s async ecosystem allows every agent to execute concurrently. The latency of a request should therefore be dominated by the slowest LLM call, not by the total number of agents. At least, that’s what we expected.  
乍一看，这种架构看起来很理想。几乎所有工作都是I/O绑定的，因此使用Python的异步生态系统允许每个代理并发执行。因此，请求的延迟应由最慢的LLM调用决定，而不是代理的总数。至少，我们原本是这么预期的。

To test that, we used a local FastAPI server. Don’t worry, we verified that it responds quickly and is not the bottleneck. We used a small random async sleep range in the server endpoint to simulate the time it takes for an LLM to respond. We sent large payloads to the server and received large payloads back, which is quite common in LLM workflows. The results can vary depending on the payload size, the async sleep range, and the number of sub-agents per agent. So, don’t focus on the absolute numbers themselves, focus on the overall trend.  
为了测试这一点，我们使用了一个本地FastAPI服务器。别担心，我们验证了它响应迅速，不是瓶颈。我们在服务器端点使用了较小的随机异步睡眠范围来模拟LLM响应所需的时间。我们向服务器发送了大型负载，并收到了大型响应，这在LLM工作流中很常见。结果可能因负载大小、异步睡眠范围和每个代理的子代理数量而异。因此，不要关注绝对数值，而要关注整体趋势。

**Zooming In**  
**深入分析**

Why could that be? We were using async exactly as we should. All of the agents and sub-agents were supposed to run simultaneously, so in theory there should have been little to no difference between running one agent and running 10,000 agents. When we run the code in debug mode, we see event loop lag warnings, which means the task is finished, i.e the LLM returns a result, but the event loop can’t acquire a worker to look at it because it is busy doing something else, and in extreme cases this can lead to the timeout errors we were seeing in production.  
为什么会这样？我们正确地使用了异步。所有代理和子代理本应同时运行，因此理论上运行一个代理和运行10,000个代理之间几乎没有区别。当我们在调试模式下运行代码时，看到了事件循环延迟警告，这意味着任务已完成（即LLM返回了结果），但事件循环无法获取工作线程来处理它，因为它正忙于其他事情，在极端情况下这会导致我们在生产中看到的超时错误。

```
WARNING:root: event-loop lag: 230 ms (loop was busy, not waiting)
WARNING:root: event-loop lag: 137 ms (loop was busy, not waiting)
WARNING:root: event-loop lag: 146 ms (loop was busy, not waiting)
WARNING:root: event-loop lag: 177 ms (loop was busy, not waiting)
WARNING:root: event-loop lag: 283 ms (loop was busy, not waiting)
...
```

But why can’t the event loop acquire a worker to handle the LLM response? In Python, because of the GIL, only one thread can execute code at a time. That means the event loop must be busy executing some CPU-bound work elsewhere. But what could it be? To figure that out, we should use a profiler.  
但为什么事件循环无法获取工作线程来处理LLM响应？在Python中，由于GIL（全局解释器锁），一次只能有一个线程执行代码。这意味着事件循环必须正忙于在其他地方执行某些CPU密集型工作。但会是什么呢？为了弄清楚这一点，我们应该使用分析器。

**The Easy Wins**  
**快速优化**

After reviewing the profiler, we can identify a few things. First, we waste a lot of time serializing and deserializing the body. By default, aiohttp uses the built-in json library, but we can significantly speed that up by using orjson.  
回顾分析器后，我们可以识别出几件事。首先，我们浪费了大量时间在序列化和反序列化主体上。默认情况下，aiohttp使用内置的json库，但通过使用orjson我们可以显著加快速度。

*Benchmark from the orjson documentation comparing deserialization and serialization performance*  
*orjson文档中比较反序列化和序列化性能的基准测试*

Second, we are not actually able to run all the I/O requests simultaneously. It turns out that the aiohttp session limits the number of HTTP connections we can open, and by default, that limit is 100. So, we tried to increase and optimize this number, but not indefinitely. At some point, continuing to increase this value will hurt performance. And if we rerun the tests, we see the results are much better. However, even now, as the number of agents increases, the latency also increases.  
其次，我们实际上无法同时运行所有I/O请求。原来aiohttp会话限制了我们可以打开的HTTP连接数，默认情况下，这个限制是100。因此，我们尝试增加和优化这个数字，但不是无限增加。在某个点之后，继续增加这个值会损害性能。如果我们重新运行测试，我们看到结果好多了。然而，即使是现在，随着代理数量的增加，延迟也在增加。
